3. Cloud Service Models

📖 Cloud Computing: On-demand delivery of IT resources (compute, storage, networking, applications) over the internet with pay-as-you-go pricing. Defined by NIST SP 800-145 with five essential characteristics: on-demand self-service, broad network access, resource pooling, rapid elasticity, and measured service.

💡 Exam Tip: The "pizza analogy" helps: SaaS = dining out (provider does everything), PaaS = meal kit (you cook from prepared ingredients), IaaS = grocery store (you buy raw materials and do everything). Remember: as you move from SaaS → PaaS → IaaS, customer responsibility INCREASES.

4. Cloud Deployment Models

ModelDescriptionUse CaseSecurity Consideration
PublicInfrastructure owned by provider; shared among multiple tenants over the internetGeneral business apps, web hosting, developmentMulti-tenancy risks; data sovereignty; dependence on provider security
PrivateInfrastructure dedicated to a single organization; on-premises or provider-hostedRegulated industries (healthcare, finance, government)More control but higher cost; still need proper security configuration
HybridCombination of public and private cloud with orchestration between themBurst capacity; keep sensitive data on-premisesComplex security policies across environments; data in transit between clouds
CommunityShared infrastructure for organizations with common concerns (mission, compliance, policy)Government agencies, healthcare consortiums, research institutionsShared governance; trust among community members; common security controls
🔑 Multi-Tenancy: Multiple customers share the same physical resources (compute, storage, networking) while maintaining logical separation. This is a fundamental characteristic of public cloud. Risks include data leakage between tenants, noisy neighbor (resource contention), and side-channel attacks. Mitigations: strong isolation at hypervisor/network/storage layers, encryption, and strict access controls.

5. Shared Responsibility Model

📖 Shared Responsibility Model: A framework that defines what the cloud provider secures (security "of" the cloud) versus what the customer secures (security "in" the cloud). Responsibility shifts based on the service model.
Security LayerIaaSPaaSSaaS
Data Classification & AccountabilityCustomerCustomerCustomer
Identity & Access ManagementCustomerCustomerShared
Application SecurityCustomerCustomerProvider
Network ControlsCustomerSharedProvider
Operating SystemCustomerProviderProvider
Virtualization / HypervisorProviderProviderProvider
Physical SecurityProviderProviderProvider
⚠️ Critical Principle: Data classification and accountability ALWAYS remain the customer's responsibility, regardless of the service model. You cannot outsource accountability. Even if the provider loses your data, your organization is accountable to regulators and data subjects.
💡 Exam Tip: If the question asks "Who is responsible for patching the OS in PaaS?" — the answer is the provider. If it asks about IaaS — the answer is the customer. Data responsibility is ALWAYS the customer. This is the #1 tested cloud concept.

6. Cloud Security Challenges

ChallengeDescriptionMitigation
Data SovereigntyData is subject to the laws of the country where it's stored; cloud providers have data centers globallyContractual data residency requirements; choose regions carefully; encryption with customer-managed keys
Vendor Lock-InDifficulty migrating between cloud providers due to proprietary formats and APIsUse open standards; multi-cloud strategy; containerization; contractual exit clauses
Insecure APIsCloud services are accessed via APIs; vulnerable APIs = exposed attack surfaceAPI authentication (OAuth 2.0); rate limiting; input validation; API gateway
MisconfigurationThe #1 cause of cloud breaches — exposed storage, open security groups, overly permissive IAMCloud Security Posture Management (CSPM); automated compliance scanning; infrastructure as code
Loss of ControlOrganizations lose direct control over infrastructure and security configurationsSLA contracts; right-to-audit clauses; third-party attestations (SOC 2, ISO 27001)
Account HijackingCompromised cloud credentials grant access to all cloud resourcesMFA; least privilege; credential monitoring; privileged access management
⚠️ Misconfiguration — The #1 Cloud Breach Cause: The Capital One breach (2019) resulted from a misconfigured WAF that allowed SSRF, exposing 100+ million customer records from AWS S3. Always implement CSPM tools and treat infrastructure configuration as code with automated security checks.

7. CSA Cloud Controls Matrix & CASB

CSA Cloud Controls Matrix (CCM)

📖 CSA CCM: A cybersecurity control framework by the Cloud Security Alliance specifically for cloud computing. It maps controls to major compliance frameworks (ISO 27001, NIST 800-53, PCI DSS, HIPAA, COBIT) and is organized into 17 domains covering cloud-specific security requirements.

Key CSA CCM domains include: Application & Interface Security, Audit Assurance & Compliance, Data Security & Privacy, Encryption & Key Management, Identity & Access Management, Infrastructure & Virtualization, and Supply Chain Management.

Cloud Access Security Broker (CASB)

📖 CASB: A security policy enforcement point placed between cloud service consumers and cloud service providers. CASBs combine and interject enterprise security policies as cloud-based resources are accessed.
CASB PillarFunction
VisibilityDiscover shadow IT; identify all cloud services in use
ComplianceEnforce regulatory requirements (GDPR, HIPAA) across cloud services
Data SecurityDLP, encryption, tokenization of data in cloud
Threat ProtectionDetect anomalous behavior, compromised accounts, malware in cloud
🔑 CASB Deployment Modes:
• Forward Proxy: Intercepts traffic from user to cloud (requires endpoint agent or network config)
• Reverse Proxy: Sits in front of cloud service (no agent needed; agentless)
• API-Based: Connects directly to cloud service APIs (out-of-band; no inline traffic inspection)

8. Cloud-Native Security

ConceptDescription
Cloud Security Posture Management (CSPM)Continuously monitors cloud environments for misconfigurations and compliance violations; auto-remediation
Cloud Workload Protection Platform (CWPP)Secures workloads (VMs, containers, serverless) across cloud environments
Infrastructure as Code (IaC)Define infrastructure in code (Terraform, CloudFormation); enables security scanning before deployment
DevSecOps in CloudIntegrating security into CI/CD pipelines; shift-left security; automated testing
Zero Trust in CloudNever trust, always verify; microsegmentation; continuous authentication; identity-centric perimeter
💡 Exam Tip: CSPM = configuration monitoring (are my cloud resources properly configured?). CWPP = workload protection (are my running VMs/containers secure?). CASB = access brokering (who's accessing which cloud services and what are they doing?). Know the difference.

9. Memory Tricks & Mnemonics

🧠 "SPI" for Service Models: SaaS → PaaS → IaaS. Going down this list, customer responsibility INCREASES. "SPI" = Software, Platform, Infrastructure.
🧠 "Data is ALWAYS Yours": Regardless of cloud model (IaaS, PaaS, SaaS), data classification and accountability are ALWAYS the customer's responsibility. You can outsource operations, never accountability.
🧠 "CASB = Cloud's Security Guard": Visibility (who's going in?), Compliance (are they following the rules?), Data Security (what are they carrying?), Threat Protection (are they dangerous?). Four pillars = V-C-D-T.
