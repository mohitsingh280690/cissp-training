3. Introduction to Security Engineering Principles

📖 Security Engineering Principles: Foundational design guidelines that govern how secure systems should be conceptualized, built, and maintained. First formally articulated by Jerome Saltzer and Michael Schroeder in 1975.

These principles are not specific technologies — they are design philosophy. They guide architects and engineers in making decisions that minimize risk, reduce attack surface, and create systems that degrade safely when compromised.

The original 1975 paper "The Protection of Information in Computer Systems" defined eight principles. Several additional principles have since joined the canon and are tested on CISSP.

4. Saltzer & Schroeder's Eight Principles

4.1 Least Privilege

🔑 Principle: Every subject should operate with the minimum set of privileges necessary to perform its authorized task — no more, no less.

In practice: A database application user account should have SELECT permission only on the tables it needs, not DBA-level access. An employee in HR should not have access to engineering source code repositories.

Key distinction: Least privilege applies to both users and processes. A web server process should run under a minimal service account, not root/SYSTEM.

4.2 Fail-Safe Defaults

🔑 Principle: Access decisions should be based on permission (allow-listing) rather than exclusion (deny-listing). When a system fails, it should default to a secure state — deny access.

In practice: A firewall should deny all traffic by default and only allow explicitly permitted flows. If an authentication server is unreachable, users should be denied access — not granted it.

💡 Exam Tip: "Fail-safe" means fail to a safe/secure state (access denied). "Fail-open" means the system allows access on failure — this violates fail-safe defaults. The exam WILL test this distinction.

4.3 Economy of Mechanism

🔑 Principle: Security mechanisms should be as simple and small as possible. Simpler designs are easier to verify, test, and maintain.

In practice: A single, well-tested authentication library is preferable to five different authentication mechanisms scattered across an application. Complex systems have more potential failure points and are harder to audit.

4.4 Complete Mediation

🔑 Principle: Every access to every resource must be checked for authorization. No caching of previous authorization decisions without revalidation.

In practice: A web application should verify authorization on every request, not assume that because a user was authenticated at login, they have permission to access any endpoint. A reference monitor implements this principle by mediating all access between subjects and objects.

⚠️ Common Violation: Session token caching without re-checking permissions. If a user's role is revoked but their cached session token still grants access, complete mediation is violated.

4.5 Open Design

🔑 Principle: Security of a system should not depend on the secrecy of its design or implementation — only on the secrecy of its keys or passwords. (Also known as Kerckhoffs' Principle in cryptography.)

In practice: AES encryption is publicly documented yet remains secure because security relies on key secrecy, not algorithm secrecy. Contrast with proprietary "security through obscurity" algorithms that, when reverse-engineered, are often trivially broken.

💡 Exam Tip: Open design ≠ open source. Open design means the security mechanism doesn't depend on secrecy of the mechanism itself. A proprietary product can follow open design if its security doesn't rely on hidden algorithms.

4.6 Separation of Privilege

🔑 Principle: A system should not grant permission based on a single condition. Multiple conditions (or multiple parties) should be required for critical operations.

In practice: Nuclear launch requires two-person integrity (two-man rule). Banking transactions above a threshold need dual authorization. Code deployment requires both developer and reviewer approval. Multi-factor authentication implements this at the individual level.

4.7 Least Common Mechanism

🔑 Principle: Minimize the amount of mechanism shared between users. Shared resources create potential covert channels and increase the risk that one user's actions affect another's security.

In practice: Separate security domains should have separate processing environments. Shared file systems, shared memory, and shared communication channels between different trust levels create information leakage risks. Virtual machine isolation addresses this by giving each tenant separate resources.

4.8 Psychological Acceptability

🔑 Principle: Security mechanisms should not make resources more difficult to access than if the mechanisms were not present. If security is burdensome, users will circumvent it.

In practice: A password policy requiring 30-character passwords changed weekly will lead to passwords on sticky notes. SSO reduces friction while maintaining security. Biometric authentication can be both more secure and more convenient than passwords.

💡 Exam Tip: When an exam scenario describes users finding workarounds for security controls, psychological acceptability is the violated principle. Users bypassing controls is a design failure, not a user failure.

5. Additional Engineering Principles

5.1 Defense in Depth (Layered Security)

No single control is sufficient. Multiple overlapping security controls ensure that if one layer fails, others still protect the asset. Example: firewall + IDS + host-based antivirus + endpoint encryption + user training.

5.2 Zero Trust

Never trust, always verify. Every access request is authenticated and authorized regardless of network location. Assumes breach and minimizes blast radius. Aligns with complete mediation.

5.3 Privacy by Design

Build privacy protections into systems from inception, not bolted on after deployment. Required by GDPR Article 25. Seven foundational principles by Ann Cavoukian.

5.4 Secure Defaults

Systems should ship in the most secure configuration. Users should have to explicitly reduce security, not explicitly enable it. Relates closely to fail-safe defaults.

8. Memory Tricks & Mnemonics

🧠 "LESS COPS" — Saltzer & Schroeder's 8 Principles:
L – Least privilege
E – Economy of mechanism
S – Separation of privilege
S – (fail-) Safe defaults
C – Complete mediation
O – Open design
P – Psychological acceptability
S – (least common mechanism) Shared resources minimized
🧠 Fail-Safe vs. Fail-Open: "Safe = Shut." If a system fails and it shuts down access → fail-safe (good). If it fails and stays open → fail-open (bad, violates the principle).
🧠 Complete Mediation = "Check Every Time": Think of a bouncer who checks every person's ID every time they enter — not just once at the beginning of the night.
🧠 Open Design = "Kerckhoffs' Principle": "The enemy knows the system" — security relies on the key, not the lock's design being secret.
