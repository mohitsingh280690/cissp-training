2. Real-World Organizational Relevance

Business Impact

Digital forensic procedures are invoked during: data breach investigations (mandatory under many regulations), insider threat cases, intellectual property theft, litigation support (e-discovery), and malware analysis. The difference between a successful prosecution and a failed case often comes down to whether proper forensic procedures were followed.

Common Procedural Failures

  • Shutting down the system prematurely: Destroys volatile evidence (RAM contents, running processes, network connections, encryption keys in memory)
  • No write-blocker: Connecting a drive to a normal system modifies access times, altering the evidence
  • Using unvalidated tools: Forensic tools must be validated and accepted by the forensic community; custom scripts may not withstand legal scrutiny
  • Single hash algorithm: Best practice is dual hashing (MD5 + SHA-256) to prevent any argument of hash collision

4. Disk Imaging & Hashing

Forensic Imaging

Forensic Image (Bit-Stream Copy): An exact, bit-for-bit duplicate of the entire storage medium — including deleted files, file slack, unallocated space, and partition tables. This is NOT the same as a normal file copy (which only copies active files).

Imaging vs Copying

FeatureForensic Image (Bit-Stream)Normal Copy
Captures deleted filesYes — copies entire disk surfaceNo — only copies active files
Captures slack spaceYesNo
Captures unallocated clustersYesNo
Captures hidden dataYes (HPA, DCO areas)No
Hash verifiableYes — bit-for-bit identical to sourceNo — different hash than source device
Court admissibleYes — meets best evidence ruleLikely challenged

Hashing

AlgorithmHash LengthStatusForensic Use
MD5128 bits (32 hex chars)Known collision vulnerabilitiesStill used in forensics for backward compatibility; pair with SHA-256
SHA-1160 bits (40 hex chars)Deprecated for securityAvoid — use SHA-256 instead
SHA-256256 bits (64 hex chars)Current standardPrimary forensic hash algorithm
Dual Hashing Best Practice: Calculate BOTH MD5 and SHA-256 at time of acquisition. Verify both hashes match when accessing the forensic image. Dual hashing eliminates arguments about hash collisions (the probability of two different inputs producing collisions in BOTH algorithms simultaneously is astronomically small).

Write-Blockers

Write-Blocker: A hardware device or software tool that allows read access to a storage device while blocking all write operations. Essential for forensic imaging — prevents the forensic workstation from modifying the evidence drive (even auto-mount, journaling, or time stamp updates).

Hardware write-blockers are preferred over software write-blockers because they are independent of the operating system and harder to bypass. They sit between the evidence drive and the forensic workstation.

5. Volatile vs Non-Volatile Data & Order of Volatility

Volatile Data: Data that is lost when power is removed or the system is shut down (RAM, cache, running processes, network connections).
Non-Volatile Data: Data that persists without power (hard drives, SSDs, USB drives, flash memory, optical media).

RFC 3227 – Order of Volatility

RFC 3227 (Guidelines for Evidence Collection and Archiving) establishes the order in which evidence should be collected — from MOST volatile (lost soonest) to LEAST volatile:

#Data SourceVolatilityExamples
1CPU registers & cacheNanosecondsCurrent instruction pointer, register contents
2Routing table, ARP cache, process table, kernel stats, memory (RAM)Seconds to minutesRunning processes, network connections, memory-resident malware, encryption keys
3Temporary file systems (/tmp)MinutesTemp files that may be cleared on reboot
4Disk / hard drivePersistent until overwrittenFiles, logs, databases, swap space
5Remote logging & monitoring dataPersistent until rotationSIEM logs, syslog server, IDS/IPS logs
6Physical configuration & network topologySemi-permanentNetwork diagrams, switch configs, firewall rules
7Archival media (backups, tapes)YearsBackup tapes, archived disk images
Exam Favorite: "What should be captured FIRST during a forensic investigation?" Answer: the MOST volatile data — CPU registers/cache, then RAM. This is why live forensics (capturing data from a running system) is critical — once the system is powered off, volatile data is gone forever.
⚠️ Never Pull the Plug: Unplugging a system destroys ALL volatile evidence. The forensic examiner should perform live acquisition of volatile data BEFORE the system is shut down. The only exception: if the system is actively destroying evidence (e.g., a wipe process is running), immediate power disconnection may be warranted.

6. Timeline Analysis & Artifact Correlation

Timeline Analysis

Timeline analysis reconstructs the chronological sequence of events by correlating timestamps from multiple sources:

  • File system timestamps: MAC times — Modified, Accessed, Created (or Changed on Linux); note: Windows NTFS stores Created, Modified, Accessed, Entry Modified
  • Log files: System logs, application logs, security logs, web server logs
  • Registry entries (Windows): Last access times, USB connection history, program execution history
  • Network logs: Firewall logs, IDS alerts, DNS queries, proxy logs
  • Email headers: Sent/received timestamps, mail server relay times

MAC Times

TimestampUpdated WhenForensic Significance
ModifiedFile content is changedWhen was the file last written to?
AccessedFile is read or openedWhen was the file last viewed? (may be disabled for performance on modern OS)
Created (NTFS) / Changed (ext)File is first created (NTFS) or metadata changes (ext)When was the file placed on this volume? NOTE: A "Created" date AFTER the Modified date indicates the file was COPIED from elsewhere
Timestomping: An anti-forensic technique where an attacker deliberately modifies file timestamps to hide when files were created or accessed. Forensic examiners can detect this by comparing file system timestamps with entries in the MFT (Master File Table) journal, log entries, or other timestamped artifacts.

7. Network Forensics

Network Forensics: The capture, recording, and analysis of network traffic for the purpose of detecting intrusions, investigating security incidents, or gathering evidence. Unlike disk forensics, network forensics often deals with real-time or near-real-time data.

Two Approaches

ApproachDescriptionProsCons
Catch-it-as-you-canCapture ALL network traffic (full packet capture) and store for later analysisComplete record; thorough analysis possibleMassive storage requirements; performance impact; privacy concerns
Stop, look, and listenAnalyze traffic in real-time and only save relevant data; flag anomalies on the flyLess storage; faster alerting; reduced privacy exposureMay miss relevant data not flagged; can't go back to review past traffic

Key Network Evidence Sources

  • Packet captures (PCAP): Full content of network communications — the gold standard for network forensics
  • NetFlow / IPFIX / sFlow: Summary records of network conversations (source, destination, port, bytes, duration) — no payload content
  • IDS/IPS alerts: Records of detected suspicious or malicious network activity
  • Firewall logs: Allowed and denied connection attempts
  • DNS logs: Domain resolution requests — critical for detecting C2 (command and control) communication, DNS tunneling, DGA (domain generation algorithm) activity
  • Proxy/web filter logs: URLs accessed, content categories, blocked requests

8. Anti-Forensics Techniques

TechniqueDescriptionDetection Method
Data wipingOverwriting data with zeros or random patterns (DoD 5220.22-M standard recommends multiple passes)Look for large blocks of zeros/patterns; check for discrepancies between file system records and actual disk content
SteganographyHiding data within other files (images, audio, video) without visible alterationStatistical analysis of carrier files; steganalysis tools; file size anomalies
EncryptionEncrypting files or entire volumes to prevent access without the keyCapture RAM before shutdown (keys may be in memory); legal compulsion; key escrow
TimestompingModifying file timestamps to disguise when files were created/modifiedCompare filesystem timestamps with MFT journal entries, $LogFile, $UsnJrnl
Slack space hidingStoring data in file slack space (gap between end of file and end of cluster)Forensic tools analyze slack space; file carving can recover hidden data
Log tamperingDeleting or modifying log entries to erase evidence of activityRemote logging (attacker can't easily modify); log integrity monitoring; SIEM correlation
Trail obfuscationUsing proxies, VPNs, Tor, or compromised intermediary systems to hide originCorrelation across multiple log sources; timing analysis; traffic pattern analysis
Exam Point — Best Defense Against Anti-Forensics: Centralized, remote logging (send logs to a separate, hardened SIEM/syslog server). An attacker who compromises a system can tamper with local logs but cannot easily modify logs already sent to a remote server.

9. Memory Tricks & Mnemonics

🧠 Forensic Process — "CEAR"
Collection → Examination → Analysis → Reporting
"Collect Evidence, Analyze, Report" (NIST SP 800-86)
🧠 Order of Volatility — "CPU Runs Through Disk, Remote, Physical, Archive"
1. CPU registers/cache
2. RAM (memory, routing table, ARP, processes)
3. Temp file systems
4. Disk (hard drive, SSD)
5. Remote logging
6. Physical topology/config
7. Archival media
"Collect from fastest-disappearing to most-persistent"
🧠 MAC Times
Modified = content changed
Accessed = last opened/read
Created = first placed on volume
"If Created date is AFTER Modified date → file was COPIED here"
🧠 Write-Blocker Rule
"Never plug evidence into a naked port." Always use a write-blocker between evidence media and forensic workstation.
