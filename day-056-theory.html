3. TCP – Transmission Control Protocol

📖 TCP (Transmission Control Protocol): A connection-oriented, reliable Layer 4 (Transport) protocol. TCP guarantees delivery, ordering, and error checking of data between two endpoints. Defined in RFC 793.

TCP Characteristics

CharacteristicMechanismSecurity Implication
Connection-OrientedRequires a 3-way handshake before data transferCreates state — attackers can exploit the handshake (SYN flood)
Reliable DeliverySequence numbers + acknowledgements (ACKs); retransmits lost segmentsSequence number prediction enables session hijacking
Ordered DeliverySequence numbers ensure segments are reassembled in orderFragmentation attacks (Teardrop) exploit reassembly
Flow ControlSliding window adjusts the amount of data sent based on receiver capacityWindow size manipulation in DoS attacks
Full-DuplexData can flow in both directions simultaneouslyBidirectional interception in MITM attacks
Error Checking16-bit checksum on each segment headerChecksum does not provide integrity against manipulation

4. The TCP Three-Way Handshake

📖 Three-Way Handshake: The process by which TCP establishes a connection between client and server. Requires three messages — SYN, SYN-ACK, and ACK — before any data is transferred. This creates a stateful, agreed-upon connection with synchronized sequence numbers.
Client Server │ │ │──── SYN (seq=x) ─────────────▶│ Client: "I want to talk; my seq# is x" │ │ │◀─── SYN-ACK (seq=y, ack=x+1)──│ Server: "OK; my seq# is y; I got your x" │ │ │──── ACK (ack=y+1) ────────────▶│ Client: "Got it; confirming your seq y" │ │ │════════ DATA TRANSFER ════════│ │ │
🔑 Why Sequence Numbers Matter (Security): Sequence numbers are critical for session security. If an attacker can predict the initial sequence number (ISN), they can spoof packets as either party — this is the basis of TCP session hijacking. Modern OS implementations use cryptographically random ISNs to prevent this. Early systems (before RFC 6528) used predictable ISNs, enabling famous attacks like the 1994 Mitnick attack on Tsutomu Shimomura.

SYN Flood Attack (TCP Handshake Exploitation)

AspectDetail
How it worksAttacker sends thousands of SYN packets with spoofed source IPs. The server replies SYN-ACK to non-existent addresses. Since the final ACK never arrives, the server's TCP state table fills up with half-open (SYN_RCVD) connections.
ImpactServer exhausts connection table — legitimate connections are refused (DoS)
DefenceSYN cookies (cryptographic ISN that encodes session state, eliminating the need to store half-open connections); rate limiting; firewall thresholds
DetectionHigh rate of SYN packets; large number of connections in SYN_RCVD state; asymmetrical traffic
💡 Exam Tip: The SYN flood exploits the stateful nature of TCP. The answer to "why does SYN flood work?" is that the server must maintain state for every half-open connection. SYN cookies solve this by encoding the connection state in the ISN, so the server doesn't need to store anything until the handshake completes.

5. TCP Flags & Connection Termination

TCP Flags (Control Bits)

FlagFull NamePurpose & Security Context
SYNSynchronizeInitiates a connection; SYN-only scan (nmap -sS) is the most common port scan technique — sends SYN and waits for response
ACKAcknowledgeConfirms receipt; used in all established connections; ACK scans can bypass some stateless firewalls
FINFinishGracefully closes a connection; FIN scan (nmap -sF) sends FIN without prior connection — closed ports reply RST, open ports ignore
RSTResetAbruptly closes a connection; used by firewalls to reject connections; RST injection can terminate established sessions
PSHPushInstructs receiver to pass data to application immediately without buffering
URGUrgentMarks urgent data; Urgent Pointer field indicates where urgent data ends
NULLNo flags setUsed in NULL scans (nmap -sN) — no flags set; RFC compliant hosts close port responds RST, open ports ignore (Unix systems only)
XMASFIN+PSH+URG setXMAS scan (nmap -sX) — like NULL scan; works only on Unix; Windows ignores flag combinations, making it unreliable

TCP Four-Way Termination

Client Server │ │ │──── FIN ──────────────────────▶│ "I'm done sending" │◀─── ACK ───────────────────────│ "Got your FIN" │ │ (Server may still be sending) │◀─── FIN ───────────────────────│ "I'm done sending too" │──── ACK ──────────────────────▶│ "Got your FIN" │ │ │ [TIME_WAIT on client] │
🔑 TIME_WAIT State: After sending the final ACK, the client remains in TIME_WAIT for 2×MSL (Maximum Segment Lifetime, typically 2 minutes) to ensure the final ACK was received. This prevents delayed duplicate packets from a previous connection from being misinterpreted by a new connection on the same port.

6. UDP – User Datagram Protocol

📖 UDP (User Datagram Protocol): A connectionless, unreliable Layer 4 protocol. UDP sends datagrams without establishing a connection, without guaranteeing delivery, and without ordering. Defined in RFC 768. It is the "fire and forget" protocol.
AspectTCPUDP
ConnectionConnection-oriented (3-way handshake)Connectionless
ReliabilityGuaranteed delivery (ACKs + retransmission)Best-effort, no guarantees
OrderingOrdered (sequence numbers)Unordered (application must handle if needed)
SpeedSlower (overhead of handshake, ACKs, retransmission)Faster (no overhead)
Header Size20–60 bytes8 bytes (fixed)
Flow ControlYes (sliding window)No
Use CasesHTTP/S, FTP, SMTP, SSH, TelnetDNS, DHCP, VoIP, video streaming, TFTP, SNMP, NTP
State on serverStateful (socket maintained)Largely stateless
🔑 Why Use UDP at All? Speed and overhead. For real-time applications like VoIP or video streaming, a retransmitted packet arrives too late to be useful — it's better to drop it and continue. DNS uses UDP for initial queries (speed); falling back to TCP for large responses (>512 bytes in DNS traditionally, now >4096 with EDNS0). DTLS (Datagram TLS) adds security to UDP for VPNs (e.g., used in WebRTC, QUIC).

UDP Security Concerns

ThreatMechanismExample / Mitigation
Amplification DDoSAttacker spoofs victim's IP; sends small UDP request to open resolver; large response floods victim (amplification factor 50–100x)DNS amplification, NTP monlist (200x amplification); mitigated by BCP 38 (ingress filtering), response rate limiting
UDP FloodHigh-volume UDP packets sent to random ports; target sends ICMP "port unreachable" for each; overwhelms targetRate limiting at perimeter; UDP flood blocking
SpoofingSince there's no handshake, source IP is easily spoofed in UDP packetsBCP 38 (network ingress filtering); stateful firewall inspection
💡 Exam Tip: UDP amplification attacks are possible precisely because UDP is connectionless — there's no handshake to verify that the requester's IP is real. TCP's three-way handshake naturally prevents amplification because the attacker never completes the handshake to an uncontrolled destination. This is a key CISSP exam distinction.

7. ICMP – Internet Control Message Protocol

📖 ICMP (Internet Control Message Protocol): A Layer 3 supporting protocol used by network devices to send error messages and operational information. ICMP does not transport application data — it carries network-layer diagnostic and control messages. Defined in RFC 792.
⚠️ ICMP Has No Port Numbers: A very common exam trap. ICMP operates at Layer 3 (Network). It does not use TCP or UDP ports. You cannot filter ICMP by port number — you filter by ICMP type and code.

Key ICMP Message Types

TypeCode / NamePurposeSecurity Use
0Echo ReplyResponse to pingHost discovery; live host confirmed
3Destination UnreachablePacket can't reach destination; code indicates why (network, host, port, protocol)Reconnaissance — closed port returns type 3 code 3 (Port Unreachable)
5RedirectInstructs host to use a different router for a specific destinationICMP redirect attacks — redirect victim's traffic through attacker's machine (MITM)
8Echo RequestThe ping requestPing sweep (mass host discovery); often blocked at perimeters
11Time ExceededTTL expired in transitUsed by traceroute (each router decrements TTL to 0 and replies with type 11)
13/14Timestamp Request/ReplyNetwork time measurementCan be used for covert channel — timing-based data exfiltration

ICMP-Based Recon & Attacks

Attack/TechniqueICMP UsageDefence
Ping SweepSend ICMP Echo Request to a range of IPs; any Echo Reply = live hostBlock inbound/outbound ICMP Echo at perimeter; monitor for high-volume ICMP
Ping of Death (PoD)Sends an ICMP Echo Request larger than 65,535 bytes (after fragmentation reassembly) causing buffer overflowPatching; all modern OSes are immune
Smurf AttackSends ICMP Echo Request to broadcast address with victim's spoofed IP as source; all hosts on subnet reply to victim (amplification)Disable directed broadcast at routers; ingress filtering
ICMP Redirect AttackAttacker sends a forged ICMP Redirect to victim, directing traffic through attacker's hostDisable ICMP redirect acceptance on hosts; use static routes where possible
ICMP TunnelingData exfiltration hidden in ICMP payload (e.g., tools like icmptunnel)Deep packet inspection; anomaly-based IDS detecting large ICMP payloads
💡 Exam Tip: Traceroute uses UDP (on Unix/Linux — sends UDP probes to ports 33434+ with incrementing TTL) or ICMP (Windows tracert sends ICMP Echo Requests with incrementing TTL by default). Either way, the intermediate routers return ICMP Type 11 (Time Exceeded). The exam may ask: "What protocol does traceroute use?" — the answer depends on OS, but the reply is always ICMP.

8. ARP – Address Resolution Protocol

📖 ARP (Address Resolution Protocol): Maps IP addresses (Layer 3) to MAC addresses (Layer 2). When a host wants to communicate with another host on the same local network, it must know the destination MAC address. ARP provides this mapping. Defined in RFC 826.
ARP Process: Host A (10.0.0.1) Host B (10.0.0.2) │ │ │──── ARP Request (Broadcast) ──────────────────▶│ │ "Who has 10.0.0.2? Tell 10.0.0.1" │ │ Destination MAC: FF:FF:FF:FF:FF:FF │ │ │ │◀─── ARP Reply (Unicast) ───────────────────────│ │ "10.0.0.2 is at AA:BB:CC:DD:EE:FF" │ │ │ │ [Host A caches entry in ARP table] │ │══════════════ DATA FLOWS ═════════════════════▶│

ARP Poisoning / ARP Spoofing (MITM Attack)

⚠️ ARP Has No Authentication: ARP is stateless and trusts any response. A host will update its ARP cache whenever it receives an ARP reply, even if it never sent a request (gratuitous ARP). This fundamental design flaw enables ARP poisoning.
AspectDetail
How it worksAttacker sends unsolicited (gratuitous) ARP replies to both the victim and the gateway. Victim maps gateway IP → attacker MAC. Gateway maps victim IP → attacker MAC. All traffic flows through attacker (MITM).
ImpactMan-in-the-Middle (can intercept, modify, or drop traffic); credential theft if traffic is unencrypted; can redirect to malicious sites
ToolsArpspoof, arpwatch (detection), Ettercap, Bettercap
DefencesDynamic ARP Inspection (DAI) on switches — validates ARP packets against a trusted DHCP snooping binding table; static ARP entries for critical servers; encrypted traffic (TLS renders MITM ineffective for data theft); network segmentation; IDS monitoring ARP anomalies
ScopeARP is only local — it only works within the same broadcast domain (same VLAN/subnet). Cross-subnet attacks require other techniques.

RARP, Proxy ARP, and Gratuitous ARP

TypeDescriptionSecurity Note
RARPReverse ARP — maps MAC → IP (deprecated, replaced by DHCP/BOOTP)Largely obsolete; was used before DHCP
Proxy ARPA router responds to ARP requests on behalf of devices in another subnetCan hide network topology; enables ARP spoofing across subnets if misconfigured
Gratuitous ARPARP reply sent without a request — used by hosts to announce their IP/MAC (e.g., after IP change, or failover)The primary mechanism for ARP poisoning — hosts accept gratuitous ARPs unconditionally
💡 Exam Tip: ARP poisoning leads to a Man-in-the-Middle attack on the local network. The most important defence the CISSP exam expects is Dynamic ARP Inspection (DAI) combined with DHCP snooping. Static ARP entries are mentioned but impractical at scale. Encryption (TLS) is the compensating control when MITM cannot be prevented.

9. Port Numbers

📖 Port Numbers: 16-bit numbers (0–65,535) used by TCP and UDP to identify specific applications or services running on a host. Combined with an IP address, a port forms a socket (e.g., 192.168.1.1:443). Port numbers allow a single host to run many services simultaneously.

Port Number Ranges

0–1,023Well-Known (System) PortsAssigned by IANA; standard services (HTTP=80, HTTPS=443, SSH=22); on most OS, require root/admin to bind
1,024–49,151Registered PortsRegistered with IANA for specific applications (MS SQL=1433, MySQL=3306, RDP=3389)
49,152–65,535Dynamic / Ephemeral PortsAssigned dynamically by the OS for outgoing connections (client-side ports); also called private ports

🔑 Protocol Security Principle: The core protocols (TCP, UDP, ICMP, ARP) were designed in the 1970s–1980s for trusted academic networks — they were never designed with security in mind. Security has been retrofitted through: TLS (over TCP), DTLS (over UDP), DNSSEC (over DNS), IPSec (at the IP layer), and DAI (for ARP). The CISSP exam tests whether you understand these design limitations and the compensating controls.

11. Memory Tricks & Mnemonics

🧠 TCP vs. UDP — "Reliable Taxicab vs. Unreliable Bus":
TCP = Taxicab: calls ahead (handshake), guarantees you'll arrive, acknowledges receipt
UDP = Bus: departs on schedule, doesn't wait for you, no confirmation you boarded
Use TCP when delivery matters; use UDP when speed matters more than guarantee.
🧠 TCP Three-Way Handshake — "SYN SYN-ACK ACK":
"Sam Said Aye" (SYN → SYN-ACK → ACK) or
"I Synchronize → I Synchronize-Acknowledge → I Acknowledge" — always exactly 3 messages, always in this order.
🧠 TCP Flags — "UAPRSF":
Urgent Acknowledge Push Reset Synchronize Fin
Memory phrase: "Unapologetically Axing Poisonous Rabbits Since Forever"
🧠 ICMP Has No Ports:
"ICMP doesn't knock — it shouts" — ICMP is Layer 3, operates at the IP level, has no port concept. When filtering ICMP, use type and code, not port numbers.
🧠 Critical Secure Port Pairs — "Just Remember Secure Sockets":
HTTP(80) → HTTPS(443) | Telnet(23) → SSH(22) | FTP(21) → SFTP(22)
LDAP(389) → LDAPS(636) | IMAP(143) → IMAPS(993) | POP3(110) → POP3S(995)
Pattern: each has a TLS/encrypted counterpart — always choose the encrypted version.
🧠 ARP Poisoning Defence — "DAI + DHCP Snooping":
Dynamic ARP Inspection checks ARP packets against the DHCP snooping binding table.
"DHCP Snoops; DAI Inspects; Together they block ARP lies."
🧠 SYN Flood Solution — "Cookie Not State":
SYN cookies eliminate the half-open connection table. The server encodes state in the ISN.
"Can't fill a table that doesn't exist."
