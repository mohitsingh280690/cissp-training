2. Real-World Relevance

A penetration test report that lists 500 findings with no prioritization is useless. A vulnerability assessment that covers 60% of assets gives false confidence about the 40% unscanned. Test coverage analysis and structured reporting transform raw test data into actionable intelligence that drives security decisions.

3. Code Coverage Analysis

Code Coverage: A metric that measures how much of the source code is executed during testing. Higher coverage means more code has been tested, reducing the likelihood of undetected bugs. Coverage is measured by instrumented tools that track which lines/branches/paths are executed.

Coverage Types (Increasing Thoroughness)

Coverage TypeWhat It MeasuresThoroughnessExample
Statement CoverageHas each line of code been executed at least once?LowestEvery line in the function runs during testing
Branch Coverage (Decision Coverage)Has each decision point (if/else, switch) been evaluated for BOTH true and false outcomes?MediumBoth the if-branch AND the else-branch have been tested
Condition CoverageHas each individual Boolean sub-expression been evaluated to both true and false?Medium-HighIn if (A && B), A and B are each tested as true and false independently
Path CoverageHas every possible execution path through the code been tested?HighestAll combinations of all branches have been executed
CODE COVERAGE EXAMPLE function checkAccess(role, mfa) { if (role === "admin") { ← Branch 1 if (mfa === true) { ← Branch 2 return "FULL ACCESS"; ← Path 1: admin + mfa } else { return "LIMITED ACCESS"; ← Path 2: admin + no mfa } } else { return "DENIED"; ← Path 3: not admin } } Statement Coverage: Execute all 3 return statements Branch Coverage: Test both TRUE and FALSE for each if Path Coverage: Test all 3 possible paths through code Path coverage ⊇ Branch coverage ⊇ Statement coverage
100% Statement Coverage ≠ Bug-Free: A function can have 100% statement coverage but still harbor bugs in untested combinations of conditions. That's why branch and path coverage provide stronger assurance. However, 100% path coverage is often impractical for complex code due to combinatorial explosion.

Coverage Targets

Application TypeTypical Statement Coverage TargetNotes
Safety-Critical (DO-178C Level A, IEC 61508 SIL 4)100% MC/DC (Modified Condition/Decision Coverage)Aviation, medical devices, nuclear systems
Financial / Payment Processing80-90% statement; high branch coveragePCI DSS requires testing but doesn't mandate specific coverage %
General Enterprise70-80% statement coverageIndustry best practice; diminishing returns above ~80%
Prototypes / Internal ToolsNo formal targetRisk-based; critical paths should still be tested
Exam Hierarchy: Statement (weakest) → Branch → Condition → Path (strongest). Branch coverage answers: "Have I tested both TRUE and FALSE for every if/else?" Path coverage answers: "Have I tested every possible route through the code?"

4. Requirements Traceability Matrix (RTM)

Requirements Traceability Matrix: A document that maps each requirement (security or functional) to its corresponding design element, implementation, test case, and test result. It ensures nothing is missed — every requirement has a test, and every test links back to a requirement.

RTM Structure

Req IDRequirement DescriptionDesign ElementTest Case IDTest ResultStatus
SEC-001All user passwords must be hashed with bcryptAuthModule.hashPassword()TC-SEC-001PASSVerified
SEC-002Session timeout after 15 minutes of inactivitySessionManager.checkTimeout()TC-SEC-002PASSVerified
SEC-003All API endpoints require authentication tokenAPIGateway.validateToken()TC-SEC-003FAILRemediation in progress
SEC-004Sensitive data encrypted at rest (AES-256)DataLayer.encrypt()TC-SEC-004PASSVerified

RTM Benefits

  • Completeness: Identifies requirements with no corresponding test (gaps)
  • Traceability: Links every defect back to the requirement it violates
  • Coverage verification: Proves to auditors that all security requirements have been tested
  • Change impact: When a requirement changes, the RTM shows which tests must be updated
  • Audit evidence: Provides formal documentation that security requirements were verified
Exam Context: If a question asks how to verify that ALL security requirements have been tested — the answer is the Requirements Traceability Matrix. If a question asks about code-level testing completeness — the answer is code coverage metrics.

6. Risk Ratings & Prioritization

CVSS (Common Vulnerability Scoring System)

CVSS ScoreSeverityTypical SLA
9.0 – 10.0CriticalRemediate within 24-48 hours
7.0 – 8.9HighRemediate within 30 days
4.0 – 6.9MediumRemediate within 90 days
0.1 – 3.9LowRemediate within 180 days or accept risk

Risk Rating Beyond CVSS

CVSS provides a technical severity score, but organizations must also consider:

  • Asset criticality: A medium CVSS on a payment processing server is higher priority than a critical CVSS on a test server
  • Exploit availability: Known exploits in the wild escalate priority regardless of CVSS score
  • Exposure: Internet-facing systems are higher priority than internal-only systems
  • Compensating controls: Network segmentation, WAF, or IPS may reduce effective risk
  • Business context: Systems under regulatory scope (PCI, HIPAA) may require faster remediation
Risk-Based Prioritization: Effective organizations don't just sort by CVSS score — they use a risk-based approach: Risk = Vulnerability Severity × Asset Value × Threat Likelihood − Compensating Controls. This focuses effort where it matters most.

7. Remediation Tracking

Remediation Lifecycle

REMEDIATION TRACKING WORKFLOW Finding → Assigned → In Progress → Remediated → Verified Identified to Owner Fix being Fix applied Re-tested developed and confirmed │ │ └── Risk Accepted ─────────────────────────────→│ (documented & (with justification & approval) │ approved)

Tracking Elements

FieldPurpose
Finding IDUnique identifier linking to the original assessment finding
Severity / Risk RatingDetermines remediation priority and SLA
Assigned OwnerIndividual responsible for remediation (not a group — a person)
Due DateBased on severity SLA; tracked for overdue alerting
StatusOpen → In Progress → Remediated → Verified → Closed
Verification MethodHow the fix will be confirmed (re-scan, re-test, code review)
Exception / Risk AcceptanceIf not remediated: formal risk acceptance with justification, approver, and review date
Exam Tip: Remediation is not complete until VERIFIED. Marking a finding as "fixed" without verification creates a false sense of security. Re-scanning or re-testing must confirm the fix actually works.

8. Memory Tricks & Mnemonics

🧠 Coverage Hierarchy — "S-B-C-P" (Simple → Complete)
Statement (weakest — just "did this line run?")
Branch (both sides of every if/else)
Condition (each Boolean sub-expression)
Path (every possible route — strongest but hardest)
"Statement → Branch → Condition → Path" = weakest to strongest
🧠 RTM Purpose
"Every Requirement gets a Test; Every Test traces to a Requirement" — bidirectional traceability ensures nothing falls through the cracks
🧠 CVSS Severity Thresholds
9 = Critical (emergency)
7 = High (urgent)
4 = Medium (scheduled)
0 = Low (when convenient)
"9-7-4-0" — just remember the cutoff points
🧠 Remediation Rule
"Fixed ≠ Verified" — always re-test to close the loop
