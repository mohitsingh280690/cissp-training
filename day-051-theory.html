3. Cryptanalytic Attacks

📖 Cryptanalytic Attack: An attack that targets the mathematical properties of the cryptographic algorithm or protocol to break or weaken the encryption, rather than attacking the implementation or human factors.
AttackAttacker HasGoalExample
Ciphertext Only (COA)Only ciphertextDeduce plaintext or keyFrequency analysis on substitution ciphers; simplest but hardest attack
Known Plaintext (KPA)Some plaintext-ciphertext pairsDeduce the keyWWII Enigma — Allies knew standard message openings ("Heil Hitler"); used this to reverse the daily key
Chosen Plaintext (CPA)Ability to encrypt chosen plaintextsDeduce the keyBEAST attack (TLS 1.0 CBC) — attacker injects chosen blocks to recover session cookie
Chosen Ciphertext (CCA)Ability to decrypt chosen ciphertextsDeduce the keyBleichenbacher attack on RSA PKCS#1 v1.5 padding — send crafted ciphertexts and observe error responses
Frequency AnalysisCiphertext of substitution cipherMap cipher letters to plaintext lettersIn English, 'E' is most common (~13%); in ciphertext, the most frequent letter likely maps to 'E'
Birthday AttackAccess to hash functionFind any two inputs with the same hashFor n-bit hash, collision in ~2^(n/2); effective against MD5 (2^64)
💡 Exam Tip — Attack Hierarchy: The attacks go from least to most powerful: Ciphertext-Only → Known-Plaintext → Chosen-Plaintext → Chosen-Ciphertext. Each gives the attacker more power. Remember: "C-K-CP-CC" = increasing attacker capability.
🔑 Key Clustering: When two different keys produce the same ciphertext from the same plaintext. This is a weakness because it reduces the effective key space. Modern algorithms are designed to minimize key clustering.

4. Implementation & Side-Channel Attacks

AttackExploitsExample
Timing AttackMeasures time taken by cryptographic operations; different key bits cause different execution pathsComparing password hashes byte-by-byte — early termination on mismatch leaks which bytes are correct
Power AnalysisMonitors power consumption during cryptographic operations; different operations draw different amounts of powerSimple Power Analysis (SPA) and Differential Power Analysis (DPA) against smart cards
Electromagnetic (EM) AnalysisCaptures electromagnetic emissions from a device during cryptographic computationSimilar to power analysis but measured from a distance via EM radiation
Fault InjectionDeliberately introduces errors (voltage glitches, laser pulses, clock manipulation) to cause faulty outputs that reveal the keyBellcore attack on RSA-CRT (Chinese Remainder Theorem) — a single faulty signature reveals the private key
Cache TimingExploits CPU cache behavior — cache hits are faster than misses, leaking information about which memory addresses were accessedSpectre/Meltdown-style attacks; Flush+Reload against AES T-tables
⚠️ Side-Channel ≠ Cryptanalysis: Side-channel attacks do NOT target the mathematical strength of the algorithm. The algorithm may be perfectly secure, but the physical implementation leaks information. Countermeasures include constant-time coding, power filtering, electromagnetic shielding, and HSMs.

5. Social & Physical Attacks on Cryptography

AttackDescriptionCountermeasure
Rubber-Hose AttackCoercion or torture to extract the key from a person (a human factors attack, not a technical one)Plausible deniability (hidden volumes, e.g., VeraCrypt), duress codes, M-of-N key splitting
Social EngineeringManipulating people into revealing keys or passphrases through deceptionSecurity awareness training; strict key handling procedures
Shoulder SurfingObserving someone entering a password or encryption passphrasePrivacy screens, awareness, not entering keys in public
Dumpster DivingSearching through discarded materials for written-down keys or passwordsSecure document destruction; clean desk policies
💡 Exam Tip: The CISSP exam loves the rubber-hose attack as a distractor. It tests whether you understand that not all attacks are technical. If the question describes physical coercion to extract a key, the answer is rubber-hose attack.

6. Key Management Lifecycle

📖 Key Management: The policies, procedures, and mechanisms for generating, distributing, storing, using, rotating, archiving, and destroying cryptographic keys. NIST SP 800-57 is the primary reference for key management recommendations.
PhaseDescriptionBest Practice
1. GenerationCreate keys using cryptographically secure random number generators (CSPRNG)Use FIPS 140-2/3 validated modules; never use predictable seeds
2. DistributionSecurely transport keys to where they're neededUse key exchange protocols (DH, ECDHE); never send keys in plaintext; out-of-band for symmetric keys
3. StorageProtect keys at restHSMs, key vaults, encrypted key stores; never hard-code keys in source code; split knowledge for critical keys
4. UsageUse keys for their intended purpose onlySeparate encryption keys from signing keys; enforce key usage restrictions via certificates (Key Usage extension)
5. RotationPeriodically replace keys (cryptoperiod)Rotate before expiration; shorter cryptoperiods = more re-encryption but less risk; automate rotation
6. ArchivalRetain old encryption keys to decrypt previously encrypted dataArchive encryption keys securely; do NOT archive signing keys
7. DestructionPermanently destroy keys when no longer neededZeroize key material; use FIPS-compliant destruction methods; ensure no copies remain; destroy in HSMs
🔑 Cryptoperiod: The time span during which a specific key is authorized for use. NIST SP 800-57 defines recommended cryptoperiods based on key type. Shorter cryptoperiods reduce the impact of key compromise (less data encrypted under one key). Example: TLS session keys change per session; CA root keys may have 20+ year cryptoperiods.
⚠️ Split Knowledge & Dual Control:
Split Knowledge: No single person knows the entire key — each person knows only their portion (e.g., M-of-N key shares with Shamir's Secret Sharing).
Dual Control: Two or more people must collaborate to perform a key operation — no single person can act alone.
These are different concepts but often used together. Both implement separation of duties for cryptographic operations.

7. Key Stretching

📖 Key Stretching: The process of taking a weak key (like a user's password) and making it more resistant to brute-force attacks by applying a computationally expensive function that produces a stronger derived key.
AlgorithmMechanismNotes
PBKDF2Applies HMAC-SHA1/SHA256 repeatedly (configurable iterations, e.g., 600,000+)NIST recommended (SP 800-132); used in WPA2 (4,096 iterations); CPU-intensive only
bcryptBased on Blowfish cipher; configurable cost factor that doubles computation per incrementMemory-hard by design; widely used for password hashing; includes built-in salt
scryptRequires large amounts of memory in addition to CPU time; configurable CPU and memory costDesigned to resist GPU/ASIC attacks by being memory-intensive; used in cryptocurrency
Argon2Winner of the Password Hashing Competition (2015); configurable time, memory, and parallelismCurrent best practice for new designs; three variants: Argon2d (GPU resistant), Argon2i (side-channel resistant), Argon2id (hybrid)
💡 Exam Tip: Know that PBKDF2 and bcrypt are key stretching algorithms that make password brute-forcing slower. Argon2 is the newest and most recommended. The key concept: stretch a weak password into a strong derived key by making computation deliberately expensive.

8. Memory Tricks & Mnemonics

🧠 Attack Hierarchy — "Climb the Knowledge Ladder":
Ciphertext-Only → Known-Plaintext → Chosen-Plaintext → Chosen-Ciphertext
Each rung gives the attacker more power. Remember: "CK-CP-CC"
🧠 Key Management Phases — "Good Dogs Stay Until Routinely Assessed Daily":
Generation → Distribution → Storage → Usage → Rotation → Archival → Destruction
🧠 Side-Channel — "The POWER of TIME": Side-channel attacks leak info through Power, Timing, and Electromagnetic emissions — not through the algorithm's math.
🧠 Split Knowledge vs. Dual Control: "Split = knowledge (what you know is divided). Dual = action (two people must act together)."

ConceptKey Point

Known/Chosen PlaintextAttacker has or can create plaintext-ciphertext pairs; more powerful
Chosen CiphertextMost powerful — attacker can decrypt chosen ciphertexts and analyze results
Side-ChannelAttacks on implementation (timing, power, EM); not on algorithm math
Rubber-HosePhysical coercion to extract keys — not a technical attack
Key Management7 phases: Generate → Distribute → Store → Use → Rotate → Archive → Destroy
CryptoperiodTime a key is authorized for use; shorter = less data at risk per compromise
Split KnowledgeKey divided among multiple people; no one knows the full key
Key StretchingPBKDF2, bcrypt, scrypt, Argon2 — make weak passwords computationally expensive to brute-force
HSMsTamper-resistant hardware for key generation, storage, and operations
