3. IoT Architecture & Constrained Devices

📖 Internet of Things (IoT): A network of physical objects ("things") embedded with sensors, software, and connectivity that enables them to collect and exchange data. IoT devices range from consumer wearables to industrial sensors to medical implants.

IoT Architecture Layers

LayerFunctionSecurity Concern
Perception (Device)Sensors and actuators that interact with the physical worldPhysical tampering, weak authentication, default credentials
Network (Communication)Transmits data between devices and cloud/serversEavesdropping, replay attacks, protocol vulnerabilities
Application (Service)Processes data and delivers services to usersInsecure APIs, insufficient authorization, data privacy

Constrained Devices

🔑 Why IoT Is Hard to Secure:
• Limited CPU/Memory: Cannot run full security stacks, antivirus, or complex encryption
• Limited Power: Battery-operated; heavy crypto drains battery faster
• No User Interface: Cannot prompt for credentials or display warnings
• Long Lifespans: Deployed for 10–20 years; may outlive vendor support
• Difficult to Patch: Many lack over-the-air (OTA) update capability
• Massive Scale: Organizations may have thousands/millions of devices
• Default Credentials: Many ship with admin/admin and users never change them
⚠️ Mirai Botnet (2016): Exploited IoT devices (cameras, DVRs, routers) using default credentials. Created a massive botnet that launched a 1.2 Tbps DDoS attack against Dyn DNS, taking down Twitter, Netflix, Reddit, and other major services. Lesson: Default credentials on IoT = weaponizable at scale.

4. SCADA / ICS Security

📖 SCADA (Supervisory Control and Data Acquisition): A system that monitors and controls industrial processes (power plants, water treatment, manufacturing). SCADA collects data from Remote Terminal Units (RTUs) and Programmable Logic Controllers (PLCs) and presents it to operators via a Human-Machine Interface (HMI).
📖 ICS (Industrial Control Systems): A broader term encompassing SCADA, Distributed Control Systems (DCS), and PLCs that control physical processes in critical infrastructure.
ICS ComponentFunction
PLC (Programmable Logic Controller)Controls physical equipment (valves, motors, switches) based on programmed logic
RTU (Remote Terminal Unit)Field device that collects sensor data and sends it to SCADA; can operate autonomously
HMI (Human-Machine Interface)Operator dashboard that displays process data and allows manual overrides
DCS (Distributed Control System)Controls processes within a single facility (e.g., a chemical plant); local loops
HistorianDatabase that records process data over time for analysis and compliance

ICS/SCADA Security Challenges

ChallengeDescription
Legacy SystemsMany run decades-old OSes (Windows XP) with no patches available
Availability PriorityUptime is paramount — systems cannot be taken offline to patch
Insecure ProtocolsModbus, DNP3, and OPC lack built-in authentication or encryption
Air-Gap ErosionOnce isolated networks are now connected to IT networks and internet
Safety ImplicationsCompromise can cause physical damage, environmental harm, or loss of life
⚠️ Stuxnet (2010): First known cyber weapon targeting ICS. Destroyed ~1,000 Iranian nuclear centrifuges by modifying PLC code to spin centrifuges at incorrect speeds while displaying normal readings on the HMI. Demonstrated that cyber attacks can cause physical destruction of equipment.
💡 Exam Tip: SCADA/ICS priorities differ from IT: Safety → Availability → Integrity → Confidentiality (not CIA). Availability matters more than confidentiality because downtime can endanger lives. The Purdue Model (ISA/IEC 62443) defines ICS network zones and security levels.

5. Cyber-Physical Systems (CPS)

📖 Cyber-Physical Systems (CPS): Systems where computational elements closely interact with physical processes, with feedback loops where physical processes affect computations and vice versa. CPS are a superset that includes IoT and ICS.
CPS ExampleCyber ComponentPhysical ComponentSafety Risk if Compromised
Autonomous VehiclesAI, sensors, navigation softwareSteering, braking, accelerationVehicle crashes, loss of life
Medical DevicesEmbedded controllers, wireless interfacesInsulin pumps, pacemakers, infusion pumpsIncorrect dosage, cardiac events
Smart GridSCADA, demand response systemsPower generation, transmission, distributionBlackouts, equipment damage
Industrial RobotsControl software, network interfacesMechanical arms, welding, assemblyWorker injury, product defects
🔑 CPS Security Principle: Because CPS directly interact with the physical world, compromises can lead to kinetic impacts — physical injury, environmental damage, or loss of life. This makes safety the #1 priority in CPS security, ahead of traditional CIA concerns.

6. Edge Computing Security

📖 Edge Computing: Processing data near the source of data generation (at the "edge" of the network) rather than sending it to a centralized cloud. Reduces latency, bandwidth usage, and enables real-time decision-making.
BenefitSecurity Consideration
Reduced latency for real-time processingPhysical security of distributed edge nodes is harder to maintain
Less data transmitted to cloudEach edge node is a potential attack target
Local data processing (privacy)Patching and updating distributed nodes is complex
Operates during cloud connectivity lossTamper-resistant hardware needed for exposed locations
💡 Exam Tip: Edge computing = "process data closer to where it's generated." Fog computing is an extension where processing happens between the edge and the cloud (an intermediate layer). For CISSP, know that edge computing reduces latency but increases the physical attack surface.

7. IoT Protocols

ProtocolPurposeKey FeaturesSecurity
MQTT
(Message Queuing Telemetry Transport)		Lightweight publish/subscribe messagingLow bandwidth; runs over TCP; designed for constrained devices and unreliable networksSupports TLS and username/password auth; no built-in encryption by default
CoAP
(Constrained Application Protocol)		RESTful protocol for constrained devicesRuns over UDP; similar to HTTP but lighter; supports GET, PUT, POST, DELETEDTLS (Datagram TLS) for security; optional — not enforced by default
ZigbeeShort-range, low-power mesh networkingIEEE 802.15.4; mesh topology; 250 kbps; range ~10-100m; up to 65,000 nodesAES-128 encryption built-in; key distribution is a known vulnerability
Z-WaveHome automation mesh networkSub-GHz frequency (less interference); mesh topology; range ~30m per hopAES-128 encryption (S2 framework); early versions had weak key exchange
BLE
(Bluetooth Low Energy)		Short-range, low-power communicationBluetooth 4.0+; 1 Mbps; range ~50m; used for wearables, beacons, sensorsAES-128; pairing vulnerabilities; BLE sniffing attacks
LoRaWANLong-range, low-power WANRange up to 15 km; very low data rate; star topology; excellent for remote sensorsAES-128 encryption at network and application layers
💡 Exam Tip: MQTT = publish/subscribe, TCP-based, lightweight messaging (most common IoT protocol on exam). CoAP = UDP-based REST for constrained devices. Zigbee = mesh network, IEEE 802.15.4. Know the transport layer: MQTT = TCP, CoAP = UDP.

8. Firmware Security

📖 Firmware: Software permanently programmed into a device's read-only memory (ROM/flash) that provides low-level control. In IoT, firmware controls the device's core behavior and is a prime target for persistent attacks.
Firmware RiskDescriptionMitigation
BackdoorsHardcoded credentials or hidden access points in firmwareCode review; firmware analysis tools; vendor vetting
No Secure UpdateDevice lacks OTA update mechanism or uses unsigned updatesRequire signed firmware updates; secure boot chain
ExtractionAttacker dumps firmware from flash chip for reverse engineeringEncrypt firmware; use secure boot; hardware tamper detection
Supply ChainFirmware modified during manufacturing or distributionSigned firmware; hardware root of trust (TPM/secure element)
🔑 Secure Boot Chain:
• Hardware Root of Trust: Immutable key stored in hardware (TPM, secure element) — the trust anchor
• Secure Boot: Each stage of boot verifies the signature of the next stage before executing it
• Measured Boot: Records measurements (hashes) of each boot component into TPM PCRs for remote attestation
• If any component is tampered with, the signature check fails and boot is halted

9. Memory Tricks & Mnemonics

🧠 "SCADA Priorities: SAIC not CIA": Safety → Availability → Integrity → Confidentiality. In ICS/SCADA, safety comes first because people can die. Remember: "SAIC, not CIA."
🧠 "IoT = Insecurity of Things": Default creds, no patches, constrained resources, long lifespans, massive scale = a perfect storm for insecurity. The Mirai botnet proved it.
🧠 "MQTT = Message Queue, TCP-based": MQTT uses TCP (reliable). CoAP uses UDP (lightweight). Remember: MQTT = "Must Queue, TCP" — it queues messages reliably. CoAP = "Constrained Over UDP."
