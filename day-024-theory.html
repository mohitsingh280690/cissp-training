⚠️ Exam Trap: Different frameworks use different lifecycle models (some have 4 stages, some 6, some 7). CISSP most often uses the 6-stage model: Create → Store → Use → Share → Archive → Destroy. Some sources merge Share and Transmit into one stage.

2. Real-World Relevance

Every data breach represents a failure at a specific lifecycle stage: data stolen in transit (failed Share controls), data leaked from an unsecured archive (failed Archive controls), sensitive data found on improperly disposed hardware (failed Destroy controls). Mapping controls to lifecycle stages helps organisations identify and fill security gaps systematically.

3. The Six Stages Overview

1. Create → 2. Store → 3. Use → 4. Share → 5. Archive → 6. Destroy

Each stage has unique security risks and requires specific controls. Classification established at creation should govern controls applied at all subsequent stages.

4. Stage 1: Create / Collect

Data enters the organisation — created internally or collected from external sources (customers, partners, sensors).

Security Controls at Creation

  • Data classification: Assign classification at creation — classification drives all downstream controls. If not classified at creation, it often defaults to sensitivity levels that may be inappropriate.
  • Data minimisation: Collect only what is necessary for the stated purpose (GDPR principle; also reduces liability — data you don't have cannot be breached).
  • Purpose specification: Document why the data is being collected and for what use — prevents later scope creep and regulatory violations.
  • Input validation: Sanitise data entering systems to prevent injection attacks and data integrity issues.
  • Data labelling: Apply classification labels to documents and data stores at creation.

5. Stage 2: Store

Data is persisted — on file systems, databases, object storage, or removable media. This is the stage where data rests at rest.

Security Controls at Storage

  • Encryption at rest: AES-256 for files, databases, volumes; Full Disk Encryption (BitLocker, FileVault) for endpoints; TDE (Transparent Database Encryption) for databases.
  • Access controls: File system permissions, database access controls — least privilege. Only authorised users and processes should have read/write access.
  • Backup and recovery: Regular backups, offsite storage, encryption of backup media, regular restoration testing.
  • Media controls: Control removable media, sanitise decommissioned storage before disposal.
  • Physical security: Secure server rooms, locked cabinets for sensitive printed material.

6. Stage 3: Use

Data is actively processed — queried, analysed, displayed, modified. This is the stage where data is in memory (RAM, CPU cache).

Security Controls at Use

  • Access controls: Role-based access control (RBAC), attribute-based access control (ABAC) — enforce least privilege at the application layer.
  • Data Loss Prevention (DLP): Monitor and prevent unauthorised copying, screen capture, printing, or emailing of sensitive data.
  • Logging and monitoring: Audit logs of data access and modification; SIEM integration; anomaly detection for unusual access patterns.
  • Screen/view masking: Redact sensitive fields in user interfaces for users who don't need full values (e.g., showing last 4 digits of SSN).
  • Secure coding: Parameterised queries, input validation to prevent injection when data is processed by applications.

7. Stage 4: Share / Transmit

Data moves — between systems, organisations, users, or across networks. This is the stage where data is in transit.

Security Controls at Sharing

  • Encryption in transit: TLS 1.2/1.3 for web/application traffic; IPSec for network-layer encryption; SSH for remote access; SFTP/FTPS for file transfer; SMTPS or S/MIME for encrypted email.
  • Data Loss Prevention (DLP): Network DLP to inspect and block unauthorised data exfiltration.
  • Third-party agreements: Data Processing Agreements (DPAs) under GDPR; NDAs; data sharing agreements defining use, controls, and return/destruction obligations.
  • Authorisation checks: Ensure the receiving party is authorised before sharing; need-to-know and classification compatibility.
  • Secure channels: Use only approved, encrypted channels — not public email for classified data.

8. Stage 5: Archive

Data is no longer actively used but must be retained for regulatory, legal, or business reasons.

Security Controls at Archiving

  • Long-term encryption: Ensure encryption algorithms used are still strong for the full retention period (e.g., avoid DES for 10-year archives — use AES-256 with key management for the archive period).
  • Retention policies: Define and enforce when archived data will be destroyed — avoid indefinite retention.
  • Access controls: Archive access should be more restricted than active data — limited to those with a business need to access historical records.
  • Integrity verification: Hashes/checksums to detect corruption or tampering over long storage periods.
  • Offsite/geo-redundant storage: Disaster recovery protection for archived records.
  • Media longevity: Monitor media degradation (tapes degrade; optical discs oxidise); refresh archives to new media as needed.

9. Stage 6: Destroy

Data has met its retention period or is no longer needed. It must be destroyed in a manner commensurate with its classification.

Security Controls at Destruction

  • Secure sanitisation: Use NIST SP 800-88 methods: Clear, Purge, or Destroy based on media type and data sensitivity (covered in depth on Day 30).
  • All copies: Destruction must include all copies — backups, archives, shadow copies, cloud replicas.
  • Certificate of destruction: Documented evidence that destruction occurred, including method used, media description, date, and authorised witness.
  • Remove from inventory: Update data inventory/asset registry to reflect destruction.
  • Vendor verification: If a third-party custodian handles destruction, require certificates of destruction in Data Processing Agreements.
⚠️ Exam Trap: Standard OS deletion (Delete key / recycle bin emptying) does NOT destroy data — it only removes the file system pointer. The data remains on disk and is recoverable by forensic tools. Secure destruction requires overwriting, degaussing, or physical destruction.

10. GDPR Lifecycle Alignment

GDPR's data protection principles map directly onto the data lifecycle:

GDPR PrincipleLifecycle StageRequirement
Purpose LimitationCreate / Use / ShareData collected for a specific purpose must be used only for that purpose — not repurposed without new consent or legal basis
Data MinimisationCreate / CollectCollect only what is necessary for the specified purpose
Storage LimitationArchive / DestroyData must not be retained longer than necessary for the purpose — defines when destruction is required
Integrity and ConfidentialityStore / Use / ShareAppropriate technical and organisational measures to protect data throughout its active life
AccuracyCreate / UsePersonal data must be accurate and kept up to date

11. Memory Tricks & Mnemonics

🧠 The Six Stages — "Can Somebody Use Some Archives Daily?"
Create → Store → Use → Share → Archive → Destroy
🧠 What Protects Data In Each State?
At REST → Encryption at rest (AES-256, FDE)
In TRANSIT → TLS, IPSec, VPN
In USE → Access controls, DLP, secure enclaves
"REST needs a blanket (encryption), TRANSIT needs a tunnel (TLS), USE needs a guard (access control)"
🧠 GDPR on Lifecycle:
Collect only what you need (minimisation)
Use only for the purpose said (purpose limitation)
Delete when done (storage limitation)
