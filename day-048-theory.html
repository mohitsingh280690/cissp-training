3. Asymmetric Cryptography Fundamentals

📖 Asymmetric Cryptography (Public-Key Cryptography): A cryptographic system that uses two mathematically related keys — a public key (shared openly) and a private key (kept secret). What one key encrypts, only the other can decrypt. Invented conceptually by Diffie and Hellman (1976), first implemented practically by Rivest, Shamir, and Adleman as RSA (1977).

How the Two Keys Work

OperationKey UsedPurpose
EncryptionRecipient's public keyConfidentiality — only the recipient's private key can decrypt
Digital SignatureSender's private keyAuthentication + Non-repudiation — anyone with the sender's public key can verify
ASYMMETRIC ENCRYPTION FLOW ╔══════════════════════════════════════════════════╗ ║ ║ ║ Alice wants to send a secret to Bob: ║ ║ ║ ║ 1. Bob publishes his PUBLIC key ║ ║ 2. Alice encrypts with Bob's PUBLIC key ║ ║ 3. Only Bob's PRIVATE key can decrypt ║ ║ ║ ║ Alice wants to SIGN a message: ║ ║ ║ ║ 1. Alice signs with her PRIVATE key ║ ║ 2. Anyone verifies with Alice's PUBLIC key ║ ║ 3. Proves Alice sent it (non-repudiation) ║ ║ ║ ╚══════════════════════════════════════════════════╝
⚠️ Critical Rule: Asymmetric encryption is 1,000–10,000× slower than symmetric encryption. It is never used to encrypt bulk data directly. Instead, it encrypts a symmetric session key, which then encrypts the actual data. This is the hybrid cryptosystem model used by TLS, PGP, and S/MIME.

Mathematical Hard Problems

Each asymmetric algorithm's security relies on a specific mathematical problem that is computationally infeasible to reverse:

Diffie-HellmanDiscrete logarithm problem

4. RSA (Rivest-Shamir-Adleman)

📖 RSA: The first practical public-key cryptosystem (1977). Named after Ron Rivest, Adi Shamir, and Leonard Adleman. Security based on the difficulty of factoring the product of two large prime numbers.

RSA Key Generation Process

  1. Choose two large random primes p and q
  2. Compute n = p × q (the modulus — part of both keys)
  3. Compute φ(n) = (p−1)(q−1) (Euler's totient)
  4. Choose e (public exponent, commonly 65537)
  5. Compute d (private exponent) such that d × e ≡ 1 mod φ(n)
  6. Public key = (n, e) — shared openly
  7. Private key = (n, d) — kept secret

RSA Key Sizes

Key SizeStatusUse
1024-bit❌ Deprecated — factorable with sufficient resourcesLegacy only; must be migrated
2048-bit✅ Current minimum standardMost TLS certificates, general use
3072-bit✅ Recommended for post-2030High-security applications
4096-bit✅ Maximum common implementationRoot CA certificates, long-term secrets
🔑 RSA Can Do Everything: RSA supports encryption, digital signatures, and key exchange. This makes it uniquely versatile among asymmetric algorithms. However, Diffie-Hellman is preferred for key exchange in modern protocols because it provides Perfect Forward Secrecy (PFS).
💡 Exam Tip: The CISSP exam expects you to know that RSA's security is based on factoring the product of two large primes — not "factoring large primes" (primes by definition cannot be factored). Also know: RSA key sizes must be ≥2048 bits today.

5. Diffie-Hellman Key Exchange (DH)

📖 Diffie-Hellman: The first published public-key protocol (1976), by Whitfield Diffie and Martin Hellman. It is a key agreement protocol — not an encryption algorithm. It allows two parties to establish a shared secret over an insecure channel without ever transmitting the secret itself.

How DH Works (Simplified)

DIFFIE-HELLMAN KEY EXCHANGE ╔══════════════════════════════════════════════════╗ ║ ║ ║ Agreed publicly: prime p, generator g ║ ║ ║ ║ Alice: Bob: ║ ║ Secret: a Secret: b ║ ║ Sends: A = g^a mod p → Receives A ║ ║ Receives B ← Sends: B = g^b mod p ║ ║ ║ ║ Alice computes: s = B^a mod p ║ ║ Bob computes: s = A^b mod p ║ ║ ║ ║ Both arrive at SAME shared secret s ║ ║ (without ever transmitting s) ║ ║ ║ ╚══════════════════════════════════════════════════╝

DH Variants

VariantDescriptionKey Feature
Static DHLong-term key pairsNo Perfect Forward Secrecy
Ephemeral DH (DHE)New key pair per session✅ Perfect Forward Secrecy — if long-term key is compromised, past sessions remain safe
ECDHEEphemeral DH using Elliptic Curves✅ PFS + smaller keys + faster — preferred in TLS 1.3
⚠️ DH Vulnerability: Basic DH is vulnerable to Man-in-the-Middle (MitM) attacks because it provides no authentication. An attacker can intercept both sides and establish separate shared secrets with each party. Solution: combine DH with digital signatures or certificates (as TLS does).
🔑 Perfect Forward Secrecy (PFS): A property where compromise of long-term keys does not compromise past session keys. Achieved by using ephemeral (temporary) DH keys for each session. TLS 1.3 mandates PFS — only DHE and ECDHE cipher suites are allowed.

6. ElGamal

📖 ElGamal: An asymmetric cryptosystem (1985) by Taher ElGamal based on the Diffie-Hellman discrete logarithm problem. Unlike DH (key exchange only), ElGamal extends the concept to provide both encryption and digital signatures.
FeatureDetail
Based OnDiscrete logarithm problem (same as DH)
CapabilitiesEncryption + Digital signatures
Key DrawbackCiphertext is twice the size of the plaintext (message expansion)
Used InGnuPG/PGP (as an alternative to RSA), some government systems
Patent StatusWas never patented (unlike RSA, which was patented until 2000)
💡 Exam Tip: ElGamal is less commonly tested than RSA or DH. Know that it's based on discrete logarithms (like DH) and that its ciphertext doubles in size. The exam is more likely to test you on RSA, DH, and ECC.

7. Elliptic Curve Cryptography (ECC)

📖 ECC: A modern approach to public-key cryptography based on the algebraic structure of elliptic curves over finite fields. Proposed independently by Neal Koblitz and Victor Miller in 1985. Provides equivalent security to RSA with much smaller key sizes.

Key Size Comparison: ECC vs. RSA

RSA Key SizeEquivalent ECC Key SizeSecurity Level (bits)
102416080
2048224112
3072256128
7680384192
15360512256
🔑 ECC Advantages:
• Smaller keys = less storage, bandwidth, and computational overhead
• Faster operations = ideal for mobile devices, IoT, smart cards
• Same security = ECC-256 ≈ RSA-3072
• Used in: TLS 1.3 (ECDHE), Bitcoin (ECDSA with secp256k1), smart cards, mobile

ECC-Based Algorithms

AlgorithmPurposeStandard
ECDH / ECDHEKey exchange (ephemeral = PFS)TLS 1.3 default
ECDSADigital signaturesBitcoin, TLS certificates
EdDSA (Ed25519)Digital signaturesSSH, Signal Protocol — faster and simpler than ECDSA
⚠️ Curve Selection Matters: Not all elliptic curves are equal. Some curves (e.g., NIST P-256) have been questioned for potential backdoors. Curve25519 (used by Ed25519/X25519) is considered highly trustworthy because its parameters are "nothing up my sleeve" numbers with no hidden structure.

8. Hybrid Cryptosystems

📖 Hybrid Cryptosystem: A practical approach that combines asymmetric and symmetric cryptography to get the best of both: the key distribution advantage of asymmetric and the speed of symmetric encryption.
HYBRID ENCRYPTION (How TLS Works) ╔══════════════════════════════════════════════════╗ ║ ║ ║ Step 1: KEY EXCHANGE (Asymmetric — slow) ║ ║ ┌──────────────────────────────────────────┐ ║ ║ │ Client & Server use ECDHE to establish │ ║ ║ │ a shared symmetric session key │ ║ ║ └──────────────────────────────────────────┘ ║ ║ ║ ║ Step 2: DATA ENCRYPTION (Symmetric — fast) ║ ║ ┌──────────────────────────────────────────┐ ║ ║ │ All application data encrypted with │ ║ ║ │ AES-256-GCM using the session key │ ║ ║ └──────────────────────────────────────────┘ ║ ║ ║ ║ Result: Secure key exchange + fast bulk data ║ ║ ║ ╚══════════════════════════════════════════════════╝

Real-World Hybrid Examples

ProtocolAsymmetric ComponentSymmetric Component

PGP/GPGRSA or ElGamal (encrypt session key)AES, CAST5, or Twofish (encrypt message)
S/MIMERSA (encrypt session key + digital signature)AES (encrypt email body)
SSHECDH/DH (key exchange) + Ed25519/RSA (host key auth)AES-256-CTR or ChaCha20

9. Asymmetric vs. Symmetric Comparison

FeatureSymmetricAsymmetric
KeysOne shared secret keyKey pair (public + private)
SpeedFast (1,000–10,000× faster)Slow — computationally intensive

Scalabilityn(n−1)/2 keys for n users2n keys for n users
Confidentiality✅ Yes✅ Yes

💡 Exam Tip: The key scalability formula is frequently tested. With symmetric encryption, 10 users need (10 × 9)/2 = 45 keys. With asymmetric, the same 10 users need 10 × 2 = 20 keys. The more users, the bigger the advantage of asymmetric key management.

10. Memory Tricks & Mnemonics

🧠 "RSA Does It All": RSA can do Encryption, Digital signatures, and Key exchange. DH can only do key exchange. ECC does key exchange + signatures but not direct encryption for CISSP purposes.
🧠 Hard Problems — "RIDE":
RSA = Integer factorization
DH / ElGamal = Discrete logarithm
ECC = Elliptic curve discrete logarithm
🧠 Key Size Rule of Thumb: "ECC punches above its weight" — ECC-256 ≈ RSA-3072. To remember: divide the RSA key size by roughly 12 to get the equivalent ECC size.
🧠 Symmetric Key Count: n(n−1)/2 → "Everyone shakes hands with everyone" (handshake problem). Asymmetric: 2n → "Everyone gets a pair of gloves."
