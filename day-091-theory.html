2. Real-World Organizational Relevance

Organizational Ownership

Application security (AppSec) teams own the code review program. Developers perform peer reviews and fix findings. DevSecOps pipelines integrate SAST/DAST into CI/CD (Jenkins, GitLab CI, GitHub Actions). Security architects define coding standards and review checklists.

Implementation Reality

Mature organizations run SAST scans on every code commit (shift-left security). DAST runs against staging environments before production deployment. Enterprise SAST tools (Checkmarx, Fortify, SonarQube, Snyk Code) scan millions of lines of code daily. The cost of fixing a vulnerability found in development is 6× cheaper than fixing it in production (NIST).

Failure Modes

  • SAST overload: Tool generates thousands of findings; developers ignore all of them due to alert fatigue
  • DAST-only approach: Finds runtime issues but misses code-level design flaws that SAST catches
  • No manual review: Automated tools miss business logic flaws (e.g., "user can approve their own expense report") — only human reviewers catch these

3. Static Application Security Testing (SAST)

SAST (Static Application Security Testing): Analysis of application source code, bytecode, or binary code WITHOUT executing the program. SAST tools examine code structure, data flow paths, and control flow to identify security vulnerabilities. Also called "white-box testing" because it requires access to the code internals.

How SAST Works

  • Input: Source code (Java, C#, Python, etc.), compiled bytecode, or binary
  • Analysis types: Data flow analysis (tracks tainted input through the code), control flow analysis (examines execution paths), pattern matching (known vulnerable code patterns)
  • Execution: Does NOT run the application — analyzes the code as text/structure
  • Output: List of potential vulnerabilities with exact code location (file, line number), CWE classification, severity rating

What SAST Finds Well

  • SQL injection (unsanitized user input flowing to database queries)
  • Cross-site scripting (XSS) — unsanitized output in HTML responses
  • Buffer overflows — array bounds not checked (C/C++)
  • Hard-coded credentials (passwords, API keys in code)
  • Insecure use of cryptographic APIs (weak algorithms, short keys)
  • Null pointer dereferences, resource leaks, race conditions

What SAST Cannot Find

  • Runtime/environment-specific vulnerabilities (misconfigured servers, exposed ports)
  • Authentication and session management flaws (requires running application)
  • Business logic flaws (requires human understanding of intent)
  • Issues in third-party components if only binary is available (no source code)

4. Dynamic Application Security Testing (DAST)

DAST (Dynamic Application Security Testing): Testing of a running application from the outside — sending crafted inputs and analyzing responses to find vulnerabilities. DAST does NOT need source code access; it interacts with the application exactly as an attacker would (via HTTP requests, form submissions, API calls). Also called "black-box testing."

How DAST Works

  • Input: URL/endpoint of a running application (web app, API, service)
  • Method: Automated crawling of the application; sends attack payloads (SQL injection strings, XSS payloads, path traversal sequences) to every input field
  • Execution: Application MUST be running — DAST interacts with the live application
  • Output: Identified vulnerabilities with the HTTP request that triggered them, response evidence, severity rating

What DAST Finds Well

  • Server/configuration vulnerabilities (exposed admin panels, default credentials, missing headers)
  • Authentication issues (broken authentication, session fixation)
  • Runtime SQL injection, XSS, CSRF
  • SSL/TLS misconfigurations
  • Error handling issues (verbose error messages leaking info)

What DAST Cannot Find

  • Code-level vulnerabilities not reachable through the interface
  • Dead code paths that the crawler doesn't trigger
  • Exact code location of the vulnerability (no source code access)
  • Complex business logic flaws requiring multi-step sequences

5. Interactive Application Security Testing (IAST)

IAST (Interactive Application Security Testing): A hybrid approach that combines elements of both SAST and DAST. An agent/sensor is installed INSIDE the running application (instrumentation) that monitors code execution in real time while the application processes requests. IAST sees both the internal code flow AND the external behavior simultaneously.

How IAST Works

  • Agent-based: A library/agent is deployed alongside the application (e.g., Java agent, .NET module)
  • Monitors from inside: Tracks data flow through the application in real time as requests are processed
  • Confirms vulnerabilities: Can pinpoint the exact code line AND the exact HTTP request that triggers it
  • Low false positives: Because it sees both code and runtime, it produces fewer false positives than SAST or DAST alone

Advantages of IAST

  • Combines SAST's code-level detail with DAST's runtime context
  • Lowest false positive rate of the three approaches
  • Works during QA testing (no separate scanning phase needed)
  • Provides exact code location + exact request that triggers the flaw

Limitations of IAST

  • Requires instrumentation — agent must be installed in the app (not always possible)
  • Language/framework specific — doesn't cover all platforms
  • Only finds vulnerabilities in code paths actually exercised during testing (coverage depends on test completeness)
  • Can introduce performance overhead in the instrumented application

Defense in Depth for Code Security: Best practice is to use ALL THREE in combination: SAST during development (shift-left), DAST in staging before deployment, IAST during QA testing. No single tool catches everything. Manual code review adds human judgment for business logic flaws.
⚠️ Exam Trap — SCA vs. SAST: Software Composition Analysis (SCA) is NOT the same as SAST. SCA scans dependencies and third-party libraries for known vulnerabilities (checks component versions against CVE databases). SAST scans YOUR code for flaws. Both are important but test different things.

7. Manual Code Review & Peer Review

Manual Code Review: A human reviewer reads and analyzes source code to find security vulnerabilities, logic errors, and design flaws that automated tools miss. Particularly effective for catching business logic vulnerabilities, authorization bypass flaws, and design-level issues.

Types of Manual Review

TypeDescriptionWhen to Use
Peer ReviewDeveloper's code reviewed by a colleague before merge; informal or lightweight processEvery code change (pull request review); standard DevOps practice
Formal Inspection (Fagan Inspection)Structured process: planning → overview → preparation → inspection meeting → rework → follow-up. Roles: moderator, author, reader, recorder, inspectorSafety-critical code; regulated environments; high-risk components
WalkthroughAuthor presents code to reviewers informally; less structured than formal inspectionKnowledge sharing; early design feedback
Security-Focused ReviewReviewer specifically looks for security flaws using a checklist (e.g., OWASP Code Review Guide)Security-critical modules; authentication, authorization, encryption code

Code Review Checklist (Security Focus)

  • ✅ Input validation on ALL external inputs (user input, API parameters, file uploads, environment variables)
  • ✅ Output encoding to prevent XSS (HTML encoding, URL encoding, JavaScript encoding)
  • ✅ Parameterized queries for ALL database operations (prevents SQL injection)
  • ✅ No hard-coded secrets (passwords, API keys, tokens) — use secrets management
  • ✅ Proper error handling — no stack traces or internal details exposed to users
  • ✅ Secure session management — proper token generation, expiration, invalidation
  • ✅ Authorization checks on EVERY endpoint (not just the UI — server-side enforcement)
  • ✅ Cryptographic functions using approved algorithms and adequate key lengths
  • ✅ Logging of security events WITHOUT logging sensitive data (no passwords or PII in logs)
Fagan Inspection: The most formal and structured code review method — 6 steps with defined roles. Named after Michael Fagan (IBM, 1976). The CISSP exam may reference this as the "formal inspection" or "Fagan inspection." Key point: defects are found but NOT fixed during the inspection meeting — rework happens separately.

8. CWE Top 25 – Most Dangerous Software Weaknesses

CWE Top 25: Published annually by MITRE, the CWE Top 25 ranks the most dangerous software weaknesses based on prevalence and severity. These are the weakness CATEGORIES (not specific vulnerabilities) that most frequently lead to exploitable vulnerabilities. Used as a baseline for secure coding standards and code review checklists.

CWE vs. CVE vs. CVSS (Reminder from Day 87):
CWE = weakness CATEGORY (e.g., "SQL injection is a type of weakness")
CVE = specific VULNERABILITY instance (e.g., "CVE-2021-44228 is a specific flaw in Log4j")
CVSS = SCORING system for severity (e.g., "this CVE has a CVSS score of 10.0")

9. Memory Tricks & Mnemonics

🧠 SAST / DAST / IAST — "SDI"
SAST = Source code (Static, no execution) = white box
DAST = Deployed app (Dynamic, running) = black box
IAST = Instrumented app (Inside agent) = gray box
False positives: S=High, D=Medium, I=Low (SDI order = descending FPs)
🧠 Fagan Inspection Steps — "POPIR-F"
Planning → Overview → Preparation → Inspection → Rework → Follow-up
"People Often Practice Inspecting, Reworking, Fixing"
🧠 Top 3 CWE Weaknesses
#1: Out-of-bounds Write (buffer overflow)
#2: XSS (script injection into pages)
#3: SQLi (database query injection)
"Overflows, Scripts, and Queries" — the three horsemen of web/app security
