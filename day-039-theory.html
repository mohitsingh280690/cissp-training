3. TCSEC (Orange Book)

📖 TCSEC (Trusted Computer System Evaluation Criteria): Published in 1983 by the U.S. DoD as DoD 5200.28-STD, nicknamed the "Orange Book" (part of the Rainbow Series). First widely-used security evaluation framework. Now superseded by Common Criteria but still appears on CISSP exams as foundational knowledge.

TCSEC Division Levels

DivisionLevelNameKey Characteristics
DDMinimal ProtectionNo security — system evaluated but failed to meet higher requirements
C (Discretionary)C1Discretionary Security ProtectionUser-level access controls; identification and authentication; separation of users and data
C2Controlled Access ProtectionObject reuse protection; individual accountability; audit trails; DAC with finer granularity
B (Mandatory)B1Labeled Security ProtectionSensitivity labels on all subjects and objects; MAC enforced; Bell-LaPadula implemented; security policy model
B2Structured ProtectionFormal security model; trusted path; covert channel analysis; design must be verifiable; least privilege enforced
B3Security DomainsReference monitor; small TCB; highly resistant to penetration; recovery procedures; full audit
AA1Verified DesignFormally verified design; mathematical proofs; trusted distribution; same as B3 but with formal methods
🔑 Key Facts:
• D (lowest) → A1 (highest) — higher letter/number = more security
• Division C = Discretionary (DAC-based)
• Division B = Mandatory (MAC-based, labels required)
• Division A = Verified (formal mathematical proofs)
• C2 was the practical minimum for DoD systems (Windows NT was C2-certified)
• Only evaluates confidentiality — no integrity or availability
⚠️ Limitation: TCSEC addresses ONLY confidentiality. It does not evaluate integrity or availability. This was a major criticism that led to ITSEC and eventually Common Criteria.

4. ITSEC (European Criteria)

📖 ITSEC (Information Technology Security Evaluation Criteria): Published in 1991 by the European Commission (France, Germany, Netherlands, UK). Addressed TCSEC's limitation by evaluating both functionality and assurance separately. Evaluated confidentiality, integrity, AND availability.

ITSEC Assurance Levels

LevelAssuranceTCSEC Equivalent
E0Inadequate assuranceD
E1Informal description of designC1
E2Informal description with testingC2
E3Source code and hardware auditB1
E4Formal security modelB2
E5Close correspondence between design and codeB3
E6Formal methods for complete verificationA1
🔑 Key Differences from TCSEC:
• ITSEC separates functionality from assurance — you rate each independently
• Evaluates confidentiality, integrity, AND availability (TCSEC only evaluated confidentiality)
• Functionality classes define WHAT the system does; assurance levels define HOW WELL it's verified
• E0 (lowest) through E6 (highest)
💡 Exam Tip: ITSEC's key innovation was separating functionality from assurance. If an exam question asks what ITSEC added over TCSEC, the answer is evaluating integrity/availability and separating functionality from assurance.

5. Common Criteria (ISO 15408)

📖 Common Criteria (CC): The international standard (ISO/IEC 15408) that replaced both TCSEC and ITSEC. Developed collaboratively by the U.S., Canada, UK, France, Germany, and the Netherlands. Mutually recognized via the Common Criteria Recognition Arrangement (CCRA).

Core Concepts

ConceptDefinition
Target of Evaluation (TOE)The product or system being evaluated
Protection Profile (PP)A document defining security requirements for a category of products (e.g., "firewalls"); written by the customer/community
Security Target (ST)A document written by the vendor describing how their specific product meets a PP; vendor's security claims
Security Functional Requirements (SFRs)Specify WHAT security functions the TOE provides
Security Assurance Requirements (SARs)Specify HOW WELL the TOE's security is assured (level of rigor)

Evaluation Assurance Levels (EAL 1–7)

EALNameDescriptionRigor
EAL 1Functionally TestedBasic functional testing; lowest meaningful assuranceMinimal
EAL 2Structurally TestedAnalysis of security functions using developer design docsLow
EAL 3Methodically Tested and CheckedGray-box testing; developer testing and evidence reviewModerate
EAL 4Methodically Designed, Tested, and ReviewedFull design review; independent testing; highest commercially practical levelHigh
EAL 5Semiformally Designed and TestedSemiformal design; covert channel analysis; modular design requiredVery High
EAL 6Semiformally Verified Design and TestedStructured implementation; comprehensive analysisVery High
EAL 7Formally Verified Design and TestedFormal mathematical verification; highest assurance; used for extreme riskMaximum
🔑 Key Facts:
• EAL 4 = highest level commercially practical (most COTS products aim for this)
• EAL 7 = formal mathematical proofs (equivalent to TCSEC A1)
• Protection Profile (PP) = customer says "I need..." (category of products)
• Security Target (ST) = vendor says "My product does..." (specific product)
• TOE = the thing being evaluated (Target of Evaluation)
💡 Exam Tip: Common Criteria replaced BOTH TCSEC and ITSEC. Remember: PP = what the customer wants (community requirements), ST = what the vendor claims (product-specific). EAL 4 is the practical ceiling for commercial products.

CC Process Flow

  Customer/Community        →  Creates Protection Profile (PP)
         ↓                      "What security do we need?"
  Vendor                    →  Creates Security Target (ST)
         ↓                      "Here's how my product meets the PP"
  Independent Lab           →  Evaluates TOE against ST
         ↓                      Tests and verifies claims
  Certification Body        →  Assigns EAL rating
         ↓                      "Product meets EAL X"
  CCRA Members              →  Mutually recognize the evaluation
                                (26+ countries accept each other's results)

⚠️ Common Misconception: Higher EAL ≠ "more secure." EAL measures the rigor of evaluation, not the strength of security features. An EAL 7 product evaluated with weak requirements could be less secure in practice than an EAL 4 product with strong requirements.

7. Memory Tricks & Mnemonics

🧠 "Orange = Old, E = European, EAL = Everyone Agreed Levels":
• Orange Book (TCSEC) = oldest, U.S. only
• ITSEC (E levels) = European
• Common Criteria (EAL) = internationally agreed
🧠 "TCSEC: Do Computers Bring Assurance?": D → C → B → A (lowest to highest). Division letters go up alphabetically as security increases.
🧠 "PP = People's Profile, ST = Seller's Target":
• Protection Profile = what the community/customer wants
• Security Target = what the seller/vendor claims
• TOE = the actual product being tested
🧠 "EAL 4 = For Commerce": EAL 4 is the highest commercially practical level. The "4" sounds like "for" — EAL 4 is "for" commercial use.
