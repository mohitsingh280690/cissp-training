3. DNS – Domain Name System

📖 DNS (Domain Name System): A distributed, hierarchical naming system that translates human-readable domain names (e.g., www.example.com) into IP addresses (e.g., 93.184.216.34). Often called "the phone book of the internet." Operates primarily over UDP port 53 (small queries) and TCP port 53 (large responses, zone transfers).

DNS Hierarchy

. (Root) │ ┌───────────────┼───────────────┐ .com .org .net ← Top-Level Domains (TLDs) │ │ example.com wikipedia.org ← Second-Level Domains │ www.example.com mail.example.com ← Subdomains / FQDNs DNS Server Types: ┌──────────────────┬────────────────────────────────────────────────────────┐ │ Root Name Server │ 13 root server clusters (A–M); knows TLD locations │ │ TLD Name Server │ Controls .com, .org, .net etc.; knows authoritative NS │ │ Authoritative NS │ Holds actual DNS records for a domain │ │ Recursive Resolver│ Client-facing; queries hierarchy on client's behalf │ │ Caching Resolver │ Stores recent answers to reduce query volume │ └──────────────────┴────────────────────────────────────────────────────────┘
🔑 Authoritative vs. Recursive: An authoritative DNS server holds the actual zone records for a domain and answers definitively. A recursive resolver (like your ISP's DNS or 8.8.8.8) goes through the hierarchy on your behalf and caches results. Most attacks target recursive resolvers, not authoritative servers, because poisoning a recursive resolver affects all clients that use it.

⚠️ Zone Transfer Risk (AXFR): If a DNS server responds to AXFR requests from any IP address, an attacker can obtain a complete list of all hostnames, IPs, and service records in your domain — essentially a roadmap for reconnaissance. Always restrict AXFR (TCP/53) to authorized secondary nameservers by IP address.

5. DNS Resolution Process

DNS Query Resolution (Iterative from Recursive Resolver): Client Recursive Resolver Root NS TLD NS (.com) Authoritative NS │ │ │ │ │ │─ "Where is example.com?" ▶│ │ │ │ │ │─ "Where is .com?" ────▶│ │ │ │ │◀─ "Ask .com TLD NS" ───│ │ │ │ │─ "Where is example.com?" ────────────▶│ │ │ │◀─ "Ask example.com's NS" ─────────────│ │ │ │─ "What is www.example.com?" ──────────────────────────▶│ │ │◀─ "93.184.216.34 (TTL=3600)" ─────────────────────────│ │◀─ "93.184.216.34" ────────│ │ [Client caches answer] │ [Resolver caches answer for TTL duration]
🔑 TTL (Time To Live) in DNS: Each DNS record has a TTL value (in seconds) that tells resolvers how long to cache the answer. Short TTLs (60–300 seconds) allow rapid changes but increase query load. Long TTLs (86400 = 1 day) reduce load but mean changes take longer to propagate. In a DNS poisoning attack response, organizations should lower TTLs to flush cached malicious records faster.

DNSSEC – DNS Security Extensions

📖 DNSSEC: A suite of extensions to DNS (RFC 4033–4035) that adds cryptographic signatures to DNS records. Responses are signed with a private key; clients verify with the corresponding public key. DNSSEC provides data origin authentication and data integrity — but NOT confidentiality (records are not encrypted).
DNSSEC ComponentPurpose
RRSIGResource Record Signature — cryptographic signature covering a set of DNS records
DNSKEYContains the public key used to verify signatures; stored in the zone
DSDelegation Signer — hash of the child zone's DNSKEY, stored in the parent zone to build the chain of trust
NSEC/NSEC3Next Secure Record — proves that a domain does NOT exist (authenticated denial of existence)
Chain of TrustRoot zone → TLD zone → Authoritative zone, each signed and verifiable up to the trusted root anchors
💡 Exam Tip: DNSSEC prevents DNS cache poisoning by cryptographically authenticating responses. However, DNSSEC does not encrypt DNS traffic (DNS over HTTPS/DoH or DNS over TLS/DoT provides confidentiality). DNSSEC = integrity/authenticity; DoH/DoT = confidentiality. Know the distinction.

7. DHCP – Dynamic Host Configuration Protocol

📖 DHCP (Dynamic Host Configuration Protocol): Automatically assigns IP addresses and network configuration (subnet mask, default gateway, DNS server, lease time) to hosts on a network. Defined in RFC 2131. Uses UDP port 67 (server) and UDP port 68 (client). The assignment process is known as DORA.

The DORA Process

DHCP DORA Process: Client DHCP Server │ │ │──── DISCOVER (Broadcast) ──────────────────────────▶│ "Anyone have an IP for me?" │ src: 0.0.0.0:68, dst: 255.255.255.255:67 │ │ │ │◀─── OFFER (Unicast or Broadcast) ───────────────────│ "I offer 192.168.1.50, lease=8h" │ │ │──── REQUEST (Broadcast) ───────────────────────────▶│ "I'll take 192.168.1.50 please" │ (Broadcast so other servers know the offer taken) │ │ │ │◀─── ACK (Unicast or Broadcast) ─────────────────────│ "Confirmed — it's yours for 8h" │ │ │ [Client configures IP, subnet, gateway, DNS] │
🔑 DHCP Options (Security-Relevant): DHCP does more than assign an IP. Option 3 = Default Gateway; Option 6 = DNS Servers; Option 15 = Domain Name; Option 43 = Vendor-specific info; Option 82 = DHCP Relay Agent Information (used with DHCP snooping). An attacker controlling DHCP can set malicious DNS servers and a rogue gateway — redirecting all traffic immediately.

DHCP Lease Management

EventTimingDescription
RenewalAt 50% of lease time (T1)Client unicasts REQUEST to the original server to extend its lease
RebindAt 87.5% of lease time (T2)Client broadcasts REQUEST if renewal failed — accepts any DHCP server
ExpiryAt 100% of lease timeClient must release IP and restart DORA process
ReleaseClient-initiatedClient sends DHCPRELEASE when done using the address (e.g., on shutdown)

8. DHCP Attacks & Defences

AttackMechanismImpactDefence
DHCP StarvationAttacker floods network with DHCP DISCOVER packets using thousands of spoofed MAC addresses, exhausting the server's IP address poolLegitimate clients cannot obtain IP addresses — network DoS; often precedes rogue DHCP server attackDHCP snooping with rate limiting per port; 802.1X port authentication; port security (limit MACs per port)
Rogue DHCP ServerAttacker connects unauthorized DHCP server to network. When clients send DISCOVER, rogue server replies faster than the legitimate server, assigning malicious gateway/DNSMITM via gateway manipulation; DNS hijacking for entire network segmentDHCP snooping — designates trusted ports (uplinks to authorized DHCP servers) vs. untrusted ports (client-facing); blocks DHCP OFFERs from untrusted ports
🔑 DHCP Snooping (The Key Defence): DHCP snooping is a Layer 2 switch feature that intercepts DHCP messages. It builds a binding table mapping MAC address → IP address → switch port → VLAN. Trusted ports (connected to real DHCP servers) can send any DHCP message. Untrusted ports (client ports) are blocked from sending DHCP OFFERs and ACKs. This table also feeds Dynamic ARP Inspection (DAI) — making DHCP snooping the foundation of Layer 2 security.
💡 Exam Tip: The CISSP exam expects you to know that DHCP snooping is the primary defence against both DHCP starvation and rogue DHCP servers. It creates a binding table that also supports DAI (for ARP poisoning defence). The technologies are linked: DHCP Snooping → binding table → DAI → ARP Inspection.

9. NTP – Network Time Protocol

📖 NTP (Network Time Protocol): Synchronizes clocks of networked systems to within a few milliseconds of Coordinated Universal Time (UTC). Defined in RFC 5905. Operates over UDP port 123. Uses a hierarchical system of stratum levels (0–15). Stratum 0 = atomic clocks / GPS; Stratum 1 = servers directly connected to stratum 0; each level adds one stratum.

Why NTP is a Security Control

DependencyWhy Clock Accuracy Matters
Kerberos AuthenticationKerberos tickets have timestamps and will be rejected if the clock skew between client and KDC exceeds 5 minutes (default). NTP required for AD environments.
Digital Certificates / TLSCertificates have validity periods (notBefore / notAfter). A clock that is wrong may cause certificates to be rejected as expired or not-yet-valid.
Log Correlation / ForensicsIncident investigation depends on accurate timestamps across all systems. A 10-minute drift between systems makes event correlation unreliable or impossible.
Audit TrailsRegulations (PCI-DSS Requirement 10.4, SOX) explicitly require synchronized time stamps for log integrity.
Two-Factor Authentication (TOTP)Time-based One-Time Passwords (TOTP, RFC 6238) expire within 30 seconds. Without NTP, TOTP codes will be invalid.

NTP Security Threats

ThreatMechanismImpactDefence
NTP Amplification DDoSNTP's "monlist" command (Mode 6/7) returns a list of the last 600 clients. Attacker spoofs victim IP in small request; enormous response floods victim (amplification ~200x)Volumetric DDoS — historically one of the largest DDoS amplifiersUpgrade to NTPd v4.2.7+; disable monlist (noquery directive); BCP38 ingress filtering
NTP SpoofingAttacker sends forged NTP responses to shift a target's clock forward or backward (time shift attack)Kerberos auth failure; certificate validation bypass; log timestamp manipulation
NTP MITMIntercept NTP traffic and provide false timeSame as spoofing — time manipulationNTP authentication using symmetric keys or autokey; use NTPsec or Chrony with authentication
⚠️ NTP vs. SNTP: SNTP (Simple NTP) is a simplified version that provides less accuracy and no loopback stability. Many IoT devices and embedded systems use SNTP. It is less secure and less accurate than full NTP. For security-sensitive environments, use NTPv4 with authentication.

10. SNMP – Simple Network Management Protocol

📖 SNMP (Simple Network Management Protocol): A protocol for monitoring and managing network devices (routers, switches, servers, printers). An SNMP manager polls SNMP agents on managed devices. Agents expose data through a Management Information Base (MIB). Uses UDP port 161 (queries to agent) and UDP port 162 (traps from agent to manager).

SNMP Architecture

SNMP Versions Comparison

VersionAuthenticationPrivacy (Encryption)Security Assessment
SNMPv1Community string (cleartext)None❌ Critically insecure — community string sent in plaintext; passive sniffing gives full access
SNMPv2cCommunity string (cleartext)None❌ Same as v1 for security; added bulk operations and improved error handling, not security
SNMPv3USM (User-based Security Model) — HMAC-MD5 or HMAC-SHADES (deprecated) or AES-128/192/256✅ Secure — authentication + encryption available; mandatory for sensitive environments

SNMPv3 Security Models

Security LevelAuthenticationPrivacyUse When
noAuthNoPrivNoneNoneNever — equivalent to SNMPv1/v2c in security
authNoPrivHMAC-SHA/MD5NoneWhen authentication integrity is needed but confidentiality is not critical
authPrivHMAC-SHA/MD5AES/DESAlways preferred — full authentication + encryption
⚠️ Community Strings (SNMPv1/v2c): Default community strings are "public" (read-only) and "private" (read-write). These are the most commonly unchanged defaults in network environments. If an attacker can read SNMP data using "public", they get full network topology (routing tables, interface details, system descriptions). Write access with "private" can allow reconfiguration of network devices.

SNMP Security Best Practices

PracticeDetail
Upgrade to SNMPv3 authPrivOnly version with authentication AND encryption
Change default community stringsNever use "public"/"private" in production
Use access control / ACLsRestrict SNMP access to authorized NMS IP addresses only
Block UDP 161/162 at perimeterSNMP should never be accessible from the internet
Use read-only community (if v1/v2c unavoidable)Disable write community strings to limit damage; still read ONLY when v3 is not possible
Monitor SNMP trap trafficUnexpected traps may indicate device failures or attacks
💡 Exam Tip: The CISSP exam consistently tests: "Which version of SNMP provides both authentication and encryption?" → SNMPv3 with authPriv security level. SNMPv2c added useful features but no security improvements over v1. Know that SNMP uses UDP 161 (queries) and UDP 162 (traps from agent to manager).

11. Memory Tricks & Mnemonics

🧠 DHCP DORA:
Discover → Offer → Request → Acknowledge
Memory phrase: "Dogs Often Run Away" — always in this exact order, no exceptions.
🧠 DNS Record Types — "AAAA MX NS PTR SOA TXT":
Memory phrase: "All Amazing Mail Never Prevents Stalking Online Targets Thoroughly"
Focus pairs: A = IP address | MX = mail | NS = name server | PTR = reverse | SOA = zone admin | TXT = SPF/DKIM/DMARC
🧠 DNSSEC vs. DoH/DoT:
DNSSEC = signs records → integrity + authenticity (no encryption of traffic)
DoH/DoT = encrypts the query/response → confidentiality (no signing of records)
"DNSSEC signs; DoH/DoT hides. You need both for full DNS protection."
🧠 SNMP Version Security — "1 and 2c = Clear; 3 = Care":
v1 and v2c = community strings in cleartext = NO security
v3 = authentication (HMAC) + privacy (AES) = YES security
Always choose v3 authPriv for managed environments.
🧠 NTP Kerberos 5-Minute Rule:
"Kerberos dies at 5 minutes of clock drift."
If your clock is off by more than 5 minutes from the KDC, all Kerberos authentication fails. NTP is not optional in Active Directory environments.
🧠 DNS Cache Poisoning Defence Chain:
"DNSSEC + Source Port Randomization + Query ID Randomization = Kaminsky defences"
The Kaminsky bug (2008) needed all three to be properly mitigated. DNSSEC is the definitive long-term solution.
🧠 DHCP Snooping Feeds DAI:
"Snooping builds the table; DAI uses the table."
DHCP Snooping → binding table (MAC/IP/port/VLAN) → Dynamic ARP Inspection validates ARP against it.
