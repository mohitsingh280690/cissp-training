1. Where This Fits in CISSP

AspectDetail
Domain5 – Identity and Access Management (13%) — Authentication methods and credential management
Why TestedAuthentication is the linchpin of access control. CISSP tests which factors belong to which types, the distinction between step-up vs. step-back authentication, password policy components (specifically what "aging" and "history" do), token types (synchronous vs. asynchronous), and common authentication protocol weaknesses.
Typical Question Styles"A hard token generates a value every 60 seconds — which type is this?"; "What is the PRIMARY weakness of passwords?"; "SMS OTP is which MFA type?"; "Which authentication protocol sends passwords in cleartext?"
Cross-Domain LinksDay 70 (IAAA framework); Day 72 (biometrics & MFA in depth); Day 73 (Kerberos authentication flow); Day 74 (federated authentication); Domain 8 (OWASP A07); Day 38 (cryptography in PKI — certificate-based authentication)
Key Exam TrapsSMS OTP is Type 2, NOT Type 3 (it's something you HAVE — your phone). Two-factor ≠ two-step (two-step can use the same factor type twice). Synchronous tokens are time-based; asynchronous tokens are challenge-response based. PAP sends passwords in cleartext; CHAP never sends the password itself.

2. Real-World Organizational Relevance

IssueContext
Password re-use across platformsAverage user has 100+ online accounts. Studies show 50–60% of users re-use the same password. A single breach of a low-security site (credential stuffing) enables attackers to access high-value accounts. Enterprise single password reuse is equally dangerous.
Phishing as auth bypassPhishing attacks don't break cryptography — they steal credentials directly. Traditional MFA (TOTP, SMS) can be bypassed by real-time phishing proxies (EvilProxy, Modlishka). Only phishing-resistant MFA (FIDO2/WebAuthn, PIV/CAC) binds the credential to the origin domain.
Password manager adoptionPassword managers enable unique, complex, long passwords for every site without memorization burden. NIST 800-63b explicitly recommends password managers and eliminates mandatory rotation policies — rotate only on evidence of compromise.
Regulatory MFA mandatesPCI-DSS 4.0: MFA required for all admin access and all remote access to CDE. CISA/NSA: phishing-resistant MFA for all federal agencies. Healthcare: MFA recommended for all electronic PHI access.

3. Type 1 – Something You Know

MechanismDescriptionSecurity Notes
PasswordA string of characters known only to the user and the system (or its cryptographic representation)Most common; most attacked. Vulnerable to guessing, brute force, dictionary attacks, phishing, keyloggers, shoulder surfing, credential stuffing
PINTypically 4–8 digit numeric code. Often paired with a physical token (card) to form MFA.Very short — security depends on accompanying lock-out policy. PIN alone is weak; PIN + smart card = strong MFA
PassphraseA sentence or series of words used as authentication. Longer than passwords, often easier to remember.Recommended by NIST and Bruce Schneier. Entropy from length beats complexity. Example: "correct-horse-battery-staple"
Security questionsKnowledge-based authentication (KBA) — answers to pre-set questionsWeakest knowledge factor: answers often guessable from public social media. NIST 2017 guidance recommends AGAINST security questions as authentication factors.
PatternSwipe pattern on touchscreenVulnerable to smudge attacks (greasy fingerprint traces on screen reveal pattern); entropy is lower than it appears (limited starting points)

4. Password Management & Policies

Password Policy Components (CISSP Must-Know)

Policy ElementDefinitionPurpose
Minimum lengthMinimum number of characters required (NIST recommends ≥ 8, ideally ≥ 15 for privileged accounts)Increases entropy; makes brute force computationally expensive
ComplexityRequires mix of uppercase, lowercase, numbers, and special charactersIncreases character space; however NIST 800-63b now discourages mandatory complexity in favour of length (users create predictable patterns: P@ssw0rd1)
Aging / ExpirationMaximum number of days before password must be changedLimits window of exposure if password is silently compromised. NIST now recommends against periodic forced rotation — rotate only on breach evidence.
HistoryPrevents re-use of N previous passwordsPrevents users cycling through passwords to return to their favourite. Must be combined with aging to be effective.
Minimum ageMinimum number of days before a password can be changed againPrevents users from immediately cycling through history list to return to previous password
Lockout thresholdNumber of failed attempts before account is lockedPrevents online brute force attacks. Must balance security vs. DoS risk (attacker could intentionally lock out users)
Lockout durationHow long account remains locked (or until admin unlocks)Time lockout: auto-unlock after N minutes. Administrative unlock: requires IT helpdesk — higher security but creates DoS risk
💡 Exam Tip — NIST 800-63b Modern Guidance:
NIST SP 800-63b (2017, updated 2024) made significant changes that appear in CISSP questions:
✓ Remove mandatory periodic password changes (change only on evidence of compromise)
✓ Don't require specific complexity rules (complexity encourages predictable substitution patterns)
✓ Do require minimum length ≥ 8, recommend ≥ 15
✓ Check passwords against known-compromised-password lists (HIBP database)
✓ Allow pasting passwords (supports password manager adoption)
✓ Eliminate security questions as sole authentication factor

Password Storage (How Passwords Should Be Stored)

MethodSecurity LevelNotes
Plaintext storage❌ Never acceptableComplete security failure — one breach exposes all passwords
Encrypted storage❌ InadequateEncryption key can be stolen; decryption possible — same result as plaintext if key compromised
MD5 / SHA-1 hash⚠️ UnacceptableFast hash functions: billions of attempts/second GPU-accelerated; precomputed rainbow tables exist; no salt = identical passwords have identical hashes
Salted SHA-256/512⚠️ MarginalSalt prevents rainbow tables; but SHA-2 is still fast — brute force is feasible on GPUs
bcrypt / scrypt / Argon2✅ CorrectSlow-by-design adaptive hash functions; work factor can be increased with hardware improvements; Argon2 won PHC 2015 and is NIST-recommended

5. Type 2 – Something You Have

Token TypeHow It WorksSecurity Notes
Synchronous token (time-based)Generates OTP using current time + shared secret seed. Synchronized with server clock (±30–60 sec tolerance). OTP changes every 30–60 seconds.TOTP (RFC 6238). Requires time sync; if clocks drift, authentication fails. Vulnerable to real-time phishing (attacker relays OTP immediately). Examples: Google Authenticator, Authy, RSA SecurID (hybrid)
Asynchronous token (challenge-response)Server issues a random challenge; token computes response using shared secret + challenge. No time dependency.HOTP (RFC 4226) — counter-based variant. Server sends challenge; token/app generates HMAC-based OTP. More resistant to replay (each challenge unique). RSA SecurID original design.
Smart card (contact)Contains a microprocessor and cryptographic key stored on card. Performs signing operations on-chip (private key never leaves card). Requires physical insertion into reader.PIV (Personal Identity Verification — US Govt FIPS 201), CAC (Common Access Card — DoD). PIN + card = strong MFA. Card alone cannot authenticate (requires PIN). Best protection against remote attacks — attacker needs physical card.
Contactless smart cardISO/IEC 14443 NFC-based communication. Same cryptographic capability as contact card.Risk: relay attacks (attacker relays communication between card and rogue reader). Distance limitations mitigate (active relay attacks extend range). Proximity-based authentication.
Mobile soft token / authenticator appSoftware implementation of TOTP/HOTP on smartphone. Tied to device via enrollment (secret key installed at provisioning time).Convenience; device loss = token loss; phone compromise (malware) could extract secret. Better than SMS. Push-based MFA (Duo, Microsoft Authenticator) adds usability but MFA fatigue attacks exploit push notifications.
SMS OTPOne-time code sent via text message to registered phone number.Type 2 (something you have — your phone). Weakest Type 2 form: subject to SIM-swapping, SS7 protocol attacks, and real-time phishing. NIST deprecated SMS OTP as authenticator in 2017 (SP 800-63b). Still common but not recommended for high-risk use cases.
Security key (FIDO2/U2F)Hardware device (USB/NFC/Bluetooth) that performs public-key cryptography. Different key pair per site. Challenge-response signs challenge with site-specific private key.Most secure Type 2. Phishing-resistant: key pair bound to origin domain — works only on the registered site. Examples: YubiKey, Google Titan Key. Recommended for high-privilege accounts. FIPS 140-2 validated versions available.

6. OTP: HOTP and TOTP

HOTP (Counter-Based) — RFC 4226 ──────────────────────────────────────── Server: shared secret + counter → HMAC-SHA1 → truncate → 6-8 digit OTP Token: same shared secret + counter → same OTP Both server and token increment counter on successful use. Risk: counter de-sync if OTP generated but not used (server allows look-ahead window) TOTP (Time-Based) — RFC 6238 (extends HOTP) ──────────────────────────────────────── Server: shared secret + floor(Unix_time / 30) → HMAC-SHA1 → truncate → OTP Token: same shared secret + same time window → same OTP OTP valid for 30-second window (±1 window for clock drift tolerance) Risk: real-time phishing can relay valid OTP within the 30-second window

7. Type 3 – Something You Are (Preview)

📖 Biometrics (Full coverage in Day 72): Authentication using physical or behavioral characteristics unique to the individual. Examples: fingerprint, iris, retina, facial recognition, voice, signature dynamics, gait analysis.
Key ConceptDefinitionCISSP Signal
FRR – False Rejection RateRate at which legitimate users are incorrectly denied access. Also: Type I error.High FRR = inconvenient for users; fingerprint scanner that rejects 1 in 5 legitimate scans
FAR – False Acceptance RateRate at which unauthorized users are incorrectly granted access. Also: Type II error.High FAR = security risk; security determines acceptable FAR threshold
CER / EERCrossover Error Rate — the point where FAR = FRR. Used to compare biometric systems. Lower CER = better system.When comparing two biometric systems, the one with lower CER is the better system overall

8. Multi-Factor Authentication Deep Dive

🔑 True MFA Rule: Two or more factors from DIFFERENT categories. Same-category factors = multi-step, NOT multi-factor.
ScenarioMFA?Reasoning
Password + PIN❌ NoBoth Type 1 (something you know). Two-step, same category.
Password + TOTP code✅ YesType 1 (know) + Type 2 (have — phone with auth app)
Smart card + PIN✅ YesType 2 (have — card) + Type 1 (know — PIN)
Fingerprint + iris scan❌ NoBoth Type 3 (something you are). Two-step biometric only.
Password + fingerprint✅ YesType 1 (know) + Type 3 (are)
FIDO2 key + fingerprint-to-unlock✅ YesType 2 (have — key) + Type 3 (are — fingerprint unlocks on-device)

MFA Attack Vectors

AttackHow It WorksDefence
Real-time phishing proxyAttacker proxies the real site; victim authenticates to attacker who relays credentials + OTP in real-time to real site before OTP expiresPhishing-resistant MFA (FIDO2/WebAuthn); origin-binding prevents proxy relay
MFA fatigue (push bombing)Attacker has username + password; floods victim with push MFA requests until user accidentally approvesNumber matching in push notifications; context-aware push (show IP, location); FIDO2
SIM swappingAttacker socially engineers telco to port victim's phone number to attacker's SIM; intercepts SMS OTPsDon't use SMS OTP for high-value accounts; use authenticator app or FIDO2
SS7 exploitationAttacker exploits SS7 protocol weaknesses in the global telephone signaling system to intercept SMS messagesSame as above — avoid SMS OTP

9. Authentication Protocols

ProtocolTypeSecurityNotes
PAP (Password Authentication Protocol)Password-based❌ Sends password in cleartext across the link. Completely insecure on public networks.PPP-based; legacy; never use unless link is independently encrypted (IPsec tunnel)
CHAP (Challenge-Handshake Authentication Protocol)Challenge-response✅ Password never transmitted; server challenges client; client responds with hash(challenge + secret)PPP-based; susceptible to replay only if same challenge reused; MS-CHAPv2 is deprecated (MPPE key derivation weakness)
EAP (Extensible Authentication Protocol)FrameworkVaries by EAP type (EAP-TLS = very strong; EAP-MD5 = weak)Used in 802.1X wired/wireless authentication; tunneled EAP types (PEAP, EAP-TTLS) protect inner authentication
RADIUS (Remote Authentication Dial-In User Service)AAA protocolEncrypts password in Access-Request; uses MD5-based obfuscation (only for password field)UDP-based; ports 1812/1813; single shared secret between NAS and RADIUS server; widely used in 802.1X, VPN, wireless
TACACS+AAA protocolEncrypts entire payload (not just password). Separates Authentication, Authorization, AccountingTCP port 49; Cisco proprietary; preferred in network device (router/switch) administration vs. RADIUS; provides per-command authorization
💡 Exam Tip — RADIUS vs. TACACS+:
RADIUS: UDP, encrypts only password, combines AuthN and AuthZ, better for network access (wireless, VPN)
TACACS+: TCP, encrypts entire body, separates AuthN/AuthZ/Accounting, better for network device administration (per-command authorization for routers/switches)

10. Authentication Attack Methods

AttackDescriptionCountermeasure
Brute forceTry every possible password combination until the correct one is found. Offline brute force (hash cracking) vs. online brute force (live login attempts).Account lockout (online); long passwords + slow hash (offline); CAPTCHA (online)
Dictionary attackTry a list of common words, phrases, and known passwords. Much faster than pure brute force.Long passphrases; complexity rules (limited); check against HIBP known-compromised-password lists
Rainbow table attackPrecomputed table of hash→password mappings. Fast lookups of hashed passwords stolen from database.Salting (unique per-user random value combined with password before hashing) — makes precomputed tables useless
Credential stuffingAutomated testing of username/password pairs from breached databases against other sites, exploiting password reuse.MFA; breach notification + forced reset; bot detection; device fingerprinting
Password sprayingTry a few common passwords (Password1, Welcome1) against many accounts, avoiding lockout thresholds.Account lockout; anomalous login detection; MFA; FIDO2
PhishingTrick user into submitting credentials to a fake site. Captures plaintext credentials regardless of password complexity.Phishing-resistant MFA (FIDO2); security awareness training; anti-phishing DNS / email filtering; DMARC enforcement
Pass-the-hashIn Windows NTLM, steal the hash from memory (via Mimikatz) and use it directly to authenticate without knowing the plaintext password.Credential Guard (Windows); Protected Users group; disable NTLM; use Kerberos; privileged access workstations (PAW)
Replay attackCapture authentication message/token and retransmit it later to authenticate as the victim.Nonces (one-time numbers in challenges); timestamps; challenge-response protocols (CHAP, Kerberos)

11. Memory Tricks & Mnemonics

🧠 Authentication Factor Types — "Know, Have, Are / KHA":
Type 1 = Know (passwords, PINs, passphrases) — cognitive
Type 2 = Have (tokens, smart cards, FIDO2 keys) — possession
Type 3 = Are (biometrics) — inherence

MFA = different letters. Password + PIN = same letter (KK) = NOT MFA.
🧠 RADIUS vs. TACACS+:
RADIUS = Rapid UDP, encrypts only password, for Network Access (wireless, VPN)
TACACS+ = TCP, Total encryption, separates AuthN/AuthZ/Accounting, for device Administration (routers/switches)
🧠 Password Storage Quality Ladder:
Plaintext → Encrypted → Fast Hash (MD5/SHA) → Salted Fast Hash → Slow Hash (bcrypt/Argon2) ← Only this is correct

Type 2 – HaveSomething you have: smart card, FIDO2 key, TOTP app, SMS (weakest Type 2); SMS OTP subject to SIM-swap/SS7
Type 3 – AreSomething you are: biometrics (full detail Day 72); cannot be "changed" if compromised
True MFATwo+ factors from DIFFERENT categories; password + PIN is NOT MFA
TOTP vs. HOTPTOTP = time-based (synchronous, 30-sec window); HOTP = counter-based (asynchronous); both HMAC-SHA1 based
PAP vs. CHAPPAP = cleartext password (dangerous); CHAP = challenge-response (password never transmitted)

Password storageOnly bcrypt/scrypt/Argon2 (adaptive slow hashes) are acceptable; always salt; never store plaintext or fast hashes
NIST 800-63bNo mandatory password rotation; no mandatory complexity; check against breach lists; minimum length ≥ 8/recommend ≥ 15
