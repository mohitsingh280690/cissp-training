3. PKI Components

📖 Public Key Infrastructure (PKI): A framework of policies, procedures, hardware, software, and people that manages the creation, distribution, storage, use, and revocation of digital certificates. PKI binds public keys to identities through a trusted third party.
ComponentRoleAnalogy
Certificate Authority (CA)Issues, signs, and revokes digital certificates; the root of trustLike a government passport office — trusted authority that issues identity documents
Registration Authority (RA)Verifies the identity of certificate requestors before the CA issues the certificate; does NOT sign certificatesLike a post office that verifies your identity before forwarding your passport application
Certificate RepositoryStores and distributes certificates and CRLs; typically uses LDAP or HTTPLike a phone directory for public keys
Certificate Revocation List (CRL)A signed list of serial numbers of certificates that have been revoked before expirationLike a "lost/stolen passport" list published by the passport office
OCSP ResponderOnline Certificate Status Protocol — provides real-time certificate status checks (good/revoked/unknown)Like calling the passport office to check if a specific passport is still valid
PKI HIERARCHY ╔════════════════════════════════╗ ║ Root CA (Offline) ║ ← Signs subordinate CAs ║ Self-signed certificate ║ ← Trust anchor ╚══════════════╦═════════════════╝ │ ┌──────────┴──────────┐ ▼ ▼ ╔═══════════╗ ╔═══════════╗ ║ Sub-CA 1 ║ ║ Sub-CA 2 ║ ← Issue end-entity certs ║ (Issuing) ║ ║ (Issuing) ║ ╚═════╦═════╝ ╚═════╦═════╝ │ │ ┌───┴───┐ ┌───┴───┐ ▼ ▼ ▼ ▼ [User] [Server] [Code] [Email] ← End-entity certificates
💡 Exam Tip: The Root CA should be kept offline (air-gapped) and only brought online to sign subordinate CA certificates. This protects the root trust anchor. If the Root CA is compromised, the entire PKI hierarchy must be rebuilt.

4. X.509 Digital Certificates

📖 X.509: The ITU-T standard that defines the format for public key certificates. Currently on version 3 (X.509v3). Used in TLS/HTTPS, S/MIME, IPSec, and code signing.

Key Fields in an X.509v3 Certificate

FieldDescription
VersionCertificate version (typically v3)
Serial NumberUnique identifier assigned by the CA; used in CRLs for revocation
Signature AlgorithmAlgorithm used by the CA to sign the certificate (e.g., sha256WithRSAEncryption)
IssuerDistinguished name (DN) of the CA that issued the certificate
Validity PeriodNot Before and Not After dates — the certificate's active window
SubjectDistinguished name of the entity the certificate identifies (CN=www.example.com)
Subject Public Key InfoThe public key and the algorithm it uses
Extensions (v3)Key Usage, Extended Key Usage, Subject Alternative Name (SAN), Basic Constraints, CRL Distribution Points
CA SignatureThe CA's digital signature over the entire certificate content
🔑 Subject Alternative Name (SAN): A v3 extension that allows a single certificate to be valid for multiple domain names (e.g., example.com, www.example.com, mail.example.com). This has largely replaced wildcard certificates in modern deployments.

5. Certificate Lifecycle

PhaseDescription
1. Enrollment / RequestSubject generates a key pair and submits a Certificate Signing Request (CSR) containing the public key and identity information
2. VerificationThe RA verifies the requester's identity (domain ownership, organizational identity, or extended validation)
3. IssuanceThe CA signs the certificate with its private key and returns it to the subject
4. Distribution / PublicationCertificate is installed on the server/device and optionally published to a certificate repository
5. UsageCertificate is used for TLS, email signing, code signing, authentication, etc.
6. Renewal / Re-keyBefore expiration, the certificate is renewed (same key) or re-keyed (new key pair)
7. Revocation (if needed)Certificate is revoked before expiration if the private key is compromised, employee leaves, or information changes
8. Expiration / ArchivalCertificate reaches its Not After date; keys may be archived for future decryption of old data
⚠️ Renewal vs. Re-key: Renewal extends the validity period with the same key pair. Re-keying generates a new key pair with a new certificate. After a key compromise, you must re-key (not just renew) because the old key pair is no longer trusted.

6. Certificate Revocation

CRL vs. OCSP

FeatureCRL (Certificate Revocation List)OCSP (Online Certificate Status Protocol)
TypeOffline — periodic downloadOnline — real-time query
FreshnessDepends on publishing interval (hours to days); can be staleReal-time or near-real-time status
SizeGrows over time; large CRLs are bandwidth-heavySmall per-request responses
PrivacyNo privacy concern — client downloads entire listOCSP responder sees which certificates the client is checking (privacy concern)
SolutionDelta CRLs (incremental updates)OCSP Stapling — server pre-fetches OCSP response and includes it in TLS handshake
💡 Exam Tip: OCSP Stapling solves BOTH the latency problem (no extra round trip to the OCSP responder) and the privacy problem (the CA doesn't see which sites the client is visiting). The web server periodically fetches its own OCSP response and "staples" it to the TLS handshake.

Revocation Reasons

ReasonWhen Used
Key CompromisePrivate key is stolen, leaked, or suspected compromised
CA CompromiseThe issuing CA's private key is compromised → all certs from that CA are suspect
Affiliation ChangedEmployee leaves organization; certificate no longer represents a valid relationship
SupersededNew certificate replaces the old one (re-key or updated information)
Cessation of OperationEntity no longer operates; certificate has no purpose
Certificate HoldTemporary suspension — certificate may be "un-held" later; useful for investigation

7. Trust Models

ModelDescriptionStrengthsWeaknesses
Hierarchical (Tree)Single root CA at the top; subordinate CAs branch below; trust flows from the root down through the certificate chainClear authority; scalable; easy chain validationRoot = single point of failure; compromise requires full rebuild
Mesh / Cross-CertificationMultiple CAs certify each other through cross-certificates; peer-to-peer trustNo single point of failure; flexible for mergers/partnershipsComplex trust paths; harder to manage and audit
Bridge CAA neutral CA acts as a trust broker between multiple separate PKI hierarchies (used by US Federal PKI Bridge)Connects disparate PKIs without requiring each to cross-certify with all othersBridge becomes a high-value target; complex policy mapping
Web of Trust (PGP/GPG)No central CA — users sign each other's keys to establish trust; trust builds through connectionsDecentralized; no single authority neededDoesn't scale; no formal revocation; requires user effort
🔑 Certificate Chain Validation: When a browser receives a server certificate, it walks the chain: End-Entity Cert → Subordinate CA → Root CA. The browser trusts the chain ONLY if the Root CA is in its trusted root store. Each certificate in the chain must be valid (not expired, not revoked) and the signatures must verify.
⚠️ Web of Trust vs. Hierarchical PKI: PGP's Web of Trust has NO central authority — users decide whom to trust. Hierarchical PKI has a CA that makes trust decisions for everyone. The CISSP exam may ask you to compare these — the key difference is centralized authority vs. decentralized trust.

8. Key Escrow & Recovery

📖 Key Escrow: A copy of a user's private key (typically an encryption key) is held by a trusted third party so the organization can recover data if the user is unavailable. Also known as key recovery or key archival.
AspectDetail
When UsedOrganization needs to recover encrypted data if employee leaves, is terminated, or loses their key
Never EscrowSignature keys should NEVER be escrowed — this would allow someone else to forge your digital signature and destroy non-repudiation
Dual ControlEscrow systems should require multiple parties (M-of-N key splitting) to reconstruct the key, preventing single-party abuse
Government ContextControversial — the US Clipper chip (1993) proposed government key escrow; it was abandoned due to privacy concerns and technical weaknesses
💡 Exam Tip: Remember: Escrow encryption keys, NEVER signature keys. Escrowing a signature key would allow the escrow agent to forge digital signatures, which destroys non-repudiation — one of the primary security services of digital signatures.

9. Certificate Pinning

📖 Certificate Pinning: A technique where an application associates (pins) a specific certificate or public key to a host, bypassing the standard CA trust chain. If the server presents a different certificate — even one signed by a trusted CA — the connection is rejected.
TypeDescription
Static PinningHard-code the expected certificate/public key in the application; requires app update to change pins
HTTP Public Key Pinning (HPKP)Server sends pin via HTTP header on first visit; browser remembers for future visits. Note: HPKP is deprecated due to risk of bricking sites if pins are misconfigured
Certificate TransparencyThe modern replacement approach — all issued certificates are logged in public, append-only logs that can be monitored for unauthorized certificate issuance
🔑 Certificate Transparency (CT): Rather than pinning, the industry has moved toward Certificate Transparency. CAs must log all certificates they issue to public CT logs. Domain owners can monitor these logs to detect unauthorized certificates. All major browsers now require CT for publicly trusted certificates.

10. Memory Tricks & Mnemonics

🧠 PKI Components — "CARROT":
CA — Certificate Authority (signs certs)
A — (is the) Authority
RA — Registration Authority (verifies identity)
Repository — stores certs & CRLs
OCSP — real-time revocation checks
Trust anchor — root CA is the trust foundation
🧠 CRL vs. OCSP: "CRL is a List (batch download). OCSP is a Question (real-time query). Stapling is the server pre-answering the question."
🧠 Never Escrow Signatures: "Encryption keys may be escrowed. Signature keys? NEVER. If someone else has your signing key, they can sign as you — destroying non-repudiation."
🧠 Trust Models: "Hierarchy (tree) = centralized, Web of Trust (PGP) = decentralized, Bridge = connector, Mesh = cross-certify."
