2. Real-World Organizational Relevance

Organizational Ownership

The vulnerability management team (often part of Security Operations) owns the scanning program. System owners are responsible for remediating findings on their systems. IT operations/patching teams apply patches. Security leadership sets remediation SLAs and reports metrics to the board.

Implementation Reality

Enterprises typically run automated vulnerability scanners on weekly or monthly cycles. PCI DSS requires quarterly external scans by an Approved Scanning Vendor (ASV). Organizations maintain vulnerability management platforms (Tenable.io, Qualys VMDR, Rapid7 InsightVM) that integrate with ticketing systems (ServiceNow, Jira) for remediation tracking. The average organization discovers 870 new vulnerabilities per month (Tenable Research, 2023).

Failure Modes

  • Scan but don't remediate: 60-day-old critical finding still unpatched — common in organizations without formal SLAs
  • Scanning exclusions: Legacy systems excluded from scans due to "fragility concerns" — often the highest-risk systems
  • Unauthenticated-only scanning: Misses 70%+ of vulnerabilities that require credentials to detect

3. Vulnerability Assessment Overview

Vulnerability Assessment: A systematic process of identifying, quantifying, and prioritizing (ranking) vulnerabilities in a system. The goal is to produce a comprehensive list of weaknesses and their risk level — WITHOUT attempting to exploit them. It answers: "What vulnerabilities exist?"

Vulnerability Assessment vs. Penetration Test

AttributeVulnerability AssessmentPenetration Test
GoalIdentify and categorize vulnerabilitiesExploit vulnerabilities to prove real-world risk
Exploitation?NoYes (with authorization)
DepthBroad coverage; many systems quicklyDeep dive; focused on specific targets
False positives?Yes — scanners have false positivesLow — exploitation confirms the finding
FrequencyFrequent (weekly/monthly automated)Annual or periodic; resource-intensive
OutputList of vulnerabilities + CVSS scoresNarrative report + exploitation evidence + impact proof
Exam Tip: Vulnerability assessment DOES NOT exploit — it only identifies. Penetration testing DOES exploit to prove impact. When the exam asks "what is the BEST first step?" — vulnerability assessment before pen testing.

4. Scanning Types — Authenticated vs. Unauthenticated

Unauthenticated (Credentialless) Scanning

Unauthenticated Scan: Scanner connects to target systems as an external, unprivileged entity — no login credentials provided. Identifies only vulnerabilities visible from the network (open ports, exposed services, banner information, publicly known vulnerabilities based on service versions).
  • Simulates: External attacker with no credentials
  • Finds: Network-exposed service vulnerabilities, open ports, SSL/TLS weaknesses, HTTP misconfigurations
  • Misses: Internal vulnerabilities (missing patches, application flaws, privilege configuration) that require login to detect
  • Advantage: Can be run without system access negotiation; simulates real attack surface

Authenticated (Credentialed) Scanning

Authenticated Scan: Scanner logs into target systems using valid credentials (typically a service account) and performs internal inspection — checking installed software versions, patch levels, registry settings, file permissions, user accounts, and service configurations.
  • Simulates: Insider or attacker who has gained initial access
  • Finds: Missing patches, weak configurations, unsafe software versions, excessive privileges, default credentials, all unauthenticated vulnerabilities PLUS internal ones
  • More comprehensive: Consistently finds 70-90% more vulnerabilities than unauthenticated scanning
  • Risk: Scanner credentials must be secured — if compromised, attacker has privileged access to all scanned systems
⚠️ Exam Trap: Authenticated scanning finds MORE vulnerabilities, not fewer. A common distractor is the idea that unauthenticated is "more thorough" because it tests what an outside attacker sees. In terms of total vulnerability count, authenticated is more thorough.

Nmap is NOT a vulnerability scanner — it's a port/service scanner. Nmap identifies what ports are open and what services are running. Vulnerability scanners take that information further and check those services against known vulnerability databases. Knowing this distinction matters on the CISSP exam.

6. CVSS Scoring System

CVSS (Common Vulnerability Scoring System): An open industry standard (maintained by FIRST — Forum of Incident Response and Security Teams) for rating the severity of security vulnerabilities. Produces a numerical score from 0 to 10 and a qualitative rating. Current major version: CVSS v3.1 (widely used); CVSS v4.0 released November 2023.

CVSS v3.1 Score Groups

Score RangeSeverity RatingAction Priority
9.0 – 10.0CriticalImmediate emergency patching; 24-48 hour SLA
7.0 – 8.9HighUrgent; patch within 7-30 days
4.0 – 6.9MediumImportant; patch within 30-90 days
0.1 – 3.9LowInformational; patch in next maintenance cycle
0.0None/InformationalAdvisory; no immediate action required

CVSS Metric Groups

GroupMetricsDescription
Base ScoreAttack Vector, Attack Complexity, Privileges Required, User Interaction, Scope, Confidentiality/Integrity/Availability ImpactIntrinsic characteristics of the vulnerability; does not change based on environment; published by vendor/researcher
Temporal ScoreExploit Code Maturity, Remediation Level, Report ConfidenceChanges over time as exploits are published, patches released; reduces or raises urgency
Environmental ScoreModified Base metrics + Confidentiality/Integrity/Availability RequirementsCustomized for a specific organization's environment; affected system criticality adjusts the score
Exam Tip — CVSS Score Customization:
The Base Score is universal — same for everyone.
The Environmental Score is customized per organization — if a critical system is affected, the score goes UP; if the system is isolated/low-impact, it goes DOWN.
This is why a CVSS 7.5 CVE on a honeypot might actually be lower priority than a CVSS 5.0 CVE on the payment processing system.

DatabaseWhat it ContainsMaintained byExample
CVE — Common Vulnerabilities and ExposuresDictionary of publicly known information security vulnerabilities. Each entry is a specific flaw in a specific product with a unique CVE-YYYY-NNNNN identifier.MITRE Corporation (funded by CISA)CVE-2021-44228 (Log4Shell in Apache Log4j)

CWE — Common Weakness EnumerationCategorizes types of software and hardware weaknesses (root causes). CWE is a class of problem; CVE is a specific instance of that problem in a product.MITRE CorporationCWE-79 (Cross-Site Scripting); CWE-89 (SQL Injection)
CVSS — Common Vulnerability Scoring SystemScoring framework for vulnerability severity. Not a database — used alongside CVE/NVD to rate severity.FIRST.orgCVE-2021-44228 has CVSS 10.0 (Critical)
CVE vs. CWE — Specific vs. Category:
CVE = "There is a SQL injection vulnerability in Product X version 2.3" (specific instance)
CWE = "SQL injection is a type of input validation weakness" (weakness class/category)
CWE-89 (SQL injection) ← applies to hundreds of CVEs across many products

8. Vulnerability Prioritization & Remediation

Risk-Based Prioritization

Not every vulnerability can be patched immediately — organizations must prioritize based on risk:

  • Exploitability: Is there a known public exploit? Is it being actively exploited in the wild? (CISA KEV — Known Exploited Vulnerabilities catalog)
  • Asset criticality: Is the vulnerable system in the DMZ? On a payment network? In a development lab?
  • CVSS environmental score: How does this CVE affect your specific environment?
  • Network exposure: Is the vulnerable service exposed to the internet or only accessible internally?
  • Ease of exploitation: Does it require authentication? Physical access? Or is it remotely exploitable with no credentials?
Vulnerability Prioritization Matrix HIGH Asset Criticality │ │ ← PATCH IMMEDIATELY Critical │ High + High │ Priority ────────────┼──────────── Low Immediate │ Scheduled Criticality │ │ remediation ▼ ▼ LOW Asset Criticality

Remediation Options

OptionDescriptionWhen to Use
Patch / UpdateApply vendor-released fix to eliminate the vulnerabilityDefault first choice when patch is available and testable
Configuration ChangeDisable vulnerable feature, restrict access, change settingsWhen patch unavailable or would break compatibility
Virtual PatchingIPS/WAF rule blocks exploitation of unfixed vulnerability without changing the systemLegacy systems that cannot be patched; immediate protection while patch is tested
Risk AcceptanceDocument and formally accept the risk as-isLow criticality; remediation cost exceeds risk; compensating controls in place
Isolation / SegmentationNetwork-isolate vulnerable system to reduce exposureHigh-risk legacy systems; OT/ICS systems that cannot be patched
Vulnerability Management Lifecycle: Discover → Prioritize → Remediate → Verify → Monitor → Repeat
"Verify" step is critical — confirm the patch was applied successfully and the vulnerability is gone. Re-scanning after remediation is mandatory.

9. Memory Tricks & Mnemonics

🧠 Vuln Assessment vs. Pen Test — "FE Rule"
Vulnerability Assessment = Find only (no exploitation)
Penetration Test = Find + Exploit (prove impact)
VA first → found weaknesses → PT confirms which ones are actually exploitable
🧠 Authenticated > Unauthenticated
Authenticated scan = inside (more findings)
Unauthenticated scan = outside view only (external attacker perspective)
More comprehensive = Authenticated (always finds more)
🧠 CVSS Score Groups — "CHMLI"
Critical (9-10)
High (7-8.9)
Medium (4-6.9)
Low (0.1-3.9)
Informational (0)
"Criminals Hack Many Low-importance systems"
🧠 CVE vs. CWE
CVE = Vulnerability (a specific flaw in product X)
CWE = Weakness (a class/category of flaw type)
CWE-89 = type "SQL injection"; CVE-2021-XXXXX = specific SQL injection in specific product

  • Vulnerability assessment = identify; penetration test = exploit and prove impact
  • Authenticated scans are MORE comprehensive than unauthenticated scans
  • CVSS Base Score is universal; Environmental Score is customized per organization's risk context
