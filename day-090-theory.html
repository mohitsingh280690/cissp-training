2. Real-World Organizational Relevance

Logs are the forensic trail of all system activity. Without logs, incident responders cannot determine the scope of a breach, verify the attacker's path, or produce evidence for legal proceedings. The FBI and regulatory bodies (SEC, HIPAA, PCI DSS) all require organizations to maintain logs for specific periods.

Modern enterprises generate terabytes of log data daily. Organizations use SIEM platforms to make this data searchable and actionable: Splunk, IBM QRadar, Microsoft Sentinel, and Elastic SIEM are the dominant platforms. The average cost of "not logging" during an incident investigation is $1M+ in extended containment time (IBM Cost of a Data Breach Report).

Log Coverage Rule: Security events require correlation across MULTIPLE log sources. Example: investigating a successful SQL injection requires web server logs (request URL) + application logs (error) + database audit logs (what data was returned). No single log source tells the complete story.

Syslog Protocol

Syslog (RFC 5424): The standard protocol for sending log messages over IP networks, typically using UDP port 514 (or TCP 6514 for TLS-encrypted syslog). Syslog messages include a facility code (source type) and severity level (0=Emergency to 7=Debug). Most Unix/Linux systems and network devices support syslog natively.
Syslog Severity LevelCodeDescription
Emergency0System is unusable
Alert1Immediate action required
Critical2Critical conditions
Error3Error conditions
Warning4Warning conditions
Notice5Normal but significant
Informational6Informational messages
Debug7Debug-level messages
NTP Time Synchronization for Logs: All log sources MUST be synchronized to a common time source (NTP — Network Time Protocol). Without synchronized timestamps, correlating events across log sources becomes impossible. A firewall log showing an event at 14:35:00 and an application log showing 14:38:00 are actually the same event if clocks are 3 minutes apart.

5. SIEM Fundamentals

SIEM (Security Information and Event Management): A platform that combines Security Information Management (SIM — historical log storage and compliance reporting) and Security Event Management (SEM — real-time threat detection and alerting) into a single system. SIEM ingests logs from all sources, normalizes them to a common format, correlates events, generates alerts (using rules and ML), and provides search and reporting capabilities.

Core SIEM Functions

FunctionDescription
Log AggregationCollects logs from all sources (network, endpoints, applications, cloud)
NormalizationConverts logs from different formats into a common schema for correlation
CorrelationLinks related events across sources to identify complex attack patterns (e.g., failed logins + successful login + privilege escalation = account compromise chain)
AlertingRule-based and ML-based alerts on suspicious activity; generates tickets for SOC analysts
Dashboards & ReportingReal-time and historical views of security posture; compliance reports (PCI DSS, HIPAA)
Retention & SearchLong-term storage of log data; full-text search for incident investigation and forensics
Threat Intelligence IntegrationEnriches events with external threat intel (IOCs — Indicators of Compromise) from OSINT, commercial feeds

6. Log Retention Policies

Log Retention: The organizational policy and technical implementation defining how long log records must be kept. Driven by regulatory requirements, legal litigation hold obligations, and operational needs (forensic investigations may need historical logs going back months).

Common Regulatory Retention Requirements

Regulation / StandardMinimum Log RetentionScope
PCI DSS (Requirement 10.7)12 months (3 months immediately available)All systems in CDE (Cardholder Data Environment)
HIPAA6 years (security rule documentation); audit logs: typically 90 days minimum recommendedPHI systems; covered entities
SOX (Sarbanes-Oxley)7 yearsFinancial systems; publicly traded US companies
GDPRDoes not specify minimum; "as long as necessary" for stated purpose; data minimization principleEU personal data
NIST SP 800-92Guidance only (not mandate); recommends tiered retention based on data sensitivityFederal systems; guidance framework
FedRAMP90 days immediately accessible; 1 year totalCloud service providers serving federal agencies
⚠️ Log Deletion Risk: Deleting logs before the retention period ends can result in compliance violations, destruction of evidence charges (spoliation), and inability to investigate past incidents. Organizations should have a documented retention policy and test it regularly.

7. Log Integrity & Tamper-Proofing

Logs are only valuable if they are trustworthy. An attacker who gains access to a system will typically attempt to delete or modify logs to hide their tracks. Log integrity controls prevent or detect this tampering.

Log Integrity Controls

Each log entry or batch signed with private key. Verification with public key proves authenticity and integrity. Used in advanced log systems.
ControlHow It WorksStrength
Separate Log ServerLogs are sent immediately to a remote server the attacked system cannot write to (send-only, remote syslog). Attacker compromises source system but cannot access the log server.High — depends on log server access control
Write-Once Media / WORMLogs written to Write-Once, Read-Many (WORM) storage. Once written, data cannot be modified or deleted. Common for long-term archival.Very high — physical cannot be modified
Cryptographic HashingHash each log entry (SHA-256) or log file periodically. Any modification changes the hash; comparison detects tampering.High — detects modification but does not prevent it
Digital SignaturesVery high — proves both integrity and origin
Time Stamps from Trusted SourceNTP-synced timestamps from a trusted authoritative source; secure NTP (RFC 8915) prevents time manipulation attacks.Medium — prevents timestamp manipulation
Immutable Cloud LoggingCloud logging services (AWS CloudTrail with S3 Object Lock, Azure Activity Log, GCP Cloud Logging) offer native immutability features.High — cloud provider controls prevent deletion without explicit unlock
Legal Admissibility of Logs: For logs to be admissible as evidence in legal proceedings, they must be:
  • Authentic: Proven to come from the claimed source (digital signatures, chain of custody)
  • Unmodified: Integrity proven via hashing or write-once media
  • Complete: No unexplained gaps in the log timeline
  • Timely collected: Gathered before potential tampering; chain of custody maintained

8. Continuous Monitoring

Continuous Monitoring: The ongoing, automated collection and evaluation of security controls and system activities to maintain situational awareness and detect deviations from expected behavior in real time (or near-real time). Required by NIST RMF (SP 800-137) and essential for FedRAMP-authorized systems.

Continuous Monitoring Components

  • Configuration monitoring: Continuous detection of unauthorized configuration changes (SCCM, Chef, Puppet, AWS Config, Azure Policy)
  • Vulnerability scanning: Automated, continuous vulnerability scanning (Tenable.io continuous assessment plugins) rather than periodic point-in-time scans
  • Log monitoring (SIEM): Real-time log analysis and alerting
  • User behavior analytics (UEBA): ML-based detection of anomalous user behavior (unusual login times, data access patterns, lateral movement indicators)
  • Network traffic analysis (NTA): Continuous analysis of network flow (NetFlow, IPFIX) for anomalies; C2 beaconing detection
  • Endpoint Detection and Response (EDR): Continuous endpoint telemetry collection and analysis (CrowdStrike Falcon, Microsoft Defender for Endpoint)

NIST SP 800-137 (Information Security Continuous Monitoring)

Defines ISCM (Information Security Continuous Monitoring) as "maintaining ongoing awareness of information security, vulnerabilities, and threats to support organizational risk management decisions." Requires:

  • Defined metrics and performance indicators
  • Monitoring strategy based on risk
  • Automated data collection where possible
  • Analysis and reporting to appropriate officials
  • Responding to findings (remediation or risk acceptance)

9. Memory Tricks & Mnemonics

🧠 Log Integrity — "SAVE Logs"
Separate log server (remote — attacker can't reach it)
Authenticate logs (digital signatures)
Verify hashes (SHA-256 of each entry/file)
Ensure write-once media (WORM for archival)
🧠 SIEM — "SIM + SEM"
SIM = Security Information Management (historical storage, compliance reports)
SEM = Security Event Management (real-time detection, alerting)
SIEM = SIM + SEM = both historical AND real-time
🧠 Why NTP Matters for Logs
Without synchronized clocks → logs from different systems can't be correlated
"Time is the spine of forensics" — without accurate timestamps, event sequencing is impossible
🧠 Continuous Monitoring vs. Periodic Assessment
Periodic assessment = snapshot (what did it look like at point in time)
Continuous monitoring = movie (what is happening RIGHT NOW and over time)
Continuous > periodic for detecting evolving threats and configuration drift

Top Exam Points:
  • Logs stored ONLY on the source system are vulnerable to attacker deletion — remote/separate log server is required
  • NTP synchronization is not optional — without it, cross-source log correlation is unreliable
  • SIEM combines SIM (historical/compliance) + SEM (real-time detection); it does NOT block attacks alone
  • PCI DSS requires 12 months log retention; SOX requires 7 years
  • WORM storage is the strongest log integrity control (physical impossibility to modify)
  • Continuous monitoring (NIST SP 800-137) is "maintain ongoing awareness" — not just periodic scans
  • Log admissibility as evidence requires authenticated, unmodified, complete, and timely-collected logs
