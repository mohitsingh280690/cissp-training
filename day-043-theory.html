3. Type 1 vs. Type 2 Hypervisors

📖 Hypervisor (Virtual Machine Monitor - VMM): Software that creates and manages virtual machines by abstracting hardware resources and distributing them among multiple guest operating systems.
FeatureType 1 (Bare-Metal)Type 2 (Hosted)
Runs OnDirectly on hardware (no host OS)On top of a host operating system
PerformanceHigher — direct hardware accessLower — extra OS layer overhead
SecurityMore secure — smaller attack surfaceLess secure — depends on host OS security
Use CaseEnterprise data centers, production serversDevelopment, testing, desktop virtualization
ExamplesVMware ESXi, Microsoft Hyper-V, Xen, KVMVMware Workstation, VirtualBox, Parallels
CPU RingRing -1 (VMX root mode)Ring 0 within host OS context

  │ Guest  │ │ Guest  │           │ Guest  │ │ Guest  │

  ┌────────────────────┐          ┌────────────────────┐
  │   Hypervisor       │          │    Hypervisor       │

  │   Hardware         │          │   Host OS           │

💡 Exam Tip: Type 1 (bare-metal) = MORE secure, runs directly on hardware. Type 2 (hosted) = runs on a host OS. When the exam says "enterprise virtualization," think Type 1. Memory aid: "1 = first on hardware."

4. Virtualization Threats

ThreatDescriptionImpact
VM EscapeAttacker breaks out of a guest VM and gains access to the hypervisor or other VMsCritical — complete host compromise; access to all VMs
VM SprawlUncontrolled proliferation of VMs; many become forgotten, unpatched, and unmanagedExpanded attack surface; compliance violations; wasted resources
Inter-VM AttacksOne compromised VM attacks another VM on the same host via shared resourcesLateral movement; data exfiltration
Resource ExhaustionOne VM consumes excessive CPU/memory/disk, starving other VMs (DoS)Availability impact on co-hosted VMs
Hypervisor CompromiseAttacker exploits vulnerability in the hypervisor itselfControl over all VMs on the host
Data RemnantsSensitive data from decommissioned VMs persists on storageData leakage; compliance violations
⚠️ VM Escape: The most critical virtualization threat. If an attacker escapes a guest VM to the hypervisor layer, they control ALL VMs on that host. Real examples: Venom (CVE-2015-3456) exploited the floppy disk controller in QEMU. Mitigations: patch hypervisors immediately, minimize attack surface in VMs, use hypervisor introspection.
🔑 VM Sprawl Prevention:
• Implement VM lifecycle management policies
• Require approval for new VM creation
• Track all VMs in a CMDB (Configuration Management Database)
• Automatically scan and decommission unused VMs
• Apply the same patch management to VMs as physical servers

5. Container Security

📖 Containers: Lightweight, OS-level virtualization that packages an application with its dependencies. Unlike VMs, containers share the host OS kernel — they are NOT separate OS instances.
FeatureVirtual MachinesContainers
IsolationFull OS per VM; hardware-level isolationShared kernel; namespace + cgroup isolation
SizeGigabytes (includes full OS)Megabytes (app + dependencies only)
StartupMinutesSeconds
Security BoundaryStronger — hypervisor enforcedWeaker — shared kernel is larger attack surface
DensityTens of VMs per hostHundreds/thousands of containers per host

Container Security Concerns

RiskDescriptionMitigation
Container BreakoutEscaping container isolation to access host kernelKeep kernel patched; use security profiles (seccomp, AppArmor)
Image PoisoningMalicious code injected into container imagesScan images; use signed/trusted base images; private registries
Privilege EscalationRunning containers as root grants host-level accessRun containers as non-root; drop unnecessary capabilities
Insecure ConfigurationOverly permissive network, volume mounts, or capabilitiesUse pod security policies; restrict network policies in Kubernetes
Supply Chain RisksVulnerable dependencies in container imagesRegular image scanning; dependency updates; SBOM
🔑 Container Orchestration (Kubernetes):
• Kubernetes manages container deployment, scaling, and networking
• Security controls: RBAC, Network Policies, Pod Security Standards, Secrets management
• Key risks: misconfigured RBAC, exposed API server, privileged pods, default namespace abuse

6. Serverless & Microservices

Serverless Computing

📖 Serverless: Cloud execution model where the provider manages all infrastructure. Developers deploy functions (code) that run in response to events. Examples: AWS Lambda, Azure Functions, Google Cloud Functions.
BenefitSecurity Concern
No server management → reduced OS patching burdenLimited visibility into runtime environment
Auto-scaling → handles load spikesEvent injection (manipulating triggers)
Pay-per-use → cost efficientOverly permissive function IAM roles
Short-lived functions → reduced attack windowInsecure third-party dependencies

Microservices Architecture

📖 Microservices: An architectural style where an application is built as a collection of small, independent services that communicate via APIs. Each service handles one business function and can be deployed independently.
🔑 Microservices Security Concerns:
• API Security: Each service exposes an API — all must be authenticated and authorized
• Service-to-Service Authentication: mTLS (mutual TLS) for inter-service communication
• Increased Attack Surface: More services = more endpoints to secure
• Service Mesh: Istio, Linkerd provide transparent security (mTLS, policy enforcement)

7. Virtualization Security Best Practices

PracticeDescription
Patch Hypervisors PromptlyHypervisor vulnerabilities enable VM escape — treat with highest priority
Limit Hypervisor AccessRestrict management interfaces; use strong authentication; separate management network
Monitor VM LifecycleTrack creation, modification, and decommissioning of all VMs
Encrypt VM Disk ImagesProtect data at rest within VM storage
Scan Container ImagesUse vulnerability scanners before deploying any container image
Implement Network SegmentationUse virtual switches and network policies to isolate VMs/containers
Avoid Running Containers as RootUse non-root users; implement pod security standards

8. Memory Tricks & Mnemonics

🧠 "Type 1 = First on the Floor (Hardware)": Type 1 hypervisors sit directly on the hardware — they're first on the floor. Type 2 sits on top of an OS — it's on the second floor.
🧠 "VM Escape = Prison Break": The VM is the prison cell. VM escape = the prisoner (attacker) breaks out of their cell and has access to the whole building (host + all VMs).
🧠 "Containers Share, VMs Don't Care": Containers share the host kernel (so a kernel bug affects all containers). VMs don't care about each other — each has its own OS and kernel.
