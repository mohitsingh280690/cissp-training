2. Real-World Organizational Relevance

AspectOrganizational Impact
Who owns itSOC (Security Operations Centre) team receives and triages alerts; network security team manages placement and tuning; security engineering configures rules and signatures
Where deployedNIDS: at internet perimeter, inside DMZ, on internal core segments, on cloud VPC spans; HIDS: on critical servers, endpoints with EDR agents, cloud instances
Alert fatiguePoorly tuned IDS generating thousands of alerts per day means analysts ignore real events. Tuning IDS — reducing false positives while preserving true positive detection — is a core operational challenge. SIEM correlation rules reduce noise.
CompliancePCI-DSS Requirement 11 mandates IDS/IPS on all cardholder data environment network boundaries and critical internal segments. ISO 27001 Annex A.12.4 requires monitoring; IDS is a primary technical control.
Limitations in practiceEncrypted traffic (TLS) is opaque to a network IDS without SSL inspection. Insider threats on authorized ports/protocols bypass signature detection. Volume-based DDoS can overwhelm IPS engines. Zero-day exploits bypass signature-based IDS entirely.

3. IDS vs. IPS – Core Distinction

📖 IDS (Intrusion Detection System): A passive monitoring system that analyses copies of network traffic or host activity, identifies suspicious patterns, and generates alerts. It does not interfere with traffic — it only observes and reports.
📖 IPS (Intrusion Prevention System): An inline active system placed in the path of traffic that can detect and block malicious traffic in real time by dropping packets, resetting connections, or blocking source IPs. If an IPS fails, traffic may be blocked (fail-closed) or pass uninspected (fail-open).
AttributeIDSIPS
Traffic pathOut-of-band (copy/span port, TAP)Inline — traffic passes through the device
Response to attackAlert/log only; humans must respondAutomatically drops/blocks malicious traffic
Impact on trafficNone — passive observationAdds latency; false positive → legitimate traffic blocked
Failure modeFail-open (traffic unaffected if IDS fails)Fail-open or fail-closed depending on config — fail-closed stops all traffic
False positive riskAnalyst alert noiseService disruption — blocking legitimate traffic is an outage
Use caseDetection, forensics, compliance loggingReal-time attack prevention; complements firewall
💡 Exam Tip: "IDS is like a burglar alarm — it detects and notifies. IPS is like a security guard with a lock — it actively stops the threat." On the CISSP exam, whenever a question says "without disrupting traffic flow" or "monitor without impacting operations", the answer is IDS (not IPS). Whenever the question requires automatic blocking, the answer is IPS.

4. Network-Based IDS/IPS (NIDS/NIPS)

📖 NIDS/NIPS: Monitors network traffic at the packet level for the entire network segment it observes. Deployed at strategic network chokepoints to inspect all traffic passing that point.
NIDS Deployment (Passive – Span Port / TAP): Internet ─── [Firewall] ─── Core Switch ─── Internal Network │ [SPAN Port / TAP] │ [NIDS Sensor] │ [Management Console] No traffic passes THROUGH the NIDS — it receives a COPY. A failure of the NIDS does not affect traffic. An attacker who knows the NIDS is out-of-band may not worry about evading it. ────────────────────────────────────────────────────────────── NIPS Deployment (Inline – Active Prevention): Internet ─── [Firewall] ─── [NIPS] ─── Core Switch ─── Internal ↑ Traffic PASSES THROUGH NIPS NIPS can DROP malicious packets inline NIPS failure impacts traffic (fail-open vs. fail-closed)

Placement Strategies

PlacementWhat It SeesRationale
Outside perimeter firewallAll inbound internet traffic before filteringCaptures attack volume statistics; sees full threat landscape; generates high alert volume
Inside perimeter firewall (DMZ)Traffic that passed firewall policyDetects attacks targeting DMZ servers; recommended placement for PCI compliance
Internal network segmentsEast-West (lateral movement) trafficCritical for detecting insider threats, lateral movement after initial compromise
In front of critical serversTraffic specifically targeting high-value assetsFocused detection; fewer false positives; highest-value placement

5. Host-Based IDS/IPS (HIDS/HIPS)

📖 HIDS/HIPS: Runs as an agent on an individual host (server, workstation, cloud instance). Monitors operating system events, file system changes, registry modifications, process activity, log files, and network connections from that specific host's perspective.
What HIDS MonitorsExamples of Detectable Events
File system integrityUnauthorized modification of system files, web shells written to disk, configuration file tampering — core of HIDS functionality. Classic HIDS tools: Tripwire, AIDE, OSSEC.
Registry / config changesMalware adding persistence keys (HKLM\Run), privilege escalation via sudoers file, service installation
Process activitySuspicious processes (PowerShell spawned by Word), process injection, unexpected parent-child relationships
Log analysisFailed authentication spikes, privilege escalation events in OS logs, sudo abuse
Network connections (host view)Outbound C2 connections, unexpected listening ports opened by malware
AttributeNIDS/NIPSHIDS/HIPS
VisibilityNetwork-level — all hosts on segmentSingle host — deep OS-level visibility
Encrypted trafficBlind to encrypted payloads (without SSL inspection)Sees traffic after decryption at the host (post-TLS)
Insider threatLimited — insider uses legitimate ports/protocolsBetter — monitors file access, privilege use, process launch
Deployment overheadCentralized — few sensors per segmentAgent on every monitored host — scalability challenge
Attack detectionNetwork-level attacks (port scans, SYN floods)Host-level attacks (privilege escalation, rootkits, file tampering)
🔑 Best Practice — Layered IDS: Deploy both NIDS and HIDS in a layered model. NIDS catches network-level attacks; HIDS catches host-level compromise. The two send alerts to a central SIEM for correlation — a host compromise that generated no network alert may still be detectable through HIDS file integrity events. Neither alone provides complete coverage.

6. Detection Methods

MethodHow It WorksStrengthsWeaknesses
Signature-Based
(Knowledge-Based)		Compares traffic/activity against a database of known attack patterns (signatures). Each signature represents a specific known attack. Like antivirus for network traffic.Very low false positive rate; fast processing; precise identification of known attacks; clear explanationsCannot detect zero-days or unknown attacks; requires constant signature updates; attackers can modify payloads to evade signatures (polymorphism)
Anomaly-Based
(Behaviour-Based)		Establishes a baseline of "normal" behaviour (traffic volume, protocols, user patterns) and alerts on deviations from that baseline. Can detect zero-days if they cause abnormal behaviour.Can detect unknown/zero-day attacks; good for detecting insider abuse deviating from normal patterns; adaptiveHigh false positive rate — anything non-baseline triggers alerts; noisy environments (seasonal patterns) cause false alarms; baselining phase requires weeks of clean traffic
Stateful Protocol Analysis
(Specification-Based)		Compares protocol usage against vendor-defined protocol specifications (RFC standards). Detects protocol anomalies — e.g. HTTP request with unusually long URI, DNS over port 80, malformed TCP flags.Effective against protocol abuse and evasion; vendor-neutral; catches zero-days that violate specResource-intensive (deep packet inspection); legitimate non-standard implementations may trigger alerts; requires protocol-specific decoders
Heuristic / Rule-BasedUses expert-defined rules or heuristic scoring (multiple indicators combined) rather than exact signatures. Example: alert if process is spawned by Office app AND connects outbound AND writes to system32.More flexible than pure signatures; can detect variants; reduced false positives over anomalyRule maintenance overhead; complex tuning; may miss novel attack chains that don't match defined heuristics
💡 Exam Tip — Detection Method vs. Scenario:
"Detect zero-day attacks" → Anomaly-based (or stateful protocol analysis). Zero-days have no signature.
"Lowest false positive rate" → Signature-based. Known patterns = precise matches.
"Detect insider deviating from normal work hours" → Anomaly-based (baseline deviation).
"Alert on malformed HTTP headers" → Stateful protocol analysis (spec violation).
This detection-method-to-scenario mapping is a classic CISSP question pattern.

7. False Positives, False Negatives & Accuracy Metrics

Detection Outcome Matrix: ACTUAL STATE ┌─────────────┬─────────────┐ │ Attack IS │ Attack NOT │ │ happening │ happening │ ALERT ───────┼─────────────┼─────────────┤ GENERATED│ YES │ True Pos. ✓ │ False Pos. ✗│ ← False alarm (wolf cry) │ NO │ False Neg. ✗│ True Neg. ✓ │ ← Missed attack (silent breach) └─────────────┴─────────────┘ True Positive (TP): Attack happens, alert fires → CORRECT detection True Negative (TN): No attack, no alert → CORRECT silence False Positive (FP): No attack, alert fires → False alarm (analyst fatigue) False Negative (FN): Attack happens, no alert → MISSED ATTACK (most dangerous) Key Metrics: False Positive Rate = FP / (FP + TN) ← How often clean traffic triggers alarm False Negative Rate = FN / (FN + TP) ← How often attacks are MISSED Detection Rate = TP / (TP + FN) ← Sensitivity / Recall
🔑 Which is Worse — FP or FN?
From a security perspective: False Negative is worse — a real attack goes undetected. This is the definition of a security control failure.
From an operational perspective: False Positive is costly — analyst alert fatigue leads to real alerts being ignored (alert fatigue paradox), which indirectly causes false negatives.
The CISSP exam perspective: False Negative is the more dangerous condition, because it represents a breach that was not detected. However, a tuning strategy must balance both — excessive tuning to eliminate FPs can introduce FNs.
ScenarioTuning DirectionTrade-off
Too many false positives (alert fatigue)Tighten thresholds / narrow signatures → reduce sensitivityRisks increasing false negatives (missing real attacks)
Missing known attacksLower thresholds / add signatures → increase sensitivityRisks increasing false positives (alert noise)
Zero-day attacksAdd anomaly/heuristic rules → no signature tuning helpsAnomaly rules inherently produce more FPs

8. IDS/IPS Deployment Modes

ModeDescriptionProsCons
Inline (IPS mode)Device sits in the active traffic path; physically inserted between switch and firewall or between network segmentsCan actively block; real-time prevention; authoritative enforcementAdds latency; single point of failure; false positives cause outages; must be HA (high availability)
Passive / Out-of-Band (IDS mode)Receives a copy of traffic via SPAN port (switch port mirroring) or network TAP (hardware splitter)No traffic impact; no failure risk; can send TCP RST to disrupt sessions (limited response)Cannot block in-line; high-throughput environments may drop copied packets; slightly delayed visibility
SPAN PortSwitch forwards copies of frames from monitored ports to a designated sensor portCheap (software feature on switch); easy to configureMay drop frames under high load; SPAN port bandwidth limited; monitoring one segment at a time
Network TAPHardware device physically inserted into the cable that passively copies all signals to the IDS — a true tapNever drops traffic; hardware-level copy; completely passive; preferred for forensicsMore expensive; physical hardware insertion required; separate TAP per link
💡 Exam Tip — TAP vs. SPAN: For highly accurate, lossless traffic capture (forensics, compliance), the preferred method is a network TAP — hardware-based, passive, never drops frames. SPAN ports are software-based and may drop frames under load. When the question asks "most reliable method to capture all traffic for a network IDS", answer is network TAP.

9. Evasion Techniques

⚠️ IDS/IPS Evasion: Attackers deliberately craft traffic to bypass detection systems. Understanding evasion is critical to appreciating IDS/IPS limitations.
Evasion TechniqueHow It WorksCountermeasure
FragmentationSplit attack payload across many small IP fragments. The attack signature only appears when fragments are reassembled — IDS checking individual fragments may miss it.IDS reassembles fragments before analysis (stream reassembly)
Polymorphism / ObfuscationModify attack code or payload encoding (URL encoding, Unicode, base64, hex) so it doesn't match the signature pattern, but achieves the same effect on the target.Normalise and decode payloads before signature matching
Session SplicingSend attack data in multiple tiny TCP segments, exploiting the fact that the IDS may not track the full TCP stream while the target reassembles it correctly.Full application-layer stream reassembly in IDS engine
TTL ManipulationCraft packets with short TTLs that reach the IDS but expire before the target — IDS sees a complete (malicious) stream; target sees different (sanitized) stream.Topology-aware IDS; normalize TTL-based evasions
Encrypted TrafficHTTPS/TLS encrypts payload; network IDS cannot inspect without SSL inspection capability.SSL/TLS inspection proxy decrypts traffic before IDS analysis
Low-and-Slow AttacksSpread attack activity over days/weeks, staying below rate-based anomaly detection thresholds.Extended baseline analysis; UEBA (User and Entity Behaviour Analytics)

🔑 Snort Rule Anatomy (for context — not memorized for CISSP):
alert tcp any any → 192.168.1.0/24 80 (msg:"HTTP GET flood"; content:"GET"; threshold:type threshold, track by_src, count 100, seconds 1; sid:1000001;)
Action = alert; Protocol = TCP; Source = any; Destination = 192.168.1.0/24 port 80; Content match; Threshold; Unique Signature ID. The CISSP exam does not require rule writing but understanding the structure demonstrates detection method knowledge.

11. Memory Tricks & Mnemonics

🧠 IDS vs. IPS — "Detect = Doorbell; Prevent = Door Lock":
IDS = Doorbell — it rings (alerts) when someone arrives; you still have to open the door. Traffic flows through unimpeded.
IPS = Door Lock + Guard — it physically prevents entry (blocks traffic) before the door opens.
🧠 Detection Methods — "SASH":
Signature = known attack patterns (like antivirus)
Anomaly = baseline deviation (catches zero-days; high FP)
Stateful Protocol = spec-violation detection
Heuristic = multi-indicator rule scoring
"SASH the intruder if they deviate from the pattern."
🧠 False Negative = The Missed Attack:
"The alarm didn't go off — the burglar got in."
False Negative = security failure — the dangerous outcome. False Positive = false alarm — annoying but not a breach.
"FN = Failed to Notice"
🧠 TAP vs. SPAN — "TAP = True All Packets":
TAP = hardware; True lossless copy; All packets; Passive hardware device.
SPAN = software mirror; may drop frames under load; cheaper; easier to configure.
