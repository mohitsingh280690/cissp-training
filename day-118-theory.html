2. Real-World Organizational Relevance

Application vulnerabilities are the #1 attack vector. Verizon DBIR consistently shows web application attacks as a top breach pattern. Fixing a vulnerability in production costs 30-100x more than fixing it during requirements or design. Organizations that integrate security into the SDLC (shift-left) dramatically reduce vulnerabilities, remediation cost, and time-to-market delays from late-stage security findings.

3. SDLC Phases & Security Activities

SDLC PhaseSecurity Activities
1. Requirements / InitiationDefine security requirements; identify regulatory/compliance needs; initial risk assessment; classify data to be processed; define security SLAs
2. Design / AnalysisThreat modeling (STRIDE); security architecture design; attack surface analysis; define security controls; design review; secure design patterns
3. Development / ImplementationSecure coding standards (OWASP); code reviews; static analysis (SAST); developer security training; use of approved libraries/frameworks
4. Testing / VerificationDynamic testing (DAST); penetration testing; fuzz testing; vulnerability scanning; security regression testing; test for OWASP Top 10
5. Deployment / ReleaseSecurity configuration review; hardening; final security sign-off; secure deployment checklist; environment separation
6. Operations / MaintenancePatch management; vulnerability monitoring; incident response; security monitoring; change management; periodic reassessment
7. Disposal / End-of-LifeSecure data destruction; credential revocation; decommission documentation; archival per retention policy
Shift Left: Moving security activities earlier in the SDLC (to the "left" on the timeline). The earlier a vulnerability is found, the cheaper and easier it is to fix. Security should start in requirements, not testing.
Cost of Fixing Defects: Fixing a security flaw costs approximately:
• Requirements phase: 1x (baseline)
• Design phase: 5x
• Development: 10x
• Testing: 20x
• Production: 30-100x
This exponential increase is why "shift left" is so important.

4. SDLC Models

ModelDescriptionStrengthsWeaknesses
WaterfallSequential phases; each phase must complete before next begins; no going backSimple; well-documented; clear milestones; good for stable requirementsInflexible; no backtracking; late testing; poor for changing requirements
V-ModelExtension of waterfall; each development phase has a corresponding test phaseTesting planned early; clear relationship between dev and test phasesStill sequential and inflexible; expensive to change
Iterative / IncrementalBuild system in repeated cycles; each iteration adds functionalityFeedback incorporated; risk reduced per iteration; partial deliveryScope creep risk; requires strong planning
SpiralCombines iterative with risk analysis; four quadrants: Plan → Risk Analysis → Develop → EvaluateRisk-driven; good for large, complex projects; flexibilityComplex; expensive; requires risk expertise
AgileShort iterations (sprints); continuous delivery; adaptive to change; collaboration focusedFlexible; rapid delivery; customer involvement; responds to changeDocumentation may suffer; requires experienced team; security can be overlooked without discipline
PrototypingBuild quick prototype → get feedback → refine → build actual systemUser feedback early; reduces misunderstanding of requirementsPrototype may become production (without proper security); throwaway work
Waterfall vs Agile — Exam Favorite:
• Waterfall = best when requirements are well-defined and stable upfront; sequential; documentation-heavy
• Agile = best when requirements are evolving; iterative; collaboration-heavy
• Spiral = best for large, risky projects — uniquely emphasizes RISK ANALYSIS in each cycle

5. Security Requirements & Threat Modeling

STRIDE Threat Model

ThreatDescriptionSecurity Property Violated
SpoofingPretending to be someone/something elseAuthentication
TamperingModifying data or codeIntegrity
RepudiationDenying having performed an actionNon-repudiation
Information DisclosureExposing information to unauthorized partiesConfidentiality
Denial of ServiceMaking a system unavailableAvailability
Elevation of PrivilegeGaining unauthorized access/permissionsAuthorization

Other Threat Modeling Approaches

ApproachFocus

VASTVisual, Agile, Simple Threat modeling — scales for Agile/DevOps environments

6. Software Acquisition & Third-Party Security

Acquisition TypeSecurity Considerations
COTS (Commercial Off-The-Shelf)Vendor reputation; patch history; known vulnerabilities; configuration hardening; vendor lock-in; no source code access
Open SourceCommunity review; license compliance; known vulnerabilities (CVE); maintenance status; supply chain risk
Custom developedFull control; security built in; most expensive; requires in-house expertise
Outsourced developmentContract must include security requirements; code ownership; escrow agreements; right to audit; security testing requirements
⚠️ Software Supply Chain Risk: Third-party libraries, open-source components, and vendor software can introduce vulnerabilities. Software Composition Analysis (SCA) tools identify known vulnerabilities in dependencies. Software Bill of Materials (SBOM) provides a complete inventory of all components — increasingly required by regulation (e.g., US Executive Order 14028).

8. Memory Tricks & Mnemonics

🧠 STRIDE Threats
Spoofing = fake identity
Tampering = modify data
Repudiation = deny action
Information disclosure = leak data
Denial of service = crash system
Elevation of privilege = gain access
"STRIDE through the threats"
🧠 SDLC Security — "Shift Left = Start Security at the Start"
Requirements → Design → Code → Test → Deploy → Maintain → Dispose
Security starts at Requirements (leftmost) and continues through every phase.
"The further RIGHT you find a bug, the more it COSTS to fix"
🧠 Spiral Model = "RISK Spiral"
Spiral is the ONLY model that explicitly incorporates risk analysis in every cycle. "Spiral around the risk."
🧠 DREAD
Damage, Reproducibility, Exploitability, Affected users, Discoverability — risk scoring for each threat
