2. Real-World Organizational Relevance

Why Zero Trust Now

Traditional perimeter security ("castle-and-moat") assumed everything inside the network was trustworthy. Modern reality invalidates this: users work remotely, applications are in the cloud, BYOD devices are common, and attackers operate inside the perimeter (via phishing, insider threats, compromised VPNs). Zero Trust addresses this by treating every access request as potentially hostile regardless of network location.

Government Mandate

US Executive Order 14028 (May 2021, "Improving the Nation's Cybersecurity") mandated Zero Trust adoption across federal agencies. OMB Memorandum M-22-09 published specific federal ZTA requirements in 2022. CISA published a Zero Trust Maturity Model (2023). ZTA is now a regulatory expectation for U.S. government contractors and is rapidly becoming an industry standard.

Real-World Drivers

The Colonial Pipeline ransomware attack (2021) — attackers entered through a legacy VPN without MFA. Zero Trust with continuous verification and enforced MFA would have blocked initial access or contained lateral movement. SolarWinds (2020) demonstrated that trusted internal networks enabled devastating lateral movement — ZTA's micro-segmentation and least-privilege access would have constrained blast radius.

3. Zero Trust Principles

Zero Trust: A security paradigm that eliminates implicit trust in any component (user, device, network, application) regardless of location. Every access request must be explicitly authenticated, authorized, and continuously validated before access is granted — and must be limited to the minimum required.

NIST's Seven ZTA Tenets (SP 800-207)

  1. All data sources and computing services are considered resources. (Networks, endpoints, SaaS apps, IoT devices)
  2. All communication is secured regardless of network location. (Internal traffic is not inherently trusted)
  3. Access to individual enterprise resources is granted on a per-session basis. (No standing persistent access)
  4. Access to resources is determined by dynamic policy. (Identity, device health, other behavioral signals)
  5. Enterprise monitors and measures the integrity and security posture of all owned and associated assets.
  6. All resource authentication and authorization are dynamic and strictly enforced before access is allowed.
  7. Enterprise collects as much information as possible and uses it to improve security posture.

Three Core Principles Summarized

4. NIST SP 800-207 ZTA Framework

NIST SP 800-207: "Zero Trust Architecture" — Published August 2020. The authoritative federal reference for ZTA. Defines the conceptual model, components (PE/PA/PEP), deployment scenarios, and interaction with existing security frameworks (NIST CSF, NIST RMF).

ZTA Logical Components

┌─────────────────────────────────────────┐ │ Zero Trust Control Plane │ │ │ │ ┌──────────────┐ ┌──────────────────┐ │ │ │ Policy │ │ Policy │ │ │ │ Engine (PE) │◄──│ Administrator │ │ │ │ │ │ (PA) │ │ │ └──────┬───────┘ └──────────────────┘ │ │ │ Trust decision │ └─────────┼───────────────────────────────┘ │ ┌─────────▼───────────────────────────────┐ │ Policy Enforcement Point (PEP) │ │ (Gateway, proxy, or agent) │ └───────────────────┬─────────────────────┘ │ Subject ────────────► Enterprise Resource (User + Device) (App, Data, Service)

Component Roles

ComponentAbbreviationRole
Policy EnginePEThe decision-maker. Evaluates access requests against policies and signals (identity, device health, location, threat intelligence) and grants/denies access. The "brain" of ZTA.
Policy AdministratorPACommunicates decisions from PE to PEP. Sets up or tears down communication paths. Executes the PE's decision operationally.
Policy Enforcement PointPEPThe gate — enforces access decisions from the PA. Sits between the subject (user/device) and the resource. Can be a gateway, reverse proxy, or agent.
PE + PA = Policy Decision Point (PDP)
The NIST model separates the decision logic (PE) from the execution of the decision (PA). Together they form the control plane's Policy Decision Point. The PEP is the enforcement (data plane).
Exam Tip — PE vs. PA vs. PEP:
PE = decides (the judge)
PA = instructs enforcement (the bailiff)
PEP = blocks or allows (the guard at the door)

5. ZTA Core Components

Signal Sources for Trust Decisions

The Policy Engine evaluates multiple signals before making a trust decision:

  • Identity: MFA authentication, user account risk level, account behavior anomalies
  • Device health: Is the device managed? Current on patches? Has it passed compliance checks (MDM/EDR)?
  • Network: Is the request from a known IP? A Tor exit node? A foreign country?
  • Application: Which resource is being accessed? What sensitivity?
  • Threat intelligence: Is the IP/user/device associated with known threats?
  • Data sensitivity: Is the data being accessed highly classified? Additional factors may be required
  • Time and behavior: Is this a normal access pattern? Unusual time of day?

ZTA Implementation Approaches

6. Micro-Segmentation & Software-Defined Perimeter

Micro-Segmentation

Micro-Segmentation: Dividing the network (or compute infrastructure) into small, isolated zones where each workload or application segment controls its own inbound/outbound traffic policy. Prevents lateral movement — compromise of one segment does not grant access to adjacent segments.

Implemented via software-defined networking (SDN), hypervisor-based firewalls (NSX), or host-based firewall agents (Illumio ASP). Traditional network segmentation uses VLANs (Layer 2); micro-segmentation operates at Layers 3–7, often with application-identity awareness.

Exam Tip: Micro-segmentation addresses east-west (lateral) traffic — the communication between workloads within the data center. Traditional perimeter firewalls only address north-south (ingress/egress) traffic. ZTA requires both.

Software-Defined Perimeter (SDP)

Software-Defined Perimeter (SDP): An access control approach in which the network infrastructure (controllers and gateways) is invisible to unauthorized entities. Services are not accessible until the user is authenticated and authorized by the SDP Controller, then connected to an Application Connector — the service is "dark" to the internet until needed.

SDP originated from the Cloud Security Alliance (CSA) and is the foundation for modern Zero Trust Network Access (ZTNA) products. ZTNA (Day 68) replaces VPN by brokering per-application access rather than network-level access.

SDP Components: Initiating Host (client) → SDP Controller (verifies identity + device) → Accepting Host (application gateway) — all connections established only after controller authorization.

7. Google BeyondCorp Model

BeyondCorp: Google's internal Zero Trust initiative, begun in 2011 after Operation Aurora (the 2009 Google China breach). Published in a series of peer-reviewed papers (2014–2018). Eliminated the trusted corporate network — all applications accessible only via an identity-aware proxy; device inventory and posture enforced.

BeyondCorp Key Elements

  • Device Inventory Database: Every device is enrolled and its security posture continuously assessed
  • User/Group Database: Authoritative identity directory
  • Device Certificate: X.509 certificate issued to every managed device — proves device identity
  • Access Proxy / Identity-Aware Proxy (IAP): All traffic to internal apps routes through the proxy; enforces authentication and authorization
  • Trust Tier: Devices rated into trust tiers based on posture; application access granted based on minimum trust tier requirement

BeyondCorp Enterprise is now a Google Cloud product. The model inspired most modern ZTNA products and heavily influenced NIST SP 800-207.

BeyondCorp's Core Insight: Access should depend on WHO you are (identity) and WHAT device you're using (device posture) — not WHERE you are (network location). This breaks the fundamental assumption of perimeter security.

⚠️ ZTA Misconceptions:
  • ZTA does NOT eliminate the need for firewalls or network security — it supplements and refines them
  • ZTA is NOT a single product — it's an architectural philosophy requiring multiple coordinated technologies
  • ZTA does NOT mean zero security — it means zero implicit trust (explicit, verified trust on every request)
  • ZTA does NOT prevent all breaches — it minimizes blast radius and increases attacker difficulty

PA = Policy Administrator (instructs)
