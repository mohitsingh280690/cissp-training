2. Real-World Organizational Relevance

Buffer overflows remain one of the most exploited vulnerability classes (CWE-120, CWE-787). Memory-unsafe languages (C/C++) power critical infrastructure — operating systems, embedded systems, network devices. Understanding why these vulnerabilities exist and how language choice affects security is essential for security architects making technology decisions.

4. Object-Oriented Programming (OOP) Security

OOP ConceptDefinitionSecurity Relevance
EncapsulationBundling data and methods together; hiding internal state; access only through defined interfacesInformation hiding — internal implementation details not exposed; reduces attack surface; enforces access boundaries
AbstractionShowing only essential features; hiding complexityUsers interact with simplified interface; internal complexity hidden from potential attackers
InheritanceChild class inherits properties/methods from parent classSecurity concern: child may inherit vulnerabilities or overly permissive methods from parent; must review inherited behaviors
PolymorphismObjects of different classes can be treated as objects of a common parent; same method name, different behaviorSecurity concern: polymorphic malware changes form to evade detection; in code, unexpected method behavior if not properly controlled
Encapsulation = Information Hiding: This is the most security-relevant OOP concept for the CISSP. Encapsulation enforces that objects can only be interacted with through their public interface — internal state is protected. Think: "Encapsulation = access control for code."

Other OOP Terms

TermDefinition
ObjectInstance of a class; has state (data) and behavior (methods)
ClassBlueprint/template for objects; defines properties and methods
MethodFunction/behavior of an object
Message passingObjects communicate by sending messages (calling methods on other objects)
CouplingDegree of interdependence between modules; LOOSE coupling is preferred (modules can change independently)
CohesionDegree to which elements within a module belong together; HIGH cohesion is preferred (module does one thing well)
Design Principle: Loose coupling + High cohesion = better security. Loosely coupled modules limit blast radius of vulnerabilities. Highly cohesive modules are simpler and easier to review and test.

5. Memory Management & Buffer Overflows

Buffer Overflow

A buffer overflow occurs when a program writes data beyond the boundary of allocated memory (buffer). This can overwrite adjacent memory, including return addresses on the stack, allowing an attacker to redirect program execution to malicious code.
TypeDescription
Stack overflowOverflow in stack memory (local variables, return addresses); most common type; can redirect execution flow
Heap overflowOverflow in heap memory (dynamically allocated); can corrupt data structures; harder to exploit but possible
Integer overflowArithmetic result exceeds variable's maximum value; wraps around; can lead to buffer overflow if result used for memory allocation size

Buffer Overflow Defenses

DefenseHow It Works
Bounds checkingValidate that data fits within buffer before writing; use safe functions (strncpy vs strcpy)
ASLRAddress Space Layout Randomization — randomizes memory addresses; attacker can't predict where code is loaded
DEP / NX bitData Execution Prevention / No-Execute — marks memory regions as non-executable; prevents executing injected code in data areas
Stack canariesRandom value placed before return address; if overwritten (by overflow), program detects and terminates
Memory-safe languagesUse Java, C#, Rust, Python — built-in bounds checking eliminates most buffer overflows
Buffer Overflow = Failure of Input Validation / Bounds Checking: The root cause is always that the program didn't check whether the input fits in the allocated buffer. The fix is always input validation (bounds checking) at the code level, supported by OS-level protections (ASLR, DEP, canaries).

6. Common Software Vulnerabilities

VulnerabilityDescriptionPrevention
Race condition (TOCTOU)Time-of-Check to Time-of-Use: condition changes between when it's checked and when it's used; attacker exploits the gapAtomic operations; locks/mutexes; minimize time between check and use
Covert channelUnauthorized communication path using shared resources; Timing (observing time differences) or Storage (using shared storage)Audit resource access; reduce shared resources; noise injection
Back door / trap doorHidden access mechanism bypassing normal authentication; may be intentional (developer) or maliciousCode review; integrity monitoring; remove before production
Privilege escalationGaining higher permissions than authorized; vertical (user→admin) or horizontal (user→another user)Least privilege; input validation; access control enforcement
Memory leakProgram allocates memory but never frees it; eventually exhausts available memory (DoS)Proper resource management; garbage collection; code review
TOCTOU (Race Condition): The exam loves TOCTOU scenarios. Example: "A program checks if a user has permission to access a file, then opens the file. Between the check and the open, a symlink is swapped in pointing to /etc/passwd." The fix: atomic operations that check AND use in a single uninterruptible step.

7. Code Quality & Security Concepts

ConceptDescription
Input validationVerify ALL input before processing; whitelist (allow known-good) preferred over blacklist (block known-bad)

8. Memory Tricks & Mnemonics

🧠 OOP Security — "EAIP"
Encapsulation = information Hiding (most security-relevant)
Abstraction = hide complexity
Inheritance = inherits parent features (and vulnerabilities!)
Polymorphism = many forms (polymorphic malware changes shape)
🧠 Buffer Overflow Defenses — "BANDS"
Bounds checking
ASLR (address randomization)
NX / DEP (no-execute)
Data type validation
Stack canaries
🧠 TOCTOU — "Check-Gap-Use = Exploit"
Check (permission verified) → Gap (attacker swaps resource) → Use (wrong resource accessed)
Fix: make check-and-use ATOMIC (no gap)
🧠 Coupling & Cohesion — "Low C, High C"
Low Coupling = good (independent modules)
High Cohesion = good (focused modules)
"L comes before H — Low coupling, High cohesion"
