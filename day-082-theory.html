2. Real-World Organizational Relevance

Organizational Ownership

The application development team is responsible for secure session management and credential storage. The security architecture team sets standards (e.g., "use bcrypt with cost factor 12"). The IAM team governs MFA and passwordless rollout. The SOC monitors for session anomalies.

Implementation Reality

OWASP's A07:2021 (Identification and Authentication Failures) highlights that weak credential storage and broken session management remain top-10 web vulnerabilities year after year. Leading organizations now mandate bcrypt/Argon2 for password hashing, HttpOnly+Secure cookie flags, and are actively migrating to passkeys (FIDO2) for consumer and employee authentication. Google and Apple both support passkeys natively in their platforms since 2022-2023.

Failure Modes

  • RockYou 2009: 32 million passwords stored in plaintext — enabled mass account takeovers across the internet via credential stuffing
  • LinkedIn 2012: 117 million passwords hashed with unsalted SHA-1 — cracked within days using rainbow tables
  • Session fixation in banking portals: Attacker sends a known session token via link; victim authenticates → attacker's session is now authenticated

Business Impact

Credential database breaches trigger regulatory notification obligations (GDPR: 72 hours), class action lawsuits, and reputational damage. Proper password storage (bcrypt/Argon2) dramatically reduces the window of exploitation if a database is stolen — a 12-round bcrypt hash takes ~100ms to crack per candidate vs. microseconds for MD5.

3. Session Tokens & Cookies

Session Token: A cryptographically random, opaque identifier issued by a server after successful authentication. It represents the authenticated session and is sent with every subsequent request to authorize access — replacing the need to re-transmit credentials.

Cookie Security Attributes

AttributePurposeWhat It Prevents
SecureCookie only sent over HTTPSCookie theft via network sniffing (MITM)
HttpOnlyJavaScript cannot access the cookieCross-Site Scripting (XSS) session theft
SameSite=StrictCookie not sent on cross-site requestsCross-Site Request Forgery (CSRF)
SameSite=LaxSent on same-site navigation but not cross-site subresource requestsPartial CSRF protection
DomainLimits cookie scope to specified domainCookie leakage to subdomains
PathLimits cookie to specific URL pathCookie leakage to unrelated paths
Max-Age / ExpiresCookie lifetimeSession persistence beyond intended duration

Session Token Requirements

  • Sufficient entropy: At least 128 bits of randomness (OWASP recommendation)
  • Generated server-side: Client must never generate or predict its own session ID
  • Regenerated on privilege elevation: New session token issued after login, after role change, after password change
  • Server-side invalidation: Logout must destroy the server-side session record — client cookie deletion alone is insufficient
Exam Tip: A session is only truly terminated when the server-side session record is destroyed. If only the client-side cookie is deleted, the session token remains valid and can be replayed.

4. Session Fixation & Session Hijacking

Session Fixation

Session Fixation: The attacker establishes a known session token before the victim authenticates, then waits for the victim to log in using that token. Once authenticated, the attacker uses the same known token to access the victim's authenticated session.
1. Attacker visits site → gets session ID (SESS=ABC123) 2. Attacker sends victim a link with that session ID embedded: https://bank.com/login?SESS=ABC123 3. Victim clicks, logs in — server authenticates the session ABC123 4. Attacker uses ABC123 → now authenticated as victim! Fix: Issue a NEW session token immediately after successful authentication

Session Hijacking

Session Hijacking: The attacker captures an already authenticated session token and presents it to the server to impersonate the victim. The session was legitimate — the attacker steals it mid-flight or from storage.

Methods of session token theft:

  • Network sniffing: Intercepting unencrypted (HTTP) cookie traffic — prevented by Secure flag + HTTPS
  • Cross-Site Scripting (XSS): document.cookie exfiltration — prevented by HttpOnly flag
  • Cross-Site Request Forgery (CSRF): Tricks browser into sending authenticated requests to a different site — prevented by SameSite flag + CSRF tokens
  • Man-in-the-Middle (MITM): SSL stripping to downgrade HTTPS to HTTP
Session FixationSession Hijacking
When does attacker act?Before victim authenticatesAfter victim authenticates
What does attacker provide?A known session token to the victimSteals an existing valid token
Primary defenseRegenerate session ID post-authenticationSecure + HttpOnly cookies, HTTPS, short session lifetime

5. Credential Storage — Hashing & Salting

Never Store Passwords in Plaintext or With Reversible Encryption

Applications must store only a one-way hash of the password, never the plaintext. If a database is compromised, hashed passwords cannot be directly used (unlike plaintext). Reversible encryption is inadequate because the decryption key may also be compromised.

The Role of Salting

Salt: A random value generated uniquely per user and stored alongside the hash. Concatenated with the password before hashing, ensuring identical passwords produce different hashes. Defeats rainbow tables and prevents identical-password accounts from being identified by matching hashes.

Password Hashing Algorithms Comparison

AlgorithmTypeKey PropertyRecommended?
MD5Cryptographic hash (fast)~10 billion/sec on GPU — trivially crackable❌ Never
SHA-1 / SHA-256Cryptographic hash (fast)Fast hashing — wrong tool for passwords❌ Never for passwords
bcryptAdaptive, CPU-hardBuilt-in salt; cost factor adjusts work; 128-bit salt✅ Yes
scryptAdaptive, memory-hardMemory + CPU intensive; resists GPU/ASIC attacks✅ Yes
Argon2Adaptive, memory-hardPHC winner (2015); configurable memory, parallelism, iterations✅ Best practice
PBKDF2Key derivation (iterative)NIST/FIPS approved; used in WPA2, iOS; iteration count adjustable✅ Acceptable (with high iterations)
CPU-hard vs. Memory-hard:
bcrypt and PBKDF2 are CPU-hard — each hash requires CPU cycles; GPUs offer marginal speedup
scrypt and Argon2 are memory-hard — require large amounts of RAM; ASICs and high-GPU-count rigs struggle because memory bandwidth is the bottleneck
Memory-hard is considered stronger against custom hardware attacks
⚠️ CISSP Exam Note: bcrypt is CPU-hard, NOT memory-hard. scrypt and Argon2 are memory-hard. This distinction appears in exam questions. From Day 51 correction: bcrypt does NOT require large memory.

6. Token Management (JWT & OAuth Tokens)

JSON Web Tokens (JWT)

JWT (JSON Web Token): A compact, URL-safe token format consisting of three base64url-encoded parts: Header.Payload.Signature. The signature proves the token was issued by a trusted party and has not been tampered with.

JWT structure:

  • Header: Algorithm type (e.g., HS256, RS256) and token type
  • Payload (Claims): Asserted data — e.g., sub (subject/user ID), exp (expiration), iss (issuer), custom roles
  • Signature: HMAC-SHA256 of header+payload using a shared secret, or RSA/ECDSA signed
⚠️ JWT Security Pitfall — "alg: none" Attack: If the server does not validate the alg header, an attacker can set "alg": "none", remove the signature, and the server may accept the unsigned token as valid. Always enforce expected algorithm server-side.

OAuth 2.0 Token Types

Token TypePurposeLifetimeSecurity Note
Access TokenGrants access to a resource on behalf of the userShort (minutes to hours)Transmitted with every API call; must be kept confidential
Refresh TokenObtains new access tokens without re-authenticationLong (days to months)Higher value target; must be stored securely, rotated on use
ID Token (OIDC)Authentication assertion containing user identity claimsShortUsed by client to verify who authenticated; JWT format

Token Storage Best Practices

  • Access tokens in browser: Memory (JavaScript variable) preferred over localStorage (vulnerable to XSS) or cookies (if not HttpOnly)
  • Refresh tokens: HttpOnly cookies (not accessible to JavaScript) or secure server-side storage
  • Token rotation: Issue a new refresh token on every use; invalidate used tokens (prevent replay)
  • Token binding: Bind tokens to TLS session or client certificate to prevent token theft and replay

7. Password Managers

Password managers generate, store, and auto-fill unique, high-entropy passwords for each service. They dramatically reduce credential reuse (the root cause of credential stuffing).

Types

TypeStorageExampleSecurity Consideration
Standalone / LocalEncrypted local vaultKeePassUser responsible for backup; no breach risk from cloud provider
Cloud-basedEncrypted on provider's servers1Password, Bitwarden, DashlaneEnd-to-end encryption; master password never leaves device; provider cannot decrypt

Exam Tip: The main security benefit of a password manager is enabling unique, high-entropy passwords per service — defeating credential stuffing. The master password itself is the single point of failure; augmenting with MFA is essential.

8. Passwordless Authentication — FIDO2 & WebAuthn

FIDO2: A set of open authentication standards developed by the FIDO Alliance and W3C. Consists of two components: WebAuthn (the W3C browser/server API) and CTAP2 (the Client-to-Authenticator Protocol between browser and hardware authenticator). Enables passwordless and phishing-resistant authentication.

How FIDO2/WebAuthn Works

REGISTRATION: User registers authenticator (hardware key / device biometric) with service Authenticator generates a public/private KEY PAIR per service (origin-bound) Public key stored on server; private key NEVER leaves the authenticator AUTHENTICATION: Server sends a cryptographic CHALLENGE Authenticator signs challenge with PRIVATE KEY (requires user presence/biometric) Server verifies signature against stored public key → No password, no shared secret, no phishing risk (origin bound)

Passkeys

Passkeys are FIDO2 credentials that can sync across devices (e.g., stored in Apple Keychain, Google Password Manager, or Microsoft Authenticator). The private key is protected by device TPM/secure enclave and synced end-to-end encrypted. Make passwordless authentication accessible to mainstream users without a hardware key.

Why FIDO2 is Phishing-Resistant

The credential (key pair) is cryptographically bound to the website's origin. When a user is phished to evil-bank.com, the authenticator refuses to sign — the registered origin is bank.com. The challenge-response is unique per authentication and tied to the TLS session — replay attacks and MITM attacks are defeated.

Passwordless vs. MFA Fatigue: Push notification MFA is vulnerable to "MFA fatigue" (bombing the user with push requests until they approve). FIDO2 eliminates this because user presence (touching the authenticator) is required and the challenge is cryptographically bound.

9. Memory Tricks & Mnemonics

🧠 Cookie Flags — "SHS"
Secure — HTTPS only (network sniffing prevention)
HttpOnly — no JavaScript access (XSS prevention)
SameSite — no cross-site requests (CSRF prevention)
Miss any one → you leave a door open
🧠 Session Fixation vs. Hijacking — Timeline Rule
Fixation = attacker acts before authentication (fixes the ID they know)
Hijacking = attacker acts after authentication (steals the established session)
Fix = pre-auth | Jack = post-auth
🧠 Password Hashing Hierarchy — "Don't Use Fast Hashes"
Bad: MD5, SHA-1, SHA-256 (designed to be FAST → bad for passwords)
Good: bcrypt (CPU-hard), scrypt/Argon2 (memory-hard)
Argon2 won the Password Hashing Competition (PHC) 2015 → top choice
🧠 FIDO2 = WebAuthn + CTAP2
WebAuthn = browser ↔ server (web API, W3C standard)
CTAP2 = browser ↔ authenticator hardware (USB/NFC/BLE)
Together = FIDO2 = phishing-resistant passwordless auth

Top Exam Points:
  • Server must destroy the server-side session on logout — deleting the client cookie alone is insufficient
  • Session token must be regenerated after login (fixes session fixation) and after privilege change
  • Fast hash algorithms (MD5, SHA-256) are inappropriate for password storage — use adaptive slow hashes
  • FIDO2 = WebAuthn (W3C) + CTAP2 (hardware protocol); passkeys are synced FIDO2 credentials
  • Only FIDO2/WebAuthn/smart cards provide phishing resistance — TOTP and SMS OTP are bypassable via AiTM
