2. Real-World Organizational Relevance

Web applications are the primary interface between organizations and users, making them the #1 attack vector. Verizon DBIR consistently shows web application attacks account for the majority of breaches. The shift to cloud-native, API-driven architectures has expanded the web attack surface. Understanding these vulnerabilities is essential for security architects reviewing application designs and for managers approving deployment decisions.

3. OWASP Top 10 Overview

OWASP (Open Worldwide Application Security Project): A nonprofit organization producing free, open resources for web application security. The OWASP Top 10 is the most widely recognized awareness document for web application security risks, updated periodically (latest: 2021).
#2021 CategoryDescription
A01Broken Access ControlUsers act outside intended permissions; #1 most common vulnerability
A02Cryptographic FailuresFailures related to cryptography leading to sensitive data exposure
A03InjectionSQL, NoSQL, OS, LDAP injection when untrusted data sent to interpreter
A04Insecure DesignMissing or ineffective security controls in design; not implementation bugs
A05Security MisconfigurationDefault configs, open cloud storage, unnecessary features, verbose errors
A06Vulnerable ComponentsUsing components with known vulnerabilities; unpatched libraries/frameworks
A07Authentication FailuresBroken authentication allowing credential stuffing, brute force, session attacks
A08Software & Data Integrity FailuresAssumptions about software updates, CI/CD pipelines without integrity checks
A09Security Logging & Monitoring FailuresInsufficient logging making breach detection/forensics impossible
A10Server-Side Request Forgery (SSRF)App fetches remote resource without validating user-supplied URL
OWASP 2021 Key Change: Broken Access Control moved to #1 (was #5 in 2017). Injection dropped to #3 (was #1 in 2017). "Insecure Design" (A04) and "SSRF" (A10) are NEW categories added in 2021.

4. Injection Attacks

SQL Injection

SQL Injection: An attacker inserts malicious SQL statements into input fields that are concatenated directly into database queries. This can bypass authentication, read/modify/delete data, and even execute OS commands.

How it works:

Application builds query: SELECT * FROM users WHERE name = 'USER_INPUT'

Attacker enters: ' OR '1'='1

Resulting query: SELECT * FROM users WHERE name = '' OR '1'='1' — always true, returns all rows.

Types of SQL Injection

TypeDescription
Classic/In-bandAttacker uses same channel to inject and receive results (error-based, UNION-based)
BlindNo visible output; attacker infers data by observing behavior differences (boolean-based) or time delays (time-based)
Second-orderMalicious input stored first, executed later when a different operation retrieves and uses it

Other Injection Types

TypeTargetExample
LDAP injectionLDAP directory queriesManipulating authentication lookups in Active Directory
XML injection (XXE)XML parsersInjecting malicious XML entities to read files, SSRF
OS command injectionSystem shellInserting ; rm -rf / into a field passed to a system command
XPath injectionXML data queriesManipulating XPath expressions that query XML documents

Injection Defenses

DefenseEffectiveness
Parameterized queries (prepared statements)#1 best defense — separates SQL code from data; input NEVER interpreted as SQL
Stored proceduresGood if properly parameterized; still vulnerable if they use dynamic SQL internally
Input validation (whitelist)Defense in depth — validate type, length, range, format; whitelist > blacklist
Output encodingEncode special characters before rendering; prevents interpretation
Least privilegeDatabase account used by app should have minimal permissions; never use DBA/sa account
WAFWeb Application Firewall — additional layer; NOT a replacement for secure coding
Common Exam Trap: WAFs and input validation are defense-in-depth measures, but parameterized queries are ALWAYS the best answer for preventing SQL injection on the CISSP exam.

5. Cross-Site Scripting (XSS)

XSS (Cross-Site Scripting): An attacker injects malicious client-side scripts (typically JavaScript) into web pages viewed by other users. The victim's browser executes the script because it trusts the website. XSS attacks the USER, not the server.
TypeHow It WorksPersistence
Reflected (Non-persistent)Malicious script is embedded in a URL/link; server reflects it back in the response; victim clicks crafted linkNot stored; one-time delivery per click
Stored (Persistent)Malicious script stored on server (e.g., in a forum post, comment, profile); executes for every user who views the pagePermanently stored; affects ALL visitors — MOST DANGEROUS
DOM-basedPayload executed by modifying the DOM in victim's browser; vulnerability is in client-side JavaScript, not server responseNot stored on server; executed entirely client-side

XSS Defenses

DefenseDescription
Output encodingEncode all untrusted data before rendering in HTML (< → &lt;, > → &gt;); BEST defense
Input validationWhitelist acceptable characters; reject or sanitize dangerous input
Content Security Policy (CSP)HTTP header telling browser which sources of scripts are allowed; blocks inline scripts
HttpOnly cookiesPrevents JavaScript from reading session cookies — limits session hijacking via XSS
X-XSS-Protection headerLegacy browser-side XSS filter (deprecated in modern browsers; CSP is replacement)
XSS vs SQL Injection:
• SQL injection targets the server/database — server-side attack
• XSS targets the user/browser — client-side attack
• SQL injection defense: parameterized queries
• XSS defense: output encoding

6. Cross-Site Request Forgery (CSRF)

CSRF (Cross-Site Request Forgery): Also called XSRF or "session riding." Forces an authenticated user's browser to submit a forged request to a vulnerable site. Exploits the SITE'S TRUST in the user's browser. The user is already logged in; the attacker tricks their browser into performing an action (e.g., transferring money) without their knowledge.

CSRF vs XSS

AspectXSSCSRF
Exploits trust of…User trusts the website → browser runs script from "trusted" siteWebsite trusts the user's browser → processes authenticated request
Attack targetThe user/browserThe web application (performs actions as victim)
Requires script execution?Yes — script runs in victim's browserNo — just needs an HTTP request from the victim's authenticated session
Best defenseOutput encoding, CSPAnti-CSRF tokens (synchronizer tokens), SameSite cookies

CSRF Defenses

DefenseHow It Works
Anti-CSRF tokensUnique, unpredictable token embedded in each form; server verifies token on submission; attacker cannot guess the token
SameSite cookiesCookie attribute preventing browser from sending cookies with cross-site requests; Strict or Lax mode
Re-authenticationRequire password/MFA for sensitive actions (transfers, password changes)
Referer/Origin header checkingVerify the request originated from the application's own domain

7. Insecure Direct Object References & Access Control Failures

IDOR (Insecure Direct Object Reference): Application exposes internal object identifiers (e.g., database IDs) in URLs or form parameters. Attacker changes the ID to access another user's data. Example: /account?id=12345 → change to /account?id=12346 to view someone else's account.

Broken Access Control Patterns

PatternDescription
IDORManipulating object references to access unauthorized resources
Horizontal privilege escalationAccessing another user's data at the same privilege level
Vertical privilege escalationPerforming actions reserved for higher-privilege roles (user → admin)
Missing function-level access controlAdministrative URLs accessible without proper authorization checks
Path traversalUsing ../ in file paths to access files outside intended directory
Defense: Enforce access control on every request, server-side. Never rely on client-side checks. Use indirect references (map user-facing IDs to internal IDs). Deny by default. Log all access control failures.

Unpatched softwareKnown vulnerability exploitationRegular patching; vulnerability scanning; SBOM tracking

9. Memory Tricks & Mnemonics

🧠 XSS Types — "RSD"
Reflected = Request (one-time, via URL)
Stored = Server stores it (persistent, most dangerous)
DOM-based = Direct browser manipulation (client-side only)
🧠 "XSS exploits user's trust in the site; CSRF exploits site's trust in the user"
This single sentence distinguishes the two on the exam every time.
🧠 Best Defenses Quick Reference
SQL injection → Parameterized queries
XSS → Output encoding
CSRF → Tokens (anti-CSRF)
Mnemonic: POT — Parameterized, Output encode, Tokens
