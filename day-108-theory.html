2. Real-World Organizational Relevance

The majority of successful breaches exploit known vulnerabilities with available patches. According to Verizon DBIR data, unpatched vulnerabilities remain a top initial access vector. Preventive measures — when consistently applied — eliminate the most common attack paths before adversaries can use them.

3. Security Baselines & CIS Benchmarks

Security Baseline: A documented, minimum security configuration standard for a specific technology (OS, application, network device). Systems must meet or exceed the baseline before deployment to production. Baselines are compared against during audits and monitoring.

CIS Benchmarks

FeatureDetail
PublisherCenter for Internet Security (CIS)
ContentDetailed, prescriptive configuration guides for 100+ technologies (Windows, Linux, AWS, Azure, databases, network devices)
ProfilesLevel 1 (basic, minimal business disruption), Level 2 (defense-in-depth, may impact functionality)
Tool supportCIS-CAT Pro scans systems against benchmarks; many vulnerability scanners include CIS benchmark checks

4. Patch Management

Patch Management Lifecycle

StepActivityDetails
1InventoryKnow what you have: complete, accurate asset inventory including OS versions, applications, and configurations
2MonitorTrack vendor patch releases, security advisories, CVE announcements
3AssessEvaluate patch urgency/priority based on vulnerability severity (CVSS), asset criticality, exploit availability, and business impact
4TestTest patches in non-production environment first; verify no breaking changes; test with critical business applications
5DeployRoll out patches through change management process; phased deployment (pilot → broader rollout)
6VerifyConfirm patches applied successfully; re-scan systems; validate functionality; document completion
Exam Trap — Emergency Patches: Critical zero-day patches may require an emergency change process that bypasses normal testing. This should be pre-approved in the change management policy with defined criteria (e.g., active exploitation of critical vulnerability). Even emergency patches should be tested on at least ONE non-production system before broad deployment.
⚠️ Virtual Patching: When patches cannot be immediately applied (legacy systems, vendor delays, testing requirements), virtual patching provides interim protection. Virtual patches are WAF rules, IPS signatures, or firewall rules that block the specific exploit vector WITHOUT modifying the vulnerable system. Virtual patching is a compensating control, NOT a substitute for actual patching.

5. Anti-Malware & Endpoint Detection

Evolution of Endpoint Protection

GenerationTechnologyDetection MethodLimitations
1st GenAntivirus (AV)Signature-based — matches known malware file hashes/patternsCannot detect unknown (zero-day) malware; requires constant signature updates
2nd GenNext-Gen AV (NGAV)Heuristic + behavioral analysis — examines code behavior, not just signaturesHigher false positive rate; still primarily passive
3rd GenEDR (Endpoint Detection & Response)Continuous monitoring + behavioral analytics + automated response; records endpoint telemetryRequires skilled analysts; generates significant data volume
4th GenXDR (Extended Detection & Response)Integrates EDR + NDR + cloud + email — cross-domain correlationVendor lock-in risk; integration complexity
EDR vs Traditional AV: Traditional AV answers "Is this file malicious?" (point-in-time scan). EDR answers "What is this endpoint doing?" (continuous behavioral monitoring). EDR can detect living-off-the-land attacks (using legitimate tools maliciously), fileless malware, and post-compromise activity that AV misses entirely.

6. Application Whitelisting / Allowlisting

Application Allowlisting: A security control that permits ONLY approved applications to execute on a system. Everything not explicitly allowed is blocked by default. This is the inverse of blocklisting/blacklisting (which blocks known-bad and allows everything else).

Allowlisting vs Blocklisting

FeatureAllowlisting (Whitelist)Blocklisting (Blacklist)
Default stanceDeny all; permit approvedAllow all; block known-bad
Zero-day protectionStrong — unknown code is blocked by defaultNone — unknown malware is not on blocklist
Management overheadHigh — must maintain approved app list; every legitimate update requires approvalLower — update signature database automatically
User flexibilityRestrictive — users cannot install unapproved softwareFlexible — users can run anything not on blocklist
Best forServers, kiosks, critical infrastructure, fixed-function systemsGeneral workstations where users need flexibility
Exam Favorite: "Which is the MOST effective control against unknown/zero-day malware?" Answer: Application allowlisting. Because unknown malware is NOT on any blocklist, blocklisting fails. Allowlisting blocks everything not explicitly approved, including zero-day malware. This is a frequent exam question.

7. Host-Based Firewalls & IPS

ControlFunctionDeployment
Host-based firewallFilters network traffic TO/FROM the individual endpoint; controls inbound and outbound connections per application or portBuilt into OS (Windows Firewall, iptables/nftables on Linux); managed via GPO/MDM for enterprise consistency
Host-based IPS (HIPS)Monitors system behavior and network traffic at the host level; blocks suspicious activity in real-timePart of EDR or standalone agents; monitors process behavior, file changes, registry modifications
Defense in Depth — Network vs Host: Network firewalls protect the perimeter; host-based firewalls protect individual systems even when the network perimeter is bypassed (insider threats, lateral movement, VPN-connected remote workers). Both are needed — this is defense in depth.

8. Hardening & Least Functionality

System Hardening: The process of reducing the attack surface by removing unnecessary services, ports, protocols, accounts, and features. The goal is to leave ONLY what is needed for the system's intended function.

Hardening Checklist

ActionDetails
Remove unnecessary servicesDisable services not required for system function (Telnet, FTP, SNMP v1/v2, print services on non-print servers)
Close unnecessary portsBlock all ports not required for operation; use host firewall
Remove default accountsDisable or rename default admin accounts; remove guest accounts
Apply security updatesCurrent patches for OS, applications, and firmware
Enable loggingConfigure appropriate log levels; forward to SIEM
Configure access controlsPrinciple of least privilege; remove unnecessary user rights
Enable encryptionEncrypt data at rest (FDE); use TLS for data in transit
Disable autorun/autoplayPrevent automatic execution of media (USB, CD)
BIOS/UEFI securitySet BIOS password; secure boot; boot order (no USB/CD boot)

9. Memory Tricks & Mnemonics

🧠 Patch Management — "IMATDV"
Inventory → Monitor → Assess → Test → Deploy → Verify
"I Must Always Test, Deploy, Verify"
🧠 Allowlisting vs Blocklisting Default
Allowlist = "You're not on my list, you can't come in" (deny by default)
Blocklist = "You're on my naughty list, you're blocked" (allow by default)
🧠 Endpoint Protection Evolution — "ANSE-X"
AV (signatures) → NGAV (heuristics) → Sandbox → EDR (behavioral) → XDR (cross-domain)
"Each generation sees more"
🧠 Hardening = Removal
"When in doubt, take it out." Hardening is about removing — services, ports, accounts, features. If the system doesn't need it, disable or remove it.

9. Insider Threat Detection & Prevention Program

Insider Threat Program Framework

Insider threats are employees, contractors, or partners with legitimate access who misuse it. Detection requires a multi-faceted program combining personnel security, monitoring, and investigation:

Program ComponentFocusTools/Methods
Threat AssessmentIdentify at-risk individuals; profiles; motivation patternsHR intel (discipline, termination plans), behavioral analysis, UEBA baselines
Behavioral MonitoringDetect deviations from normal activity; unusual access patternsUEBA, DLP, file access logs, email monitoring, USB activity logging
Technical ControlsPrevent exfiltration of sensitive dataDLP policies, USB restrictions, file integrity monitoring, egress filtering
Personnel SecurityVetting, access controls, job rotation, mandatory vacationBackground checks, SoD, access reviews, job rotation (no consecutive access to critical functions)
Reporting & InvestigationDetect suspicious activity; conduct thorough investigationAnonymous reporting channel, forensic investigation team, security briefings

Red Flags for Insider Threat

CategoryRed Flags
BehavioralUnusual working hours (after-hours access when not typical); excessive sick leave; personality changes; financial stress signs; fascination with security systems or bypass methods
TechnicalExcessive file access (downloading beyond job scope); mass data transfer; access to unrelated systems; clearing logs; attempting privilege escalation; unusual USB activity
SituationalPending termination or demotion; job dissatisfaction complaints; competitor employment contact; foreign travel to sensitive countries; divorce or financial hardship
Access Pattern ChangesAccessing data outside normal business hours or from unusual locations; VPN from new geography; accessing data they previously didn't use

Investigation & Response

  • Initial Report: Anonymous tip (ethics hotline) or alert from monitoring. DO NOT immediately confront the suspect; preserve evidence.
  • Triage: Is this a real threat (data actually exfiltrated)? Assess immediacy; does suspect still have access?
  • Evidence Preservation: Engage IT to pull logs, create forensic images of systems used by suspect, preserve email, file access history, USB history.
  • Legal Engagement: Notify HR and legal counsel; determine what can be disclosed to suspect vs. kept confidential for investigation.
  • Suspension/Restrictions: May revoke access, restrict to monitored workstation, or conduct monitored interview (with legal present).
  • Root Cause Analysis: Post-investigation, determine how insider got access to what data, where data went, and control failures that allowed it.
💡 Exam Tip: Insider threat scenarios on CISSP often test whether you know the difference between suspicion and proof. You suspect theft (technical monitoring shows unusual access) but need evidence (DLP logs showing data was copied to USB, or email showing exfiltration). The scenario might ask: "An analyst accessed 100 HR records last night. Is this insider threat?" Answer: Not yet — you have suspicious access, but you need to investigate: Was data copied? Where did it go? Does the analyst have a business reason (payroll batch job)? This is why UEBA + DLP together are powerful; behavioral (UEBA) + data movement (DLP) = confirmation.
