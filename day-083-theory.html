2. Real-World Organizational Relevance

Organizational Ownership

The IAM team designs and operates the IDaaS platform. The network/infrastructure team manages directory synchronization (e.g., Azure AD Connect). Application owners integrate their apps via SAML or OIDC. Security leadership governs access policies, MFA requirements, and conditional access rules.

Implementation Reality

The majority of enterprises (>80%, per Gartner) now use some form of IDaaS for employee or customer identity. Microsoft Entra ID (formerly Azure AD) has 300M+ monthly active users. Okta serves 17,000+ enterprise customers. The COVID-19 pandemic accelerated adoption massively — remote work demands cloud-based identity rather than on-prem VPN-dependent solutions.

Failure Modes

  • Lapsus$ attacks (2022): Hackers repeatedly compromised IDaaS providers and enterprise tenants via social engineering against help desks — bypassing MFA via helpdesk account resets
  • Directory sync errors: Stale accounts synchronized from on-prem AD keep cloud access alive for terminated employees
  • Overly permissive conditional access: Policies that trust any managed device regardless of risk score enable compromised laptops to access cloud resources

Business Impact

IDaaS reduces IT helpdesk costs (self-service password reset saves $30+/call), improves user productivity via SSO, and enables rapid application onboarding. However, the IDaaS provider becomes a critical single point of failure — SLA guarantees (99.99% = ~52 min/year downtime) must be evaluated against business continuity requirements.

3. IDaaS Overview & Core Components

Identity as a Service (IDaaS): A cloud-delivered IAM platform providing identity management, authentication, SSO, MFA, and lifecycle governance for users accessing cloud, on-premises, and hybrid resources. Eliminates the need to build and maintain custom IAM infrastructure.

5. SCIM — Identity Provisioning Protocol

SCIM (System for Cross-domain Identity Management): An open standard (RFC 7644/7643) for automating the exchange of user identity information between identity providers and cloud applications. Enables automated provisioning and deprovisioning of user accounts across systems.

SCIM vs. SAML / OIDC

⚠️ Critical Distinction — SCIM vs. SAML/OIDC:
SCIM = identity provisioning (creating/updating/deleting accounts, syncing attributes)
SAML/OIDC = identity authentication/federation (proving identity at login time)
These are complementary — SCIM provisions the account; SAML/OIDC authenticates the user to it

SCIM in Practice

  • HR system triggers: New hire in Workday → SCIM creates accounts in Entra ID, Slack, Salesforce, GitHub automatically
  • Termination: HR terminates employee in Workday → SCIM disables/deletes all downstream accounts within seconds
  • Attribute sync: Department, title, manager changes propagate to all apps automatically
  • Just-in-time provisioning (JIT): SAML-based alternative — account created on first login from IdP assertion, using SAML attributes

SCIM Technical Overview

AspectDetail

6. Hybrid Identity Models

Most enterprises have a mix of on-premises and cloud resources, requiring a hybrid identity strategy that unifies both into a single identity plane.

Model 1: Cloud-Only Identity

All identities exist only in the cloud directory (e.g., Entra ID). Suitable for cloud-first/cloud-native organizations. No on-premises AD dependency. Simplest model but not viable for organizations with on-premises applications or OT systems requiring AD authentication.

Model 2: Directory Synchronization (Password Hash Sync)

On-premises AD remains the source of truth. A sync tool (e.g., Azure AD Connect) copies user objects and a hash of the password hash to the cloud directory. Authentication can happen in the cloud without calling back to the on-prem AD. Advantage: Uses on-prem passwords for cloud auth. Risk: Credential compromise in AD propagates to cloud.

Model 3: Pass-Through Authentication (PTA)

Authentication occurs against the on-premises AD in real time via a lightweight agent. The password hash is NOT stored in the cloud. Advantage: On-prem password policies instantly enforced in cloud. Risk: Requires on-prem infrastructure availability; if on-prem AD is unavailable, cloud auth fails.

Model 4: Federation (ADFS / SAML)

On-premises Active Directory Federation Services (ADFS) issues SAML tokens. The cloud directory trusts tokens from ADFS. Full on-prem control; supports advanced claims transformation. Complex; most organizations are moving away from ADFS toward Azure AD Connect cloud sync.

7. Benefits & Risks of IDaaS

Benefits

  • Rapid application integration: Pre-built connectors for thousands of SaaS apps (SAML/OIDC/SCIM)
  • Reduced infrastructure: No servers to patch, no HA clusters to maintain for identity services
  • Automatic updates: Provider handles security patches, new protocol support, feature rollout
  • Global scalability: Multi-region availability by default; cloud SLAs typically 99.9–99.99%
  • Advanced capabilities: AI-powered risk signals, behavioral analytics, impossible travel detection out-of-box

Risks

RiskDescriptionMitigation
Vendor dependencySingle IDaaS provider is a single point of failure for all authenticationSLA review, offline bypass procedures, multi-IdP architecture
Data sovereigntyIdentity data (names, attributes, auth logs) stored in provider's data centersChoose providers with regional data residency options; review DPA/DPAs (Data Processing Agreements)
Account takeover at providerCompromise of the IDaaS provider affects all tenantsPhishing-resistant MFA, IP restrictions, hardware keys for admin accounts
Shadow IT integrationDevelopers bypass IDaaS, create direct user stores in SaaS appsPolicies requiring SSO; CASB to detect direct SaaS auth
Misconfigured policiesOverly permissive conditional access or missing MFA requirementsRegular policy review; automated compliance scanning

8. Cloud IAM Best Practices

  • Enforce MFA for all users — phishing-resistant (FIDO2) for admins and high-risk users
  • Implement Conditional Access — device compliance, location, risk score evaluation before granting access
  • Automate lifecycle management via SCIM — no manual deprovisioning; tie to HRMS
  • Audit and certify access regularly — quarterly access reviews; immediate revocation on termination
  • Apply least privilege in cloud roles — no standing admin roles; use just-in-time elevation (PIM/PAM)
  • Monitor sign-in logs — integrate IDaaS logs into SIEM; alert on anomalous sign-ins, impossible travel, new device registration
  • Protect admin accounts separately — dedicated admin accounts; privileged access workstations (PAWs); break-glass accounts with offline storage
  • Use app-specific service accounts with minimal scope — avoid global admin service principals; rotate secrets via managed identities where possible
Privileged Identity Management (PIM) in IDaaS: Allows eligible users to activate elevated roles just-in-time with time limits and approval workflows. Eliminates standing admin accounts — aligns with PAM concepts from Day 79. Major IDaaS platforms include PIM capabilities (Azure PIM, Okta Admin Roles, Google Cloud Privileged Access Manager).

9. Memory Tricks & Mnemonics

🧠 IDaaS Core Functions — "SSM-CLD"
Single Sign-On (one login, many apps)
SCIM provisioning (automate lifecycle)
MFA enforcement (second factor mandatory)
Conditional Access (context-aware policies)
Lifecycle governance (join/move/leave)
Directory sync (on-prem ↔ cloud)
🧠 SCIM vs. SAML — Provisioning vs. Authentication
SCIM = "Set up the account" (before login) — provisioning
SAML = "Sign in with the account" (at login) — authentication
SCIM creates the room; SAML opens the door
🧠 Hybrid Identity Models — Password Location
Cloud-Only: Password lives in cloud
Password Hash Sync: Hash-of-hash lives in cloud
Pass-Through Auth: Password stays on-prem (validated live)
Federation (ADFS): Token-based, password stays on-prem
Security preference: Pass-Through ≥ Federation > Hash Sync (for on-prem control)

Top Exam Points:
  • SCIM automates provisioning; SAML/OIDC handles authentication — they serve different purposes and are complementary
  • IDaaS does not eliminate identity governance responsibility — the organization remains accountable
  • Third-party IDaaS introduces supply chain risk — requires vendor risk assessment, SLA, DPA
  • Just-in-Time provisioning (JIT) via SAML creates accounts on first login using IdP attributes — alternative to SCIM pre-provisioning
  • Managed identities and service principals in cloud platforms eliminate the need for static credentials in service accounts
