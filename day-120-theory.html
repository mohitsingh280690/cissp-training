2. Real-World Organizational Relevance

Organizations deploying code hundreds of times per day cannot rely on manual security reviews for each release. DevSecOps automates security checks within the CI/CD pipeline — SAST at commit, SCA at build, DAST in staging, and compliance verification before production. This enables "security at the speed of DevOps" without manual bottlenecks.

3. DevOps → DevSecOps Evolution

ConceptDefinition
DevOpsCultural and technical practice unifying Development and Operations; automates software delivery; aims for faster, more reliable releases through collaboration, automation, and continuous feedback
DevSecOpsExtension of DevOps that integrates Security as a shared responsibility throughout the entire pipeline. Security is not a separate team/gate but embedded into every stage
DevSecOps Core Principle: "Everyone is responsible for security." Security is not the security team's job alone — developers write secure code, operations deploy securely, and security provides tools, training, and automation to enable both. Security is a shared responsibility, not a bottleneck.

4. CI/CD Pipeline & Security Gates

CI/CD Defined

TermDefinition
Continuous Integration (CI)Developers frequently merge code into shared repository; each merge triggers automated build and tests. Catches issues early
Continuous Delivery (CD)Code is always in a deployable state; deployment to production requires manual approval
Continuous DeploymentEvery change that passes automated tests is automatically deployed to production — no manual approval

Security Gates in CI/CD Pipeline

Pipeline StageSecurity ActivityTool Type
Code commitPre-commit hooks; secret scanning; linting for security patternsSecret scanners, linters
BuildStatic Application Security Testing (SAST); code quality analysisSAST tools
Dependency checkSoftware Composition Analysis (SCA); vulnerability check in librariesSCA tools
TestUnit security tests; integration security tests; DAST in stagingDAST, security test frameworks
Container buildContainer image scanning; base image verification; Dockerfile analysisContainer scanners
ReleaseCompliance verification; security sign-off; SBOM generationCompliance as code
DeployConfiguration audit; infrastructure verification; secrets injectionIaC scanning, config mgmt
RuntimeRuntime Application Self-Protection (RASP); monitoring; WAFRASP, APM, SIEM
SAST vs DAST:
• SAST (Static) = analyzes SOURCE CODE without running it; finds bugs early (build stage); white-box; fast; many false positives
• DAST (Dynamic) = tests RUNNING APPLICATION from outside; finds runtime vulnerabilities; black-box; fewer false positives but later in pipeline
• IAST (Interactive) = combines both; instrumented agent inside running app; most accurate but requires runtime

5. Infrastructure as Code (IaC) Security

Infrastructure as Code (IaC): Managing and provisioning infrastructure through machine-readable definition files rather than manual configuration. Examples: Terraform, CloudFormation, Ansible, Puppet, Chef.
Security BenefitSecurity Risk
Consistent, repeatable configurations (eliminates drift)Misconfiguration in code propagates at scale
Version controlled — changes tracked and auditableSecrets accidentally committed to repositories
Can be scanned for security issues before deploymentOverly permissive defaults may be templated
Enables immutable infrastructure (replace, don't patch)Complexity of IaC tools themselves
Immutable Infrastructure: Instead of patching running servers, replace them entirely with new instances built from updated IaC templates. Eliminates configuration drift and ensures every server matches the known-good template. "Cattle, not pets" — servers are disposable and replaceable, not individually maintained.

6. Container & Microservices Security

ConcernSecurity Practice
Base image vulnerabilitiesUse minimal, verified base images; scan images in CI/CD; update regularly
Image integritySign container images; verify signatures before deployment; use trusted registries
Runtime isolationContainers share host kernel — NOT full isolation like VMs; use namespaces, cgroups, seccomp profiles
Secrets in containersNever bake secrets into images; inject at runtime from secrets vault
Network segmentationMicroservice-to-microservice communication should be encrypted (mTLS); network policies limit lateral movement
Orchestration securityKubernetes RBAC; pod security policies; namespace isolation; API server authentication
⚠️ Containers ≠ VMs: Containers share the host OS kernel. A kernel vulnerability affects ALL containers on that host. Containers provide process isolation (namespaces), not hardware isolation (hypervisor). For true isolation between untrusted workloads, use VMs or microVMs.

7. Secrets Management

Secrets: Any credential, key, certificate, or token that grants access — API keys, database passwords, encryption keys, SSH keys, TLS certificates, OAuth tokens.

Secrets Management Best Practices

PracticeDescription
Never in codeSecrets must NEVER be hardcoded in source code or committed to version control
Secrets vaultUse dedicated secrets management (HashiCorp Vault, AWS Secrets Manager, Azure Key Vault); centralized, encrypted, access-controlled
Inject at runtimeSecrets delivered to applications at runtime via environment variables or vault API calls — never baked into images/artifacts
RotationAutomated secret rotation; short-lived credentials; revoke immediately on compromise
AuditLog all secret access; monitor for unusual access patterns
Pre-commit scanningAutomated scanning of commits for accidentally included secrets (git-secrets, truffleHog)
Exam Essential: If a question mentions API keys, passwords, or tokens stored in source code — the answer is ALWAYS to move them to a secrets vault and inject at runtime. Hardcoded credentials are one of the most common and critical vulnerabilities.

9. Memory Tricks & Mnemonics

🧠 DevSecOps = "Dev + Sec + Ops = Security is EVERYONE's job"
Not a team, not a gate — a culture. Security embedded in every pipeline stage.
🧠 SAST vs DAST — "S = Source code (Static); D = Deployed app (Dynamic)"
SAST = looks at code (white-box, early, fast, more false positives)
DAST = tests running app (black-box, later, slower, fewer false positives)
IAST = inside the app (best of both, requires runtime)
🧠 CI/CD — "CI = Code In (frequently); CD = Code Delivered (always ready)"
Continuous Integration = merge + build + test frequently
Continuous Delivery = always deployable (manual approval)
Continuous Deployment = automatically deployed (no manual gate)
🧠 Secrets — "Never In Code, Always In Vault"
NICAIV — secrets NEVER in code repos; ALWAYS in a secrets vault; inject at runtime; rotate regularly.
