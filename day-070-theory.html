1. Where This Fits in CISSP

AspectDetail
Domain5 – Identity and Access Management (13%)
Why TestedIAM is one of the highest-tested domains. Every security control ultimately comes back to identity — who is doing this, are they allowed, can we prove it happened? CISSP tests fundamental IAM concepts (IAAA), identity lifecycle, access control models, and modern IAM technologies (MFA, SSO, federation, PAM).
Typical Question Styles"A user presents a username — which IAM step is this?"; "What is required to hold a user accountable for actions?"; "Which principle states users should only have the minimum access needed?"; "What is the difference between authentication and authorization?"
Cross-Domain LinksDomain 1 (least privilege, SoD — governance principles); Day 60 (firewalls enforce access between zones); Domain 7 (audit logs enable accountability); Domain 8 (IAM in application development — OWASP A07: Identification and Authentication Failures)
Key Exam TrapsIdentification ≠ Authentication. Identification = claim identity (present username). Authentication = PROVE identity (present password/factor). Authorization = what you CAN DO. Accountability requires authentication to work (can't hold anonymous user accountable). Non-repudiation is the strongest accountability mechanism (cryptographic proof).

2. Real-World Organizational Relevance

IssueContext
Identity is the new perimeterTraditional network perimeters (firewalls at the edge) have dissolved with cloud, remote work, and SaaS. Identity is now the primary security control. If an attacker has valid credentials, they can access resources legitimately — no network perimeter to bypass.
Access sprawlEmployees accumulate permissions over time (privilege creep). Without regular access reviews and deprovisioning, users retain access they no longer need — violating least privilege. Orphan accounts (accounts of departed employees) are among the most dangerous.
IAM is breach enabler #1Verizon DBIR consistently shows that compromised credentials are involved in 80%+ of breaches. MFA alone blocks 99%+ of credential-based automated attacks. IAM hardening (MFA + PAM + just-in-time access) is the highest ROI investment after basic hygiene.
Regulatory requirementsGDPR: access controls for personal data. HIPAA: minimum necessary standard mirrors least privilege. PCI-DSS: unique user IDs, MFA for all admin access. SOX: separation of duties in financial systems. All require demonstrable access controls and audit trails.

3. IAAA – The Four Pillars

📖 IAAA: The four core functions of an identity and access management system: Identification, Authentication, Authorization, Accountability. Each is distinct — failure at any step weakens the entire IAM chain.
PillarQuestionDefinitionExample
IdentificationWho are you claiming to be?The user claims an identity — presents a unique identifier. Not yet verified.Typing a username; presenting an employee badge number; entering an email address in a login form
AuthenticationCan you prove who you are?The system verifies the claimed identity — proves the claim is genuine using one or more factors.Entering a password; scanning a fingerprint; entering an OTP; presenting a smart card
AuthorizationWhat are you allowed to do?After proven identity, the system determines what resources and actions the authenticated identity may access. Enforced by access control policies.User can read file A but not write it; user can access HR system but not finance; role-based permissions
AccountabilityWhat did you do, and can we prove it?All actions by authenticated users are logged and attributable to the specific identity. Enables auditing, forensics, and non-repudiation.Audit logs recording every database query; security event logs with user ID timestamps; SIEM correlation
🔑 IAAA depends on the previous step:
Authorization can only work after authentication (can't enforce permissions for an unknown identity).
Accountability can only work after authentication (audit logs only meaningful if identity is proven).
Non-repudiation (the strongest accountability) requires cryptographic proof — digital signatures prove the specific private key holder performed an action.

4. Identification

AspectDetail
FunctionAllows each user to be uniquely distinguished from all others. The identifier must be unique within the system. No security guarantee at this stage — anyone can claim any identity.
Unique identifiersUsername, user ID, email address, employee number, certificate Distinguished Name, public key
RequirementsEach identifier must be unique. Shared/generic accounts ("admin", "user1") break accountability — cannot attribute actions to a specific individual. Unique IDs are a PCI-DSS requirement.
Non-security roleIdentification alone is not a security function — it simply labels the subject so the system knows which credentials to check and which authorization profile to apply.

5. Authentication

Authentication Factor TypeCategoryExamplesWeakness
Type 1 – Something you knowKnowledge-basedPassword, PIN, passphrase, security question, patternCan be guessed, phished, shoulder-surfed, stolen from database; weakest factor type
Type 2 – Something you havePossession-basedSmart card, hardware token (RSA SecurID), software OTP (TOTP/HOTP), physical key, mobile deviceCan be lost, stolen, cloned (depending on implementation); SIM-swapping Attack on SMS
Type 3 – Something you areBiometricFingerprint, iris scan, retina scan, facial recognition, voice recognition, gait analysisCannot be changed if compromised; false acceptance/rejection; requires enrollment; cultural/legal privacy concerns

Multi-Factor Authentication (MFA)

🔑 True MFA requires TWO OR MORE factors from DIFFERENT type categories.
Password + PIN = NOT MFA (both are Type 1 – something you know)
Password + OTP = MFA (Type 1 + Type 2)
Password + fingerprint = MFA (Type 1 + Type 3)
Smart card + PIN = MFA (Type 2 + Type 1)

6. Authorization

ConceptDefinition
AuthorizationThe process of determining what an authenticated identity is permitted to do. Separate from authentication — a user can be authenticated but unauthorized for a specific resource.
Access control policySet of rules defining what subjects (users, processes, roles) can access which objects (files, databases, systems) and what operations (read, write, execute, delete) they may perform.
Implicit denyUnless explicitly granted, access is denied by default. The correct security posture: whitelist approach. The opposite (allow by default) is blacklisting — insecure baseline.
Need-to-knowEven within their clearance level, individuals may only access information they have a specific need for their assigned duties. Limits damage from insider threats and compromised accounts.
Least privilegeUsers, processes, and systems are granted only the minimum permissions necessary to perform their function. Reduces attack surface and blast radius of compromise.
Separation of duties (SoD)No single individual has enough access to commit fraud or sabotage undetected. Critical operations require multiple people (e.g., both request and approval of a wire transfer). Also: dual control, two-man rule.
💡 Exam Tip — Least Privilege vs. Need-to-Know: These are often used together. Least privilege applies to functional capabilities (minimum permissions for the role). Need-to-know applies to information access within a clearance level — even if you can technically access the data, you should only see it if you need it for your work.

7. Accountability

ConceptDetail
Audit trails / loggingRecords of who accessed what, when, from where, and what they did. Required for accountability. Logs must be protected from modification (write-once, log server separate from logged system).
Non-repudiationThe inability to deny having performed an action. Requires strong authentication (especially digital signatures — private key uniquely identifies the signer). Higher standard than simple audit logging.
Shared accounts undermine accountabilityIf 10 people share one "admin" account, you cannot determine which person performed a malicious action. Shared accounts must be prohibited for any accountable system.
Clipping levelA threshold below which minor security events are not logged or reported. For example, 3 failed login attempts trigger no alert; 10 failed attempts trigger a lockout and alert. Reduces noise while catching genuine attacks.
Log integrityLogs must be tamper-evident and tamper-resistant. An attacker who compromises a system often deletes logs. Solutions: write-once media (WORM), remote log server (SIEM), cryptographic log chaining, append-only log services.

8. Identity Lifecycle Management

📖 Identity Lifecycle: The complete lifecycle of a user account from initial creation through modification to eventual deletion. All phases require controls to maintain least privilege, remove stale access, and ensure timely termination.
PhaseDescriptionKey Control
ProvisioningCreating a new account with appropriate access based on role/job function. Access should be based on role (RBAC) — not ad-hoc requests.Role-based provisioning workflow; manager approval; baseline role definition
ModificationUpdating access when role changes, promotions, transfers, additional responsibilities.Access must be reviewed at role change — old access removed, new access granted (not just added)
Privilege creepAccumulation of permissions over time through role changes without removing old permissions. User ends up with far more access than their current role requires.Access recertification (periodic review of all access); entitlement reviews
SuspensionTemporary disable of account (leave of absence, investigation, suspected compromise).Disable without deletion to preserve audit trail; immediate suspension on security concern
TerminationImmediate disable of all access when employee leaves. Critical control — departing employees (especially those leaving involuntarily) are high risk actors.HR-IT integration: account disable triggered on HR termination event; same-day as departure; disable BEFORE notifying if involuntary
Orphan accountsAccounts belonging to former employees or unused service accounts that still exist with active permissions. A primary attack vector for lateral movement.Regular account audits comparing HR records to active accounts; automated deprovisioning

9. Subjects and Objects

ConceptDefinitionExamples
SubjectAn active entity that requests access to a resource. Subjects initiate actions.User, process, service, program, application, device acting on behalf of a user
ObjectA passive entity that contains or receives information. Objects are acted upon.File, database record, printer, network share, API endpoint, memory segment, process (when being accessed by another)
Access rightA specific permitted operation that a subject may perform on an object. Examples: read, write, execute, delete, append, change permissions (own)User (subject) has read permission on file (object)
A subject can become an objectA running process (subject) can be inspected by a debugger (making it an object of the debugger process). Context determines role.Process A writes to Process B's memory — Process B is the object in that transaction

10. Access Control Principles

PrincipleDefinitionCISSP Relevance
Least privilegeGrant only the minimum permissions required to perform the job functionCore principle; applies to users, processes, services, and applications. Reduces blast radius of compromise.
Need-to-knowEven with clearance, access only to information needed for current dutiesEspecially relevant in government/classified contexts; complements least privilege for information access
Separation of duties (SoD)No one person can complete a critical transaction alonePrevents fraud (finance), prevents single-admin sabotage. Different from separation of privilege (requires two keys/conditions to be met, not two people)
Dual controlTwo or more individuals must cooperate simultaneously to perform an actionNuclear launch codes; safe deposit boxes; crypto key ceremonies; database decryption
Two-man ruleTwo authorized individuals must be present for sensitive operations (mutual oversight)Nuclear weapons handling; physical security for critical assets; backup tape retrieval
Job rotationPeriodically rotate people through positions; builds resilience and detects fraudExposes fraudulent activities (requires ongoing maintenance vs. periodic); also provides cross-training
Mandatory vacation / forced leaveForce employees to take leave; temporary replacement may discover ongoing fraudClassic banking control — embezzlement often requires constant presence to maintain the cover

11. Memory Tricks & Mnemonics

🧠 IAAA — "I Always Actively Audit":
Identification → Authentication → Authorization → Accountability
Each step depends on the previous. No accountability without authentication.
🧠 Authentication Factors — "Know, Have, Are":
Type 1 = Something you Know (password, PIN) — weakest, can be shared/stolen
Type 2 = Something you Have (token, smart card, phone) — can be lost
Type 3 = Something you Are (biometric) — can't be changed if compromised
MFA = two or more from DIFFERENT type categories
🧠 Identity Lifecycle — "Arrive, Change, Leave":
Provision on hire → Modify on role change (remove old, add new) → Suspend on leave → Terminate on departure
Danger zones: Privilege creep (no removal of old access) + Orphan accounts (termination not executed)
