Domain 1 — Security and Risk Management

15% of exam · ~22 questions

CIA Triad & Core Concepts

PropertyGoalControlsAttack (DAD)
ConfidentialityPrevent unauthorised disclosureEncryption, access control, classification, steganographyDisclosure — eavesdropping, data exfiltration
IntegrityPrevent unauthorised modificationHashing, digital signatures, HMAC, version control, input validationAlteration — MitM tampering, SQL injection
AvailabilityEnsure reliable accessRedundancy, clustering, backups, DDoS mitigation, BCP/DRPDestruction — DoS/DDoS, ransomware
Exam Tip: Non-repudiation, Authenticity, and Accountability extend CIA. Non-repudiation requires asymmetric crypto only. HMAC provides integrity + authentication but NOT non-repudiation (shared key = either party could have created it).
Ransomware trap: Ransomware violates all three CIA properties, but its primary goal is Availability (to extort) or Confidentiality (if data is exfiltrated).

Defense in Depth (Layered Security)

Multiple independent controls arranged in layers: Policies/Awareness → Physical → Perimeter → Network → Host → Application → Data. Control diversity across layers is essential — same technology at every layer = single failure defeats all.

Due Care vs. Due Diligence

Due Care — Doing the right thing (action). Patching known vulnerabilities, encrypting customer data. Failure = negligence.
Due Diligence — Verifying it's working (ongoing investigation). Running vulnerability scans to confirm patching; auditing encryption; reviewing logs.

Policy Hierarchy

Policy (mandatory, senior mgmt signed) → Standard (specific mandatory requirements) → Procedure (step-by-step instructions) → Guideline (recommended, optional). Acceptable Use Policy (AUP) must be signed before granting access.

(ISC)² Code of Professional Ethics — 4 Canons (in priority order)

  1. Protect society, the common good, necessary public trust and confidence, and the infrastructure
  2. Act honourably, honestly, justly, responsibly, and legally
  3. Provide diligent and competent service to principals
  4. Advance and protect the profession
Priority: Canon 1 (public/society) always beats Canon 4 (profession). When legal and ethical conflict — ethics wins. When employer asks you to do something unethical — refuse and report.

Risk Management & Analysis

Quantitative Risk Formulas (Must Memorise)

SLE = AV × EF
ALE = SLE × ARO
CBA = ALE(before) − ALE(after) − Annual Control Cost  [positive = justified]
ROSI = (ALE Reduction − Control Cost) / Control Cost × 100%
Availability = MTBF ÷ (MTBF + MTTR)
VariableMeaningNote
AVAsset Value (dollars)Total value including data, revenue generation, replacement cost
EFExposure Factor (0–1 decimal)% of AV lost per event: 50% = 0.50
SLESingle Loss Expectancy = AV × EFDollar loss per single event occurrence
AROAnnual Rate of OccurrenceExpected frequency per year; ARO = 0.1 = once per 10 years
ALEAnnualised Loss Expectancy = SLE × AROExpected annual dollar loss — justify control costs against this
MTBFMean Time Between FailuresHigher = more reliable; measures reliability
MTTRMean Time To RepairLower = better; measures recovery speed

Qualitative Analysis

Uses expert judgment: High/Medium/Low or 1–5 scales. Risk Matrix (Heat Map): Likelihood × Impact. Delphi Technique: anonymous iterative expert consensus — eliminates groupthink and dominant-person bias.

Supply Chain Risk

Business Continuity Planning (BCP) & Business Impact Analysis (BIA)

BCP: How to keep the business running during disruption — proactive and concurrent. Broader scope.
DRP: How to restore IT systems after a disaster — reactive and sequential. IT focus. BCP drives DRP.

Domain 2 — Asset Security

10% of exam · ~15 questions

Data Classification & Roles

Government Classification (Highest → Lowest)

  1. Top Secret — Exceptionally grave damage to national security
  2. Secret — Serious damage to national security
  3. Confidential — Damage to national security
  4. Unclassified — No damage

SCI (Sensitive Compartmented Information): NOT a classification level — a control system layered on top of a TS clearance; requires clearance + formal compartment access approval + need-to-know. SAP (Special Access Programme): Similarly a restricted-access control program requiring code-word authorisation; both SCI and SAP restrict access within the TS classification, not above it.

Commercial Classification (Highest → Lowest)

  1. Confidential (or Restricted) — Trade secrets, M&A data
  2. Private — Employee HR records, medical
  3. Sensitive — Internal financial data, strategic plans
  4. Public — Press releases, marketing materials
Exam Trap: Government "Confidential" ≠ Corporate "Confidential". Government Confidential is the lowest classified level. Corporate Confidential is the highest commercial level.

Classification Principles

  • Classification is determined by the Data Owner (business executive, not IT)
  • Declassification requires Data Owner approval — not custodian's decision
  • Data should be labelled/marked at the highest classification of any element it contains
  • Aggregation: Combining unclassified data items can create a higher classification

Data Loss Prevention (DLP)

  • Network DLP: Monitors traffic at gateway; blocks sensitive data leaving network
  • Endpoint DLP: Agent on workstation; controls USB, printing, screenshots, copy-paste
  • Cloud/CASB DLP: Cloud Access Security Broker; enforces policy for cloud app usage; four modes: API, proxy, agent, out-of-band
IRM/DRM (Information Rights Management / Digital Rights Management): Persistent protection embedded in the content itself — travels with the document; enforces who can read/edit/print/forward regardless of where the file resides.

SSD Critical Difference: Wear levelling causes overwrites to go to different blocks — simple overwriting does NOT erase all data from SSDs. Cloud data destruction: You cannot physically destroy cloud media yourself. The primary customer-controlled option is cryptographic erasure — ensure all data was encrypted with customer-managed keys, then destroy the keys. Cloud providers handle physical media disposal through their own sanitisation processes.

Digital Signature Process: (1) Hash the message with SHA-256 → (2) Encrypt the hash with sender's PRIVATE key → (3) Recipient decrypts hash with sender's PUBLIC key → (4) Compare to hash of received message. Provides: Integrity + Authenticity + Non-repudiation. Encrypted with PUBLIC key = confidentiality (only private key holder can decrypt). Encrypted with PRIVATE key = signature (anyone with public key can verify).

Password Hashing (Secure Storage)

  • Salting: Unique random value per password prepended before hashing → prevents rainbow table attacks (same password = different hash for each user)
  • Key Stretching: Deliberately slow hashing algorithms → bcrypt, Argon2, scrypt, PBKDF2 — makes brute-force computationally expensive
  • NEVER store passwords as plain MD5 or SHA-1 (even salted) — use bcrypt or Argon2 minimum

Quantum Computing Threat

  • Shor's Algorithm: Breaks RSA, ECC, DH (asymmetric) — renders current public-key crypto obsolete
  • Grover's Algorithm: Halves effective key length of symmetric ciphers → AES-256 effectively becomes AES-128
  • Post-Quantum Cryptography (PQC): NIST selected standards in 2024: CRYSTALS-Kyber (key encapsulation), CRYSTALS-Dilithium (signatures); lattice-based
  • Harvest Now, Decrypt Later: Adversaries storing encrypted traffic today to decrypt once quantum computers available — classified/long-lived data must be PQC-protected now

Key Management Lifecycle

Generation → Distribution → Storage → Use → Rotation/Renewal → Revocation → Destruction. Keys must NEVER leave HSMs in plaintext. Key custodians use M-of-N split knowledge to reconstruct keys — no single person has full key.

Password Security Controls

  • Minimum length (12–16+ characters) = most effective single control for password strength
  • Password history (e.g., remember last 12): prevents reuse of recent passwords
  • HOTP: HMAC-based OTP — counter-based; token and server sync counter; token does not expire until used
  • TOTP: Time-based OTP — 30–60 second window; Google Authenticator, Microsoft Authenticator; requires time synchronisation (NTP)
  • Synchronous tokens: Time-based (TOTP) — token and server must have synchronised clocks
  • Asynchronous tokens: Challenge-response — server sends challenge; user enters into token; token generates response
  • FIDO2/WebAuthn: Passwordless; public key cryptography; private key on device/security key; phishing-resistant (domain binding); strongest MFA approach

Log Management

  • Log sources: OS, applications, firewalls, IDS/IPS, DNS, DHCP, authentication systems, cloud APIs
  • Centralise logs on a separate system — logs on same system as attack target = attacker can tamper
  • Log integrity: Write-once media (WORM) or forward-only syslog to secured SIEM; cryptographic hashing for tamper evidence
  • Chain of custody: Maintained throughout log collection and preservation; required for legal admissibility
  • SIEM: Aggregates, normalises, correlates logs from multiple sources; generates alerts on correlated events; enables retrospective investigation
  • Log retention: SOX = 7 years; HIPAA = 6 years; PCI-DSS audit logs = 1 year (90 days immediately accessible)

"Best" or "most important" first stepUsually: Risk assessment / BIA first; then policy; then technical controls. Or for IR: Containment, then Eradication, then Recovery.
Incident response: what do you do first?Contain (isolate) → Preserve evidence → Eradicate → Recover → Document. In ransomware: ISOLATE first.
Vendor or third-party risk?SOC 2 Type II + Right-to-audit clause + SBOM + contractual security requirements + regular reviews.
New system or project: what first?BIA (Business Impact Analysis) or Risk Assessment first. Then requirements. Then design. Security cannot be designed without knowing the business impact.
Data classification questionData Owner (executive) decides; Custodian (IT) implements. Never IT-decides-classification.
"Which is the MOST effective control?"Administrative controls (policy, training) address root causes. Technical controls enforce policy. Physical controls protect assets. Prioritise root cause address over symptoms.
Someone discovers something suspiciousPreserve evidence first. Notify supervisor/CSIRT. Do NOT confront the suspect. Do NOT proceed without authority. Consult legal before law enforcement.

Common Exam Traps (Most Frequently Missed)

TrapCorrect Understanding
HMAC = non-repudiation?NO — HMAC uses shared secret; either party could have created it. Non-repudiation requires digital signature with PRIVATE key.
OAuth 2.0 = authentication?NO — OAuth 2.0 is AUTHORISATION. OIDC adds authentication. "Sign in with Google" = OIDC, built on OAuth.
IDS blocks attacks?NO — IDS only DETECTS and alerts. IPS is INLINE and BLOCKS.
Degaussing works on SSDs?NO — degaussing only affects magnetic media. Use cryptographic erasure or physical destruction for SSDs/flash.
Encryption at rest prevents SQL injection?NO — TDE encrypts the data file; data is decrypted at query time. SQL injection runs inside the database after decryption.
Data Owner = IT person?NO — Data Owner is a business executive (VP, Director). IT is the Custodian who implements controls.
All security controls must pass CBA?NO — regulatory and compliance requirements must be implemented regardless of CBA. Legal obligation overrides cost-benefit.
WPA2 is secure with PSK?PARTIALLY — WPA2-PSK is vulnerable to offline dictionary attacks (captured 4-way handshake; weak passphrases crackable offline). KRACK (2017) is a separate key reinstallation attack against the handshake regardless of passphrase strength. Use WPA3-SAE or WPA2-Enterprise (802.1X).
Vulnerability assessment = penetration test?NO — VA identifies; pen test exploits. VA = comprehensive list of vulnerabilities. Pen test = demonstrate impact by exploiting selected vulnerabilities.
SOC 2 Type I = strong assurance?NO — Type I only shows "controls are designed." Type II shows "controls operated effectively" over 6+ months. Always prefer Type II.
Risk transfer eliminates legal liability?NO — Cyber insurance transfers financial risk only. Legal liability (GDPR fines, negligence suits) stays with the organisation.
Compliance = security?NO — compliance sets a minimum baseline. You can be compliant and still be breached. Security goes beyond compliance.
