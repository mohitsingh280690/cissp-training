1. Domain 5 – Identity & Access Management (13%)

Authentication Factors

FactorTypeExamples
Type 1 — Something You KnowKnowledgePassword, PIN, passphrase, security question
Type 2 — Something You HavePossessionSmart card, token, phone (OTP), PIV card
Type 3 — Something You AreBiometricFingerprint, retina, iris, voice, facial recognition
Type 4 — Somewhere You AreLocationIP address, GPS, geolocation
Type 5 — Something You DoBehaviorTyping rhythm, gait analysis, signature dynamics
MFA requires factors from DIFFERENT categories. Two passwords = NOT MFA (both Type 1). Password + smart card = MFA (Type 1 + Type 2). MFA is ALWAYS the best answer for improving authentication security.

Biometric Error Rates

MetricDefinitionExam Focus
FRR (Type I)False Rejection Rate — rejects authorized userInconvenience (too sensitive)
FAR (Type II)False Acceptance Rate — accepts unauthorized userSecurity risk (too lenient)
CER/EERCrossover Error Rate — where FRR = FARLower CER = more accurate system; BEST comparison metric
Type II (FAR) is always the MORE DANGEROUS error — it lets unauthorized users in. Type I (FRR) is annoying but not a security breach. CER is the gold standard for comparing biometric systems.

Access Control Models

ModelWho DecidesBasisExample
DACData ownerIdentity of requesterNTFS permissions, Unix file perms
MACSystem/policyLabels/clearancesMilitary classified systems (Top Secret, Secret)
RBACAdministratorJob role/functionDoctor role, Nurse role → different access
ABACPolicy engineAttributes (user, resource, environment, action)IF dept=finance AND time=business_hours → allow
Rule-basedRules/ACLsPredefined rulesFirewall rules (allow/deny by IP/port)
DAC = most common, least restrictive. MAC = most restrictive, most secure. RBAC = best for large organizations (role-based). ABAC = most flexible, granular (attribute-based policies).

SSO Technologies

ProtocolUseKey Fact
KerberosEnterprise SSO (AD)Ticket-based; port 88; TGT from KDC; mutual auth; vulnerable to pass-the-ticket, golden ticket
SAMLWeb SSO (enterprise)XML-based; IdP sends assertions to SP; browser redirect flow
OAuth 2.0Authorization (APIs)NOT authentication; grants access tokens; delegated authorization
OpenID ConnectAuthenticationBuilt ON TOP of OAuth 2.0; adds ID token (JWT); "Login with Google"
RADIUSNetwork AAAEncrypts password only; UDP; combined auth + authz; 802.1X backend
TACACS+Network AAAEncrypts ENTIRE payload; TCP; separates AAA functions; Cisco preferred
RADIUS vs TACACS+: If the question asks about encrypting the full payload or separating AAA → TACACS+. If asking about 802.1X backend → RADIUS. TACACS+ = TCP + full encryption + separate AAA. RADIUS = UDP + password-only encryption + combined auth/authz.

Identity Lifecycle

Provisioning → Review → Modification → Deprovisioning

  • Provisioning: Create account, assign minimum permissions (least privilege)
  • Review: Periodic access reviews/recertification (manager attestation)
  • Modification: Role changes; REMOVE old access (prevent privilege creep/aggregation)
  • Deprovisioning: Disable IMMEDIATELY on termination; delete after retention period

4. Authentication & Authorization Master Table

ConceptAuthenticationAuthorization
Question answered"Who are you?""What can you do?"
OccursFirstAfter authentication

OAuth 2.0 = AUTHORIZATION only. It's the most commonly confused protocol. OAuth grants access tokens for API authorization. OpenID Connect (built on OAuth) adds AUTHENTICATION via ID tokens.
