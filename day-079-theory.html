3. PAM Concepts & Core Principles

📖 PAM (Privileged Access Management): A comprehensive security discipline and set of technologies for managing, controlling, monitoring, and auditing the access and activities of accounts with elevated rights — including system administrators, network engineers, DBA roles, cloud account root users, and service accounts with elevated permissions.
PAM PrincipleDescription
Least privilege for privileged accountsEven privileged users only get the minimum elevated access needed for a specific task. A Windows sysadmin does not need Domain Admin rights for basic file-server management tasks. Privilege scope is scoped to the task, not the person.
No standing privilegeIdeal PAM state: no accounts have permanently elevated privileges. Access is elevated only when needed (JIT) and automatically revoked when the task is complete. Eliminates the standing target for attackers.
Separation of privileged & standard accountsAdmins have two accounts: a standard account for email, web browsing, general tasks; and a separate privileged account for admin work. Privileged account is not used for daily browsing (reduces phishing exposure of admin credentials).
Strong authentication for all privileged accessPrivileged accounts require MFA at every login, even from trusted networks. Phishing-resistant MFA (FIDO2, hardware key) preferred for highest-privilege accounts.
Full session audit trailAll privileged sessions are recorded (keystrokes, screen, commands). Any admin action is attributable, reviewable, and usable as evidence. Admins know they are being monitored — deterrence value as well as forensic value.

5. Just-in-Time (JIT) Access

📖 Just-in-Time (JIT) Access: Privileged access is granted dynamically on demand for a limited time window (e.g., 2 hours, 1 day) to complete a specific task. After the window expires, the elevated access is automatically and completely revoked. No standing privileged access exists between task windows.
JIT Access Flow: ──────────────────────────────────────────────────────────────────── Admin needs to patch server PROD-SQL-01 │ ▼ [1] Admin submits JIT request: • Target: PROD-SQL-01 • Access level: Local Admin • Duration: 4 hours • Justification: "Monthly patch window KB5027231" │ ▼ [2] Approval workflow: • Auto-approved for standard patch tasks (pre-authorized) • Requires manager approval for production Domain Admin (Large orgs use automated approval for low-risk, human approval for high-risk) │ ▼ [3] PAM system provisions: • Adds admin account to "Administrators" group on PROD-SQL-01 • Issues temporary vault credential / sets up session proxy • Timer starts: T+0 │ ▼ [4] Admin works on PROD-SQL-01 via bastion host (session recorded) │ ▼ [5] T+4 hours: PAM automatically removes admin from "Administrators" group • Credential rotated • Session recording archived • Task log submitted for review ──────────────────────────────────────────────────────────────────── Result: Admin had privilege ONLY WHEN NEEDED. Standing privilege = zero.
JIT BenefitDetail
Reduces attack surfaceAttacker who compromises admin credential during off-hours finds no elevated group memberships — nothing to leverage. Privilege only exists during approved task window.
Enforces least privilegeJIT is the operational implementation of least privilege over time — not just "minimum scope" but "minimum duration."
Creates audit trailEvery JIT request has justification, approval, and time bounds — complete accountability for all privilege use.

6. Privilege Bracketing

📖 Privilege Bracketing: A programming and operational technique where elevated privileges are acquired immediately before the specific operation that requires them and immediately released/dropped as soon as that operation completes. Privileges are held only for the minimum duration required — "bracketed" around the exact operation.
Without Privilege Bracketing (Dangerous): ────────────────────────────────────────────── Process starts with root/admin privileges [Reads config file] ← privileged (ok) [Parses user input] ← privileged (dangerous — if parsing is exploited, attacker has root) [Writes log file] ← privileged (dangerous — unnecessary privilege) [Opens network conn]← privileged (dangerous — any exploit has full root access) Process exits with root With Privilege Bracketing (Secure): ────────────────────────────────────────────── Process starts as low-privilege user [Parses user input] ← low privilege (exploit = low damage) acquire admin → ← elevate ONLY for the operation that needs it [Reads /etc/shadow] ← privileged (needed) drop admin → ← IMMEDIATELY drop elevated privilege [Continues as low privilege]← parsing, networking back as low-privilege acquire admin → [Writes to protected log] ← privileged (needed) drop admin → Process exits as low privilege
💡 Privilege Bracketing in Context: Privilege bracketing appears both as an operational PAM concept (JIT access windows are a macro-level form of bracketing) and as a software security principle (programs acquiring/releasing OS privileges tightly around operations). CISSP may test either context.

7. Bastion Hosts & Jump Servers

📖 Bastion Host / Jump Server: A hardened, highly monitored gateway server that is the ONLY allowed path from a less-trusted network zone to privileged administrative access in a more-trusted zone (e.g., production environment, OT network). All admin sessions are proxied through the bastion — direct admin access from end-user workstations is blocked.
Network Architecture with Bastion Host: ────────────────────────────────────────────────────────────────────────────────────── [Admin Workstation] [Production Servers] Corporate Network Isolated VLAN / DMZ │ │ │ SSH/RDP to bastion (with MFA, vault credential) │ ▼ │ ┌─────────────────────────────────────────────────────────────┤ │ BASTION HOST / JUMP SERVER │ │ • Hardened OS (minimal attack surface) │ │ • All sessions recorded (keystrokes, screen, commands) │ │ • Session proxy: admin never gets direct credential │ │ • Integrates with PAM vault for credential injection │ │ • No internet access; no email; no general purpose apps │ │ • MFA required at bastion login │ └─────────────────────────────────────────────────────────────┤ │ │ │ Port-forwarded session to target server │ ▼ ▼ [Admin access to PROD-SQL-01, PROD-WEB-02, Network Switches] Firewall rule: DENY all direct SSH/RDP to production EXCEPT from bastion host IP. Admin workstations cannot directly reach production systems. ──────────────────────────────────────────────────────────────────────────────────────
Bastion Host FeatureSecurity Value
Single point of entry for admin accessConcentrates all privileged access activity at one monitored gateway. Instead of monitoring thousands of direct SSH connections across the environment, monitor one bastion. Reduces attack surface dramatically.
Credential injection (never revealed)PAM vault provides credentials directly to the bastion session proxy. The admin never sees the actual password of the target system — cannot exfiltrate it, cannot reuse it outside the PAM system.
Network segmentation enforcementFirewall rules enforce that production systems are UNREACHABLE from user workstations except via bastion. Even if admin credentials are compromised, attacker must also compromise the bastion to use them.
Hardened attack surfaceBastion runs only the PAM/jump software. No email, no browser, no office apps. Minimal OS with all unnecessary services disabled. Regularly patched. Limited blast radius if compromised.

8. Session Recording & Monitoring

AspectDetail
What is recordedALL privileged sessions end-to-end: keystrokes (all commands typed), screen video (full visual stream), session metadata (who, when, what target, duration), file transfers, clipboard content. Modern PAM solutions record at protocol level (SSH protocol parsing, RDP video stream).
Immutable audit trailSession recordings stored in tamper-evident format, ideally in write-once/append-only storage (WORM) separate from the systems being administered. Even a compromised domain admin should not be able to delete their own session recording.
Real-time monitoring & alertingPAM platforms analyze session activity in real-time: alert on high-risk commands (rm -rf, DROP TABLE, net user /add, powershell encoded commands), unusual transfer volumes, access outside approved JIT window, connections to unexpected systems.
User behavior analytics (UBA)ML-based analysis of privileged session activity patterns. Baseline normal behavior per admin — alert on deviations (unusual working hours, commands never seen before, lateral movement patterns, access to systems outside admin's scope).
Deterrence valueKnown monitoring of sessions changes insider threat behavior — the "observer effect" in security. Employees and contractors who know their sessions are fully recorded are deterred from policy violations and malicious actions.
Detective, not preventiveSession recording is a detective control — it records what happened for forensic analysis and compliance demonstration. It does NOT prevent a privileged admin from doing something malicious. Must be combined with preventive controls (JIT, least privilege, approval workflows).

9. Credential Vaulting

📖 Credential Vaulting: Storing privileged account credentials (passwords, SSH keys, API tokens, certificates) in a centralized, encrypted, access-controlled repository (the vault). Users access credentials via the vault with MFA and full audit logging — the vault provisions credentials to sessions without the user ever seeing them directly.
Vaulting FeatureSecurity Benefit
Centralized encrypted storageCredentials are not stored in spreadsheets, sticky notes, config files, or emails. Encrypted at rest with hardware-backed keys (HSM integration in enterprise deployments). Single authoritative source of truth for all privileged credentials.
Automated credential rotationVault automatically changes privileged account passwords on a schedule (e.g., every 24 hours, after each use, or after JIT session expiry). Stolen credentials have a short validity window. Service account passwords changed without human knowing the new password.
Zero-knowledge credential injectionFor session-proxied access: vault provides credential directly to the session connection — admin connects via vault, vault authenticates to target on their behalf. The admin never sees the actual password. Prevents password exfiltration, shoulder surfing, clipboard theft.
Access loggingEvery credential retrieval is logged: who accessed, which credential, when, from where. Enables complete audit trail of privileged account usage. Alerts on unusual access patterns (credential retrieved outside business hours, credentials retrieved for systems outside admin's normal scope).
MFA required for vault accessAccessing the vault itself requires strong authentication. Attacker who steals admin's domain password cannot access vault credentials without also compromising the MFA factor.

10. Emergency Break-Glass Procedures

📖 Break-Glass (Emergency Access): A predefined process for accessing emergency privileged credentials when normal authentication systems are unavailable (e.g., PAM vault down, MFA infrastructure failure, identity provider outage). Named after the "break glass in emergency" concept — the seals on the emergency access are broken only in genuine emergencies.
Break-Glass ElementDetail
The problem it solvesIf the PAM vault is down in a disaster scenario, how do admins access systems to fix the disaster? If Azure AD is the IDP and it is experiencing an outage, how do admins access cloud resources? Break-glass accounts provide a bypass path for genuine emergencies.
Credential storageBreak-glass credentials stored in a PHYSICAL sealed envelope (literal break-glass in a physical safe) or in a separate offline vault that is not dependent on the primary PAM system or IDP that may be failing. For cloud: separate local administrator account with credentials stored offline.
Dual-custody (two-person integrity)Two separate people must cooperate to access break-glass credentials. CISO holds half; IT Director holds half. Neither can access alone — prevents single insider from using break-glass inappropriately. Physical seal provides evidence of whether access occurred.
Automatic alerting on ANY useAny use of break-glass credentials must immediately and automatically alert senior security leadership (CISO, CTO, Security Operations). No use should go undetected or unreported. Alert must be sent to a channel that does NOT depend on the system being recovered.
Mandatory post-use reviewEvery break-glass event requires a formal review: verify usage was justified, document what was done, rotate credentials immediately after use. If unauthorized use detected, treat as a security incident. No break-glass use is "routine."
TestingBreak-glass procedures should be tested periodically (e.g., annually) to confirm they work — without actually using real production credentials. Tabletop exercise minimum; full drill if possible.
⚠️ Break-Glass Misuse Risk: Break-glass accounts are prime insider-threat attack vectors. If credentials are not properly sealed, rotated, and monitored, a malicious insider can use break-glass accounts to gain unlimited access while appearing to operate within a sanctioned emergency procedure. Every control must apply: dual-custody + alerting + review + rotation after use.

12. Memory Tricks & Mnemonics

🧠 PAM Core — "No Standing Privilege":
The goal of PAM = ELIMINATE standing privileged accounts
Zero standing privilege → attackers find nothing to steal when they compromise credentials
JIT = implement this operationally; Credential Rotation = implement this for service accounts
🧠 JIT = "Ticket to the Show":
Think of JIT access like a ticket: it grants specific access for a specific show (task), at a specific time; after the show (task is done), the ticket expires.
Without PAM: permanent VIP backstage pass = always elevated risk.
With JIT: day-pass for 4 hours only.
🧠 Bastion = "The Only Door":
The bastion host is the ONLY door into the privileged zone. All admin traffic must go through this door. The door has a guard (MFA), a camera (session recording), and an inspector (PAM vault injection).
Firewall blocks all OTHER doors to production — only the bastion door is open.
🧠 Break-Glass — 3 Rules:
1. Two people to open it (dual custody)
2. Alert immediately on any use (automatic)
3. Rotate credentials immediately after use
Break-glass that is not monitored = secret master key with no audit trail = worst possible privileged account design.
🧠 Session Recording = Detective, NOT Preventive:
Recording what admin did DOESN'T stop a malicious admin. It creates evidence and deters.
Pair with JIT (preventive) + vault injection (preventive) + bastion gateway (preventive) for defense-in-depth.
