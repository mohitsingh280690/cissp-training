2. Real-World Organizational Relevance

IssueContext
DAC is what you use every dayWindows NTFS, Linux file permissions, SharePoint folder sharing — all are DAC. You (the owner) decide who can access your files. Flexible and user-friendly but dangerous: you can accidentally grant overly broad access, and malware running as you can access everything you can access.
MAC is the government standardAll US government classified information handling (Top Secret, Secret, Confidential) is governed by MAC principles. Personnel must have both the clearance AND need-to-know. Systems enforce these rules — users cannot override them regardless of ownership.
MAC in commercial contextsSELinux (server hardening), AppArmor (Ubuntu), INTEGRITY OS (military avionics), Multics (historical MAC OS). MAC is used wherever system integrity and confidentiality requirements exceed what user discretion can ensure.
Trojan Horse risk in DACClassic DAC weakness: malware running with user's privileges inherits all the user's permissions. A user with broad discretionary access accidentally runs malicious code — the malicious code can access, exfiltrate, or modify everything the user could. MAC prevents this by enforcing labels regardless of who (or what) is requesting access.

4. Discretionary Access Control (DAC)

📖 DAC – Discretionary Access Control: An access control model where the owner (creator) of a resource has discretion over who can access it. The owner sets permissions at their own discretion. Permission grants can be passed to others (access rights can be delegated).
CharacteristicDetail
Identity-basedAccess decisions based on who is requesting (user identity or group membership). Access Control List (ACL) maps users/groups to permitted operations on each resource.
Owner controls accessThe creator of a file/resource automatically becomes the owner. Owner can grant or revoke permissions for other users. Owner can delegate ownership.
Permissions can be delegatedA user granted access can pass that access to others (in the most permissive DAC implementation). This is the "discretionary" property — discretion is in the hands of the owner.
ImplementationAccess Control Lists (ACLs) per object. Each object has a list of (subject, permission) entries. Examples: NTFS ACLs, UNIX file permissions (user/group/other), AWS S3 bucket policies.
Primary weakness: Trojan HorseA malicious process running with the user's identity inherits all the user's permissions. The access control checks the requester's identity — the Trojan horse IS the user from the system's perspective.
Confinement problemIn DAC, there is no mechanism to prevent a user from passing copies of data to unauthorized users (data "leaks" downstream through authorized copies).

DAC Access Control List (ACL) Example

DAC — File Access Control List ───────────────────────────────────── file: /projects/budget.xlsx owner: alice Subject Permission ───────────── ───────── alice Read, Write, Delete, Change Permissions bob Read finance-group Read, Write marketing-group (none — access denied) everyone (none — implicit deny) ───────────────────────────────────── alice (as owner) can modify this ACL at will. She can grant bob Write, or remove finance-group entirely.

5. Mandatory Access Control (MAC)

📖 MAC – Mandatory Access Control: An access control model where the operating system or security policy (not the data owner) enforces access based on security labels. Users cannot grant or modify access permissions — the system makes all access decisions based on the classification label of the resource and the clearance level of the subject.
CharacteristicDetail
Label-basedEvery subject (user, process) and object (file, resource) is assigned a security label. The label has two components: classification level (hierarchical) + compartments (non-hierarchical categories).
System enforces policyThe operating system/security kernel enforces the access policy. Owners CANNOT override it — even the owner of a file cannot grant access to someone with insufficient clearance.
No discretion by usersUsers cannot change security labels on objects. Cannot grant access to others beyond what policy allows. Cannot declassify data.
Clearance + Need-to-knowUsers must have BOTH the appropriate security clearance (level ≥ object classification) AND need-to-know (compartment access) to access an object under MAC.
Prevents Trojan HorseEven if a Trojan Horse runs with user credentials, it cannot read data above the user's clearance (MAC check) or write data below the user's level without declassification (prevents data leakage downward).
EnvironmentsGovernment/military classified systems, healthcare systems with strict privacy requirements, high-assurance financial systems

6. Sensitivity Labels and Compartments

ComponentDefinitionExample
Classification level (hierarchical)Defines the sensitivity tier. Higher level requires higher clearance to access. Ordered: Top Secret > Secret > Confidential > Unclassified (US Government scale)A document labeled SECRET can only be accessed by users with Secret or Top Secret clearance.
Compartments (non-hierarchical)Additional categories or compartments beyond the classification level. Even with correct clearance level, a user needs explicit access to the relevant compartment (need-to-know).SECRET/NOFORN (No Foreign Nationals); TOP SECRET/SCI/HUMINT — TS clearance + SCI access + HUMINT compartment all required
Combined labelA subject must dominate the object's label to access it. "Dominates" = classification level ≥ object's level AND subject has access to ALL compartments of the object.User with label [Secret, {SIGINT, HUMINT}] can access file labeled [Secret, {HUMINT}] but NOT [Secret, {SIGINT, TALENT}] (missing TALENT).
MAC Label Structure ───────────────────────────────────────────────────── Label = [Classification Level] / [Compartment(s)] Examples: TOP SECRET / SCI (high level, one compartment) SECRET / NOFORN (mid level, NOFORN = no foreign national access) SECRET / HUMINT, SIGINT (mid level, two compartments) UNCLASSIFIED (lowest level, no compartments) Dominance rule: Subject S dominates Object O iff: S.level >= O.level AND S.compartments ⊇ O.compartments ─────────────────────────────────────────────────────

7. MAC Security Models (Bell-LaPadula, Biba)

Bell-LaPadula Model (Confidentiality)

📖 Bell-LaPadula: A formal MAC security model focused on protecting confidentiality. Prevents unauthorized disclosure of classified information. Government/military origin (developed 1973-1976 for USAF/MITRE).
PropertyRulePurpose
Simple Security Property (ss-property)"No Read Up" — Subject cannot READ an object at a higher classification level (e.g., a Secret-cleared user cannot read a Top Secret document)Prevents unauthorized READ of sensitive data (confidentiality)
Star Property (*-property)"No Write Down" — Subject cannot WRITE to an object at a lower classification level (e.g., a Top Secret user cannot write to a Secret file — prevents data leakage)Prevents declassification/downgrading of sensitive data through write operations
Strong Tranquility PropertySecurity labels never change during operationPrevents dynamic label manipulation attacks

Biba Model (Integrity)

📖 Biba: A formal MAC security model focused on protecting integrity. The inverse twin of Bell-LaPadula — protects against unauthorized modification. Important for financial data, software integrity, audit logs.
PropertyRulePurpose
Simple Integrity Property"No Read Down" — Subject cannot READ from a lower integrity level (prevents contamination of high-integrity processes with low-integrity data)Prevents low-integrity data from corrupting high-integrity processes
Star Integrity Property (*-property)"No Write Up" — Subject cannot WRITE to a higher integrity level (prevents low-integrity data from being injected into high-integrity sources)Prevents low-integrity data from polluting trusted data stores
💡 Exam Tip — Bell-LaPadula vs. Biba Summary:
ModelProtectsNo ReadNo Write
Bell-LaPadulaConfidentialityNo Read UpNo Write Down
BibaIntegrityNo Read DownNo Write Up
Memory: Bell-LaPadula = B = Blue (classified info is sky-high — can't read UP to the stars)
Biba = "integrity matters from above" — can't write UP to contaminate trusted sources

8. Lattice-Based Access Control

📖 Lattice-Based Access Control: A mathematical framework that formalizes MAC using a lattice structure. A lattice is a partially ordered set where every pair of elements has a least upper bound (join) and greatest lower bound (meet). Security labels are organized into a lattice, and access decisions are based on dominance relationships in the lattice.
Security Label Lattice (simplified) ───────────────────────────────────────────────────── TOP SECRET │ SECRET / \ CONFIDENTIAL SECRET/NOFORN \ / UNCLASSIFIED Dominance: A label L1 "dominates" L2 if L1 ≥ L2 in the lattice. Subject's label must dominate Object's label to allow READ (BLP) or WRITE (Biba). ─────────────────────────────────────────────────────
ConceptMeaning
Upper bound (join)Least upper bound of two labels = the minimum label that dominates both. Used to determine the classification of a document created from two sources.
Lower bound (meet)Greatest lower bound = maximum label dominated by both. The "safe" floor for shared operations.
BLP implementationBell-LaPadula is a lattice-based model. Access allowed when subject's lattice position dominates the object's position for reads.

9. Rule-Based Access Control

📖 Rule-Based Access Control (RuBAC): Access control decisions are made based on a set of rules or conditions defined by the administrator — not by the owner, not by label dominance. Rules define conditions under which access is granted or denied. Often combined with other models.
CharacteristicDetail
Rule-driven, not role-drivenRules can be time-based, location-based, network-based, or based on any attribute. Example: "Employees may access payroll system only Monday–Friday, 8am–6pm, from corporate network."
Firewall ACLs are rule-basedClassic example: network firewall access control lists. "Allow TCP 443 from any to 10.1.1.5/32; deny all other traffic." Rules applied in order; first match wins.
Not to be confused with RBACRBAC = Role-Based Access Control (users assigned to roles, roles have permissions — Day 76). Rule-based = access decisions per rules/conditions. "Rule" vs. "Role" — different concepts despite similar names.
Often layered with RBACMany systems combine: RBAC grants baseline access by role; rule-based conditions add further restrictions (time, location, device state). Together = context-aware, adaptive access control.

10. SELinux – MAC in Practice

📖 SELinux (Security-Enhanced Linux): A Linux kernel security module that implements MAC using a mandatory access control policy. Every process and file has a security context (label). Policies define what each labeled process type may do with each labeled resource. Default-deny: if not explicitly permitted, access is denied.
FeatureDetail
Security contextEvery process and file has a label: user:role:type:level (e.g., system_u:system_r:httpd_t:s0). The type component is most important — determines what the process CAN do.
Type Enforcement (TE)Primary policy mechanism. Defines which process types may perform which operations on which file types. Apache (httpd_t) can read web content (httpd_sys_content_t) but NOT read /etc/shadow (shadow_t).
MCS (Multi-Category Security)Extension for MAC with sensitivity labels. Used in containers (each container gets a unique MCS label) to prevent cross-container access even when running as the same Linux uid.
Policy modesEnforcing (full MAC policy active), Permissive (log violations but don't block — used for policy development/testing), Disabled (SELinux off)
Security benefitIf Apache is compromised (remote code execution), the attacker runs as httpd_t — SELinux confines the attack to only what httpd_t is permitted to do. Cannot read /etc/passwd, cannot write to system binaries, cannot bind privileged ports.

11. Memory Tricks & Mnemonics

🧠 DAC vs. MAC — "Owner vs. System":
DAC = Discretionary = Data owner decides = flexible = Trojan Horse weakness
MAC = Mandatory = Must follow policy = system enforces = rigid = highest security
"D" for "Discretion" — YOU choose. "M" for "Mandatory" — you have NO choice.
🧠 Bell-LaPadula — "No Read Up, No Write Down" (BLP = Blue sky = confidentiality):
Can't read up (secret agent can't read TS documents)
Can't write down (TS agent can't post TS info on the unclassified bulletin board)
BLP protects reading classified info UP; prevents writing it DOWN.
🧠 Biba — "No Read Down, No Write Up" (Biba = integrity, inverted BLP):
Can't read down (trusted process can't read untrusted data — garbage in, garbage out)
Can't write up (untrusted input can't corrupt trusted records)
Biba protects integrity — untrusted can't poison trusted (up), trusted can't be poisoned by reading down).
🧠 Rule-Based ≠ Role-Based:
Rule-based = conditions/logic (firewall ACL, "access only on weekdays")
Role-based = job functions (RBAC, "HR role can access HR system") — Day 76
When exam says "firewall rules" = rule-based. When exam says "job function permissions" = RBAC.
