2. Real-World Organizational Relevance

Threat modeling is most valuable when done early in the Software Development Lifecycle (SDLC), during the design phase. Microsoft's "Security Development Lifecycle" (SDL) mandates threat modeling before code is written. Fixing security flaws at design stage costs approximately 6× less than fixing them in production.

MITRE ATT&CK has become a de facto industry standard for describing threat actor behaviour. SOC teams, threat intelligence analysts, and red teams all use ATT&CK to communicate about attack techniques using a common vocabulary.

3. What Is Threat Modeling?

Threat Modeling: A structured, systematic process of identifying potential threats to a system, understanding their likelihood and impact, and defining countermeasures — typically performed during system design rather than after deployment.

Goals of Threat Modeling

  • Identify threats early when they're cheapest to mitigate
  • Understand the system from an attacker's perspective
  • Prioritise security investments based on real threats
  • Document security design decisions for future reference
  • Satisfy compliance requirements (GDPR Art. 25 Privacy by Design, NIST SSDF)

High-Level Process

  1. Scope the system: What are we protecting? What's in scope?
  2. Decompose the application: Create Data Flow Diagrams — identify data stores, data flows, processes, external entities, and trust boundaries
  3. Identify threats: Apply threat classification framework (STRIDE, PASTA, etc.)
  4. Rate and prioritise threats: Score by impact and likelihood
  5. Define mitigations: Map controls to each threat
  6. Validate: Review and test that mitigations are effective

4. STRIDE Framework

STRIDE: A threat classification framework developed by Microsoft (Loren Kohnfelder & Praerit Garg, 1999) that categorises threats into six types, each violating a specific security property. Used to systematically identify threats to software components, services, and data flows.
CategorySecurity Property ViolatedExample ThreatExample Countermeasure
SpoofingAuthenticationAttacker impersonates a legitimate user or serviceMulti-factor authentication, digital certificates, mutual TLS
TamperingIntegrityAttacker modifies data in transit or at restDigital signatures, HMAC, checksums, database write controls
RepudiationNon-repudiationUser denies performing an action; no audit trail existsAudit logs, digital signatures, tamper-proof logging (WORM)
Information DisclosureConfidentialitySensitive data exposed to unauthorised partiesEncryption, access controls, data masking, TLS in transit
Denial of ServiceAvailabilityAttacker overwhelms system; legitimate users denied accessRate limiting, load balancing, DDoS protection, failover
Elevation of PrivilegeAuthorisationAttacker gains higher-level access than permittedPrinciple of least privilege, privilege separation, sandboxing
💡 STRIDE Application: STRIDE is applied at the Data Flow Diagram level — each arrow (data flow), box (process), cylinder (data store), and rectangle (external entity) is examined against all 6 STRIDE categories. This systematic approach ensures no threat type is overlooked.

5. PASTA (Process for Attack Simulation and Threat Analysis)

PASTA: A risk-centric, seven-stage threat modeling methodology that aligns technical threat analysis with business objectives and risk appetite. Developed by Tony UcedaVélez and Marco Morana. Considered the most comprehensive mainstream threat modeling framework.

PASTA's 7 Stages

StageNameActivities
IDefine ObjectivesAlign security requirements with business objectives; define risk appetite; identify compliance requirements
IIDefine Technical ScopeInventory components, infrastructure, dependencies; define system boundaries
IIIApplication DecompositionCreate DFDs; identify data flows, trust levels, and attack surfaces
IVThreat AnalysisIdentify threat actors (insider, competitor, nation-state); use threat intelligence feeds
VVulnerability & Weakness AnalysisCorrelate existing vulnerabilities (CVSS scores, CVE data) with threat actors
VIAttack ModellingDevelop attack trees; map MITRE ATT&CK techniques; model likely attack paths
VIIRisk & Impact AnalysisCalculate business risk; prioritise by impact to business objectives; define countermeasures
PASTA vs. STRIDE:
STRIDE = technical, design-time, focused on software components
PASTA = business-risk-centric, includes threat intelligence and attack simulation, outputs business risk in business language
When the exam asks "most comprehensive" or "business-aligned" → PASTA

6. Attack Trees

Attack Tree: A tree diagram where the root node represents the attacker's goal and branches represent different methods to achieve that goal. Sub-nodes can be connected with AND (all required) or OR (any one sufficient) logic. Developed by Bruce Schneier. 
        GOAL: Steal Customer Credit Card Data
                     |
         +-----------+-----------+
         |                       |
  Breach the Database      Intercept Traffic
         |                       |
    +----+----+             +----+----+
    |         |             |         |
SQL Injection Stolen     MITM Attack  TLS
              Credentials             Downgrade
    |
+---+---+
|       |
Via Web  Via API
Form     Endpoint

Attack trees help visualise compound attack scenarios, identify choke points where defences have maximum effect, and support adversarial thinking during design reviews.

💡 AND vs. OR Nodes:
OR node: attacker only needs ONE child path to succeed — each path represents a separate vulnerability
AND node: attacker needs ALL child paths to simultaneously succeed — multiple conditions required
AND nodes represent stronger security (more conditions needed to attack); OR nodes represent architectural weakness.

7. MITRE ATT&CK Framework

MITRE ATT&CK (Adversarial Tactics, Techniques, and Common Knowledge): A publicly available knowledge base documenting real-world attacker Tactics, Techniques, and Procedures (TTPs) observed in actual attacks. Maintained by MITRE Corporation. Used for threat detection, hunting, and response — NOT for design-time threat modeling.

ATT&CK Structure

LevelDefinitionExample
TacticThe attacker's why — high-level objective or goalCredential Access, Lateral Movement, Exfiltration
TechniqueThe how — specific method to achieve the tacticT1059 – Command and Scripting Interpreter
Sub-techniqueMore specific implementation of a techniqueT1059.001 – PowerShell
ProcedureSpecific observed implementation by real threat groupsAPT29 using PowerShell for initial access in SolarWinds attack

ATT&CK Use Cases

  • Threat Intelligence: Map known threat actor TTPs to your environment
  • Gap Analysis: Identify which ATT&CK techniques you cannot currently detect
  • Red Team Planning: Use ATT&CK to design realistic adversary simulation exercises
  • SOC Detection Rules: Build SIEM detections aligned to ATT&CK techniques
  • Incident Response: Document attacker behaviour using common taxonomy

MITRE D3FEND

D3FEND is the defensive counterpart to ATT&CK — it maps defensive techniques (detect, harden, isolate, deceive, evict) to the ATT&CK offensive techniques. Enables defenders to systematically address known attacker TTPs.

8. Data Flow Diagrams and Trust Boundaries

Data Flow Diagram (DFD): A diagram showing how data moves through a system — between processes, data stores, and external entities. The primary input for STRIDE-based threat modeling. Represents the system from a data-movement perspective rather than a code perspective.

DFD Elements

SymbolNameRepresentsSTRIDE Applicable?
Rectangle / SquareExternal EntitySystems or users outside the application (users, external services)Spoofing, Repudiation
Rounded Rectangle / CircleProcessApplication logic that transforms or routes dataAll 6 STRIDE categories
Cylinder / Double LinesData StorePersistent storage (database, file, cache)Tampering, Information Disclosure, DoS
ArrowData FlowMovement of data between elementsTampering, Information Disclosure, DoS
Trust Boundary: A line on the DFD that separates components with different trust levels. Whenever data crosses a trust boundary, it must be validated and authenticated. Trust boundaries are the most important security checkpoints in a system design.
Trust Boundary Examples:
• Between internet (untrusted) and DMZ (partially trusted)
• Between DMZ and internal network (trusted)
• Between user space and kernel space
• Between different user privilege levels
• Between on-premises and cloud components

9. Other Frameworks (DREAD, LINDDUN)

DREAD (Legacy Scoring)

LetterCategoryWhat It Measures
DDamageHow severe is the damage if exploited?
RReproducibilityHow easy is it to reproduce the attack?
EExploitabilityHow easy is it to exploit the vulnerability?
AAffected UsersHow many users are affected?
DDiscoverabilityHow easy is it to find the vulnerability?

Each category is scored 1–10 and averaged. DREAD was used by Microsoft until ~2008 when they found the scoring was too subjective and inconsistent. It has largely been replaced by CVSS in industry but may still appear on the exam.

LINDDUN (Privacy Threat Modeling)

LINDDUN (Linkability, Identifiability, Non-repudiation, Detectability, Disclosure, Unawareness, Non-compliance) is a privacy-specific threat modeling methodology developed at KU Leuven. It follows the same DFD-based approach as STRIDE but focuses on privacy threats rather than security threats — a natural complement to GDPR compliance and Privacy by Design (Day 6).

10. Memory Tricks & Mnemonics

🧠 STRIDE — "STRIDE is what attackers do to violate security properties":
Spoofing → violates Authentication
Tampering → violates Integrity
Repudiation → violates Non-repudiation
Information Disclosure → violates Confidentiality
Denial of Service → violates Availability
Elevation of Privilege → violates Authorisation
Memory sentence: "Six Threats Really Irritate Diligent Experts"
Note: STRIDE maps almost perfectly to CIA Triad + Authentication + Authorisation + Non-repudiation
🧠 PASTA Stages — "DO TA VV AR":
Define objectives → Objective setting
Technical scope → Application decomposition
Vulnerability analysis → Value (threat) analysis
Attack modeling → Risk assessment
Simpler: PASTA = "Purpose: Align Technical Security to Attacker goals"
🧠 Framework Selection Quick Guide:
Software component threat identification → STRIDE
Business-risk-aligned comprehensive analysis → PASTA
Visualise compound attack paths → Attack Trees
Describe/detect real attacker TTPs → MITRE ATT&CK
Privacy-specific threats → LINDDUN
Deprecated scoring but still tested → DREAD
🧠 ATT&CK Hierarchy — "Tactics Tell Techniques, Procedures Prove":
Tactic = Why (goal) → Technique = How (method) → Procedure = Who did what specifically
Think: Military analogy — Tactic is "capture the hill" / Technique is "flanking manoeuvre" / Procedure is "3rd platoon went left"
