2. Real-World Organizational Relevance

Organizations choose pen test types based on objectives. A startup with a new web application might use a white-box assessment to maximize coverage quickly with limited budget. A financial institution practicing incident response might run a full red team exercise to test how the SOC detects and responds. A large tech company might run a public bug bounty year-round to crowdsource vulnerability discovery from thousands of researchers.

The wrong test type leads to wasted resources or gaps: running a black-box test when you want full coverage misses vulnerabilities that require internal knowledge. Running only external tests when internal threats are a primary concern provides false assurance.

3. Knowledge-Based Testing Types (Black / White / Gray Box)

These terms describe how much prior knowledge of the target environment the pen testers are given BEFORE the engagement begins. This is about information, not access level.

Exam Tip — Gray Box Clarification: Gray box means partial KNOWLEDGE, not partial ACCESS. A gray-box tester might have valid user credentials (to simulate an authenticated user) and a general network diagram, but not source code or full admin credentials. This is the most common type used in practice.

4. Scope-Based Testing Types (Internal vs. External)

External Penetration Test

External Pen Test: Testing targets the organization's internet-facing attack surface — public IP addresses, web applications, email servers, VPN gateways, DNS, public APIs. The tester starts from outside the network perimeter, with no internal network access.
  • Targets: DMZ systems, web servers, email gateways, public-facing APIs, SSL/TLS configurations, DNS zone security
  • Purpose: Identify what an internet-based attacker could exploit
  • Required by: PCI DSS quarterly external scanning (ASV); annual external pen test requirement

Internal Penetration Test

Internal Pen Test: Testing conducted from INSIDE the network perimeter — simulates an attacker who has already bypassed perimeter defenses (via phishing, stolen VPN credentials, insider threat, physical access). Focuses on internal systems, active directory, lateral movement, privilege escalation.
  • Targets: Active Directory, internal servers, shared network drives, lateral movement paths, privilege escalation vectors, internal applications
  • Purpose: Assess "assume breach" posture — what can an attacker do once inside?
  • Finds: Domain admin paths, internal network segmentation failures, credential reuse, Kerberoasting, excessive service account privileges
External vs. Internal is about STARTING POSITION, not knowledge:
External = tester starts outside (internet); Internal = tester starts inside (corporate LAN)
These are orthogonal to black/gray/white box — you can have a "white box internal" or "black box external" test.

5. Physical Penetration Testing

Physical Pen Test: Testing of physical security controls — building access controls, badge readers, locks, guard procedures, tailgating prevention, secure areas. Testers attempt to gain unauthorized physical access to restricted areas.

Physical Testing Techniques

TechniqueDescription
Tailgating / PiggybackingFollowing an authorized person through a secured door without using own badge; tailgating = without their knowledge; piggybacking = with their complicit assistance
ImpersonationPosing as IT support, maintenance worker, delivery person, or contractor to gain building access
Lock picking / bypassBypassing physical locks using lock picks, bump keys, or relay attacks on RFID
Badge cloningUsing RFID/NFC readers to clone proximity cards (HID, 125kHz cards are highly vulnerable)
Dumpster divingSearching discarded materials for sensitive documents, credentials, hardware
USB drop attackLeaving malicious USB drives in parking lots or common areas to test if employees plug them in
Exam Tip — Tailgating vs. Piggybacking: Tailgating = without the person's knowledge or consent (they don't know they let an attacker in). Piggybacking = the authorized person knowingly holds the door open for the tester/attacker. Both are physical access vulnerabilities; different social engineering dynamics.

6. Social Engineering Testing

Social Engineering Testing: Authorized testing that attempts to manipulate employees into taking insecure actions — revealing credentials, opening malicious attachments, granting unauthorized access, or disclosing sensitive information.

SmishingSMS-based phishing — text messages with malicious linksMobile security awareness

7. Red Team / Blue Team / Purple Team

Red Team

Red Team: An internal or external group that simulates a full, realistic adversary attack against an organization — using ALL attack vectors (digital, physical, social engineering) over an extended period (weeks or months). The goal is to test detection and response capabilities, NOT just find vulnerabilities. The defensive team (Blue Team) does NOT know the test is occurring.
  • Uses TTPs (Tactics, Techniques, Procedures) of real threat actors (often modeled after APT groups)
  • Multi-vector attack: combines network hacking + social engineering + physical testing
  • Success is measured differently: can the red team achieve a specific objective (e.g., reach payment data, exfiltrate IP)?
  • More expensive and time-intensive than a pen test; typically 4–12 week engagements

Blue Team

Blue Team: The internal defensive security team — SOC analysts, incident responders, security architects. Their job is to detect, respond to, and recover from attacks (real or simulated). During a red team exercise, the blue team is unaware that they're being tested.

Purple Team

Purple Team: A collaborative exercise where red team and blue team work TOGETHER to improve both offensive testing coverage and defensive detection/response. The red team shares TTPs and attack methods in real-time so the blue team can practice detection and tuning of controls against known attack patterns.
Red vs. Purple Exercise — Key Difference:
Red team: Adversarial; blue team DOESN'T know they're being tested; measures real detection capability
Purple team: Collaborative; both teams work together; goal is to IMPROVE controls together, not measure them
Red team → assesses current state; Purple team → builds/improves defensive capability

8. Bug Bounty Programs

Bug Bounty Program: An organization invites independent security researchers (white-hat hackers) to test systems within a defined scope and pays rewards (bounties) for valid vulnerability reports. Unlike contracted pen tests, bug bounties are open to the public or a defined pool of researchers continuously.

Key Characteristics

AttributeBug BountyContracted Pen Test
TestersCrowd of independent researchers globallyDefined, contracted security firm or individuals
DurationOngoing / continuousFixed timeframe engagement
PaymentPer valid finding (bounty)Fixed fee for engagement
CoverageBroad — many researchers, many skills, unpredictable pathsDefined scope; methodology-driven; systematic
Scope controlLegally defining scope is critical — researchers must stay within scopeScope controlled by SOW and ROE document
Legal authorizationPublic program policy IS the legal authorization for in-scope activitySigned contract and ROE required
Best forContinuous coverage; large attack surfaces; large organizations (Google, Microsoft)Deep systematic assessment; compliance requirements; testing under NDA

Bug Bounty Platforms

  • HackerOne: Largest bug bounty platform; used by US DoD, major tech companies
  • Bugcrowd: Alternative platform; also offers "managed" programs with vetting
  • Synack: Vetted, trusted researcher pool (combines crowdsourced + curated)
⚠️ Scope Violations: Bug bounty hunters who test systems outside defined scope (even accidentally) can face legal consequences. The scope definition in the bug bounty policy IS the legal authorization — testing out-of-scope is identical to unauthorized access.

9. Memory Tricks & Mnemonics

🧠 Black/White/Gray — Knowledge Levels
Black box = Dark room (no info — must find everything yourself)
White box = Lights fully on (full info — architecture, code, credentials)
Gray box = Dim room (partial info — maybe one key, maybe a map, but not everything)
🧠 Red/Blue/Purple Team Summary
Red = Attacks (adversary, offensive)
Blue = Defends (SOC, incident response)
Purple = Both play together ("Red + Blue = Purple") — collaborative improvement
🧠 Tailgating vs. Piggybacking
Tailgating = Follows WITHOUT knowledge (like a dog following you without being invited — you don't know)
Piggybacking = Held the door (the person knowingly lets them in — they chose to let the "pig" ride along)
🧠 Bug Bounty vs. Pen Test
Bug Bounty = Open invitation (many researchers, continuous, pay-per-bug)
Pen Test = Hired contractor (specific testers, fixed scope, fixed fee)

Top Exam Points:
  • Gray box = partial KNOWLEDGE (not partial access); most common test type in real engagements
  • Red team = covert adversary simulation measuring real detection capability; not a standard pen test
  • Purple team = collaborative RED + BLUE improvement exercise (not adversarial)
  • Bug bounty is CONTINUOUS and CROWDSOURCED; pen test is FIXED SCOPE and CONTRACTED
  • Physical testing includes USB drop attacks, badge cloning, tailgating — all valid CISSP exam topics
  • Internal testing simulates "assume breach" — most dangerous and often most revealing
