2. Real-World Organizational Relevance

The 2017 Equifax breach exploited a known Apache Struts vulnerability (CVE-2017-5638) that had a patch available for two months before the breach. The 2021 Log4Shell vulnerability (CVE-2021-44228, CVSS 10.0) demonstrated the challenge of zero-day management — organizations had to identify all instances of a ubiquitous library across their entire software stack. Effective vulnerability management would have prevented the former; mature zero-day response was essential for the latter.

Lifecycle Details

PhaseActivitiesKey Outputs
1. DiscoverMaintain asset inventory; run authenticated and unauthenticated vulnerability scans; conduct penetration tests; leverage threat intelligence; participate in bug bounty programsVulnerability reports; asset database
2. PrioritizeScore vulnerabilities (CVSS); evaluate asset criticality; check exploit availability (EPSS — Exploit Prediction Scoring System); consider business contextPrioritized remediation queue
3. RemediateApply patches (through change management); implement compensating controls; accept residual risk (documented risk acceptance by asset owner)Remediation actions; risk acceptance records
4. VerifyRe-scan to confirm vulnerability is resolved; validate no regressions introduced; test system functionalityVerification scan results
5. ImproveTrack metrics (mean time to remediate, scan coverage, open vulnerability trends); refine processes; report to managementMetrics dashboard; management reports

Vulnerability Scanning Types

TypeDescriptionProsCons
Unauthenticated (network)Scans from outside with no credentials; sees what an external attacker seesSimulates external attacker perspective; no credential riskMisses many host-level vulnerabilities; high false negative rate
Authenticated (credentialed)Scans with valid system credentials; examines installed software, configurations, patch levelsMost accurate; fewest false negatives; finds configuration issuesRequires credential management; slightly higher system load
Agent-basedLightweight agent installed on each system; continuous assessmentWorks for roaming/remote devices; real-time monitoring; no network scan overheadAgent deployment and management; resource consumption on host
Exam Emphasis: Authenticated scans are ALWAYS more accurate than unauthenticated scans. When the exam asks about reducing false negatives in vulnerability scanning, "use authenticated scanning" is typically the correct answer.

4. Vulnerability Scoring — CVSS

CVSS (Common Vulnerability Scoring System): An open, industry-standard system for assessing the severity of vulnerabilities. Produces a numerical score (0.0-10.0) with a corresponding qualitative rating.

CVSS Score Ranges

ScoreRatingTypical Response
0.0NoneInformational — no action required
0.1 – 3.9LowRemediate in regular maintenance cycle
4.0 – 6.9MediumRemediate within 30 days; compensating controls
7.0 – 8.9HighRemediate within 7-14 days; escalate priority
9.0 – 10.0CriticalImmediate remediation; emergency change process

CVSS Metric Groups

Metric GroupWhat It MeasuresKey Metrics
BaseIntrinsic, unchanging characteristics of the vulnerabilityAttack Vector (network/adjacent/local/physical); Attack Complexity (low/high); Privileges Required; User Interaction; Scope; CIA Impact
TemporalCharacteristics that change over timeExploit Code Maturity (unproven/PoC/functional/high); Remediation Level (official fix/temp fix/workaround/unavailable); Report Confidence
EnvironmentalOrganization-specific contextModified Base metrics reflecting local importance; CIA requirements for the affected asset
CVSS Base Score Alone Is Insufficient: Prioritization should consider: (1) CVSS Base score, (2) asset criticality/business value, (3) exploit availability in the wild, (4) compensating controls already in place, and (5) threat intelligence (is this vulnerability being actively exploited?). A CVSS 7.0 on a critical internet-facing database server may be higher priority than a CVSS 9.0 on an isolated test system.

5. Virtual Patching & Compensating Controls

When Patches Cannot Be Applied

ScenarioCompensating Control Options
Legacy system — vendor no longer provides patchesNetwork isolation; WAF rules; enhanced monitoring; application retirement plan
Critical system — cannot take downtime for patchingVirtual patch (WAF/IPS rule); schedule patch for next maintenance window
Testing delay — patch may break business applicationVirtual patch while testing proceeds; enhanced monitoring; network segmentation
Vendor delay — no patch available yetWorkaround per vendor advisory; virtual patch; monitoring for exploitation attempts
Virtual Patching: Implementing protective rules at the network perimeter (WAF, IPS, firewall) that block the specific exploit vector for a known vulnerability — WITHOUT modifying the vulnerable application or system. The vulnerability still exists; the exploit path is blocked.
⚠️ Virtual Patches Are Temporary: Virtual patching is a compensating control, NOT a permanent fix. The actual patch must still be applied. Virtual patches only protect against KNOWN exploit vectors — a new exploit technique for the same vulnerability may bypass the virtual patch.

6. Vulnerability Disclosure

Disclosure Models

ModelDescriptionProsCons
Responsible / Coordinated DisclosureResearcher notifies vendor privately; gives vendor reasonable time (typically 90 days) to develop and release a patch; then researcher publishesVendor has time to fix; users can patch before exploit details are publicVendor may delay indefinitely; users are unaware of risk during disclosure period
Full DisclosureResearcher publishes vulnerability details publicly immediately, with or without vendor notificationPressures vendors to act quickly; informs defenders immediatelyAttackers can exploit before patch is available; creates zero-day window
Non-DisclosureResearcher does not disclose at all, or sells vulnerability on underground marketsNone (ethically problematic)Vulnerability persists; may be exploited by threat actors who purchase it
CISSP favors Responsible/Coordinated Disclosure: The exam considers coordinated disclosure as the ethical standard. The researcher gives the vendor a reasonable timeframe (commonly 90 days — Google Project Zero standard) to issue a fix before public disclosure.

Key Vulnerability Identifiers

SystemPurposeFormat Example
CVECommon Vulnerabilities and Exposures — unique identifier for each known vulnerabilityCVE-2021-44228 (Log4Shell)
NVDNational Vulnerability Database — NIST-maintained database that adds CVSS scores and analysis to CVEsNVD entry with CVSS score, references, patch info
CWECommon Weakness Enumeration — categorizes types of software weaknessesCWE-79 (Cross-Site Scripting)
CPECommon Platform Enumeration — standardized naming for IT productscpe:2.3:a:apache:log4j:2.14.1

7. Zero-Day Management

Zero-Day Vulnerability: A vulnerability that is being exploited (or that has no available patch) on the same day it becomes publicly known — the vendor has had "zero days" to develop a fix. Zero-day = no patch available.

Zero-Day Response Strategy

ActionDetails
Assess exposureIdentify all instances of the affected software/component across the environment (requires accurate asset inventory)
Apply vendor workaroundsImplement any temporary mitigations recommended by the vendor (disable affected feature, configuration change)
Virtual patchingDeploy WAF/IPS rules to block known exploit vectors
Network segmentationIsolate affected systems to limit blast radius if compromised
Enhanced monitoringIncrease logging and alerting for affected systems; monitor for IoCs associated with the vulnerability
Threat huntingProactively search for signs that the vulnerability has already been exploited in the environment
Patch immediately when availableUse emergency change process to apply vendor patch as soon as released

8. Memory Tricks & Mnemonics

🧠 Vulnerability Management Lifecycle — "DPRVI"
Discover → Prioritize → Remediate → Verify → Improve
"Discover Problems, Remediate, Verify, Improve"
🧠 CVSS Metric Groups — "BTE"
Base (intrinsic — never changes)
Temporal (changes over time — exploit maturity, patch status)
Environmental (org-specific context)
"Base is constant; Temporal evolves; Environmental is yours"
🧠 CVE vs CWE
CVE = specific Example ("CVE-2021-44228 is Log4Shell")
CWE = Enumerated weakness type ("CWE-79 is XSS")
"CVE points at ONE bug; CWE names the BUG TYPE"
🧠 Disclosure — 90 Days
Responsible disclosure standard: Give the vendor 90 days. "90 days to fix it, then the world knows."
