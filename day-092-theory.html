2. Real-World Organizational Relevance

QA teams own testing processes; developers write unit and integration tests; security teams define security test requirements and review results. Modern CI/CD pipelines run automated tests on every code commit — a failed test blocks deployment. Organizations practicing DevSecOps integrate security tests (SAST, DAST, dependency scanning) as mandatory pipeline gates alongside functional tests.

Failure modes: Deploying without regression testing after a hotfix (breaks existing features); skipping UAT for urgent releases (deploys software that doesn't meet business requirements); no load testing before launch (system crashes under real traffic).

3. Testing Levels (Unit → Integration → System → Acceptance)

Testing Levels (V-Model Alignment) Requirements ──────────────────── Acceptance Testing (UAT) │ ▲ High-Level Design ────────────── System Testing │ ▲ Detailed Design ──────────────── Integration Testing │ ▲ Coding ───────────────────────── Unit Testing

Unit Testing

Unit Testing: Tests individual components (functions, methods, classes) in isolation. Each unit is tested independently with mock objects replacing dependencies. Written by developers, typically automated, and run on every code commit.
  • Scope: Single function or method — smallest testable unit
  • Who: Developers
  • When: During coding; runs in CI pipeline on every commit
  • Security relevance: Tests input validation functions, authentication logic, encryption/decryption routines at the code level
  • Tools: JUnit (Java), pytest (Python), NUnit (.NET), Jest (JavaScript)

Integration Testing

Integration Testing: Tests how multiple units/modules work TOGETHER. Verifies that interfaces between components function correctly — data passes correctly between modules, API calls return expected responses, database queries work with real (or test) database.
  • Scope: Interactions between 2+ modules/services
  • Approaches: Top-down (test from UI layer down), bottom-up (test from data layer up), sandwich/hybrid (both simultaneously)
  • Security relevance: Tests authentication flows end-to-end, API authorization between services, encrypted communication between components

System Testing

System Testing: Tests the COMPLETE, integrated system as a whole against the original requirements. Conducted in an environment that mirrors production. Includes functional testing, performance testing, security testing, and usability testing.
  • Scope: Entire application/system in production-like environment
  • Who: QA team (independent from developers)
  • Security relevance: DAST scanning, full penetration testing, configuration review are performed at system test level

Acceptance Testing (UAT)

User Acceptance Testing (UAT): Final testing phase where actual end users or business stakeholders verify the system meets their business requirements and is ready for production deployment. UAT is the LAST gate before go-live.
  • Types: Alpha testing (at developer site by users), Beta testing (at user site in real environment)
  • Who: End users, business stakeholders, product owners
  • Pass/fail criteria: Business requirements are met — NOT technical correctness (that was verified in earlier phases)
  • Security relevance: Validates that security controls don't impede business workflows; confirms data access controls match business rules
Exam Tip — V-Model: The V-Model maps each development phase to a corresponding test level. Requirements → UAT; High-level design → System testing; Detailed design → Integration testing; Coding → Unit testing. Each left-side phase has a matching right-side verification.

4. Regression, Smoke & Sanity Testing

Test TypeDefinitionWhen RunPurpose
Regression TestingRe-running previously passing tests after a code change to ensure the change didn't break existing functionalityAfter every code change, patch, or bug fixEnsure no unintended side effects; verify stability
Smoke TestingA quick, shallow test of critical functions to verify the build is stable enough for further testing ("does it turn on and not crash?")After every new build; first test run on a new deploymentVerify basic functionality before investing time in full test suite
Sanity TestingNarrow, focused testing of a specific area after a targeted change (subset of regression testing)After a specific bug fix or feature changeConfirm the specific fix works without full regression suite
Smoke vs. Sanity:
Smoke: Broad and shallow — "does the whole thing work at a basic level?" (like smoke testing electronics — turn it on, does it smoke?)
Sanity: Narrow and focused — "does this specific change work?" (a sanity check on one area)
⚠️ Exam Trap: Regression testing is NOT about finding new bugs — it's about ensuring that changes didn't break EXISTING, previously working functionality. After a security patch, regression testing confirms the patch didn't break legitimate features.

5. Performance Testing (Load, Stress, Volume)

Test TypeDefinitionWhat It MeasuresExample
Load TestingTests system behavior under EXPECTED peak load (normal maximum users/transactions)Response time, throughput, resource utilization at expected capacity"How does the site perform with 10,000 concurrent users?" (our expected peak)
Stress TestingTests system behavior BEYOND maximum capacity — pushes until failure to find the breaking pointBreaking point, failure mode (graceful degradation vs. crash), recovery behavior"What happens if 50,000 users hit the site simultaneously?" (3× our capacity)
Volume TestingTests system behavior with large amounts of DATA (not users) — massive datasets, large database operationsPerformance with large data volumes; database query performance; storage handling"How does the database perform with 100 million records vs. 1 million?"
Endurance (Soak) TestingTests system stability over an EXTENDED TIME period under sustained loadMemory leaks, resource exhaustion, performance degradation over hours/days"Does the application stay stable after 72 hours of continuous operation?"
Spike TestingTests system response to SUDDEN, dramatic load increasesHow quickly the system handles sudden traffic bursts; auto-scaling behavior"What happens when traffic jumps from 1,000 to 20,000 users in 10 seconds?"
Security Relevance of Performance Testing: DDoS resilience is validated through stress/spike testing. Resource exhaustion attacks exploit systems that haven't been stress-tested. Memory leaks found in endurance testing can be exploited for denial of service.

6. Test-Driven Development (TDD)

TDD (Test-Driven Development): A software development methodology where developers write automated test cases BEFORE writing the actual code. The cycle: (1) Write a failing test, (2) Write minimal code to pass the test, (3) Refactor the code while keeping tests passing. Also known as "Red-Green-Refactor."
TDD Cycle — "Red-Green-Refactor" ┌─────────────────────────────────────────┐ │ │ ▼ │ RED: Write a │ failing test ──► GREEN: Write │ (test first!) minimum code │ to pass test │ │ │ ▼ │ REFACTOR: │ Clean up code │ (tests still │ pass) ─────────┘

TDD Security Benefits

  • Security requirements expressed as tests from the start (e.g., "input with SQL characters should be rejected")
  • Regression protection — any code change that breaks a security test is immediately caught
  • Forces developers to think about edge cases and failure modes before writing code
  • Building security tests into the test suite ensures they're run on every build
BDD (Behavior-Driven Development): Extension of TDD that writes tests in natural language ("Given... When... Then...") to describe expected behavior. Security-focused BDD: "Given a user without admin privileges, When they access the admin page, Then they should receive a 403 Forbidden response." Tools: Cucumber, SpecFlow.

8. Memory Tricks & Mnemonics

🧠 Testing Levels — "U-I-S-A"
Unit → Integration → System → Acceptance
"USA-I" — scope increases from smallest (unit) to largest (acceptance)
🧠 Load vs. Stress vs. Volume
Load = expected USERS (how many people can use it?)
Stress = BEYOND limits (what breaks it?)
Volume = DATA size (how much data can it handle?)
"LSV: Load=Users, Stress=Limits, Volume=Data"
🧠 TDD = "Red-Green-Refactor"
Red = test fails (you haven't written the code yet)
Green = test passes (you wrote minimal code)
Refactor = clean up (improve code quality, tests still pass)
Test FIRST, then code — the opposite of traditional development
🧠 Smoke vs. Regression
Smoke = "Can it turn on without catching fire?" (quick, basic check)
Regression = "Did we accidentally break something that was working?" (comprehensive recheck)
