2. Real-World Organizational Relevance

Most organizations have adopted Agile or hybrid methodologies. The challenge for security teams is integrating security activities into rapid sprint cycles without slowing delivery. Understanding Scrum ceremonies and how to embed security stories, threat modeling, and automated testing into sprints is essential for modern security practitioners.

Agile Does NOT Mean No Documentation: The manifesto says we value working software MORE than documentation — not that documentation is eliminated. Security documentation (threat models, security requirements, test results) remains essential even in Agile. The items on the right still have value.

5. Kanban

Kanban = "Visual signal" (Japanese). A workflow management method that visualizes work on a board, limits work-in-progress (WIP), and focuses on continuous flow rather than time-boxed iterations.
Kanban PrincipleDescription
Visualize workflowBoard with columns (To Do → In Progress → Review → Done); each work item is a card
Limit WIPMaximum items allowed in each column; prevents overloading; surfaces bottlenecks
Manage flowOptimize the flow of work through the system; reduce cycle time
Continuous improvementRegularly analyze and improve the process
Kanban vs Scrum: Kanban = continuous flow, no sprints, WIP limits, flexible. Scrum = time-boxed sprints, defined roles, ceremonies. Both are Agile. Many teams use "Scrumban" (hybrid).

6. Extreme Programming (XP)

XP PracticeDescriptionSecurity Benefit
Pair programmingTwo developers at one workstation; one codes, one reviews in real-timeContinuous code review; catches security flaws immediately
Test-driven development (TDD)Write test first → write code to pass → refactorSecurity tests written before code; ensures testability
Continuous integrationCode integrated and tested multiple times per daySecurity tests run with every integration; fast feedback
RefactoringRegularly improve code structure without changing behaviorEliminate security debt; simplify attack surface
Simple designBuild only what's needed now; avoid unnecessary complexityReduced attack surface; easier to secure

7. Capability Maturity Model Integration (CMMI)

CMMI (originally from Carnegie Mellon / SEI) measures organizational process maturity on a 5-level scale. Higher levels indicate more mature, predictable, and improvable processes.
LevelNameCharacteristics
1InitialAd hoc, chaotic; success depends on heroic individuals; unpredictable processes; no formal procedures
2Managed (Repeatable)Basic project management; processes documented for specific projects; can repeat previous successes
3DefinedOrganization-wide standardized processes; documented and followed consistently; proactive
4Quantitatively ManagedProcesses measured with metrics; data-driven decisions; statistical process control
5OptimizingContinuous process improvement; innovative practices; defect prevention focus
CMMI Levels — Exam Must-Know:
Level 1 = chaos (no formal process)
Level 2 = repeatable (project-level process)
Level 3 = defined (org-wide process) — first level where organization-wide standards exist
Level 4 = quantitative (measured with metrics)
Level 5 = optimizing (continuous improvement)
Key distinction: Level 2 = project-level processes; Level 3 = organization-wide processes

8. IDEAL Model

IDEAL is a process improvement model (also from SEI/Carnegie Mellon) that describes the steps for implementing process improvement programs.
PhaseActivities
InitiatingBusiness case; sponsorship; infrastructure setup
DiagnosingAssess current state; identify gaps between current and desired maturity
EstablishingSet priorities; develop approach; plan actions
ActingImplement improvements; pilot changes; deploy solutions
LearningAnalyze results; document lessons learned; propose future actions

9. Memory Tricks & Mnemonics

🧠 CMMI Levels — "I Must Define Quantitative Optimization"
1 = Initial
2 = Managed (repeatable)
3 = Defined
4 = Quantitatively managed
5 = Optimizing
🧠 Scrum Roles — "PO, SM, DT"
Product Owner = WHAT to build (prioritizes backlog)
Scrum Master = HOW to work (facilitates process)
Development Team = BUILD it (self-organizing)
🧠 IDEAL = "I Diagnose, Establish, Act, Learn"
Process improvement in 5 steps — assess where you are, plan where to go, act, then learn from results.
🧠 Agile Manifesto — "People, Software, Collaboration, Adaptation"
Value the left (people/software/collaboration/adaptation) MORE THAN the right (process/docs/contracts/plans) — but the right still matters.
