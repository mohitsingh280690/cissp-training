3. Zachman Framework

📖 Zachman Framework (1987): A two-dimensional classification framework (not a methodology) for organizing enterprise architecture artifacts. Created by John Zachman at IBM. Uses a 6×6 matrix of perspectives and interrogatives.

The 6×6 Matrix

Perspective (Row)WhatHowWhereWhoWhenWhy
Executive (Scope/Planner)Business context and scope definitions
Business Management (Owner)Business model and concepts
Architect (Designer)System logical models
Engineer (Builder)Technology/physical models
Technician (Implementer)Detailed implementation specifications
Enterprise (Functioning)Actual running system
🔑 Key Facts:
• 6 rows (perspectives) × 6 columns (interrogatives: What, How, Where, Who, When, Why)
• It is a taxonomy, not a methodology — it classifies but doesn't prescribe steps
• Each cell is unique and represents a specific architectural artifact
• Does NOT prescribe how to build architecture — only how to organize descriptions of it
💡 Exam Tip: Zachman = "6×6 matrix" = "classification framework" = "not a methodology." If you see any of these keywords, think Zachman.

4. TOGAF (The Open Group Architecture Framework)

📖 TOGAF: A comprehensive enterprise architecture methodology developed by The Open Group. Unlike Zachman (which classifies), TOGAF provides a step-by-step process (ADM) for developing enterprise architecture.

Architecture Development Method (ADM)

TOGAF's core is the ADM cycle — an iterative process with phases:

  • Preliminary: Define architecture principles and framework
  • A: Architecture Vision
  • B: Business Architecture
  • C: Information Systems Architecture (data + application)
  • D: Technology Architecture
  • E: Opportunities and Solutions
  • F: Migration Planning
  • G: Implementation Governance
  • H: Architecture Change Management
  • Requirements Management (center, ongoing)
🔑 Key Facts:
• IS a methodology (unlike Zachman)
• Four architecture domains: Business, Data, Application, Technology (BDAT)
• ADM is iterative and adaptable
• Maintained by The Open Group; vendor-neutral

5. SABSA (Sherwood Applied Business Security Architecture)

📖 SABSA: A security-specific enterprise architecture framework that is business-driven and risk-focused. Uses a layered model similar to Zachman's perspectives but specifically oriented toward security architecture.

SABSA Layers

LayerViewFocus
ContextualBusiness ViewBusiness requirements, risk appetite, security strategy
ConceptualArchitect's ViewSecurity principles, fundamental concepts
LogicalDesigner's ViewSecurity services, logical architecture
PhysicalBuilder's ViewSecurity mechanisms, products, technologies
ComponentTradesman's ViewDetailed specifications, standards, tools
OperationalFacilities Manager's ViewOperations, administration, monitoring
🔑 Key Facts:
• Security-specific (not general EA like Zachman/TOGAF)
• Business-driven: starts with business requirements, derives security from business needs
• Risk-focused: security controls trace back to business risk
• Uses same 6 interrogatives as Zachman (What, How, Where, Who, When, Why) per layer
💡 Exam Tip: SABSA = "Security Architecture" + "Business-Driven" + "Risk-Focused." It's the go-to answer when a question asks for a security-specific architecture framework.

6. ITIL (Information Technology Infrastructure Library)

📖 ITIL: A set of best practices for IT service management (ITSM). Not a security framework per se, but its service management approach is relevant to security operations, incident management, change management, and service continuity.

ITIL v3 Service Lifecycle

ComponentSecurity Relevance
Service StrategyAligning security services with business needs
Service DesignDesigning security controls into services
Service TransitionChange management, release management (security review of changes)
Service OperationIncident management, problem management, event monitoring
Continual ImprovementSecurity metrics, improvement cycles
🔑 Key Facts:
• IT Service Management (ITSM) framework, not security-specific
• Relevant to CISSP for change management, incident management, and service continuity
• Originally UK government; now managed by AXELOS/PeopleCert
• Complements security frameworks rather than replacing them

7. NIST Enterprise Architecture

📖 NIST Enterprise Architecture: Originally based on the Federal Enterprise Architecture Framework (FEAF), NIST's EA guidance helps U.S. federal agencies develop enterprise architectures. Five reference models provide standardized approaches.

Five Reference Models

ModelFocus
Performance Reference Model (PRM)Measuring outcomes and results
Business Reference Model (BRM)Business operations and services
Service Component Reference Model (SRM)IT service components and capabilities
Data Reference Model (DRM)Data standards, sharing, and management
Technical Reference Model (TRM)Technology standards and specifications
🔑 Key Facts:
• U.S. federal government focused
• Five reference models providing standardized vocabulary
• Supports interoperability between federal agencies
• Complements NIST SP 800 series security publications

8. DoDAF & MODAF

DoDAF (Department of Defense Architecture Framework)

📖 DoDAF: The U.S. Department of Defense enterprise architecture framework. Defines a standard set of viewpoints and models for describing military architectures, ensuring interoperability and capability integration across DoD systems.

DoDAF Viewpoints

ViewpointFocus
All Viewpoint (AV)Overarching aspects, scope, context
Capability Viewpoint (CV)Capability requirements and delivery timelines
Data and Information Viewpoint (DIV)Data relationships, structures, flows
Operational Viewpoint (OV)Operational activities, nodes, information exchanges
Project Viewpoint (PV)Project relationships to capabilities
Services Viewpoint (SvcV)Service functions, interfaces, interactions
Standards Viewpoint (StdV)Technical standards and conventions
Systems Viewpoint (SV)System functions, interconnections, resources

MODAF (Ministry of Defence Architecture Framework)

📖 MODAF: The UK Ministry of Defence architecture framework. Based on DoDAF but adapted for UK/NATO military requirements. Uses similar viewpoint concepts with UK-specific terminology and NATO interoperability requirements.
🔑 DoDAF vs. MODAF:
• DoDAF = U.S. Department of Defense
• MODAF = UK Ministry of Defence
• Both are military-focused enterprise architecture frameworks
• Both use viewpoints/views to describe architectures
• MODAF includes strategic viewpoints and acquisition perspectives specific to UK/NATO

10. Memory Tricks & Mnemonics

🧠 "Zachman = Zoo Matrix": Zachman is a 6×6 grid — a "zoo" of artifacts organized in a matrix. It classifies but doesn't tell you what to do (not a methodology).
🧠 "TOGAF = The Only Good Architecture Framework (with a Process)": TOGAF is the methodology — it tells you HOW to build architecture step by step via ADM.
🧠 "SABSA = Security Architecture By Studying Business Aims": SABSA is the ONLY security-specific framework. Business-driven and risk-focused.
🧠 "ITIL = IT Is for Lifecycle": ITIL manages IT services through their lifecycle — incident, change, continuity.
🧠 "DoDAF = DoD, MODAF = MoD": U.S. DoD uses DoDAF. UK MoD uses MODAF. Both are military architecture frameworks with viewpoints.
