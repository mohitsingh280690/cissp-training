2. Real-World Organizational Relevance

Session hijacking and authentication bypass attacks account for a large portion of data breaches. Weak session management lets attackers impersonate legitimate users without needing their credentials. With cloud/SaaS adoption, organizations increasingly depend on web-based authentication flows (OAuth, SAML, OIDC), making secure session handling and authentication architecture critical for enterprise security posture.

3. Session Management Security

Session Management: HTTP is stateless — each request is independent. Sessions provide continuity by assigning a unique session ID (token) after authentication, stored in cookies. The server validates this token with each request to maintain the authenticated state.

Session Attacks

AttackDescriptionDefense
Session hijackingStealing an active session ID (via network sniffing, XSS, log exposure) to impersonate the userHTTPS everywhere; HttpOnly cookies; Secure flag; short session timeouts
Session fixationAttacker sets a known session ID before victim logs in; after authentication, attacker uses the same IDRegenerate session ID after successful authentication (critical defense)
Session predictionGuessing valid session IDs due to weak randomness (sequential, predictable patterns)Use cryptographically secure random number generators for session IDs; sufficient length (128+ bits)
Session replayCapturing and replaying a valid session token at a later timeShort expiration; bind session to client IP/fingerprint; use nonces

Secure Cookie Attributes

AttributePurposeSecurity Benefit
SecureCookie only sent over HTTPSPrevents transmission over unencrypted HTTP; stops network sniffing of session ID
HttpOnlyCookie inaccessible to JavaScriptPrevents XSS from stealing session cookies via document.cookie
SameSiteControls cross-site cookie sendingPrevents CSRF; Strict (never cross-site) or Lax (safe methods only)
Domain/PathLimits which URLs receive the cookieRestricts cookie scope to intended application
Expires/Max-AgeControls cookie lifetimeShort-lived sessions reduce window of exploitation
Session Fixation vs Session Hijacking:
• Fixation: Attacker sets the session ID BEFORE authentication (pre-login attack)
• Hijacking: Attacker steals the session ID AFTER authentication (post-login attack)
• Fixation defense: Regenerate session ID after login
• Hijacking defense: HTTPS + HttpOnly + Secure cookie flags

4. Authentication & Credential Flaws

FlawDescriptionDefense
Credential stuffingAutomated login attempts using leaked username/password pairs from other breachesMFA; rate limiting; account lockout; CAPTCHA; credential breach detection
Brute forceSystematically trying all possible passwordsAccount lockout after N attempts; progressive delays; MFA; strong password policies
Password sprayingTrying common passwords across many accounts (avoids lockout)MFA; detect distributed login patterns; ban common passwords
Default credentialsUsing manufacturer-default usernames/passwords left unchangedForce password change on first login; remove default accounts; scan for defaults
Insecure password recoveryWeak "forgot password" — security questions with guessable answers, reset links in email without expirationToken-based reset with short expiration; MFA for recovery; no security questions

Secure Authentication Practices

  • Multi-factor authentication (MFA) — ALWAYS the best answer for authentication security on CISSP
  • Bcrypt/Argon2/PBKDF2 for password hashing (NOT MD5 or SHA alone)
  • Salting — unique random value per password prevents rainbow table attacks
  • Generic error messages — "Invalid username or password" (don't reveal which is wrong)
  • Account lockout — lock after threshold; auto-unlock after delay (not permanent)
Permanent account lockout = denial of service risk. Attackers can intentionally lock out legitimate users. Best practice: temporary lockout with progressive delays, plus alerting.

5. Server-Side Request Forgery (SSRF)

SSRF: An attacker makes the server send requests to unintended locations — internal services, cloud metadata endpoints, or other backend systems. The server acts as a proxy for the attacker, bypassing firewalls and ACLs that protect internal resources.

SSRF Attack Scenarios

ScenarioImpact
Access cloud metadata (e.g., 169.254.169.254)Steal IAM credentials, API keys, instance metadata from AWS/GCP/Azure
Scan internal networkDiscover internal services not exposed to the internet
Access internal APIsBypass authentication on internal services that trust server-to-server traffic
Read local files via file:// protocolAccess server configuration files, credentials

SSRF Defenses

DefenseDescription
Input validationWhitelist allowed URLs/domains/IP ranges; block private IP ranges (10.x, 172.16-31.x, 192.168.x, 127.x)
Network segmentationRestrict server's outbound network access to only necessary destinations
Disable unused protocolsBlock file://, gopher://, dict:// — only allow https://
IMDSv2Cloud metadata service requiring session tokens (AWS); prevents simple SSRF metadata theft
SSRF is NEW in OWASP Top 10 (2021) at #10. It's particularly dangerous in cloud environments where cloud metadata APIs are accessible from the server. The Capital One breach (2019) was a famous SSRF attack exploiting AWS metadata.

6. Clickjacking & UI Redress

Clickjacking: An attacker loads a legitimate website in a transparent iframe overlaid on a malicious page. The victim thinks they're clicking on the attacker's page but actually clicks on the hidden legitimate site, performing unintended actions (changing settings, authorizing transactions).

Defenses

DefenseDescription
X-Frame-Options headerDENY (never framed) or SAMEORIGIN (only framed by same domain); prevents embedding in iframes
CSP frame-ancestorsModern replacement: Content-Security-Policy: frame-ancestors 'self'
Frame-busting scriptsJavaScript that detects if page is in iframe; legacy approach, less reliable than headers

7. Security Headers

HeaderPurposeProtects Against
Content-Security-Policy (CSP)Whitelist allowed sources of scripts, styles, images, fontsXSS, data injection, clickjacking (frame-ancestors)
Strict-Transport-Security (HSTS)Forces browser to use HTTPS for all future requests to domainSSL stripping, downgrade attacks
X-Frame-OptionsControls whether page can be embedded in iframeClickjacking
X-Content-Type-OptionsSet to nosniff — prevents MIME type sniffingMIME confusion attacks
Referrer-PolicyControls how much referrer information is sent with requestsInformation leakage via Referer header
Permissions-PolicyControls which browser features (camera, mic, geolocation) the page can useFeature abuse, privilege escalation

8. Web Application Firewalls (WAF)

WAF: A Layer 7 (application layer) firewall that inspects HTTP/HTTPS traffic for malicious patterns. Sits between clients and the web server. Can operate in positive model (whitelist) or negative model (blacklist/signature-based).
WAF DeploymentDescription
Inline/reverse proxyWAF intercepts all traffic; can block attacks in real-time; adds latency
Out-of-band/passiveWAF monitors copy of traffic; detection only (no blocking); no latency impact
Cloud-basedManaged WAF service (AWS WAF, Cloudflare, Akamai); scales automatically; subscription-based
WAF Limitations: A WAF is defense-in-depth, NOT a replacement for secure coding. It cannot protect against all attack variations (WAF bypass techniques exist). It cannot protect against business logic flaws. On CISSP, if asked for the BEST defense, secure coding practices (parameterized queries, output encoding) always beat WAF.

9. Memory Tricks & Mnemonics

🧠 Cookie Attributes — "SHSD" (She Has Secure Data)
Secure = HTTPS only
HttpOnly = Hide from JavaScript
SameSite = Stop CSRF
Domain/Path = Define scope
🧠 Session Fixation vs Hijacking
Fixation = Fixed BEFORE login (attacker pre-sets ID)
Hijacking = Hi-jacked AFTER login (attacker steals ID)
Fix → Regenerate on login; Hijack → Protect the cookie
🧠 Security Headers — "CHXRP" (Charlie Has eXtra Rad Protection)
CSP = XSS prevention
HSTS = Force HTTPS
X-Frame-Options = Clickjacking
Referrer-Policy = Info leakage
Permissions-Policy = Feature control
