2. Real-World Organizational Relevance

Google's OSS-Fuzz project has found over 10,000 vulnerabilities in open-source projects using continuous fuzzing. Microsoft requires fuzzing of all network-facing code (SDL — Security Development Lifecycle). Abuse case modeling is required by financial regulators for fraud detection systems. NATO CCDCOE uses attack surface analysis as a prerequisite for cyberdefense planning.

Failure modes: Testing only "happy paths" (normal usage) — fails to discover what happens when attackers send unexpected input; ignoring boundary conditions — a field accepting values 1-100 crashes when given 0, -1, or 101.

3. Misuse Case & Abuse Case Testing

Misuse Case: A use case written from the attacker's perspective — describes how a malicious actor (the "misactor") would try to damage, compromise, or abuse the system. Where use cases describe what the system SHOULD do, misuse cases describe what the system should PREVENT.
Abuse Case: Similar to misuse cases but focuses specifically on how legitimate features can be ABUSED beyond their intended purpose. Often used interchangeably with misuse cases, but technically an abuse case targets feature misuse by authorized users.

Exam Tip: Misuse cases are created during the REQUIREMENTS/DESIGN phase — they identify security requirements early. They are NOT a testing technique used after development — they drive the creation of security tests and security controls during design. The exam tests this timing distinction.

4. Fuzzing (Fuzz Testing)

Fuzzing: An automated testing technique that sends random, unexpected, malformed, or semi-valid data to program inputs to cause crashes, hangs, memory corruption, or unexpected behavior. Fuzzing is exceptionally effective at finding vulnerabilities that developers never anticipated — buffer overflows, format string bugs, injection flaws, assertion failures.

Fuzzing Types

TypeHow It WorksBest For
Mutation-Based (Dumb Fuzzing)Takes valid input samples and randomly modifies them (bit flips, byte insertions, field truncation, character replacement)Quick setup; effective against binary parsers, file format parsers (PDF, JPEG, XML)
Generation-Based (Smart/Intelligent Fuzzing)Creates inputs from scratch based on knowledge of the input format (protocol specification, file format grammar). Understands structure and generates semi-valid but edge-case dataProtocol testing (HTTP, DNS, TLS); complex file formats; higher code coverage
Coverage-Guided FuzzingInstruments the target binary and uses code coverage feedback to guide input generation toward unexplored code paths (AFL, libFuzzer)Maximum code coverage; finding deep bugs in complex codebases; state-of-the-art approach

Popular Fuzzing Tools

ToolTypeNotable Uses
AFL (American Fuzzy Lop)Coverage-guidedFound thousands of bugs in open-source projects; gold standard for binary fuzzing
libFuzzerCoverage-guided (in-process)Part of LLVM project; used by Google OSS-Fuzz for continuous fuzzing
Peach FuzzerGeneration-basedProtocol fuzzing; smart fuzzing with data models
BoofuzzGeneration-based (network)Successor to Sulley; network protocol fuzzing
What Fuzzing Finds: Buffer overflows, integer overflows, format string vulnerabilities, null pointer dereferences, use-after-free, assertion failures, infinite loops, memory leaks, denial of service conditions. Fuzzing is particularly effective at finding memory corruption bugs in C/C++ code.
⚠️ Fuzzing Limitation: Fuzzing finds CRASH-level bugs — it does NOT find logic flaws, authorization bypasses, or business rule violations. A system that doesn't crash but returns wrong data won't be caught by fuzzing alone.

5. Fault Injection

Fault Injection: Deliberately introducing faults (errors, failures, unexpected conditions) into a system to test how it handles them. The goal is to verify that the system fails gracefully — securely — rather than catastrophically. Tests error handling, failover, and resilience.

Types of Fault Injection

TypeMethodWhat It Tests
Hardware Fault InjectionDisconnecting storage, introducing power fluctuations, causing memory errorsHardware resilience; RAID failover; UPS switching; redundancy validation
Software Fault InjectionModifying code to introduce errors; corrupting data in memory; forcing API failures; injecting exceptionsError handling; exception handling; graceful degradation
Network Fault InjectionIntroducing latency, packet loss, connection drops, bandwidth throttlingNetwork resilience; timeout handling; retry logic; distributed system behavior

Chaos Engineering

Chaos Engineering: The practice of deliberately injecting failures into production systems to build confidence in system resilience. Pioneered by Netflix (Chaos Monkey — randomly terminates production instances). Extends fault injection to production environments in a controlled manner.
  • Netflix Chaos Monkey: Randomly kills production compute instances to verify auto-recovery
  • AWS Fault Injection Simulator: Managed service for running chaos experiments on AWS infrastructure
  • Gremlin: Commercial chaos engineering platform
Key Security Insight: Fault injection validates that the system fails SECURELY — meaning that when errors occur, the system does not: expose sensitive data in error messages, bypass authentication, leave resources in an insecure state, or crash in a way that allows exploitation.

6. Boundary Value Analysis & Equivalence Partitioning

Boundary Value Analysis (BVA)

Boundary Value Analysis: A testing technique that focuses test cases on the EDGES (boundaries) of input domains — the minimum, maximum, and values just inside/outside the boundaries. Based on the principle that bugs cluster at boundaries: off-by-one errors, buffer overflow at exact buffer size, integer overflow at max int.

For an input field accepting values 1-100:

Test ValueCategoryExpected
0Just below minimumRejected
1Minimum boundaryAccepted
2Just above minimumAccepted
50Nominal (middle)Accepted
99Just below maximumAccepted
100Maximum boundaryAccepted
101Just above maximumRejected

Equivalence Partitioning (EP)

Equivalence Partitioning: Divides input data into CLASSES (partitions) where all values in a partition are expected to be handled the same way. Test one representative value from each partition instead of testing every possible input. Reduces test count while maintaining coverage.

For the same 1-100 input field:

PartitionRangeRepresentative Test ValueExpected Behavior
Invalid (low)< 1-5Rejected with error message
Valid1–10050Accepted and processed
Invalid (high)> 100200Rejected with error message
BVA + EP Together: Best practice is to combine both — use equivalence partitioning to identify the input classes, then use boundary value analysis to focus on the edges of each class. Together they provide maximum coverage with minimum test cases.

7. Attack Surface Analysis

Attack Surface: The total number of points (attack vectors) where an unauthorized user can attempt to enter data or extract data from a system. The attack surface includes all entry points (APIs, web forms, network ports, physical interfaces) and all data flows. A smaller attack surface = lower risk.

Attack Surface Components

Surface TypeExamples
Network Attack SurfaceOpen ports, listening services, network protocols, VPN endpoints, load balancers
Software Attack SurfaceWeb forms, API endpoints, file upload functions, authentication interfaces, admin panels
Physical Attack SurfaceUSB ports, console access, badge readers, removable media drives, network jacks
Human Attack SurfaceEmployees susceptible to phishing, help desk social engineering, executive fraud targets

Attack Surface Reduction Strategies

  • Reduce entry points: Close unnecessary ports; disable unused services; remove unused software
  • Reduce access: Implement least privilege; restrict API access; require authentication everywhere
  • Reduce code exposure: Remove debug endpoints; disable directory listing; remove default accounts
  • Network segmentation: Isolate sensitive systems; micro-segmentation (Day 84 — Zero Trust)
Exam Tip: Attack surface analysis should be performed during the DESIGN phase and repeated after each significant change. The attack surface can only be reduced, not eliminated — the goal is to minimize it to the smallest possible footprint needed to deliver required functionality.

8. Memory Tricks & Mnemonics

🧠 Fuzzing Types — "MGC"
Mutation = Mutate existing valid inputs (dumb, easy setup)
Generation = Generate from format knowledge (smart, better coverage)
Coverage-guided = Code coverage feedback (state-of-the-art, AFL/libFuzzer)
🧠 BVA Boundaries — "Just Inside, Just Outside"
For range 1-100: test 0 (just outside), 1 (boundary), 100 (boundary), 101 (just outside)
Bugs live at the EDGES — off-by-one errors, overflow at max, underflow at min
🧠 Use Case vs. Misuse Case
Use case = "What SHOULD happen" (user story)
Misuse case = "What should NOT happen" (attacker story)
Both created in requirements phase — misuse cases drive security requirements
🧠 Attack Surface = "Doors + Windows + Chimneys"
Every entry point is a potential attack vector — the fewer openings, the harder to break in
