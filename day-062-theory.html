3. VPN Fundamentals

📖 VPN (Virtual Private Network): Creates an encrypted tunnel over a public (untrusted) network — typically the internet — so that two endpoints can communicate as if directly connected on a private network. The "virtual" private network emulates the security properties of a dedicated private leased line over shared infrastructure.
VPN PropertyHow AchievedPurpose
ConfidentialitySymmetric encryption of tunnel payload (AES-256, ChaCha20)Traffic cannot be read if intercepted on the public internet
IntegrityHMAC (HMAC-SHA256, HMAC-SHA512) on each packetTampering with tunnel packets is detected and rejected
AuthenticationPKI certificates, pre-shared keys (PSK), or user credentials (MFA) for IKE / TLS handshakeEnsures both ends of the tunnel are who they claim to be
Non-repudiationCertificate-based authentication provides non-repudiation of VPN connectionsAudit trail of who connected; supports forensic investigations

4. IPSec – Architecture & Protocols

📖 IPSec (Internet Protocol Security): A suite of protocols that provides security services at Layer 3 (Network). IPSec secures all IP traffic — not just specific applications — making it transparent to applications. It is defined by IETF RFCs 4301–4309 and is mandatory in IPv6 (required in the spec, though implementation is optional in practice).

AH vs. ESP

ProtocolFull NameProtocol #ProvidesDoes NOT Provide
AHAuthentication HeaderIP Protocol 51Data origin authentication; integrity (via HMAC over packet including IP header); anti-replay (sequence numbers)Confidentiality (NO encryption) — payload is transmitted in cleartext. AH provides integrity/auth only.
ESPEncapsulating Security PayloadIP Protocol 50Confidentiality (encryption of payload); integrity; data origin authentication; anti-replay. The complete solution.Does NOT authenticate the outer IP header (in tunnel mode) — only the encapsulated payload + ESP header
🔑 AH Problem with NAT: AH signs the entire IP packet including the source IP address. NAT changes the source IP header — this breaks the AH integrity check. AH is therefore incompatible with NAT. ESP in tunnel mode does not have this problem because the original IP header is encapsulated and the outer header (subject to NAT) is not authenticated. This is why ESP is almost always used in practice, not AH.
💡 Exam Tip — AH vs. ESP:
"Which IPSec protocol provides encryption?" → ESP (NOT AH)
"Which IPSec protocol only provides authentication/integrity without encryption?" → AH
"Which is incompatible with NAT?" → AH
"Which IPSec protocol should you always use for a VPN?" → ESP (provides everything AH provides plus encryption)

6. IKE – Internet Key Exchange

📖 IKE (Internet Key Exchange): The protocol used to negotiate, establish, and manage Security Associations (SAs) for IPSec. IKE runs on UDP port 500 (and UDP 4500 when NAT traversal is needed). Without IKE, keys would have to be manually configured on every VPN endpoint — IKE automates this securely.

IKE Phase 1 vs. Phase 2

PhasePurposeOutput
Phase 1 (IKE SA)Establishes a secure authenticated channel between the two VPN peers using Diffie-Hellman key exchange + mutual authentication (certificates or PSK). Protects Phase 2 negotiation.IKE SA (bidirectional) — a secure management tunnel
Phase 2 (IPSec SA)Uses the Phase 1 tunnel to negotiate IPSec SA parameters (encryption algorithm, HMAC, SA lifetime, protocol: AH or ESP) for actual data traffic. Creates two unidirectional SAs (one per direction).Two IPSec SAs — one inbound, one outbound for data traffic

IKEv1 vs. IKEv2

AttributeIKEv1IKEv2
RFCRFC 2409 (1998)RFC 7296 (2014)
Phase 1 modesMain mode (6 messages) or Aggressive mode (3 messages — insecure)Single IKE_SA_INIT + IKE_AUTH exchange — simpler and faster
Aggressive modeSends identity before encryption — vulnerable to off-line dictionary attacks on PSKRemoved — IKEv2 has no aggressive mode
NAT traversalRequires extension (RFC 3947)Built in natively
MOBIKENot supportedSupports MOBIKE (RFC 4555) — allows endpoint IP changes (mobile devices, roaming)
RecommendationLegacy — IKEv1 aggressive mode is a known vulnerabilityPreferred — more secure, efficient, and mobile-friendly
⚠️ IKEv1 Aggressive Mode Security Risk: In aggressive mode, the initiator sends its identity hash before a secure channel is established. An attacker can capture this hash and perform an offline dictionary attack against the pre-shared key. Any organization still using IKEv1 aggressive mode with PSK authentication should migrate to IKEv2 with certificate-based authentication immediately.

7. SSL/TLS VPN

📖 SSL/TLS VPN: A VPN that uses the TLS protocol (commonly referred to as "SSL VPN" despite TLS being the current standard) to create an encrypted tunnel. Operates at Layer 5–7, typically over TCP 443. Because HTTPS is universally allowed through firewalls, SSL VPNs work where IPSec VPNs are blocked.
AttributeIPSec VPNSSL/TLS VPN
LayerLayer 3 (Network) — transparent to all applicationsLayer 5–7 — application-aware; may require client software or browser
Firewall traversalRequires UDP 500, UDP 4500, ESP (IP protocol 50) — often blockedTCP 443 (HTTPS) — universally permitted; works through NAT and restrictive firewalls
Client requirementDedicated VPN client software required on every deviceClientless option: browser-based for web apps. Full tunnel: thin client app required for all traffic.
Traffic scopeFull network-level tunnel — all IP traffic routed through VPNClientless: only browser sessions; Full tunnel: all traffic (same as IPSec)
Use caseSite-to-site VPNs; environments with full control over endpointRemote access from unmanaged devices, hotels, public Wi-Fi; clientless access to specific web apps
AuthenticationPSK, certificates (IKE)Username/password, MFA, client certificates — integrated with web-based authentication flows
SecurityBoth are secure when properly configured. IPSec uses IKE for key management.TLS 1.3 provides strong security. Clientless mode limits attack surface. Full tunnel equivalent to IPSec security level.

8. Legacy VPN Protocols (L2TP, PPTP)

ProtocolLayerSecurityStatus
PPTP
(Point-to-Point Tunneling Protocol)		L2Broken — do not use. Uses MS-CHAPv2 for authentication, which has a critical vulnerability allowing offline dictionary attacks against captured handshakes. Encryption via MPPE (RC4) is weak. NSA has been reported to decrypt PPTP in bulk. Port TCP 1723.Deprecated; must never be selected as "secure" on CISSP exam
L2TP
(Layer 2 Tunneling Protocol)		L2L2TP provides tunneling only, no encryption. It is always paired with IPSec (L2TP/IPSec) for security. L2TP/IPSec uses UDP 1701 (L2TP) + UDP 500/4500 (IKE). Without IPSec, L2TP is completely insecure.Largely superseded by IKEv2/IPSec; still common on older VPN infrastructure
⚠️ PPTP on the CISSP Exam: PPTP is the "wrong answer" whenever a question asks about secure remote access. It appears on the exam specifically as a distractor — candidates who haven't studied recognize it as a "VPN protocol" and may select it. Any question mentioning PPTP as a good security choice is incorrect. The correct answer will be IPSec, SSL/TLS VPN, or IKEv2.

10. Split Tunneling & Always-On VPN

📖 Split Tunneling: A VPN configuration where only traffic destined for the corporate network is routed through the encrypted VPN tunnel, while all other traffic (internet browsing, streaming) goes directly to the internet via the local gateway — bypassing the organization's security controls.
ConfigurationHow It WorksSecurity Implications
Full Tunneling (No Split)ALL traffic from the remote device is routed through the corporate VPN tunnel, including internet trafficMost secure — all traffic subject to corporate DLP, web filtering, IDS/IPS, and proxy policies. Higher bandwidth load on VPN gateway. Higher latency for remote users' internet browsing.
Split Tunneling (Enabled)Only traffic for corporate subnets uses the VPN; internet traffic goes direct to local gatewaySecurity risk — if remote device is compromised (malware), the attacker has a foothold on the corporate network without restrictions. Internet malware C2 traffic bypasses corporate controls. Better performance and lower gateway load.
Inverse SplitAll traffic through VPN except specified destniations (e.g., Microsoft 365, CDN traffic direct)Hybrid approach — protects most traffic; specific trusted cloud services excluded for performance
📖 Always-On VPN: VPN client automatically establishes the tunnel before user login and maintains it continuously. The device never operates "off-VPN" regardless of network location. Used to ensure corporate security policies apply even when users forget to connect, and to protect devices that may be unattended.
💡 Exam Tip — Split Tunneling Security: From a CISSP security perspective, split tunneling is a risk. It allows remote devices to access the internet without corporate visibility — a compromised device can introduce malware into the corporate network through the VPN tunnel while the malware's C2 communication goes undetected over the direct internet path. The secure default is full tunneling or inverse split with careful exclusions.

11. Memory Tricks & Mnemonics

🧠 AH vs. ESP — "AH proves who; ESP hides what":
AH = Authentication Header = proves authenticity (integrity + auth) but NO encryption
ESP = Encapsulating Security Payload = encrypts AND authenticates (the full package)
"AH is the ID check at the door. ESP is the ID check AND the private room."
🧠 Tunnel vs. Transport Mode — "Tunnel buries the body":
Tunnel mode = buries the entire original packet (including original IP header) inside a new encrypted wrapper. Used for site-to-site VPN. Original IP addresses hidden.
Transport mode = only the payload is encrypted; original IP header stays exposed. Used for host-to-host.
🧠 PPTP = "Pretty Pathetically Terrible Protocol":
Never select PPTP for "secure remote access." MS-CHAPv2 + MPPE RC4 = broken. Always choose IPSec or SSL/TLS VPN.
L2TP alone = no security. L2TP + IPSec = secured tunneling.
🧠 IKE Phases — "Phase 1: Safe room; Phase 2: Business deal":
Phase 1 = Build a secure, authenticated channel between peers (IKE SA)
Phase 2 = Inside that safe channel, negotiate the actual IPSec parameters (IPSec SA)
IKEv2 is always preferred; IKEv1 aggressive mode = offline PSK attack risk.
