🔑 SMTP Protocol Details for CISSP:
The email envelope (SMTP protocol fields: MAIL FROM, RCPT TO) is separate from the email header/body (the RFC 5322 message: From:, To:, Subject:, Date:). This distinction is fundamental to understanding SPF, DKIM, and DMARC. SPF and DKIM check the envelope or header — DMARC aligns them both to the From: header domain.

4. S/MIME – Certificate-Based Email Security

📖 S/MIME (Secure/Multipurpose Internet Mail Extensions): An IETF standard for end-to-end email encryption and digital signing using X.509 certificates and Public Key Infrastructure (PKI). Provides confidentiality, integrity, authentication, and non-repudiation at the message level — the message content is protected regardless of how many mail servers relay it.
S/MIME CapabilityHow It WorksCIA Mapping
Digital SignatureSender signs the message with their private key. Recipient verifies with sender's public key (from sender's certificate). Proves identity and detects any modification after signing.Integrity + Authentication + Non-repudiation
EncryptionSender encrypts a random symmetric session key with the recipient's public key. Message body encrypted with the symmetric key. Recipient decrypts session key with their private key and then decrypts message.Confidentiality
CombinedMost secure use: sign first, then encrypt. The signature is protected inside the encryption layer. On receipt: decrypt then verify signature.CIA + Non-repudiation

S/MIME Certificate Requirements

RequirementDetail
Certificate typeX.509 email certificate (S/MIME certificate) issued by a trusted CA. Contains the user's name, email address, and public key.
Trust anchorBoth sender and recipient must trust the issuing CA (root/intermediate CA in their trust store). Enterprise deployments typically use an internal CA for S/MIME certificates.
Certificate exchangeTo send an encrypted email to Bob, Alice needs Bob's public key certificate. Bob must first send Alice a signed email (which contains his public key) — or Alice fetches it from a directory (LDAP, public key server).
Non-repudiationBecause the private key is known only to the signer, a digitally signed S/MIME message cannot be repudiated — the signer cannot later deny having sent it, provided key management is sound.
💡 Exam Tip — S/MIME vs. TLS: TLS protects email in transit between mail servers (hop-by-hop). S/MIME provides end-to-end protection — the message is encrypted/signed before leaving the sender's client and remains protected until decrypted/verified by the recipient's client. If non-repudiation is asked about in an email context, the answer is S/MIME digital signing, not TLS.

5. PGP and GPG – Web of Trust

📖 PGP (Pretty Good Privacy) / GPG (GNU Privacy Guard): A standard for email encryption and signing that uses public-key cryptography but does NOT rely on a central Certificate Authority. Instead uses a "Web of Trust" model where individuals vouch for each other's public keys. OpenPGP is the IETF standard (RFC 4880); GPG is the free open-source implementation.
AspectPGP/GPGS/MIME
Trust modelWeb of Trust — individuals sign each other's public keys; trust is transitive through signers you trustHierarchical PKI — trust flows from root CA → intermediate CA → end-entity certificate
Certificate formatOpenPGP key format (.asc, .gpg files)X.509 certificates (.pem, .p12 files)
CA requirementNone — no central authority requiredRequires trusted CA infrastructure
Enterprise useLess common in corporate environments; dominant in open-source communities, journalists, activistsMore common in enterprise (integrates with Active Directory CA)
Key distributionPublic keyservers (keys.openpgp.org), key signing parties, direct exchangeLDAP directories, O365/Exchange GAL, trusted CA certificate transparency
OperationsSame as S/MIME: sign (private key), verify (sender's public key), encrypt (recipient's public key), decrypt (own private key)Same operations
🔑 Web of Trust Concept: Alice trusts Charlie. Charlie has signed Bob's key (vouching for Bob's identity). Therefore Alice can transitively trust Bob's key. However if Alice trusts no one who has signed Bob's key, she cannot verify Bob's identity — there is no central authority to fall back on. This is the core difference from PKI.

6. SPF – Sender Policy Framework

📖 SPF (Sender Policy Framework) — RFC 7208: A DNS-based mechanism that lets a domain owner specify which IP addresses and mail servers are authorized to send email on behalf of their domain. The receiving mail server checks the SPF record to verify whether the sending server is authorized for the domain in the SMTP MAIL FROM field.
SPF Check Flow: Inbound email arrives at recipient MTA SMTP envelope: MAIL FROM: <accounting@example.com> Sending server IP: 203.0.113.10 Recipient MTA queries DNS: DNS query: TXT record for example.com → "v=spf1 ip4:203.0.113.0/24 include:sendgrid.net -all" SPF Evaluation: 203.0.113.10 matches ip4:203.0.113.0/24 → PASS (authorized sender) If IP was 198.51.100.5 → FAIL (not in SPF record; -all = hard fail → reject/mark spam)
SPF QualifierMeaningAction on Match
+ (default)Pass — authorizedAccept message
~Soft Fail — probably unauthorizedAccept but mark as probable spam
-Hard Fail — unauthorizedReject message
?Neutral — no policy statementAccept (no penalty)
⚠️ SPF Limitation — The Header From: Gap: SPF only validates the SMTP envelope sender (MAIL FROM), not the From: header that users see in their email client. An attacker can set a legitimate MAIL FROM that passes SPF while putting a spoofed address in the user-visible From: header. This is why DMARC is required — it mandates alignment between the SPF/DKIM domain and the From: header.

7. DKIM – DomainKeys Identified Mail

📖 DKIM (DomainKeys Identified Mail) — RFC 6376: Allows the sending mail server to cryptographically sign outbound emails using a private key. The corresponding public key is published in the sending domain's DNS. Receiving servers verify the signature, proving message integrity and that it originated from the legitimate domain's mail infrastructure.
DKIM Signing and Verification: Sending Server (example.com): 1. Compute hash of specified email headers + body 2. Sign hash with DKIM private key 3. Add DKIM-Signature: header to the email DKIM-Signature: v=1; a=rsa-sha256; d=example.com; s=selector1; h=From:To:Subject:Date; bh=<body-hash>; b=<signature> Receiving Server: 1. Read DKIM-Signature header; extract domain (d=) and selector (s=) 2. DNS query: selector1._domainkey.example.com → TXT record (public key) 3. Verify signature using public key PASS: message not tampered with; originated from authorized signing server FAIL: signature invalid → tampered in transit, or forged domain
DKIM AspectDetail
What it protectsIntegrity of specified headers and message body. Proves message was sent by a server controlling the private key for the claimed domain.
What it does NOT protectDoes not check IP address (unlike SPF). Does not prevent replay of a signed message to different recipients (unless cc/bcc headers included in signed set). A DKIM-signed message from evil.com cannot impersonate example.com's DKIM.
Key rotationSelectors (e.g., "selector1", "selector2") allow multiple keys and key rotation without disrupting existing signatures. Old keys can be deactivated by removing or changing the DNS TXT record.
Survives forwarding?DKIM signature survives email forwarding (body/header unchanged). SPF typically breaks on forwarding (forwarding server's IP not in original domain's SPF). This is why DMARC can pass on DKIM alignment even when SPF fails after forwarding.

8. DMARC – Domain-Based Message Authentication

📖 DMARC (Domain-based Message Authentication, Reporting and Conformance) — RFC 7489: A policy published in DNS by the domain owner that instructs receiving mail servers what to do if an email fails SPF and DKIM alignment checks. DMARC ensures that the domain in the user-visible From: header aligns with the domain verified by SPF (envelope sender) or DKIM (signature domain). It also provides aggregate and forensic reporting back to the domain owner about authentication results.

DMARC alignment requirement

🔑 Alignment: DMARC requires that at least one of these aligns with the RFC5322.From domain (the header users see):
1. SPF alignment: MAIL FROM domain matches From: domain (exact or relaxed)
2. DKIM alignment: DKIM d= domain matches From: domain (exact or relaxed)

If neither aligns → DMARC failure → policy applied (none / quarantine / reject)
DMARC Policy (p=)Action on Failure
p=noneMonitor only — no enforcement action. Emails are delivered even if DMARC fails. Aggregated reports still sent to rua= address. Used during initial rollout to assess impact before enforcement.
p=quarantineMessages failing DMARC are quarantined — typically delivered to junk/spam folder. Partial enforcement.
p=rejectMessages failing DMARC are rejected at the receiving MTA — never delivered. Full enforcement. The target state for any mature email security posture.
DMARC ReportingDescription
rua= (aggregate reports)Daily XML reports from receiving servers listing all messages claiming to be from your domain, with SPF/DKIM pass/fail counts and authentication results. Crucial for monitoring unauthorized use of your domain.
ruf= (forensic reports)Per-message failure reports (contain full or partial message headers). Reveals exact attempts to spoof your domain.

9. SPF + DKIM + DMARC Working Together

Layered Email Authentication – How the Three Work Together: Legitimate email from ceo@example.com: SPF: MAIL FROM: ceo@example.com → sending IP in example.com's SPF record → PASS DKIM: DKIM-Signature d=example.com → public key verified → PASS DMARC: From: example.com aligns with both SPF (example.com) and DKIM (example.com) → PASS Policy: p=reject → message accepted and delivered ✓ Spoofed/phishing email claiming to be from ceo@example.com: Sent from attacker's server (not in example.com SPF): SPF: MAIL FROM: ceo@example.com → attacker IP not in SPF → FAIL DKIM: No DKIM-Signature, or signed by attacker.com (not example.com) → FAIL/No Alignment DMARC: From: example.com but neither SPF nor DKIM aligns → FAIL Policy: p=reject → message REJECTED ✗ Forwarded email (legitimate, sent through forwarder): SPF: MAIL FROM: → forwarder's IP (not in original SPF) → FAIL (SPF breaks) DKIM: Signature from original sender intact → PASS (DKIM survives forwarding) DMARC: DKIM alignment passes (d= matches From:) → DMARC PASS ✓ Message delivered (DKIM alignment saved it despite SPF failure)
💡 Exam Tip — Complementary Functions: SPF, DKIM, and DMARC are complementary; none is sufficient alone: SPF alone doesn't protect the visible From: header. DKIM alone provides integrity but no policy enforcement. DMARC without SPF or DKIM to align would always fail. Together they provide a complete email authentication framework. DMARC is the enforcement policy engine that makes the others actionable.

10. Email Gateway Security

📖 Secure Email Gateway (SEG): A dedicated appliance or cloud service that sits in the email delivery path (via MX record pointing to the SEG) and applies a layered set of security controls to inbound and outbound email before delivery to the mail server. Examples: Proofpoint, Mimecast, Microsoft Defender for Office 365.
SEG CapabilityDescription
Anti-spamContent analysis, header analysis, keyword filters, reputation-based blocking (IP reputation, domain reputation). Reduces spam volume before it reaches users' inboxes.
Anti-malwareSignature-based and heuristic scanning of attachments (PDF, Office documents, executables). Strips or quarantines malicious attachments.
SandboxingExecute suspicious attachments in an isolated virtual environment before delivery. Detonates the attachment and observes behavior (file writes, network connections, registry changes). Detects zero-day and polymorphic malware that evades signature scanning.
URL rewritingReplaces URLs in emails with proxied versions that are scanned at click-time (time-of-click analysis). Catches URLs that become malicious after delivery (retroactive protection).
SPF/DKIM/DMARC enforcementSEG validates SPF, DKIM, and DMARC on all inbound email and applies configured policies. Often provides more configurable enforcement than native mail server capabilities.
DLP (Data Loss Prevention)Scan outbound email for sensitive data patterns (PII, credit card numbers, medical data). Block or encrypt based on policy. Prevents accidental or malicious data exfiltration via email.
TLS enforcementEnforce opportunistic or mandatory TLS for specific partner domains. Provides in-transit encryption (hop-by-hop, not end-to-end).

12. Memory Tricks & Mnemonics

🧠 SPF, DKIM, DMARC — "Sends, Digitally signs, Defines policy":
SPF = who is authorized to Send email for a domain (IP allowlist in DNS)
DKIM = Digitally signs the message (private key signs; public key in DNS verifies)
DMARC = Defines the policy if SPF/DKIM don't align with From: header (none/quarantine/reject)
All three are DNS TXT records. DMARC is the policy enforcer that ties SPF and DKIM together.
🧠 S/MIME Encryption Direction — "Lock with their key, open with yours":
Encrypt to send: use recipient's public key to lock → only recipient's private key can open.
Sign to prove identity: use your private key to sign → anyone with your public key can verify.
PGP/GPG: same cryptographic operations, different trust model (Web of Trust vs. PKI hierarchy).
🧠 Email Protocol Ports — "25 relay, 587 submit, 993 IMAP secure, 995 POP secure":
25 = MTA-to-MTA relay (block at ISP for end users)
587 = authenticated client submission (STARTTLS)
143 = IMAP plain | 993 = IMAP SSL/TLS
110 = POP3 plain | 995 = POP3 SSL/TLS
🧠 DMARC Policy Progression — "None, Quarantine, Reject" = "Monitor, Warn, Block":
Start with p=none (monitor only; read rua= reports; find all legitimate senders)
Move to p=quarantine (failures go to spam; limited impact while tuning)
End goal: p=reject (spoofed emails blocked completely at receiving MTA)
