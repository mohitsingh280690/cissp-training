1. Where This Fits in CISSP

AspectDetail
Domain4 – Communication and Network Security (13%)
Why TestedCISSP expects candidates to know which protocols provide confidentiality, integrity, and authentication at each layer—and to distinguish between protocols that are secure by design, upgraded versions (TLS 1.3 vs. 1.2), and deprecated protocols (SSLv3, TLS 1.0/1.1). The exam frequently presents a protocol description and asks which is most secure, or presents a network scenario requiring a secure replacement for an insecure protocol.
Typical Question Styles"Which protocol provides in-transit encryption for web traffic?"; "An admin needs to securely manage a remote server via command line — which protocol?"; "Which secure file-transfer protocol encapsulates FTP inside SSH?"; "What replaces RTP to provide confidentiality for VoIP media streams?"
Cross-Domain LinksDay 35 (TLS uses symmetric + asymmetric crypto); Day 48 (RSA/ECC used in TLS handshake); Day 29–32 (PKI provides TLS certificates); Day 62 (VPN uses TLS or IPSec for tunneling); Day 65 (SMTP over TLS)
Key Exam TrapsSFTP ≠ FTPS. SFTP is FTP-like functionality tunneled over SSH (port 22). FTPS is FTP with TLS/SSL (ports 989/990). SSH replaces Telnet, rlogin, rsh — all unencrypted. TLS 1.3 removed RSA key exchange — all key exchanges now use ephemeral Diffie-Hellman (forward secrecy mandatory). HTTPS on its own is only as good as the TLS certificate validation; HSTS prevents downgrade.

2. Real-World Organizational Relevance

IssueOrganizational Context
TLS everywhere mandate"HTTPS by default" is now a baseline security requirement. Browsers mark HTTP sites as "Not Secure." PCI-DSS, HIPAA, and most frameworks require encryption in transit. TLS 1.0 and 1.1 are deprecated — modern compliance requires TLS 1.2+ (TLS 1.3 preferred).
SSH key managementOrganizations often accumulate thousands of unmanaged SSH keys (authorized_keys entries). Unmanaged keys represent persistent unauthorized access paths. Privileged Access Management (PAM) tools and SSH certificate authorities address this at scale.
VoIP security gapMany organizations deploy VoIP (Cisco UCM, Microsoft Teams Phone) but neglect to enable SRTP for media encryption. Eavesdropping on VoIP traffic on internal networks is trivial without SRTP — call recordings are plaintext.
Secure messagingEnd-to-end encrypted messaging (Signal, WhatsApp E2E, iMessage) has become standard for sensitive communications. Organizations increasingly allow or require Signal for incident response communications, especially during breach response where email infrastructure may be compromised.

3. TLS 1.3 – Transport Layer Security

📖 TLS (Transport Layer Security): The dominant protocol for securing point-to-point communications over the internet. Provides confidentiality (encryption), integrity (MAC/AEAD), and server authentication (PKI certificate). TLS 1.3 (RFC 8446, 2018) is the current version — significantly simplified and hardened versus TLS 1.2.

TLS 1.3 Handshake (Simplified)

TLS 1.3 Handshake — 1-RTT (1 Round-Trip): Client Server │── ClientHello ─────────────────────────►│ │ (TLS version, cipher suites, │ │ client key_share [ECDHE]) │ │ │ │◄── ServerHello ─────────────────────────│ │ (selected cipher suite, │ │ server key_share [ECDHE], │ │ Certificate + CertificateVerify, │ │ Finished [all messages MACed]) │ │ │ │── Finished ─────────────────────────────►│ │ (client MAC over all handshake msgs) │ │ │ │◄══════ Application Data (encrypted) ════│ [Session established — 1 RTT]
FeatureTLS 1.2TLS 1.3
Handshake RTT2 RTT (more round trips)1 RTT (faster)
Key exchangeRSA (static), DHE, ECDHEECDHE or DHE only — RSA key exchange removed
Forward secrecyOptional (only with DHE/ECDHE cipher suites)Mandatory (all key exchange uses ephemeral DH)
Cipher suitesMany, including weak options (RC4, 3DES, NULL)Only 5 AEAD cipher suites (AES-GCM, ChaCha20-Poly1305)
Encryption scopePost-handshake only; much of handshake is plaintextMost of handshake is encrypted (certificate now encrypted)
Removed featuresN/ARemoved: RSA key exchange, static DH, RC4, 3DES, MD5, SHA-1, compression, renegotiation
0-RTT resumptionSession resumption (session ID / ticket)0-RTT ("early data") for session resumption — replay risk acknowledged
🔑 Forward Secrecy: Even if the server's long-term private key is compromised later, past session recordings cannot be decrypted. Requires ephemeral key exchange (DHE/ECDHE) — a new temporary key pair is generated for each session and discarded after. TLS 1.3 makes this mandatory. TLS 1.2 with RSA key exchange does NOT provide forward secrecy.

4. HTTPS – HTTP over TLS

AspectDetail
What it isHTTP carried inside a TLS tunnel. Server certificate authenticates the server's identity. All HTTP content (headers, body, URL path and query string) is encrypted. Port 443 (plaintext HTTP uses port 80).
Certificate typesDV (Domain Validated): CA verifies domain ownership only — fast and cheap; browser padlock shown. OV (Organization Validated): CA verifies organization legal existence. EV (Extended Validation): Rigorous identity verification; historically showed green bar (browsers removed EV UI in 2019). All types encrypt equally well.
HSTSHTTP Strict Transport Security — server instructs browser to always use HTTPS for the domain for a set max-age (e.g., 1 year). Prevents SSL stripping attacks. HSTS preloading bakes the policy into browsers' hardcoded list — enforced even on first visit.
Certificate Transparency (CT)All publicly-trusted TLS certificates must be logged in public CT logs. Enables detection of unauthorized certificates issued for a domain. Used with CAA DNS records (limits which CAs may issue for your domain).
Certificate pinningApplication hardcodes the expected certificate or public key — rejects connections presenting any other certificate, even if signed by a trusted CA. Prevents CA compromise / MitM via rogue CA. Difficult to maintain (pin rotation complexity).

5. SSH – Secure Shell

📖 SSH (Secure Shell — RFC 4251-4254): A cryptographic network protocol for secure remote access (terminal/shell), remote command execution, secure file transfer (SFTP/SCP), and port forwarding (tunneling). Replaces plaintext equivalents: Telnet (23), rlogin (513), rsh (514). SSH uses port 22 by default.
SSH ComponentFunction
SSH Transport LayerProvides server authentication, encryption, and integrity. Key exchange (typically ECDH or Diffie-Hellman), then AES-GCM or ChaCha20-Poly1305 for data.
SSH User AuthenticationAuthenticates the client to the server. Methods: password (weak — brute-forceable), public key (preferred — client proves possession of private key corresponding to an authorized_keys entry), GSSAPI/Kerberos.
SSH Connection ProtocolMultiplexes the encrypted transport into multiple logical channels: interactive shell sessions, tunneled TCP connections, SFTP subsystem, X11 forwarding.

SSH Public Key Authentication Flow

Client setup: ssh-keygen → generates id_rsa (private) + id_rsa.pub (public) Copy id_rsa.pub to server's ~/.ssh/authorized_keys Connection: Client ─── "I want to authenticate as user 'alice' using key fingerprint X" ──► Server Server: "Is fingerprint X in alice's authorized_keys?" → YES Server ─── "Prove it: here is a challenge encrypted with your public key" ──────► Client Client: Decrypt challenge with private key; sign with private key ──────────────► Server Server: Verify signature with stored public key → Auth GRANTED
SSH Security HardeningDetail
Disable password authUse public key only (PasswordAuthentication no in sshd_config)
Disable root loginPermitRootLogin no — force users to log in as normal user then sudo
Change default portMove SSH off port 22 — reduces automated scanning noise (not a true security control, security through obscurity)
Key managementAudit authorized_keys entries; use SSH certificate authorities for scalable key management; rotate old keys
SSH tunneling cautionSSH can tunnel arbitrary TCP traffic — can be used to bypass firewall controls. Restrict with AllowTcpForwarding no for non-admin users.

6. Secure File Transfer Protocols

⚠️ Classic Exam Trap — SFTP vs FTPS vs SCP: These three are commonly confused. SFTP and SCP both run over SSH (port 22 only). FTPS runs FTP with TLS (different ports, different protocol). Plain FTP (port 21) and Telnet (port 23) are unencrypted and should never be used.
ProtocolFull NameUnderlying ProtocolPort(s)Notes
SFTPSSH File Transfer ProtocolSSH subsystem22Full-featured file management (list, delete, rename, download, upload) over SSH. No separate control/data channels. Most used secure file transfer protocol today.
SCPSecure Copy ProtocolSSH22Simple one-way file copy over SSH. No directory listing or management. Faster than SFTP for simple copies. Being replaced by SFTP in modern tools.
FTPSFTP Secure (FTP over TLS)FTP + TLS990 (implicit), 21+STARTTLS (explicit)Standard FTP protocol with TLS encryption layer. Uses two channels (control + data) — NAT-unfriendly. Explicit FTPS (AUTH TLS command on port 21) vs. Implicit FTPS (TLS from session start on port 990).
FTPFile Transfer ProtocolTCP21 (control), 20 (data)PLAINTEXT — credentials and data sent unencrypted. Never use over untrusted networks. Should be disabled on all production systems.

7. Voice Security: VoIP and SIP

📖 VoIP (Voice over IP): Transmission of voice communications as digital packets over IP networks. Enables telephone services over the internet or corporate IP networks. Introduces security challenges absent from traditional PSTN telephony.
ProtocolRoleSecurity Concern
SIP (Session Initiation Protocol)Signaling: sets up, modifies, tears down voice/video sessions. Port 5060 (UDP/TCP), 5061 (TLS). Analogous to the "dialing" function.Unencrypted SIP reveals who is calling whom (metadata). SIP can be exploited for toll fraud (unauthorized calls), registration hijacking (impersonate a user), DoS (INVITE flood). Secure SIP uses TLS (SIPS — port 5061).
RTP (Real-Time Transport Protocol)Carries actual voice/video media stream. UDP-based (low latency). No built-in security.RTP streams are plaintext — voice calls can be intercepted and reassembled from captured packets using Wireshark + VoIP call flow tools. SRTP replaces RTP to add encryption.
H.323Older VoIP signaling standard (ITU-T). Largely replaced by SIP in enterprise.Complex protocol suite; harder to firewall. Limited modern support.

VoIP Security Attacks

AttackDescription
EavesdroppingCapture RTP packets on LAN (via ARP poisoning or switch SPAN port) and reassemble call audio. Wireshark has a built-in "Telephony → RTP → Play Streams" feature. Mitigated by SRTP.
Toll FraudExploit SIP registration weaknesses to make calls billed to the victim organization. Common with misconfigured SIP trunks. Can result in five-figure phone bills overnight.
SPIT (Spam over Internet Telephony)VoIP equivalent of email spam — automated calls to VoIP endpoints.
VLAN-hopping for VoIPVoIP phones are typically on a dedicated voice VLAN. Switch misconfigurations can allow attacker on data VLAN to access voice VLAN and intercept calls.

8. SRTP – Secure Real-Time Transport Protocol

📖 SRTP (Secure Real-Time Transport Protocol — RFC 3711): An extension of RTP that adds encryption, message authentication, and replay protection to VoIP and multimedia streams. Provides confidentiality for voice/video media. Only encrypts the media (payload). SIP signaling must be separately secured with TLS.
SRTP FeatureDetail
EncryptionAES in Counter Mode (AES-CTR) or AES-f8 mode. Encrypts the RTP payload (voice data). RTP header remains unencrypted for routing.
AuthenticationHMAC-SHA1 message authentication code on each packet — detects packet tampering.
Replay protectionSequence numbers with a sliding window — replayed packets rejected.
Key managementSRTP keys are typically derived via ZRTP (peer-to-peer) or negotiated via SDP security descriptions (SDES) within SIP signaling (requires SIP/TLS to protect key exchange).
What SRTP does NOT protectCall metadata (who called whom, when, duration) — visible in SIP headers. Requires TLS-protected SIP for metadata confidentiality.

9. Secure Messaging – Signal Protocol

📖 Signal Protocol (formerly Axolotl): A cryptographic protocol developed by Open Whisper Systems providing end-to-end encryption for instant messaging with forward secrecy and break-in recovery. Used by Signal, WhatsApp, Facebook Messenger (Secret Conversations), and others.
Signal Protocol FeatureHow It Works
End-to-End Encryption (E2EE)Only sender and recipient can decrypt messages. The messaging server sees only ciphertext — cannot read messages. This contrasts with server-side encryption where the server can access plaintext.
Double Ratchet AlgorithmCombines a Diffie-Hellman ratchet (forward secrecy — new DH key per session) and a symmetric-key ratchet (new message key per message). Provides per-message forward secrecy AND break-in recovery (if a key is compromised, past and future messages remain secure).
Extended Triple Diffie-Hellman (X3DH)Asynchronous key agreement for establishing initial session keys even when one party is offline. Uses pre-published one-time prekeys (signed by the recipient). Enables asynchronous secure messaging.
Forward secrecyCompromise of today's message key does not reveal past messages — past keys are erased after use.
Break-in recovery (future secrecy)Even if an attacker obtains current session state, future messages become secure again after sufficient ratchet steps.
CISSP relevanceUnderstand E2EE vs. transit encryption and server-side encryption distinction. E2EE means only communicating parties can decrypt — not the service provider. This creates lawful intercept challenges but provides maximum confidentiality.

💡 Pattern: Nearly every plaintext protocol has a TLS-secured version. The pattern is: wrap in TLS (often add S suffix), assign a new well-known port. STARTTLS is the alternative approach — upgrade an existing plaintext connection to TLS mid-session using an explicit command (used by SMTP/IMAP/POP3).

11. Memory Tricks & Mnemonics

🧠 SFTP vs FTPS — "FTP over SSH vs FTP over SSL":
SFTP = SSH-based file transfer (NOT Standard FTP) → port 22 only, no data channel
FTPS = FTP + S(ecure) = FTP protocol with TLS overlay → ports 21 (explicit) / 990 (implicit)
SCP = "Secure CoPy" over SSH → port 22, simple one-way copy, no directory listing
🧠 TLS 1.3 Key Changes — "No RSA, All Forward":
TLS 1.3 removed RSA key exchange entirely → every session now uses ephemeral DH → forward secrecy mandatory
Also removed: 3DES, RC4, SHA-1, MD5, compression, custom DH groups
Cipher suites reduced to 5 AEAD-only options (AES-GCM, ChaCha20-Poly1305)
🧠 VoIP Security — "SIP signals, RTP delivers, SRTP secures":
SIP = Session setup/teardown (signaling). Secure with TLS → SIPS on port 5061
RTP = Real-time media stream (voice packets). Plaintext by default
SRTP = RTP + encryption (AES-CTR) + auth (HMAC-SHA1). Replaces RTP for secure calls
Both SIP/TLS and SRTP required for complete VoIP confidentiality.
