2. Real-World Organizational Relevance

TopicReal-World Context
ACLs everywhereNetwork ACLs (router/switch permit/deny rules), filesystem ACLs (Windows NTFS, Linux setfacl), cloud bucket policies (S3 bucket ACL), database object-level permissions — all are ACL implementations.
Capability tables in OS kernelsUnix process file descriptors are a capability implementation — the descriptor IS the capability token granting access rights to a file without the process needing to know the full path. Some OS security systems (seL4) build entirely on capabilities.
Database views as data firewallExposing only specific rows/columns of a table to an application or user via a view is a commonplace data access control technique. Applications use views instead of direct table access to enforce column-level security and row-level security without application-layer filtering.
Restricted interfaces in web appsHiding menu items, disabling buttons, and restricting API endpoints based on user role — all restricted interface enforcement. Requires defense-in-depth: the back-end must ALSO enforce the restriction, not rely solely on UI hiding.

3. Access Control Lists (ACLs)

📖 ACL (Access Control List): An object-centric list of access rights. Each protected object (file, network resource, database table) maintains a list of subjects and their permitted operations. The ACL answers: "Who is allowed to do what to THIS object?"
ACL for File: /hr/payroll/2024_salaries.xlsx ─────────────────────────────────────────── Subject (User/Group) Permissions ──────────────────── ───────────────── HR-Manager-Group Read, Write Payroll-Processor Read, Write, Execute CEO Read Auditor-Group Read SYSTEM/root Full Control [Implicit Deny All] ← everyone else denied ACL for Network Interface (Firewall): ─────────────────────────────────────────── Rule Source IP Dest Port Action ──── ───────────── ───────── ────── 10 10.0.0.0/8 443 PERMIT 20 10.1.5.0/24 22 PERMIT 30 ANY ANY DENY
ACL PropertyDetail
Object-centricStored with the object. To determine what a user can access, you must check every object's ACL. Good for answering "who can access this object?" quickly. Poor for auditing "what can this user access?" (must scan all ACLs)
DAC implementationACLs are the primary implementation mechanism for Discretionary Access Control (DAC). OS owners set ACL entries for their files/objects.
Implicit denyBest practice: anything NOT explicitly permitted in the ACL is denied. First-match vs. best-match evaluation rules vary by implementation (network ACLs typically first-match).
Discretionary elementIn DAC systems, the object owner can modify the ACL — grant others access. This owner-delegated modification capability is the defining feature of DAC via ACLs.

4. Capability Tables

📖 Capability Table: A subject-centric list of access rights. Each subject (user, process) holds a list of objects it is permitted to access and with what rights. The capability table answers: "What objects can THIS subject access, and how?"
Capability Table for Subject: Alice (HR-Manager) ──────────────────────────────────────────────────────────────────── Object (Resource) Permitted Operations ─────────────────────────────── ───────────────────────────── /hr/payroll/2024_salaries.xlsx Read, Write /hr/performance-reviews/ Read, Write /hr/org-chart.pdf Read HR Intranet Portal Read, Write (admin section) Employee Database (HR tables only) SELECT, UPDATE, INSERT ──────────────────────────────────────────────────────────────────── Capability Table for Subject: Bob (Finance-Analyst) ──────────────────────────────────────────────────────────────────── Object (Resource) Permitted Operations ─────────────────────────────── ───────────────────────────── /finance/budget/ Read, Write /finance/reports/ Read Accounting System View, Generate Reports ────────────────────────────────────────────────────────────────────
Capability Table PropertyDetail
Subject-centricStored with the subject. Easy to answer "what can Alice access?" — just read her capability table. Poor for answering "who can access this file?" (must scan all capability tables).
Capability as tokenSome systems implement capabilities as unforgeable cryptographic tokens. Possessing the token grants the right — no separate ACL lookup needed (capability-based security systems).
Unix file descriptorsA file descriptor is a capability granted to a process upon open(). Even if the file's permissions change later, the open descriptor continues to allow access — the capability was granted when the descriptor was issued.
Revocation difficultyRevoking a capability is harder than revoking an ACL entry — you must find and invalidate all copies of the token. ACLs centralize the permission with the object, making revocation straightforward.

5. Access Control Matrix

📖 Access Control Matrix: The conceptual, two-dimensional representation of all access rights in a system. Rows = subjects; Columns = objects; Cells = permitted operations. Theoretical model — too large to implement directly for enterprise systems; implemented as ACLs (extract columns) or capability tables (extract rows).
Access Control Matrix │ File A │ File B │ Database │ Printer ─────────────┼─────────────┼──────────────┼──────────────┼────────── Alice │ read, write │ read │ select │ print ─────────────┼─────────────┼──────────────┼──────────────┼────────── Bob │ read │ read, write │ select,update│ print ─────────────┼─────────────┼──────────────┼──────────────┼────────── Carol (Admin)│ read, write,│ read,write, │ all │ print │ execute, │ execute, │ │ │ change ACL │ change ACL │ │ ACL for File A = extract FILE A COLUMN → {Alice:r/w, Bob:r, Carol:r/w/x/chacl} Capability for Alice = extract ALICE ROW → {File A:r/w, File B:r, DB:select, Printer:print}
💡 Exam Tip — Matrix, ACL, Capability:
Access Control Matrix = theoretical complete picture
ACL = one COLUMN of the matrix (per-object, who can do what)
Capability table = one ROW of the matrix (per-subject, what can they do)
All three represent the same information in different organizational forms.

6. Content-Dependent Access Control

📖 Content-Dependent Access Control: Access decisions are based on the CONTENT of the data being accessed, not just the object's identity. The same object may be accessible to different subjects based on what data it contains at the time of access.
ExampleMechanism
Database row-level securityAlice (HR-Analyst for EMEA) can SELECT from the employee table but sees only rows WHERE region = 'EMEA'. Bob (HR-Analyst for APAC) sees only WHERE region = 'APAC'. Same table, same permission label — different visible content based on identity.
Document DRM by classificationA SharePoint library containing both Confidential and Public documents. A user classified at "Public" clearance can read the same folder but is blocked from downloading documents with a "Confidential" metadata tag — access decision depends on document content/metadata, not just folder identity.
Email content filteringEmail gateway blocks messages containing credit card numbers, SSNs, or malware signatures — this is content-dependent control at the network layer. The control depends on what's IN the data.
MAC label on dataMAC classification labels attached to objects (from Day 75) are effectively content-dependent — the label reflects the sensitivity of the content, and access is controlled by label comparison.

7. Context-Dependent Access Control

📖 Context-Dependent Access Control: Access decisions are based on the CONTEXT in which the access occurs — not just who you are or what data is involved, but when, where, how, and in what sequence the access happens. Guards against valid sequential steps performed in an invalid order.
ExampleMechanism
Transaction sequence enforcementIn a purchase-order workflow, "Approve PO" is only accessible AFTER "Create PO" and "Review PO" steps are complete. A user with Approver role cannot jump directly to approval without the prior context steps — context validates the process state.
E-commerce checkout flowCannot submit payment without having confirmed shipping address first. Accessing /checkout/confirm before /checkout/address returns 403. The access depends on context (session state / prior steps).
Clinical data access after consentHealthcare system: physician can only access patient records in context of an active patient encounter. Access outside an encounter context is blocked — even with the physician role — because context (ongoing care relationship) is required.
Time-of-day restrictionFrom Day 76 — temporal access restrictions are a specific form of context-dependent control where the environmental context (time) gates access.
💡 Content vs. Context:
Content-dependent = based on WHAT the data says (the data's values, labels, content).
Context-dependent = based on WHEN/HOW/WHERE/IN-WHAT-SEQUENCE the access occurs (environmental + procedural state).

8. Restricted Interfaces

📖 Restricted Interface: Constraining what users can see, access, and manipulate within a system by presenting only the subset of functionality or data appropriate for their role. A form of access control enforcement within the application's user interface or API layer.
Restricted Interface TypeDescriptionExample
Menu/feature restrictionUI elements (menu items, buttons, navigation links) are hidden or disabled for users who lack the required role/permission. User sees only what they are allowed to access.Standard user sees "View Reports" only; Admin user sees "View Reports + User Management + System Config". Manager's menu omits payroll tables accessible to Finance role.
API endpoint restrictionRESTful API returns 403 Forbidden or omits certain endpoints from API documentation/schema based on the caller's identity. Admin operations at /admin/* are restricted to the Admin role.GET /api/users/{id} returns full user record for Admin; returns redacted record for self-service user; returns 403 for others.
Shell restriction / constrained shellLimited command-line environment (restricted shell − rbash in Linux; or telecom switch CLI) that only exposes a small set of commands. User cannot access forbidden commands.Call-center agent given restricted shell that only allows: show account_balance, show transaction_history — cannot reach underlying OS commands.
Physical kiosk modePublic-facing terminals locked to a single application (kiosk mode). Cannot access OS, file system, or other apps.Airport check-in terminal, ATM, hotel check-in kiosk.
⚠️ Security Principle: Restricted interfaces enforce least privilege at the UI/API layer — but they are NOT a substitute for back-end enforcement. If a user bypasses the UI and directly calls the API (e.g., manipulates HTTP requests), the server must still enforce access control. UI restriction is defense-in-depth, not the primary control.

9. Database Views

📖 Database View: A virtual table defined by a SQL SELECT query. Users are granted SELECT permission on the view rather than on the underlying base tables. The view acts as a logical filter — exposing only specific rows (row-level security) or columns (column-level security) while keeping the underlying table structure hidden.
Base Table: employees ───────────────────────────────────────────────────────────────────────────────────── emp_id | name | salary | ssn | department | performance_rating | region 1001 | Alice Brown | 95000 | 555-12-3456 | HR | 4.2 | EMEA 1002 | Bob Chen | 110000 | 555-23-4567 | Finance | 3.9 | AMER 1003 | Carol Davis | 78000 | 666-34-5678 | IT | 4.5 | APAC View: hr_analyst_view (for HR Analysts – EMEA only, no salary or SSN) ───────────────────────────────────────────────────────────────────────────────────── CREATE VIEW hr_analyst_view AS SELECT emp_id, name, department, region ← columns filtered (NO salary, SSN) FROM employees WHERE region = 'EMEA'; ← rows filtered (EMEA only) HR Analyst sees: emp_id | name | department | region 1001 | Alice Brown | HR | EMEA (only EMEA rows, no salary/SSN columns — salary and SSN are fully invisible)
Database View Security FeatureDetail
Column-level access controlGrant access to only specific columns (e.g., hide salary, SSN, medical info from general HR staff while allowing access to other employee fields)
Row-level access controlRestrict which rows are visible based on user context (WHERE clause in view definition). User never sees other regions', departments', or sensitivity-level rows.
Hides schema complexityApplication queries a clean view interface without knowing the full underlying table structure. Reduces attack surface by limiting SQL injection blast radius — attacker can only enumerate what the view exposes.
Aggregation-only viewsStatistical/reporting views that show only aggregates (SUM, AVG, COUNT) never expose individual records — statistical disclosure protection.

11. Memory Tricks & Mnemonics

🧠 ACL vs. Capability — "Object vs. Subject":
ACL = Attached to the Asset (object) — "Who can touch this ASSET?"
Capability = Carried by the Client (subject) — "What can this CLIENT do?"
ACL = think "guest list at the door" (door = object holds the list)
Capability = think "concert ticket" (ticket = token the subject carries)
🧠 Content vs. Context:
Content-dependent = controls based on the data's CONTENTs (values, labels, classification)
Context-dependent = controls based on the access CONTExt (when, where, what sequence, how)
A doctor sees patient records based on encounter CONTEXT. DLP blocks emails based on SSN CONTENT.
🧠 Restricted Interface ≠ Security by Obscurity:
Restricting what the UI SHOWS reduces the attack surface and enforces least privilege in the interface layer.
But the BACK-END must also enforce access control. UI restriction alone = security theater.
"Never trust the client."
🧠 Database View = "Window on Data":
A view is a WINDOW into the table — you see only what the window frame shows (filtered rows AND columns).
Column filtering = vertical blinds (hides certain columns). Row filtering = horizontal blinds (hides certain rows).
