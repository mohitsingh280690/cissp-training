2. Security Models Summary

ModelFocusKey Rules
Bell-LaPadulaConfidentialitySimple Security (no read up), Star Property (no write down), Strong Star (read/write only at own level)
BibaIntegritySimple Integrity (no read down), Star Integrity (no write up) — exact inverse of Bell-LaPadula
Clark-WilsonIntegrity (commercial)Constrained Data Items (CDIs), Unconstrained Data Items (UDIs), Transformation Procedures (TPs), Integrity Verification Procedures (IVPs), well-formed transactions, separation of duties
Brewer-Nash (Chinese Wall)Conflict of InterestDynamic access control — once you access data in one conflict class, you're blocked from competing classes
LipnerBoth Confidentiality & IntegrityCombines Bell-LaPadula and Biba in a single implementation
Graham-DenningObject Operations8 primitive protection operations (create/delete objects and subjects, grant/revoke rights)
Harrison-Ruzzo-Ullman (HRU)Access RightsExtension of Graham-Denning; provably secure but undecidable in general case
🧠 Quick Recall: "Bell-LaPadula = no read UP, no write DOWN (protects secrets from leaking down). Biba = no read DOWN, no write UP (protects integrity from corruption flowing up)."

ChaCha20Stream256-bitN/A✅ Modern alternative; used in TLS 1.3
🔑 Block Cipher Modes:
ECB — No IV; identical blocks → identical ciphertext (insecure pattern leakage)
CBC — Chained blocks with IV; error propagates; padding oracle attack risk
CTR — Counter mode; turns block cipher into stream cipher; parallelizable
GCM — Galois/Counter Mode; provides authenticated encryption (confidentiality + integrity); used in TLS 1.3

8. Cryptographic Attacks Summary

Attack TypeWhat Attacker Has/Does
Ciphertext-OnlyOnly ciphertext; frequency analysis
Known-PlaintextSome plaintext + corresponding ciphertext
Chosen-PlaintextCan encrypt arbitrary plaintexts
Chosen-CiphertextCan decrypt arbitrary ciphertexts (most powerful)
Birthday AttackFind hash collisions in 2^(n/2)

Side-ChannelTiming, power, EM emissions — attacks implementation
Rubber-HosePhysical coercion for key extraction

9. Key Management Summary

PhaseCritical Practice
GenerationCSPRNG; FIPS 140-2/3 validated modules
DistributionKey exchange protocols; never plaintext; out-of-band for symmetric
StorageHSMs; key vaults; never in source code
UsageSeparate encryption and signing keys; enforce key usage restrictions
RotationDefine cryptoperiods; automate; re-key after compromise
ArchivalArchive encryption keys (for decrypting old data); NEVER archive signing keys
DestructionZeroize; FIPS-compliant methods; ensure no copies remain
🔑 Remember: Escrow/archive encryption keys only. Never escrow signing keys (destroys non-repudiation). Split knowledge = key divided among people. Dual control = two people must act together.
