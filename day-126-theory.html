2. Real-World Organizational Relevance

Testing is the verification side of "trust but verify." Without security testing, secure coding guidelines are unenforceable. Modern DevSecOps pipelines integrate SAST, DAST, SCA, and fuzz testing into CI/CD — finding vulnerabilities before deployment instead of after a breach. Penetration testing provides the adversarial perspective that automated tools miss, particularly for business logic flaws.

3. Testing Approaches – White/Black/Grey Box

ApproachKnowledge LevelWhat Tester KnowsBest For
White box (clear box, open box)Full knowledgeSource code, architecture, design docs, credentials, network diagramsThorough internal review; code audit; finds logic vulnerabilities deep in code
Black box (closed box)Zero knowledgeNothing — simulates external attacker with no inside infoRealistic attack simulation; tests external-facing defenses
Grey boxPartial knowledgeSome documentation, limited credentials, partial architecture knowledgeEfficient compromise — insider threat simulation; authenticated testing
White box finds MORE vulnerabilities; black box simulates REAL attacks. On the CISSP exam, white box is best for comprehensive code review; black box is best for simulating external attackers. Grey box balances both.

4. Static Application Security Testing (SAST)

SAST (Static Analysis): Analyzes source code, bytecode, or binary WITHOUT executing the application. Scans for vulnerable patterns, insecure functions, and coding standard violations. "White box" by nature — requires access to code.
AspectDetail
WhenDuring development — earliest in pipeline (shift-left); runs on every code commit in CI/CD
FindsSQL injection patterns, buffer overflows, hardcoded credentials, use of deprecated functions, insecure crypto
StrengthsFinds bugs early (cheapest to fix); covers entire codebase; no running environment needed; pinpoints exact line of code
WeaknessesHigh false positive rate; cannot find runtime issues (auth bypass, config errors); cannot test deployed environment
ToolsSonarQube, Checkmarx, Fortify, Semgrep, Bandit (Python)

5. Dynamic Application Security Testing (DAST)

DAST (Dynamic Analysis): Tests the RUNNING application by sending crafted HTTP requests and analyzing responses. "Black box" by nature — does NOT examine source code. Tests the application as an attacker would.
AspectDetail
WhenAfter deployment to testing/staging environment; later in pipeline than SAST
FindsXSS, SQL injection (runtime), authentication flaws, server misconfigurations, exposed endpoints, information leakage
StrengthsLow false positives (finds real exploitable issues); language/framework agnostic; tests actual deployed state
WeaknessesCannot pinpoint exact code location; cannot test unexecuted code paths; slower than SAST; requires running application
ToolsOWASP ZAP, Burp Suite, Acunetix, Nessus (web scanning)

6. Interactive & Runtime Testing (IAST/RASP)

MethodHow It WorksBenefit
IAST (Interactive AST)Agent embedded in running application; combines SAST + DAST — observes code execution while handling real requestsBest of both worlds: real vulnerability confirmation + exact code location; low false positives; tests during QA
RASP (Runtime Application Self-Protection)Embedded in production application; monitors behavior in real-time; can BLOCK attacks, not just detectReal-time protection; can terminate malicious sessions; deep visibility into app context; defense-in-depth layer
IAST = Testing tool (QA phase). RASP = Runtime protection tool (production). IAST finds; RASP blocks. Both are agent-based (instrumented inside the application).

7. Fuzz Testing

Fuzz Testing (Fuzzing): Sending random, malformed, unexpected, or boundary-value input to an application to discover crashes, hangs, memory leaks, and unhandled exceptions. Effective at finding vulnerabilities that manual testing misses — especially buffer overflows and input parsing errors.

Fuzzing Types

TypeDescription
Mutation-based (dumb)Takes valid input and randomly mutates it (bit flips, byte insertions, truncation); no knowledge of input format
Generation-based (smart/intelligent)Generates input based on knowledge of the protocol/format specification; more targeted; better coverage
Coverage-guidedUses code coverage feedback to guide mutation toward unexplored code paths; modern approach (AFL, libFuzzer)
Fuzzing finds: Buffer overflows, format string bugs, integer overflows, crashes, DoS conditions, unhandled exceptions. Fuzzing does NOT find: Business logic flaws, authentication bypasses, access control issues. It's a brute-force approach that excels at finding input-handling bugs.

8. Penetration Testing

Penetration Testing: Authorized, simulated attack against systems to identify exploitable vulnerabilities. Goes beyond scanning — testers actively exploit discovered vulnerabilities to demonstrate real-world impact. Requires written authorization (Rules of Engagement).

Penetration Testing Phases

PhaseActivity
1. Planning & ScopingDefine targets, rules of engagement, authorization, timeline, emergency contacts
2. ReconnaissanceInformation gathering — passive (OSINT) and active (scanning, enumeration)
3. Scanning/DiscoveryPort scanning, vulnerability scanning, service identification
4. ExploitationAttempt to exploit discovered vulnerabilities; gain access
5. Post-exploitationLateral movement, privilege escalation, data exfiltration (within scope)
6. ReportingDetailed findings, evidence, risk ratings, remediation recommendations

Pen Test vs Vulnerability Assessment

FeatureVulnerability AssessmentPenetration Test
PurposeIdentify vulnerabilitiesExploit vulnerabilities
ApproachScanning — automated, broadAttacking — manual, focused, deep
OutputList of vulnerabilities with severityProof of exploitability with impact demonstration
RiskLow (non-invasive)Higher (active exploitation may disrupt systems)
FrequencyRegular/continuousPeriodic (annually, or after major changes)
ALWAYS GET WRITTEN AUTHORIZATION before penetration testing. Without it, pen testing is legally indistinguishable from an actual attack. The Rules of Engagement document must specify: scope, targets, methods, timeframe, contacts, and data handling.

9. Code Review Methods

MethodDescriptionEffectiveness
Formal code review (Fagan inspection)Structured, multi-phase process: planning, overview, preparation, inspection meeting, rework, follow-upMost thorough; finds 60-90% of defects; time-intensive
Peer review (walkthrough)Developer explains code to peers; less formal than Fagan; reviewer checks for issuesGood balance of thoroughness and efficiency
Over-the-shoulderReviewer looks at code on developer's screen; informal, real-timeQuick but limited coverage
Pair programmingTwo developers write code together simultaneously; continuous reviewPrevents bugs from being introduced; XP practice
Tool-assisted reviewAutomated review tools (SAST) + human review of findingsScalable; reduces reviewer burden; catches pattern-based issues
Fagan Inspection: The MOST formal and MOST effective code review method. Named after Michael Fagan (IBM, 1976). Key characteristic: structured phases with defined roles (moderator, designer, coder, tester). CISSP loves asking about this as the "most thorough" review method.

10. Software Composition Analysis (SCA)

SCA: Automated analysis of third-party and open-source components in a codebase. Identifies known vulnerabilities (CVEs), license compliance issues, and outdated dependencies. Essential because modern applications are 70-90% third-party code.
SCA CapabilityPurpose
Vulnerability detectionMatch components against CVE databases (NVD); alert on known-vulnerable versions
License complianceIdentify open-source licenses (GPL, MIT, Apache); flag copyleft requirements
SBOM generationCreate Software Bill of Materials listing all components and versions
Version trackingIdentify outdated dependencies; recommend updates

11. Memory Tricks & Mnemonics

🧠 SAST vs DAST — "SAST = Source, DAST = Deployed"
SAST = Source code analysis (static, white box, early, high false positives)
DAST = Deployed app testing (dynamic, black box, late, low false positives)
🧠 Testing Tools — "SDIR" (Security Defense In Rounds)
SAST → DAST → IAST → RASP
Static → Dynamic → Interactive → Runtime
Development → Testing → QA → Production
🧠 Fagan Inspection — "POPIR-F" (People Often Plan Inspections Really Formally)
Planning → Overview → Preparation → Inspection → Rework → Follow-up
🧠 Pen Test Phases — "PRSEPR" (Professionals Recon, Scan, Exploit, Post-Exploit, Report)
Planning → Reconnaissance → Scanning → Exploitation → Post-exploitation → Reporting
