2. Real-World Organizational Relevance

IssueContext
Active Directory = KerberosActive Directory (the dominant enterprise directory) uses Kerberos as its primary authentication protocol since Windows 2000. Every Windows domain authentication uses Kerberos tickets. Understanding Kerberos is understanding how AD authentication works.
SSO drives productivityAverage enterprise has 100+ SaaS applications. Without SSO, employees manage 100 passwords, leading to reuse, weak passwords, and help-desk burden. SSO reduces authentication friction while centralizing control — allowing immediate deprovisioning from one place.
Kerberos attacks are realKerberoasting (used in 30%+ of ransomware attacks), golden ticket attacks (Mimikatz), pass-the-ticket — all exploit Kerberos weaknesses. Red teams routinely use BloodHound + Mimikatz to enumerate and exploit Kerberos in AD environments.
SSO is a high-value targetCompromising the SSO provider (Okta, Azure AD, on-prem AD) gives access to everything. This is why SSO providers are targeted (Okta breaches 2022-2023; SolarWinds/SAML golden SAML attack). The SSO server must be hardened to the highest standard.

3. Single Sign-On (SSO)

📖 Single Sign-On (SSO): An authentication system that allows a user to authenticate once and gain access to multiple independent systems or applications without re-authenticating for each one. The SSO service handles authentication and provides tokens/credentials to other systems on the user's behalf.
SSO BenefitSSO Risk
Users authenticate once — reduced password fatigueSingle point of compromise: if SSO credentials are stolen, all systems are accessible
Centralized access management — deprovisioning from one place revokes all access immediatelySingle point of failure: if SSO service is unavailable, users cannot access any systems
Enables strong authentication once (MFA at SSO layer) rather than per-applicationIncreased exposure window: one long-lived SSO session provides persistent access to all systems
Reduced help-desk burden (fewer password resets)Session hijacking: stealing SSO session token = access to all connected applications
Better audit trail: all access flows through central SSO providerTrust relationship complexity: misconfigured federation trust can grant unintended access

4. Kerberos Components

ComponentRoleKey Facts
KDC (Key Distribution Center)Central trusted authority. Contains both AS and TGS within it. Holds the master key database (all principal secrets).Most critical component. If KDC is compromised, the entire domain is compromised. KDC contains both AS and TGS — CISSP often asks which "part" of the KDC does what. UDP/TCP port 88.
AS (Authentication Service)Part of the KDC. Issues the Ticket Granting Ticket (TGT) after verifying the user's password hash.Client sends AS-REQ with username; AS-REP contains TGT encrypted with KRBTGT key and session key encrypted with user's key. User never learns the KRBTGT key.
TGS (Ticket Granting Service)Part of the KDC. Issues service tickets (ST) in exchange for a valid TGT. Client presents TGT to get access to specific services.TGS validates the TGT, confirms the service exists, issues a Service Ticket (ST) encrypted with the service account's key. User presents ST to service.
TGT (Ticket Granting Ticket)A credential issued by the AS after initial authentication. Presented to the TGS to request service tickets. Acts as a "master ticket."Encrypted by KRBTGT key (only KDC can decrypt). Contains user identity, validity period (default 10 hours), session key. Renewable up to 7 days.
ST / Service TicketA ticket issued by TGS that grants access to a specific service. Presented to the service (not the KDC).Encrypted with the service account's long-term key. Service decrypts it to verify identity without contacting KDC again.
KRBTGT accountSpecial service account whose password hash is used to encrypt/sign ALL TGTs in a Kerberos realm.The "crown jewel" of AD. Compromising KRBTGT hash enables forging any TGT (Golden Ticket attack). Should be changed at least twice during domain compromise recovery (due to replication).
PrincipalAny entity (user, service, host) that participates in Kerberos authentication. Identified by a principal name (user@REALM or service/host@REALM).SPN (Service Principal Name) is how services register with Kerberos — e.g., HTTP/webserver.corp.com. Required for Kerberos to build service tickets.

Kerberos rejects tickets reused beyond 5-minute clock skew (requires time sync).
🔑 Three critical security mechanisms in Kerberos flow:
1. Mutual auth: Steps 5-6 — service also proves its identity to the client (AP-REP), preventing impersonation of servers
2. Anti-replay: Authenticator timestamp prevents replaying captured tickets (clock skew tolerance: ±5 minutes by default)
3. No cleartext passwords: Passwords are never transmitted. All messages encrypted. KDC proves server identity by encrypting response in a key only the legitimate server has.

6. Kerberos Security Properties

PropertyHow AchievedWeakness / Limitation
Passwords never transmittedClient proves identity by decrypting a message that only a holder of the correct password key can decryptThe password HASH is still used as a long-term key — weak passwords lead to weak keys
Mutual authenticationBoth client and server prove identity to each other (AP-REQ/AP-REP exchange)Optional in some implementations; older configurations may not enforce server auth
Third-party authenticationAuthentication delegated to trusted KDC; services don't need to store passwordsKDC is SPOF — must be highly available and hardened
Time-based anti-replayTickets include timestamps; KDC maintains replay cache for 5-minute windowRequires NTP time synchronization across domain; clock skew > 5 min = authentication failures
Ticket confidentialityTickets encrypted with service/KRBTGT secretsIf service account has a weak password: Kerberoasting extracts and cracks the service ticket offline

7. Kerberos Attacks

AttackHow It WorksWhat's RequiredImpactDefence
Golden TicketAttacker compromises KRBTGT account password hash. Forges arbitrary TGTs — can impersonate ANY user, including Domain Admins, with any privilege, any duration.KRBTGT hash (domain admin-level compromise)Complete domain persistence. Golden ticket valid even after password reset of regular accounts (only valid mitigation: reset KRBTGT twice).Reset KRBTGT password twice (force replication); monitor for forged TGT attributes (invalid ticket duration, impossible SIDs)
Silver TicketCompromises a specific service account hash. Forges Service Tickets for that specific service. No KDC contact required — service validates ST itself.Service account NTLM hashAccess to that specific service, undetected (KDC not contacted, so no KDC logs). Narrower than golden ticket.Protect service account passwords (long, complex, managed by gMSA); monitor for ST without corresponding TGS events
Pass-the-Ticket (PtT)Steal a live TGT or ST from memory (using Mimikatz/lsass dump) and use it from attacker's machine. Doesn't require password cracking — uses valid live ticket.Code execution on victim's machine (to dump LSASS memory)Lateral movement using victim's identity until ticket expiresCredential Guard (protects LSASS); short ticket lifetimes; restricted admin mode; privileged access workstations (PAW)
KerberoastingAny domain user can request a TGS service ticket for any SPN. Service ticket is encrypted with the service account's password hash. Attacker requests tickets for service accounts with SPNs and cracks them offline.Any domain-authenticated user accountIf service account has weak password: account compromise. Service accounts often have elevated privileges.Service accounts: long, complex passwords (>25 chars); use gMSA (Group Managed Service Accounts — machine-managed passwords); monitor for bulk TGS requests from non-service accounts
AS-REP RoastingTargets accounts with "Do not require Kerberos preauthentication." AS-REP contains data encrypted with user's key — can be cracked offline without any authentication.No authentication required — just a domain account list or anonymous LDAP queryIf any accounts have pre-auth disabled AND weak passwords: full account compromiseNever disable Kerberos pre-authentication; audit accounts with this flag

8. SESAME

📖 SESAME (Secure European System for Applications in a Multi-vendor Environment): A successor/extension to Kerberos developed in the 1990s. Designed to address Kerberos limitations, particularly the lack of native authorization (delegation) and privilege management. Used Privilege Attribute Certificates (PACs).
FeatureKerberosSESAME
AuthenticationSymmetric key (shared secret with KDC)Asymmetric (public key) AND symmetric
AuthorizationLimited — tickets contain identity, not explicit privilegesPAC (Privilege Attribute Certificate) carries authorization attributes explicitly
Cross-domain trustRequires explicit trust between realmsBetter support for cross-domain and multi-vendor environments
SSOWithin a realmAcross multiple organizations/domains
DeploymentUbiquitous (Active Directory, MIT Kerberos)Largely obsolete in practice (superseded by SAML/OIDC for federation)
💡 Exam Tip: SESAME is tested mainly as "what did it improve over Kerberos?" — answer: added public key support and explicit privilege/authorization attributes (PACs). In practice, SESAME itself is not deployed; the concept is tested for understanding evolution toward modern federation (SAML).

9. SSO Risks and Mitigations

RiskDetailMitigation
Single point of compromiseStealing SSO credentials = access to all integrated applications. Stealing SSO session token = same result.Strong MFA at SSO layer (phishing-resistant); short session lifetimes; conditional access policies
Single point of failureSSO provider downtime = complete service disruption for all dependent applications. Business-critical.HA clustering; geographic redundancy; fallback authentication mechanisms
Session token theftAfter SSO authentication, session token (cookie/JWT) grants access. Token theft (XSS, network sniff, physical) = session hijacking across all apps.Secure cookies (HttpOnly, Secure, SameSite); short token lifetimes; token binding; IP/device binding
Overly broad accessSSO can mask the scope of access breach — user (or attacker posing as user) authenticates once and silently accesses many systems.Maintain per-application authorization (SSO handles authentication; each app still enforces authorization); audit access logs per application
Orphaned SSO integrationsApplications integrated with SSO but decommissioned at the app level; SSO account still maps to them; access persists.Application decommissioning checklist includes SSO integration removal; periodic review of SSO connected apps

11. Memory Tricks & Mnemonics

🧠 Kerberos Components — "KDC has AS and TGS; AS gives TGT; TGS gives ST":
KDC = house that contains AS (front door) and TGS (back room)
Walk in front door (AS) → Get a master key (TGT)
Take master key to back room (TGS) → Get room key (ST)
Use room key at the room (Service) → Get in
🧠 Kerberos Attacks — "Gold crowns Silver tickets, Thieves Pass, Roasters brew":
Golden Ticket = forge TGTs (need KRBTGT hash = crown jewels)
Silver Ticket = forge service tickets (need service account hash)
Pass-the-Ticket = steal live tickets from memory (need LSASS access)
Kerberoasting = request STs and crack offline (any domain user can do it)
🧠 SSO Risk — "One key opens every door":
Benefit: one authentication, many doors.
Risk: one stolen key, many open doors. + one broken lock, no doors open.
