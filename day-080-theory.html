Domain5 – Identity and Access Management (13%) — directory services are the infrastructure backbone of enterprise identity: they store identities, authenticate users, and serve as the source of truth for group membership and access policy.

3. X.500 – The Directory Standard Foundation

📖 X.500: An ITU-T family of standards defining directory services — how information about users, computers, and organizations can be stored and accessed in a hierarchical, distributed manner. Published 1988. The conceptual ancestor of LDAP and Active Directory. X.509 (TLS/SSL certificates used everywhere today) is part of the X.500 family.
X.500 ConceptDefinition
Directory Information Tree (DIT)The hierarchical tree structure of the directory. Root → Country → Organization → Organizational Unit → Objects (users, devices). Objects are identified by their position in this tree.
Distinguished Name (DN)The full unique path to an object in the DIT. Uniquely identifies any directory object. Example: CN=Alice Brown,OU=HR,DC=corp,DC=example,DC=com — reads from specific to general.
Relative Distinguished Name (RDN)The entry's identifier within its parent container (the leaf portion of the DN). CN=Alice Brown is the RDN within the OU=HR container.
SchemaDefines what object classes exist (user, group, computer, printer) and what attributes each class can have (givenName, sn, telephoneNumber, userPrincipalName). Enforces data structure. Can be extended (AD schema extensions for Exchange, Lync, etc.).
DAP vs. LDAPX.500 originally required DAP (Directory Access Protocol) — complex, heavy. LDAP (Lightweight DAP) was developed as a slim TCP/IP-based access protocol for X.500 directories. LDAP became the practical standard; full X.500/DAP is rarely implemented.

4. LDAP – Lightweight Directory Access Protocol

📖 LDAP: TCP/IP protocol for accessing and modifying directory services based on the X.500 model. Standards defined in RFC 4511. Used to query and manage directory information — authenticating to systems, looking up user attributes, checking group membership.
LDAP PropertyDetail
PortsLDAP: TCP/UDP port 389 (unencrypted). LDAPS (LDAP over SSL/TLS): TCP port 636. LDAP + STARTTLS: TCP 389 with TLS upgrade. Always use LDAPS or STARTTLS — plain LDAP on 389 transmits credentials in cleartext (LDAP bind password).
LDAP BindAuthentication operation. Client sends DN + password (or SASL token) to prove identity. Anonymous bind (no credentials) — read-only access to some directory information without authentication. Simple bind over unencrypted LDAP = password in cleartext.
LDAP SearchCore read operation. Specifies: base DN (where to start), scope (base/one-level/subtree), filter (attribute conditions), and attributes to return. Powerful query language — can extract all users in an OU, all members of a group, all computers with specific attributes.
LDAP Modify / Add / DeleteWrite operations. Modify: change attribute values (change password, update phone number). Add: create new entry. Delete: remove entry. These require appropriate directory write permissions.
DN structure (reading)DN is read right-to-left for hierarchy: CN=Alice Brown, OU=HR, DC=corp, DC=example, DC=com → domain is example.com → corp subdomain → HR OU → Alice's CN. DC = Domain Component; OU = Organizational Unit; CN = Common Name.
LDAP Distinguished Name Example: ────────────────────────────────────────────────────────────────────────────────────── DN: CN=Alice Brown,OU=HR,OU=Staff,DC=corp,DC=example,DC=com Reading left-to-right (most specific → least specific): CN=Alice Brown ← The person (Common Name) — the object OU=HR ← She is in the HR Organizational Unit OU=Staff ← HR is inside the Staff OU DC=corp ← In the "corp" subdomain DC=example ← Of the "example" domain DC=com ← Top-level domain .com Full LDAP tree path (top to bottom): DC=com DC=example DC=corp OU=Staff OU=HR CN=Alice Brown ← The entry Other DN components: O = Organization L = Locality (city) C = Country UID = User ID emailAddress = email ──────────────────────────────────────────────────────────────────────────────────────

5. Active Directory – Architecture Deep Dive

📖 Active Directory (AD): Microsoft's enterprise directory service, introduced in Windows Server 2000. Built on LDAP for data storage and querying, Kerberos (Day 73) for authentication, DNS for locating domain controllers, and Group Policy for centralized configuration management. The dominant enterprise identity and access management infrastructure.
AD ComponentDefinitionKey Security Notes
DomainThe fundamental unit of AD administration. A domain has one DNS namespace (corp.example.com), one centralized authentication database (NTDS.dit), and one set of policies. Objects (users, computers, groups) belong to a domain.Domain controllers (DCs) hold the AD database. DC compromise = domain compromise. DCs must be highly protected physical and logical assets.
Domain Controller (DC)A server running Active Directory Domain Services (AD DS). Handles authentication (Kerberos, NTLM), LDAP directory queries, Group Policy distribution, and replication with other DCs.The most critical servers in an environment. Compromise allows creation of any user, modification of any policy, golden ticket attacks (Day 73). Require strict PAM controls (Day 79) and network isolation.
NTDS.ditThe Active Directory database file. Contains all domain objects: user accounts, password hashes, group memberships, computer accounts. Located on every DC at C:\Windows\NTDS\NTDS.dit.Extracting NTDS.dit gives attacker all password hashes → offline cracking or pass-the-hash attacks. Protect with backup encryption, VSS shadow copy controls, and monitoring of Volume Shadow Copy creation by non-admins.
ForestThe highest-level container in AD. A forest is a collection of one or more domains that share a common schema, global catalog, and configuration. Forest root domain is the first domain created. The forest is the security boundary in AD.The Security Boundary in AD is the FOREST, not the domain. Domain admins in one domain of a multi-domain forest can typically reach other domains. Forest-level compromise affects all domains in the forest.
TreeA contiguous DNS namespace of domains within a forest. corp.example.com and subsidiary.corp.example.com form a tree — they share a contiguous namespace. A forest can contain multiple trees (different root DNS namespaces).Trust relationships within a tree are automatic and bi-directional (transitive). Security considerations for cross-domain trust.
Global Catalog (GC)A partial replica of ALL objects in ALL domains in the forest stored on designated GC servers. Used to look up objects across the forest without querying every domain. Required for universal group membership resolution and forest-wide searches.GC servers are high-value targets — they have data from ALL domains. Compromising a GC server exposes forest-wide object information.

6. Active Directory – Trusts, OUs, and GPOs

AD Trusts

Trust TypeDescriptionSecurity Implication
Internal (intra-forest) trustAutomatically created, bi-directional, transitive trusts between all domains in a forest. Domain A trusts Domain B which trusts Domain C → Domain A implicitly trusts Domain C (transitive).A compromised domain in a forest can be leveraged to attack other trusted domains (trust abuse, SID history attacks). Trusts must be audited and documented.
External trustManually created, non-transitive, one or two-way trust between specific domains in DIFFERENT forests. Only the explicitly trusted domain is trusted — transitivity does not flow.Explicit, controlled trust relationship. Non-transitive limits blast radius. Requires careful scoping — trust grants use of Kerberos tickets for resource access across the trust boundary.
Forest trustTransitive trust between two forest ROOT domains, enabling all domains in both forests to trust each other.Highest-risk trust relationship — effectively merges the security boundary of two forests. Compromise of either forest can cascade. Used for M&A integrations.
SID filteringA security feature on external trusts that strips SID history from cross-trust Kerberos tickets, preventing an attacker in a trusted domain from injecting high-privilege SIDs into tickets to gain privileges in the trusting domain.SID filtering SHOULD be enabled on external trusts to prevent SID history attacks. Enabled by default on external trusts since Windows Server 2003.

Organizational Units (OUs) and Group Policy Objects (GPOs)

ComponentPurposeSecurity Use
Organizational Unit (OU)A container within a domain used to organize objects (users, computers, groups). Allows delegation of administrative authority — you can grant a help desk team the ability to reset passwords ONLY for users in the Help Desk OU, without giving domain admin rights.OU structure reflects both organizational and security policy zones. Sensitive objects (privileged admin accounts, PAM servers, domain controllers) should be in separate protected OUs with restricted modification rights.
Group Policy Object (GPO)A collection of settings (security policies, software deployment, registry settings, script execution) applied to an OU, domain, or site. Settings apply automatically to all computer and user accounts within the GPO's scope.GPOs enforce enterprise security baselines at scale: screensaver lock timeout, password complexity, audit policy, AppLocker rules, Windows Defender configuration, disabling legacy protocols (NTLM, LM hash), restricting local admin rights. Misconfigurations are a common lateral movement path (writable GPOs, GPO link order manipulation).
💡 AD Security Boundary Reminder: The DOMAIN is NOT the security boundary in Active Directory — the FOREST is the security boundary. Domain Admins can gain access across domains in the same forest by default. For true isolation, separate forests are required.

7. RADIUS vs. TACACS+

📌 Cross-Reference: RADIUS and TACACS+ were covered in depth in Day 71 (Authentication Methods) in the context of AAA protocols for network access. Here they are placed in the directory services architecture context — they serve as AAA back-ends consuming identities stored in LDAP/AD.
FeatureRADIUSTACACS+
Developer / StandardIETF RFC 2865/2866 — open standardCisco proprietary (though documented)
TransportUDP — ports 1812 (auth/authz), 1813 (accounting)TCP — port 49 (reliable). Full packet encryption.
EncryptionEncrypts password ONLY in Access-Request; rest of packet in cleartextEncrypts ENTIRE packet (body) — full confidentiality including username, attributes, authorization data
AAA separationAuthentication and authorization combined in one transaction — cannot separateAuthentication, Authorization, and Accounting are SEPARATE processes — can authorize per command, per device type independently
Command authorizationNot supported — authorize access to the device or not; cannot control per-commandSupported — TACACS+ can authorize each command the admin types (show = permit; configure = deny for helpdesk role)
Primary use caseNetwork access authentication: 802.1X Wi-Fi, VPN authentication, dial-up, Network Access Control (NAC)Network device management: router/switch/firewall CLI access and per-command authorization
Multi-protocol supportCHAP, PAP, EAP (802.1X); works with any NAS vendorCHAP, PAP; primarily Cisco CLI administration
💡 RADIUS vs. TACACS+ Classic Exam Question:
"Which protocol encrypts only the password?" → RADIUS.
"Which protocol encrypts the entire payload?" → TACACS+.
"Which protocol is best for per-command authorization on network devices?" → TACACS+.
"Which protocol is used for Wi-Fi 802.1X authentication?" → RADIUS.

8. Diameter Protocol

📖 Diameter: An AAA protocol designed as the successor to RADIUS for mobile networks, broadband subscriber management, and 4G/5G LTE core network authentication. Named "Diameter" as it is "twice RADIUS" (diameter = 2 × radius). Defined in IETF RFC 6733.
FeatureRADIUSDiameter
TransportUDPTCP or SCTP — reliable, ordered delivery
Failure handlingNo built-in failoverPeer negotiation, failover, and error notification built in
Message modelRequest/ResponseMore complex peer exchange model; supports server-initiated messages
ExtensibilityLimited AVP typesUsing standard AVPs (Attribute Value Pairs) with well-defined extension mechanism; supports mobile network specific interfaces (Cx, Rx, Gx, S6a in LTE)
Primary deploymentEnterprise Wi-Fi, VPN, NACMobile network (4G LTE core, 5G NR); not common in enterprise environments
💡 Diameter on the CISSP Exam: Diameter usually appears as a distractor in RADIUS/TACACS+ questions or as the correct answer when the scenario involves mobile networks, cellular carriers, or 4G/LTE network AAA. It is NOT used for standard enterprise Wi-Fi or device management.

ServicePlatformKey Features & Notes
Microsoft Entra ID (formerly Azure AD)Microsoft AzureCloud-native identity platform for Microsoft 365, Azure, and integrated SaaS apps. Uses SAML 2.0, OIDC, OAuth 2.0 for federation (NOT LDAP for cloud apps — LDAP is legacy protocol). Supports hybrid sync with on-premises AD via Entra Connect. Conditional Access = cloud ABAC. Enables single sign-on across on-prem and cloud apps from a single identity. Password hash sync or pass-through authentication or federation (ADFS). The on-prem AD remains the authoritative identity source in hybrid models.

10. Directory Service Security Risks

RiskDescriptionMitigation
LDAP injectionAttackers inject LDAP filter syntax into application inputs to manipulate directory queries — bypassing authentication or extracting directory data. Similar to SQL injection but targeting LDAP query parser.Input validation and encoding; use parameterized LDAP queries; least-privilege LDAP service account; restrict anonymous LDAP reads. LDAP special chars: *, (, ), \, NUL, /, ÿ.
Unencrypted LDAP (port 389)LDAP operations on port 389 transmit bind credentials (username + password) in cleartext. An attacker on the same network segment can capture directory credentials via passive eavesdropping.Require LDAPS (port 636) or STARTTLS on all LDAP connections. Enforce via Domain Controller security policy: "LDAP server signing requirements = Require signing." Block port 389 on internal firewalls where possible.
DCSync attackAn attacker with Domain Admin (or equivalent) permissions calls MS-DRSR replication APIs to request domain controller replication data — effectively pulling all password hashes from the DC without logging into it or touching NTDS.dit. Used to extract the krbtgt hash for Golden Ticket attacks.Monitor for unusual replication requests from non-DC machines; restrict DCSync permissions (only Domain Controllers should have Replicating Directory Changes All); SIEM alerts on "4662 – Object was operated on" with sensitive permissions.
AD attack surface (Kerberoasting, AS-REP Roasting, PtH)Covered in Day 73 (Kerberoasting, Silver/Golden tickets) — service account SPNs exposed to any domain user; AS-REP Roasting targets accounts without Kerberos pre-authentication; PtH uses stolen NTLM hashes for lateral movement without knowing cleartext password.Strong service account passwords + regular rotation (PAM); disable accounts without pre-auth required; protect NTLM by enforcing Kerberos; LAPS for local admin accounts; MFA everywhere possible.
GPO misconfigurationsOverly permissive write access to GPOs enables privilege escalation. An attacker with "Modify" rights on a GPO linked to Domain Controllers can deploy malware to all DCs via Group Policy.Audit GPO permissions regularly (BloodHound tool useful for this). Principle of least privilege on GPO management. SYSVOL permissions (where GPO templates are stored) must be tightly controlled.

11. Memory Tricks & Mnemonics

🧠 LDAP DN — "Most Specific on the Left":
DN: CN=Alice,OU=HR,DC=corp,DC=example,DC=com
Read LEFT to RIGHT = most specific to least specific (like a street address reversed)
CN = the person; OU = the floor; DC = the building; DC = the city
The domain in DNS: corp.example.com maps to DC=corp,DC=example,DC=com (dots → DC= separated)
🧠 AD Security Boundary = FOREST (Not Domain):
"The FOREST is where the FENCE is."
Domains within a forest automatically trust each other. The fence (security boundary) is at the FOREST edge.
For true isolation: separate FORESTS, not separate domains.
🧠 RADIUS vs. TACACS+:
RADIUS = Reduced encryption (password only), Runs on UDP
TACACS+ = Total encryption (full packet), TCP-based, fine-grained per-command authorization
RADIUS → Wi-Fi and VPN (Access). TACACS+ → Cisco device CLI (Administration). "Access = RADIUS; Administration = TACACS+"
🧠 Diameter = Mobile/Cellular, NOT enterprise:
Diameter protocol belongs to the cellular/mobile world (4G LTE, 5G).
Do NOT use it for Wi-Fi or enterprise AAA answers — that is RADIUS.
"Diameter is for the cell tower, RADIUS is for the office."
🧠 LDAP Ports — "389 is plain, 636 is secure":
389 = HTTP of LDAP (unencrypted, avoid for binds)
636 = HTTPS of LDAP (LDAPS, encrypted)
