2. Real-World Organizational Relevance

According to Gartner, 80% of unplanned outages are caused by ill-planned changes. The 2017 AWS S3 outage that took down much of the internet was caused by a single command entered incorrectly during routine maintenance — a change management process failure. Proper change management prevents outages, maintains security posture, and ensures regulatory compliance (SOX, PCI DSS, HIPAA all require formal change control).

3. Change Management Process

Change Management: A structured process for requesting, evaluating, approving, implementing, and reviewing changes to IT systems. The goal is to minimize risk from changes while enabling necessary business evolution.

Change Management Steps in Detail

StepActivityKey Outputs
1. Request (RFC)Requestor submits formal Request for Change documenting: what, why, when, who, impact assessment, testing plan, back-out planRFC document
2. EvaluateChange manager/team assesses risk, impact on other systems, resource requirements, security implications, scheduling conflictsImpact analysis; risk assessment
3. ApproveCAB (or delegated authority) reviews and approves/rejects/defers. Emergency changes may use emergency CAB (ECAB)Approval decision; conditions
4. ImplementExecute the change during approved maintenance window; follow documented procedures; test after implementationImplementation records; test results
5. ReviewPost-implementation review: Did the change achieve its objective? Any unexpected impacts? Back-out executed if failedReview report; lessons learned
6. CloseUpdate CMDB with new configuration; archive RFC; close change ticketUpdated CMDB; closed RFC

Change Types

TypeDescriptionApproval Process
StandardPre-approved, low-risk, routine changes that follow a documented procedure (e.g., monthly password rotation, standard user provisioning)Pre-authorized — no individual CAB review needed
NormalChanges requiring full evaluation and CAB approvalFull RFC process through CAB
EmergencyChanges required urgently to restore service or fix critical security vulnerability (e.g., zero-day patch)ECAB (subset of CAB) for rapid approval; full documentation completed retroactively
Exam Trap — Emergency Changes: Emergency changes still require APPROVAL — they just use an expedited process (ECAB). "Emergency" does NOT mean "skip change management." After implementation, emergency changes must be documented and reviewed retroactively through the normal process.

4. Change Advisory Board (CAB)

CAB (Change Advisory Board): A group of stakeholders that evaluates and authorizes changes to IT systems. Membership typically includes IT management, security, operations, application owners, and business representatives affected by changes.
CAB CharacteristicDetail
PurposeEvaluate risk; approve/reject/defer changes; resolve scheduling conflicts; ensure business alignment
CompositionIT ops, security, development, QA, business stakeholders, service desk, network/infrastructure leads
Meeting frequencyWeekly or as needed; emergency CAB convened ad-hoc
AuthorityDerives from management — CAB typically has authority to approve or schedule changes within defined scope
Back-Out Plan Requirement: Every change MUST have a documented back-out (rollback) plan before approval. The back-out plan describes exactly how to reverse the change if it fails or causes unexpected problems. Without a viable back-out plan, the change should NOT be approved.

5. Configuration Management

Configuration Management: The process of identifying, documenting, controlling, and auditing the configuration of IT systems throughout their lifecycle. Configuration management ensures systems remain in a known, approved, secure state.

Configuration Management Activities

ActivityDescription
IdentificationIdentify and document all Configuration Items (CIs) — hardware, software, documentation, network components
ControlEnsure changes to CIs follow the change management process; prevent unauthorized modifications
Status AccountingRecord and report the current status and history of each CI (version, location, owner, changes made)
Verification & AuditPeriodically verify that actual configurations match documented configurations; identify drift

Configuration Baseline

Configuration Baseline: A formally approved configuration of a system at a specific point in time — the "known good" state. All changes are measured FROM the baseline. After approved changes, a NEW baseline is established. Configuration drift occurs when actual system state deviates from the baseline without going through change management.

6. CMDB & Configuration Items

CMDB (Configuration Management Database): A centralized database that stores information about all CIs and their relationships. The CMDB is the authoritative source for "what do we have and how is it configured."

Configuration Item (CI) Examples

CI CategoryExamples
HardwareServers, workstations, routers, switches, firewalls, storage arrays
SoftwareOperating systems, applications, middleware, firmware, scripts
DocumentationProcedures, policies, network diagrams, architecture documents
NetworkIP addresses, VLANs, firewall rules, certificates, DNS records
ServicesBusiness services, dependencies, SLAs

CMDB Relationships

The power of CMDB is in relationships between CIs — for example, knowing that Server A hosts Application B, which depends on Database C, connected via Network D. During incident response, these relationships enable rapid impact analysis: "If Server A goes down, what is affected?"

⚠️ CMDB Accuracy Challenge: A CMDB is only valuable if it is accurate and current. Stale or inaccurate CMDB data provides false confidence. Automated discovery tools should continuously reconcile actual infrastructure with CMDB records. Any discrepancy is a potential unauthorized change.

7. Unauthorized Change Detection

Detection Methods

MethodDescriptionTools
File integrity monitoring (FIM)Monitors critical files for unauthorized modifications (hashes compared to baseline)Tripwire; OSSEC; AIDE; commercial EDR
Configuration assessmentCompares actual configuration against approved baseline/benchmarkCIS-CAT; vulnerability scanners; SCAP tools
CMDB reconciliationAutomated discovery compares actual infrastructure to CMDB recordsServiceNow Discovery; BMC Discovery; Qualys
Change log analysisSIEM correlates system changes with approved change ticketsSIEM; log analysis platforms
Exam Point — Unauthorized Change = Security Incident: An unauthorized change should be treated as a potential security incident until proven otherwise. It could indicate: a compromised system, an insider threat, a policy violation, or simply a process failure. The first response is to investigate, not to simply remediate.

8. Infrastructure as Code (IaC)

Infrastructure as Code (IaC): Managing and provisioning infrastructure through machine-readable definition files rather than manual configuration. IaC applies software engineering practices (version control, testing, review) to infrastructure management.

IaC Security Benefits

BenefitDescription
ConsistencyEvery deployment is identical — eliminates configuration drift and "snowflake" servers
Version controlInfrastructure definitions stored in Git — full audit trail of who changed what and when
Peer reviewInfrastructure changes go through code review (pull requests) before deployment
Automated testingSecurity checks (policy compliance, vulnerability scanning) can be automated in the deployment pipeline
Rapid recoveryRebuild infrastructure from code in minutes rather than manually reconfiguring
Immutable infrastructureRather than patching in place, deploy new instances from updated templates and destroy old ones

9. Memory Tricks & Mnemonics

🧠 Change Management — "REAIRC"
Request (RFC) → Evaluate → Approve → Implement → Review → Close
"Request Evaluated, Approved, Implemented, Reviewed, Closed"
🧠 Change Types — "SEN"
Standard (pre-approved, routine) → Emergency (urgent, ECAB) → Normal (full CAB process)
"SEN — from least process to most process"
🧠 Config Management — "ICSV"
Identify CIs → Control changes → Status accounting → Verification/audit
"I Control System Versions"
🧠 Back-Out Plan
"No back-out, no go-out." Every change needs a rollback plan before approval.
