4. Packet Filtering (Stateless)

📖 Packet Filtering Firewall: The simplest, oldest firewall type. Examines each packet in isolation and makes allow/deny decisions based on header fields only: source IP, destination IP, protocol (TCP/UDP/ICMP), source port, destination port. Also called stateless firewall or Access Control List (ACL) on a router.

How it Works

Packet Filtering Rule Set (ACL Example): Rule | Source IP | Src Port | Dest IP | Dest Port | Proto | Action ──────────────────────────────────────────────────────────────────────────── 1 | Any | Any | 10.0.0.10 | 80 | TCP | PERMIT ← Allow HTTP to web server 2 | Any | Any | 10.0.0.10 | 443 | TCP | PERMIT ← Allow HTTPS to web server 3 | 192.168.1.0/24│ Any | Any | 22 | TCP | PERMIT ← Allow SSH from admin subnet 4 | Any | Any | Any | Any | Any | DENY ← Implicit deny all (default) Rules are evaluated TOP-TO-BOTTOM; first match wins. Each packet is evaluated independently — no memory of previous packets.

Strengths and Weaknesses

StrengthsWeaknesses
Very fast — hardware-level performance; line rate processingCannot distinguish if TCP port 80 traffic is legitimate HTTP or an attack hiding on port 80
Simple to configure for basic access controlVulnerable to IP spoofing — no verification that source IP is genuine
No context — scales to any volume of trafficDoes not track connection state — cannot detect TCP-based attacks (e.g., ACK flood, Christmas tree scan)
Supports protocol-level filtering (TCP vs. UDP vs. ICMP)Cannot inspect payload content — SQL injection on port 443 passes right through
Commonly deployed on routers as edge ACLsRules can become complex to manage at scale (rule order matters; mistakes are common)
⚠️ Stateless Firewall and Established Connections: A stateless firewall enforcing "allow inbound TCP responses" must typically allow all inbound TCP with ACK set (because return traffic from established connections carries ACK). This also allows TCP ACK scanning and can allow TCP RST/FIN packets that weren't part of a real session. Stateful firewalls solve this by tracking which connections are legitimate.

5. Stateful Inspection Firewalls

📖 Stateful Inspection Firewall: Introduced by Check Point in the early 1990s (patented as "Stateful Inspection"). Tracks the state of all network connections passing through it using a state table (connection tracking table). Packets are evaluated in the context of their connection, not just their headers.

How State Tables Work

State Table (Connection Tracking Table): Src IP │ Src Port │ Dst IP │ Dst Port │ Proto │ State │ Timeout ──────────────┼──────────┼───────────────┼──────────┼───────┼──────────────┼──────── 192.168.1.10 │ 54201 │ 93.184.216.34 │ 443 │ TCP │ ESTABLISHED │ 3600s 192.168.1.10 │ 54202 │ 8.8.8.8 │ 53 │ UDP │ ESTABLISHED │ 30s 192.168.1.20 │ 54300 │ 10.0.0.5 │ 22 │ TCP │ SYN_SENT │ 120s When a packet arrives → check if it belongs to an existing state entry → YES: permit (it's return traffic from a known connection) → NO: match against policy rules; if allowed → create new entry → No match in policy → DENY Return traffic is automatically permitted without a separate inbound rule.
🔑 Why Stateful is Better than Stateless: With stateful inspection, you don't need to allow all inbound ACK packets. Instead, the firewall only allows return traffic for connections that were initiated from a trusted side. An incoming packet that doesn't match any state table entry is dropped — even if it has ACK set. This eliminates the entire class of attacks (ACK floods, session spoofing) that bypass stateless firewalls.

Stateful Firewall Limitations

LimitationWhy It Matters
Cannot inspect application payloadMalware, SQL injection, XSS in HTTP/S passes through. Stateful only sees "TCP/443 — established connection".
State table exhaustion (State DoS)Attacker sends many half-open connections (like SYN flood) to fill state table. Solutions: SYN cookie offload, connection timeout tuning, rate limiting.
Encrypted traffic is opaqueTLS-encrypted sessions are tracked as state but content is invisible. TLS inspection (SSL/TLS proxy) needed to see inside.
DNS/ICMP state is complexUDP has no handshake, so "state" for UDP is a synthetic concept (based on timing). ICMP state is similarly indirect.

6. Proxy / Application-Level Gateways

📖 Application-Level Gateway (Proxy Firewall): Operates at Layer 7. Acts as an intermediary (proxy) between client and server — the client connects to the proxy, the proxy makes a separate connection to the server, and relays content between them. The proxy can fully inspect and filter the application payload.
Proxy Firewall (Application-Level Gateway): Client ───────────────────────────────── Server │ │ │── Request ──▶ [PROXY FIREWALL] ──────────│ │ │ │ │ ├ Terminate client TLS │ │ ├ Inspect HTTP payload │ │ ├ Apply content policy │ │ ├ Re-encrypt to server │ │ └────────────────────────▶ │ (New connection to server) │ │ │◀── Response ── [PROXY FIREWALL] ◀───────│ (Inspects response too) Key: Two separate TCP connections — client↔proxy + proxy↔server The server never directly communicates with the client. The proxy's IP appears as the client's source IP to the server.

Proxy Firewall Characteristics

AspectDetail
Full Content InspectionCan block specific HTTP verbs, MIME types, malicious payloads, URL patterns, file types, keyword filtering
Application Protocol EnforcementEnforces valid protocol usage — a DNS proxy that only accepts valid DNS queries blocks DNS tunneling that would pass a port 53 rule on a stateless firewall
AnonymizationServer sees the proxy's IP as the client; true client IP is hidden — useful for both privacy and security
Performance ImpactSignificant overhead — must terminate every connection, inspect content, and re-establish. Not suitable for high-throughput, latency-sensitive traffic.
Protocol-SpecificA separate proxy is needed per protocol (HTTP proxy, FTP proxy, SMTP proxy, etc.) — not a generalist solution like packet filtering
TLS InspectionTo inspect HTTPS, the proxy performs a TLS man-in-the-middle (with authorized certificate). Clients must trust the proxy's CA certificate.
💡 Exam Tip: When a CISSP question asks "which firewall understands HTTP commands and can detect SQL injection in the URL?", the answer relates to application-level gateway / proxy / WAF — not stateful or packet-filtering. An application-level gateway protects against attacks that are valid at L4 but malicious at L7. This is the key distinction between firewall generations.

7. Circuit-Level Gateways

📖 Circuit-Level Gateway: Operates at Layer 5 (Session). Validates the TCP handshake — ensures the three-way handshake completes before proxying the session. Once the session is established and validated, all subsequent packets are forwarded without further inspection. SOCKS proxy is the classic example.
AspectDetail
Layer5 – Session; validates connection establishment only
InspectionVerifies TCP handshake; does NOT inspect application payload after the session is established
Classic ExampleSOCKS4/SOCKS5 proxy — routes TCP/UDP connections without understanding the application protocol
vs. Application GatewayCircuit-level: validates the handshake then forwards blindly. Application-level: validates AND inspects every byte of payload throughout the session.
Security LimitationOnce a session passes the handshake validation, malicious payload can flow freely — no content filtering
AdvantageProtocol-agnostic (works with any TCP/UDP application) and faster than application-level proxies
💡 Exam Tip: The CISSP exam consistently tests the circuit-level vs. application-level distinction. Remember: circuit-level = validates the handshake/session establishment (L5); application-level = validates the payload and commands (L7). SOCKS is circuit-level. HTTP proxy is application-level. This is a favourite CISSP trap question.

8. Next-Generation Firewalls (NGFW)

📖 NGFW (Next-Generation Firewall): A third-generation firewall that combines traditional stateful inspection with application awareness, user identity, IPS/IDS functionality, TLS/SSL inspection, and threat intelligence. Introduced by Palo Alto Networks in 2007 and now the industry standard for enterprise perimeter firewalls.

NGFW Capabilities vs. Traditional Firewall

CapabilityTraditional StatefulNGFW
Application IdentificationPort/protocol only (port 80 = "HTTP")Deep packet inspection identifies actual application regardless of port (Facebook on 443, Tor, BitTorrent)
User IdentityIP address onlyMaps traffic to user accounts via AD/LDAP integration — policy per user/group
Intrusion PreventionNo inline IPSBuilt-in IPS engine with signature + anomaly detection
TLS InspectionNo — treats TLS as opaqueSSL/TLS decrypt-inspect-re-encrypt for visibility into encrypted traffic
URL FilteringNoCategory-based web filtering; real-time URL reputation
Malware DetectionNoFile sandboxing; antivirus; threat intelligence
Policy Granularity"Allow TCP/443 from any to web server""Allow YouTube video streaming for marketing team only, deny file upload"
🔑 NGFW TLS Inspection (SSL Bumping): To inspect HTTPS traffic, the NGFW acts as a TLS MITM — it decrypts the client's TLS session, inspects the plaintext, re-encrypts to the server using a certificate it generates on-the-fly, signed by a corporate CA. Clients must be configured to trust this CA. Privacy implications must be addressed in policy. Selective decryption should be applied (e.g., exempt banking sites, healthcare).
⚠️ NGFW Does Not Replace All Security Controls: NGFW is powerful but is not the only security layer needed. It does not eliminate the need for endpoint security, patch management, application security (WAF), network segmentation, IAM, or monitoring/SIEM. Defence in depth still applies — assume any single control can be bypassed.

9. Web Application Firewalls (WAF)

📖 WAF (Web Application Firewall): A specialized application-layer firewall designed specifically to protect HTTP/S-based web applications. Inspects HTTP request/response headers and bodies for attacks including OWASP Top 10 vulnerabilities. Can be deployed inline (reverse proxy), out-of-band, or as a SaaS cloud service.

WAF vs. NGFW

AspectWAFNGFW
Protected ProtocolsHTTP/HTTPS onlyAll IP protocols
Protects FromOWASP Top 10, SQLi, XSS, CSRF, SSRF, broken authentication in web appsNetwork-level attacks, malware, botnets, IPS signatures, unauthorized applications
LayerL7 – specifically HTTP/S application semanticsL2–L7 generalist
PlacementIn front of web servers (reverse proxy); CDN edgePerimeter; between network zones
Rule TypesHTTP method restrictions; request rate limiting; injection pattern signatures; response filteringIP/port ACLs; application control; IPS signatures; URL categories
Virtual PatchingYes — WAF rules can block exploitation of vulnerabilities before the application is patchedPartial via IPS signatures, but less HTTP-specific
🔑 Virtual Patching: A WAF rule that blocks exploitation of a known vulnerability (e.g., a specific SQL injection pattern targeting an unpatched CMS) without modifying the application code. This is critical when vendor patches are delayed or when a patch cannot be deployed immediately due to change management timelines. Virtual patching is a key compensating control under PCI-DSS and other standards.

WAF Deployment Modes

ModeHow It WorksPros/Cons
Inline Reverse ProxyWAF sits between client and web server; all traffic passes through it; can block attacks in real time+ Full inspection and blocking; − Single point of failure; adds latency
Transparent BridgeWAF inserted at network level; transparent to clients and servers; no IP changes needed+ Easier deployment; − Complex bypass risk if misconfigured
Out-of-Band / PassiveWAF monitors a copy of traffic (via port mirroring/TAP); does not block inline+ No performance impact; − Detection only, cannot block attacks in real time
Cloud/SaaS WAFDNS points traffic to CDN/cloud WAF service (e.g., Cloudflare, AWS WAF, Akamai); traffic is scrubbed before reaching origin+ DDoS protection included; scalable; − Cloud provider sees all traffic; DNS dependency

10. Firewall Architectures

Bastion Host

📖 Bastion Host: A hardened host exposed to the internet (or untrusted network) that is the primary point of contact. Typically a single, purpose-built system that undergoes extensive hardening — all unnecessary services removed, all network exposure minimized. Examples: a hardened jump server, an exposed SMTP relay, a screened-subnet web server.

Screened Subnet (DMZ)

📖 Screened Subnet (DMZ — Demilitarized Zone): A network zone positioned between the internet and the internal network, protected by TWO firewalls (or two firewall interfaces). Servers that must be reachable from the internet (web, mail, DNS) are placed in the DMZ. Even if compromised, an attacker in the DMZ faces a second firewall to reach the internal network.
Classic Screened Subnet (Two-Firewall DMZ): Internet ──── [Outer FW] ──── DMZ ──── [Inner FW] ──── Internal Network │ ┌───────┴────────┐ │ Web Server │ │ Mail Relay │ │ DNS (External)│ └────────────────┘ Traffic Policies: Internet → DMZ: Permitted (e.g., TCP 80/443 to web server) DMZ → Internet: Permitted (e.g., outbound SMTP from mail relay) Internet → Internal: DENIED (no direct path) Internal → DMZ: Permitted (e.g., admin SSH to DMZ servers from management zone) DMZ → Internal: CONTROLLED (only specific traffic, e.g., DB queries from web server to internal DB) Internal → Internet: Via proxy or permitted (policy-dependent) The inner firewall is the critical chokepoint — if DMZ server is compromised, the attacker must defeat the second firewall to reach internal resources.

Dual-Homed Host

📖 Dual-Homed Host: A single host with two network interfaces — one connected to the untrusted network and one to the trusted network. IP forwarding is disabled — the host does not route packets between interfaces. Applications on the host proxy traffic between the two networks. A software-only firewall approach, less common in modern architectures.

Firewall Architecture Comparison

ArchitectureConfigurationSecurity LevelCISSP Scenario
Packet Filtering RouterSingle router with ACLs; no dedicated firewallLowest — no state, no content inspectionSmall org, limited budget; adds basic perimeter but insufficient for sensitive data
Screened Host (Single Firewall)One firewall; bastion host optionally in internal network facing internetMedium — single point of failure; single firewall breach = full accessSmall-medium org; adequate for non-critical internet exposure
Screened Subnet (Dual Firewall)Two firewalls with DMZ between them; public servers in DMZHigh — attacker must breach two firewalls; lateral movement contained to DMZStandard enterprise architecture; required for PCI-DSS compliance with cardholder data on internal servers
Dual-Homed HostOne multi-NIC host acting as proxy; IP forwarding disabledMedium — entire security on one host; host compromise = network compromiseLegacy / small deployment; not recommended for modern organizations
💡 Exam Tip — "Best Architecture" Questions: CISSP scenarios placing a web server accessible from the internet → answer is screened subnet (DMZ) with the web server in the DMZ and the database server in the internal network. The exam frequently asks about the placement of servers relative to firewall zones. Public servers go in DMZ; backend databases and internal services stay behind the inner firewall.

11. Firewall Rule Design & Policy

Core Firewall Policy Principles

PrincipleDescription
Default Deny (Implicit Deny All)Every rule set should end with "deny all" — only explicitly permitted traffic is allowed. This is the foundational security posture.
Least PrivilegeRules should permit the minimum traffic necessary for business function — not ranges like "allow TCP any any"
Rule OrderingMore specific rules before general rules; permit/deny logic first; most-matched rules should be near the top (performance)
Egress FilteringFilter outbound traffic as carefully as inbound — data exfiltration, C2 beaconing, and botnet participation all require outbound connectivity
Ingress Filtering (BCP38)Drop inbound packets with source IPs from RFC 1918 ranges (impossible on internet) and the organization's own address space (spoofed)
Logging EverythingAll permit and deny actions (especially denies) must be logged with timestamp, source/dest IP, port, protocol, and action for forensics and compliance
Regular Rule ReviewRules accumulate over years; orphaned rules for decommissioned systems or expired projects expand attack surface; quarterly review minimum

What Firewalls Cannot Do

LimitationWhyCompensating Control
Stop insider threatsAuthorized users with valid credentials pass firewall rules legitimatelyDLP, UEBA, privileged access management, separation of duties
Inspect encrypted traffic (without SSL inspection)TLS content is opaque at L3/L4TLS inspection on NGFW; endpoint EDR
Prevent social engineering / phishingPhishing uses legitimate ports (HTTPS/443) and protocolsEmail gateway, security awareness training, URL filtering
Protect against zero-days in allowed servicesPermitted traffic (e.g., HTTPS to web server) may carry zero-day exploitsWAF, IPS, sandboxing, application patching
Prevent data exfiltration via allowed channelsOutbound HTTPS to cloud services is typically permittedDLP, CASB, egress filtering, application control

12. Memory Tricks & Mnemonics

🧠 Firewall Generation Order — "Packets, State, Circuit, App, Next":
1. Packet Filtering (stateless, L3/L4, headers only)
2. Stateful Inspection (state table, connection tracking, L3/L4)
3. Circuit-Level Gateway (L5 session validation — SOCKS)
4. Application-Level Gateway / Proxy (L7, full payload, protocol-specific)
5. NGFW (all of the above + app ID + user + IPS + TLS)
WAF = specialized L7 for HTTP/S only (sits alongside, not in the progression)
🧠 Circuit vs. Application Gateway — "Handshake vs. Payload":
Circuit-Level = validates the HANDSHAKE (TCP 3-way); then passes all traffic blindly; SOCKS proxy
Application-Level = validates the PAYLOAD throughout; blocks commands / content; HTTP/SMTP/FTP proxy
Memory: "Circuit: card gets swiped at the door. App: security checks your bag AND your ID throughout the journey."
🧠 DMZ Placement — "Public in DMZ; Private Behind Inner Firewall":
Web servers, mail relays, external DNS → DMZ (between two firewalls)
Databases, HR systems, internal applications → Internal (behind inner firewall)
Management systems → separate management VLAN/zone
"If the internet sees it, it lives in the DMZ."
🧠 Firewall Default Posture — "Default Deny, Explicitly Permit":
A firewall without a "deny all" at the end effectively permits everything not explicitly denied.
"If it's not on the guest list, it doesn't get in."
The implicit deny rule is the most important firewall rule — and it's invisible.
🧠 WAF vs. NGFW — "WAF = Web App; NGFW = Everything":
WAF: HTTP/HTTPS only; OWASP attacks; virtual patching; reverse proxy in front of web app
NGFW: All protocols; user + app + IPS + URL + threat intel; at the perimeter
Both needed in full security stack — they protect different layers.
