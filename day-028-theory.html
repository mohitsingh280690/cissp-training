2. Real-World Relevance

Data security controls are what make classification policy real. Without technical enforcement, a "Confidential" label is just a word on a document. These controls enforce the policies defined by data owners, custodians, and governance frameworks — preventing unauthorised access, exfiltration, and misuse.

3. Encryption Controls

Encryption at Rest

Protects data stored on physical or logical media:

  • AES-256: The standard algorithm for data at rest; used for file/folder encryption, database encryption, full disk encryption
  • Full Disk Encryption (FDE): Encrypts the entire storage volume; examples: BitLocker (Windows), FileVault (macOS), VeraCrypt (cross-platform); protects against physical theft of devices
  • Transparent Database Encryption (TDE): Database-level encryption (SQL Server, Oracle, MySQL) that encrypts data files, log files, and backups transparently — applications interact with decrypted data; encryption/decryption happens at the database engine layer
  • Self-Encrypting Drives (SEDs): Storage devices with built-in hardware encryption; cryptographic erasure is as simple as destroying/resetting the drive-internal key

Encryption in Transit

Protects data moving across networks:

  • TLS 1.2/1.3: Transport Layer Security — standard for HTTPS, SMTPS, IMAPS, and other application protocols; TLS 1.3 is preferred (TLS 1.0/1.1 are deprecated)
  • IPSec: Network-layer encryption; Tunnel mode (site-to-site VPNs — encrypts entire packet including headers); Transport mode (host-to-host — encrypts payload only)
  • SSH: Secure Shell — encrypted remote administration and SFTP file transfer
  • VPN: Virtual Private Network — extends an encrypted tunnel over public networks; uses IPSec, SSL/TLS, or WireGuard

Encryption in Use

Protects data while it is being actively processed in memory:

  • Homomorphic encryption: Allows computation to be performed on encrypted (ciphertext) data without decrypting it first; results when decrypted match results from operating on unencrypted data; currently computationally expensive but promising for cloud computing privacy
  • Secure enclaves (Intel SGX): Hardware-isolated memory regions (enclaves) within the CPU; applications run and process data inside the enclave; enclave memory is encrypted and inaccessible even to the OS, hypervisor, or BIOS; Intel SGX is the most prominent example
  • TEE — Trusted Execution Environment: Broader term for isolated secure processing environments (hardware and software); includes ARM TrustZone; used in mobile payments, DRM, and attestation

5. DRM and IRM

DRM — Digital Rights Management

DRM: Technologies and policies that control access to and use of digital content (copying, printing, forwarding, screen capture) by enforcing rights embedded in the content itself. Protection travels with the document.

DRM use cases: Music and video streaming platforms (Netflix, Spotify); e-books (Kindle); software licencing; government classified documents.

IRM — Information Rights Management

IRM: Enterprise-focused DRM for business documents. Applies persistent protection to Office documents, PDFs, and emails. Users can be restricted from copying, printing, forwarding, or editing specific documents. Access can be revoked even after distribution.

IRM examples: Microsoft Azure Information Protection (AIP) / Microsoft Purview; Adobe LiveCycle Policy Server. IRM can enforce: "View only – no print – no copy – expires 30 days."

DRM vs. IRM — The Distinction:
DRM = General term, often consumer content protection
IRM = Enterprise version for business documents
Both provide persistent protection that travels with the content regardless of where the file is saved or sent.

6. Digital Watermarking

Digital Watermarking: Embedding identifying information (ownership, recipient identity) into digital media (images, documents, audio, video) to prove ownership and trace the source of unauthorised copies or leaks.
  • Overt (visible) watermarks: Visible to the user — "CONFIDENTIAL" overlay text on a document; a company logo stamped on an image. Deters casual copying but can be cropped or removed.
  • Covert (invisible/steganographic) watermarks: Hidden within the digital content without visible distortion; detectable only by the watermarking software or algorithm; survives format conversion and printing-and-scanning. Used for forensic investigation — if a document leaks, extracting the watermark reveals the specific copy that was distributed and who received it.
⚠️ Important Distinction: Watermarking does NOT prevent unauthorised copying or distribution. It identifies the source after the fact. For preventing distribution, use DRM/IRM. For proving source after a breach, use watermarking. These are complementary controls used together.

7. Database Security Controls

  • Database Views: Virtual tables that present a subset of columns and rows to specific users; achieve least privilege at the data layer without modifying the underlying table structure. Example: finance users see a view that shows all columns; call centre users see a view with SSN masked.
  • Stored Procedures: Pre-compiled SQL code that applications call by name instead of sending raw SQL; the primary defence against SQL injection at the application layer; parameterised queries are the key technique.
  • Column-level encryption: Encrypt specific sensitive columns (SSN, credit card, passwords) rather than the entire database; even authenticated users cannot read the column values without the decryption key.
  • Row-level security (RLS): Restricts which rows a user can access based on their identity or role; e.g., a salesperson sees only their own customers' records, not all customers.
  • Database Activity Monitoring (DAM): Real-time monitoring and logging of all database queries and actions; alerts on anomalous activity (mass SELECT of sensitive columns, after-hours access, access from unexpected IP); supports forensic investigation.
  • Privilege separation: Database admin (DBA) role should be separate from Security Admin; DBAs should not have access to the application's data they administer.

8. Cloud Data Security

  • Data residency / data sovereignty: Regulations may require data to remain in specific jurisdictions; cloud providers must be contracted to respect data residency requirements
  • CSP Shared Responsibility Model: In IaaS, the cloud customer is responsible for OS security, application security, and data security; the CSP is responsible for physical infrastructure, virtualisation layer, and network. In SaaS, the CSP handles most security — the customer controls data and access management.
  • CASB — Cloud Access Security Broker: A security policy enforcement point between cloud service users and cloud service providers. CASB capabilities:
    • Visibility: Discover what cloud services are being used (including Shadow IT)
    • Data Security: Enforce DLP policies across cloud storage and sharing
    • Threat Protection: Detect anomalous user behaviour and malicious activity in cloud apps
    • Compliance: Enforce compliance policies (e.g., prevent uploading unencrypted PII to cloud storage)
  • Cloud key management: Who controls the encryption keys? Customer-managed keys (CMK) provide the highest control — the CSP cannot access data without the customer's key. Bring Your Own Key (BYOK) and Hold Your Own Key (HYOK) are variants.

9. Memory Tricks & Mnemonics

🧠 Access Control Models — "DAC = Discretionary (owner), MAC = Mandatory (system), RBAC = Role, ABAC = Attributes":
D = Discretionary = Data owner decides (most flexible, least restriction)
M = Mandatory = Military / government (system enforces labels; most restrictive)
R = Role = Enterprise standard (group-based, admin assigns)
A = Attributes = Advanced / cloud / zero-trust (policy engine evaluates multiple factors)
🧠 DRM vs. Watermarking — "Rights vs. Reveals":
DRM/IRM = Controls RIGHTS — prevents unauthorised copying, forwarding, printing
Watermarking = REVEALS the source — proves who leaked it, after the fact
"DRM stops the leak; Watermark names the leaker"
🧠 CASB = The Cloud Security Sheriff:
Visibility (what cloud services are used?)
Data security (DLP in the cloud)
Threat protection (anomalous behaviour)
Compliance enforcement (no unencrypted PII uploads)
