2. Real-World Organizational Relevance

The SolarWinds breach (2020) compromised ~18,000 organizations via a trojanized software update. Log4Shell (2021) affected millions of applications using a single open-source library. These attacks proved that your security is only as strong as your weakest dependency. Executive Order 14028 now mandates SBOM for US federal software procurement. Organizations must inventory, monitor, and govern every piece of third-party code in their environment.

3. Software Acquisition Types

TypeDescriptionSecurity Considerations
COTS (Commercial Off-the-Shelf)Pre-built commercial software purchased from vendor (Microsoft Office, SAP, Oracle)Vendor controls security; limited customization; reliance on vendor patching; evaluate vendor security posture
Open SourceSource code freely available; community-developed (Linux, Apache, OpenSSL)Transparent code (auditable); community patches; risk of abandoned projects; license compliance
Custom (in-house)Built by organization's own developersFull control; highest cost; organization responsible for all security testing and patching
OutsourcedCustom software built by external contractor/vendorContractual security requirements; code escrow; IP ownership; vendor assessment; secure coding standards in contract
Cloud/SaaSSoftware as a Service; vendor hosts and maintainsShared responsibility; limited security visibility; vendor lock-in; data sovereignty concerns
COTS vs Custom Trade-off:
• COTS: Lower cost, less customization, dependent on vendor for security patches
• Custom: Higher cost, full control, organization bears all security responsibility
• CISSP questions focus on understanding the security implications and risk ownership for each acquisition type.

4. Supply Chain Risk Management

Software Supply Chain Attack: Compromising a software product by attacking a component in its supply chain — the vendor, a library, the build process, the distribution channel, or the update mechanism. The attacker targets the weakest link to reach many downstream victims.

Notable Supply Chain Attacks

AttackYearVectorImpact
SolarWinds2020Trojanized build process; malicious update (Orion)~18,000 organizations including US government agencies
Log4Shell2021Critical vulnerability in Log4j open-source libraryMillions of Java applications worldwide; zero-day RCE
Codecov2021Compromised CI/CD script exfiltrated environment variablesCredentials leaked from hundreds of customer CI pipelines
3CX2023Supply chain compromise of desktop VoIP application600,000+ organizations; malicious DLL in signed application

Supply Chain Security Controls

Code signing verificationVerify digital signatures on software/updates before installation
Secure distribution channelsDownload only from official sources; verify checksums/hashes

Build integrityReproducible builds; SLSA framework; secure build environments
Contractual requirementsSecurity clauses in vendor contracts; right to audit; incident notification requirements

5. Software Bill of Materials (SBOM)

SBOM: A comprehensive, machine-readable inventory of all components, libraries, and dependencies that make up a software product. Like a nutrition label for software — tells you exactly what's inside.
SBOM ElementInformation
Component nameName of each library/package/module
VersionExact version number of each component
SupplierWho created/maintains the component
Dependency relationshipsHow components relate to each other (direct vs transitive dependencies)
LicenseLicense type for each component (MIT, GPL, Apache, etc.)
Hash/checksumCryptographic hash to verify component integrity

SBOM Standards

StandardDescription
SPDXSoftware Package Data Exchange — Linux Foundation; ISO/IEC 5962 standard
CycloneDXOWASP standard; designed for security use cases; supports vulnerability data
SWID TagsSoftware Identification Tags — ISO/IEC 19770-2; focused on installed software inventory
SBOM Value: When a vulnerability like Log4Shell is announced, an SBOM lets you instantly determine if your software is affected — instead of spending days or weeks investigating. US Executive Order 14028 (2021) mandates SBOM for software sold to the federal government.

6. Open Source Software Risks

RiskDescriptionMitigation
Known vulnerabilitiesCVEs in popular libraries; may persist if not patched promptlySCA scanning; automated dependency updates; vulnerability monitoring
Abandoned projectsMaintainer stops updating; no security patches for new vulnerabilitiesEvaluate project health (commit frequency, contributor count, responsiveness)
TyposquattingMalicious packages with names similar to popular ones (e.g., "requets" vs "requests")Verify exact package names; use lockfiles; SCA to detect malicious packages
Dependency confusionAttacker uploads malicious package to public registry with same name as private internal packageNamespace reservation; private registry configuration; scoped packages
License complianceCopyleft licenses (GPL) may require releasing your code as open sourceSCA license scanning; legal review; policy on acceptable licenses
Transitive dependenciesYour code depends on A, which depends on B, C, D… — vulnerabilities deep in the chainSBOM with full dependency tree; lock file management; SCA for transitive deps
Copyleft License Risk (GPL): If your proprietary software incorporates GPL-licensed code, you may be legally required to release YOUR source code under the same GPL license. This is a significant IP risk. Always have legal review open-source licenses before incorporation. Permissive licenses (MIT, Apache, BSD) are generally safe for commercial use.

7. Vendor Assessment & Contractual Controls

Vendor Assessment Methods

MethodDescription
Security questionnairesStandardized questions about vendor's security practices (SIG, CAIQ); self-reported
SOC 2 reportsIndependent auditor's report on vendor's controls (Type I = point-in-time; Type II = over time period)
ISO 27001 certificationThird-party certified ISMS; demonstrates systematic security management
Penetration testing reportsRequest or review vendor's pen test results; verify they test regularly
On-site auditsPhysical inspection of vendor's facilities and practices; most thorough; most expensive
Continuous monitoringSecurity ratings services (BitSight, SecurityScorecard); ongoing vendor risk monitoring

Contract Security Clauses

ClausePurpose
Right to auditOrganization can audit vendor's security practices and compliance
Incident notificationVendor must notify within X hours of a security breach; define what constitutes an incident
SLA for patchingVendor commits to patch critical vulnerabilities within defined timeframes
Data handling requirementsHow vendor stores, processes, transmits, and destroys organization's data
Code escrowSource code held by trusted third party; released if vendor goes bankrupt or fails to maintain
Subcontractor flow-downSecurity requirements extend to vendor's subcontractors
Termination & data returnHow data is returned/destroyed when contract ends
SOC 2 Type I vs Type II:
• Type I = Design of controls at a POINT IN TIME (snapshot)
• Type II = Effectiveness of controls OVER A TIME PERIOD (typically 6-12 months)
• Type II is always MORE valuable because it proves controls actually work over time, not just that they exist.

8. Software Composition Analysis (SCA)

SCA CapabilityWhat It Does
Dependency inventoryIdentifies all open-source and third-party components in the codebase
Vulnerability matchingMatches components against CVE/NVD databases; alerts on known-vulnerable versions
License detectionIdentifies all open-source licenses; flags copyleft or incompatible licenses
SBOM generationProduces SPDX or CycloneDX SBOM documents
Remediation guidanceRecommends version upgrades that fix vulnerabilities
Policy enforcementBlock builds that include components violating security or license policies

Popular SCA Tools: Snyk, Black Duck (Synopsys), WhiteSource (Mend), OWASP Dependency-Check, GitHub Dependabot, Sonatype Nexus.

9. Memory Tricks & Mnemonics

🧠 Software Acquisition — "COCOC" (Compare Options Carefully, Outsource Cautiously)
COTS — Buy off the shelf
Open source — Free, inspect the code
Custom — Build it yourself
Outsource — Hire someone to build
Cloud/SaaS — Rent from vendor
🧠 SOC Types — "1 = Snapshot, 2 = Sustained"
Type I = point In time (design only)
Type II = over time perIIod (design + effectiveness)
🧠 SBOM = "Software's Nutrition Label"
Just as food labels list ingredients, SBOM lists every software component. When a recall happens (vulnerability), you check the label to see if you're affected.
