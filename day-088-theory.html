2. Real-World Organizational Relevance

Penetration testing is a services-based engagement — organizations either employ internal red teams or hire external firms. A typical enterprise pen test costs $15,000–$100,000+ depending on scope and duration. PCI DSS Requirement 11.4 mandates annual penetration testing of the CDE. HIPAA requires periodic testing of "technical controls" (often interpreted as pen testing).

The output is a findings report with exploitation evidence — screenshots, proof-of-concept (PoC) code, network captures showing successful attacks. This differentiates pen test output from vulnerability scan output (which only shows what was found, not what is actually exploitable).

⚠️ Legal Warning: Penetration testing without explicit written authorization is illegal under the Computer Fraud and Abuse Act (CFAA), UK Computer Misuse Act, and equivalent laws globally. Even internal employees can violate CFAA by testing systems without written authorization from the system owner.

3. What Is a Penetration Test?

Penetration Test: An authorized, simulated cyber attack against a system, network, or application to identify vulnerabilities that can be exploited and to evaluate the effectiveness of security controls. Unlike vulnerability assessment, pen testing DOES involve exploitation to prove that a vulnerability can result in real unauthorized access or data exposure.

A penetration test's value is in proving risk — not just listing potential issues. The pen tester demonstrates: "I can log in as administrator using CVE-2021-XXXX" or "I can read the production database contents by exploiting this SQL injection."

4. Penetration Testing Frameworks

PTES — Penetration Testing Execution Standard

PTES defines seven phases: Pre-engagement interactions, Intelligence gathering, Threat modeling, Vulnerability analysis, Exploitation, Post-exploitation, Reporting. It is widely used as a comprehensive methodology for black-box and gray-box engagements. Available at pentest-standard.org.

OSSTMM — Open Source Security Testing Methodology Manual

OSSTMM (by ISECOM) is the broadest methodology — covers not only network/application testing but also physical security, social engineering, wireless, telecommunications, and data networks. It formalized the quantitative measurement of Attack Surface using "attack surface area" in its scoring. Produces quantitative metrics rather than just qualitative findings. Best for organizations needing comprehensive physical and technical combined assessment.

NIST SP 800-115 — Technical Guide to Information Security Testing and Assessment

NIST SP 800-115 is the US federal government's standard technical guide for security testing. Defines the phases: Planning, Discovery, Attack, Reporting. Widely used for government system assessments and by organizations following NIST RMF. Includes vulnerability scanning, penetration testing, and review techniques (log review, ruleset review, documentation review).

OWASP Testing Guide (OTG)

OWASP Testing Guide focuses specifically on web application penetration testing. Organized around OWASP Top 10 risk categories (injection, broken authentication, XSS, etc.). Most relevant for web application security assessments; not a general network pen test framework.

5. Phases of a Penetration Test

PHASE 1: Planning & Pre-Engagement ├── Define scope, permissions, objectives ├── Sign Statement of Work (SOW) + Rules of Engagement (ROE) └── Emergency contacts, escalation procedures PHASE 2: Reconnaissance (Intelligence Gathering) ├── Passive recon (OSINT — no direct target interaction) └── Active recon (direct probing — DNS queries, port scans) PHASE 3: Scanning / Discovery ├── Vulnerability scanning (Nessus, OpenVAS) ├── Port scanning (Nmap) └── Service enumeration; OS fingerprinting PHASE 4: Exploitation ├── Attempt to exploit discovered vulnerabilities ├── Use exploit frameworks (Metasploit, Cobalt Strike) └── Establish initial access PHASE 5: Post-Exploitation ├── Privilege escalation (local → admin → domain admin) ├── Lateral movement (pivot to other systems) ├── Persistence establishment (backdoor, C2) └── Data exfiltration simulation; demonstrate impact PHASE 6: Reporting ├── Executive summary (risk level, business impact) ├── Technical findings (vulnerability + evidence + CVSS) └── Remediation recommendations

Phase Detail — Planning

The most critical phase from a legal and organizational standpoint. Defines scope (which systems are in-scope), timing (when testing may occur), contacts (who to call if something breaks or alarms trigger), rules of engagement (what is allowed and what is off-limits), and legal authorization (signed document authorizing the testing).

Phase Detail — Exploitation

This is the key differentiator from a vulnerability assessment. The tester actively attempts to gain unauthorized access using discovered vulnerabilities. Tools include Metasploit (exploit framework), Cobalt Strike (commercial red team platform), and custom exploits. The goal is to gain an initial foothold.

Phase Detail — Post-Exploitation

After gaining initial access, the tester demonstrates what an attacker could do with that access. This includes privilege escalation, lateral movement, accessing sensitive data, and establishing persistence. This phase provides the most meaningful evidence of real-world risk — showing the business impact, not just the technical vulnerability.

6. Reconnaissance Techniques (Passive vs. Active)

TypeDefinitionExamplesDetectability
Passive Reconnaissance (OSINT)Gathering intelligence WITHOUT directly interacting with the target's systems. Uses publicly available information.Whois lookups, Google dorking, Shodan searches, LinkedIn employee enumeration, certificate transparency logs (crt.sh), DNS records via third-party services, job postings revealing technology stackVery low — no traffic sent to target systems
Active ReconnaissanceDirectly probing target systems to gather information. Generates traffic that reaches (and can be detected by) the target.Nmap port scan, DNS zone transfer attempt, web application crawling, SMTP enumeration, OS fingerprinting, banner grabbingMedium to high — IDS/SIEM can detect scanning activity
OSINT Tools for Passive Recon:
• Shodan — search engine for internet-connected devices; finds exposed cameras, ICS, servers
• Maltego — visualizes relationships between entities (domains, IPs, people, organizations)
• theHarvester — collects emails, hosts, subdomains, IPs from public sources
• crt.sh — certificate transparency log search; reveals subdomains
• Recon-ng — open-source OSINT framework
Exam Tip — Passive vs. Active: Passive recon does NOT touch the target — it uses information others have gathered (search engines, registration databases, public DNS). Active recon DOES touch the target — a port scan is active recon even though it precedes exploitation.

7. Exploitation & Post-Exploitation

Exploitation Tools

ToolTypePurpose
Metasploit FrameworkOpen source exploit frameworkContains thousands of exploits, payloads, and auxiliary modules; most widely used pen test exploitation platform
Cobalt StrikeCommercial red team platformAdvanced C2 (command and control) capabilities; beacon implants; post-exploitation automation
Burp SuiteWeb application testingIntercepts HTTP/HTTPS traffic; fuzzing; scanner; web app exploitation
SQLmapAutomated SQL injectionDetects and exploits SQL injection vulnerabilities in web apps
MimikatzCredential harvestingExtracts Windows credentials (NTLM hashes, Kerberos tickets) from memory

Post-Exploitation Objectives

  • Privilege Escalation: Move from low-privileged access to administrator/SYSTEM/root
  • Credential Dumping: Extract password hashes for offline cracking (Mimikatz, impacket)
  • Lateral Movement: Use compromised system as pivot point to attack other internal systems (impossible with VPN not in scope)
  • Persistence: Install backdoor, create rogue admin account, add scheduled task — to simulate attacker maintaining access
  • Data Exfiltration Simulation: Show that sensitive data could be extracted (demonstrate impact to business)

8. Reporting & Rules of Engagement

Rules of Engagement (ROE)

The ROE document defines everything permitted and prohibited during the test. Without ROE, the pen tester has no legal protection.

ROE ComponentExample Content
ScopeIP ranges, domains, applications in-scope; explicit out-of-scope list
Testing Window"Testing allowed 8PM–6AM weekdays and weekends only" — to minimize business impact
Authorized TechniquesSocial engineering allowed? Physical access testing allowed? DoS allowed?
Notification ContactsWho to call if a critical system is accidentally taken down
Escalation ProceduresWhat to do if active exploitation of a 3rd party found; criminal activity discovered
Emergency Stop ConditionsTester must immediately stop and notify if real attacker discovered on network
Evidence HandlingHow to secure and destroy data gathered during the test; data classification

9. Memory Tricks & Mnemonics

🧠 Pen Test Phases — "Please Run Six Exploits Past Reporting"
Planning → Reconnaissance → Scanning → Exploitation → Post-exploitation → Reporting
🧠 Framework Quick Reference
PTES = General pen test (most popular, 7 phases)
OSSTMM = Everything including Physical (Open Source = comprehensive)
NIST 800-115 = Government/federal (4 phases: Plan/Discover/Attack/Report)
OWASP OTG = Web apps only
🧠 Passive vs. Active Recon
Passive = you don't touch them (read public info — Google, Shodan, Whois)
Active = you send packets to them (port scan, DNS query, banner grab)
🧠 Why Post-Exploitation Matters
"Initial access is not impact" — the POST-exploitation phase shows whether a foothold can be used to cause real damage (domain admin, data exfil, lateral spread). Without post-exploitation, a pen test only proves entry, not consequence.

Top Exam Points:
  • OSSTMM is unique for covering physical and wireless — PTES and NIST 800-115 are more network/IT focused
  • Passive recon = no packets sent to target; active recon = packets sent to target
  • Post-exploitation proves IMPACT — the difference between "this is technically vulnerable" and "we can steal your data"
  • Rules of Engagement provide legal protection for pen testers
  • Pen testing does NOT require exploiting every vulnerability — only testing to prove a finding within scope
