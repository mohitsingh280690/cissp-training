1. Where This Fits in CISSP

AspectDetail
Domain4 – Communication and Network Security (13%)
Why TestedUnderstanding the attack landscape is essential for designing defences. CISSP tests attack mechanics to ensure candidates select appropriate countermeasures. The exam typically presents an attack scenario and asks for the correct mitigation, or describes a mitigation and asks which attack it prevents. DoS/DDoS, MitM, session hijacking, and ARP/DNS poisoning are the most heavily tested.
Typical Question Styles"A SYN flood consumes all server connection slots — which countermeasure?"; "An attacker intercepts traffic between two hosts without their knowledge — what attack?"; "Packets arrive from an IP address belonging to the internal network from the external interface — what attack and what countermeasure?"; "A session token is captured and reused — what attack?"
Cross-Domain LinksDay 56 (TCP, ICMP, ARP — the protocols being abused); Day 57 (DNS attacks detailed); Day 58 (ARP/CAM flooding); Day 61 (IDS/IPS detects these attacks); Day 60 (firewalls mitigate many of these attacks); Domain 7 (incident response to network attacks)
Key Exam TrapsDoS = single source; DDoS = distributed/botnet. Mitigating DDoS requires upstream ISP / cloud scrubbing — a single on-premise firewall cannot absorb volumetric DDoS. Smurf attack uses broadcast ICMP + spoofed source IP. SYN cookies mitigate SYN flood — not just "block all SYN." Replay attack ≠ session hijacking (replay reuses previously captured tokens; hijacking steals live sessions).

2. Real-World Organizational Relevance

Attack TrendOrganizational Context
DDoS scaleDDoS attacks have reached multi-Tbps volumes (Cloudflare mitigated a 5.6 Tbps attack in 2024). No on-premise device can absorb this — cloud scrubbing services (Cloudflare, Akamai, AWS Shield Advanced) are mandatory for internet-facing services.
MitM in enterpriseCorporate network MitM via ARP poisoning remains common in internal red team exercises. Encrypted protocols (HTTPS, SSH) mitigate most MitM impact but poor certificate validation (accepting any cert) re-opens the attack surface.
Session hijackingWeb application session hijacking via XSS (cookie theft) is an OWASP Top 10 issue. Network-level session hijacking of TCP sessions is rare on encrypted networks but still relevant for unencrypted services.
Botnet prevalenceMost DDoS attacks use compromised IoT devices and home routers. Organizations should monitor for outbound C2 traffic — compromised internal devices participating in botnets generate significant outbound traffic and expose the organization to legal liability and ISP blacklisting.

3. DoS & DDoS Attacks

📖 DoS (Denial of Service): An attack from a single source intended to render a resource unavailable to legitimate users. Exploits either a vulnerability (crash the service) or resource exhaustion (consume all CPU/memory/bandwidth/connections).
📖 DDoS (Distributed Denial of Service): DoS attack using multiple sources simultaneously (hundreds to millions of compromised devices — a botnet). Volumetric DDoS can saturate internet uplinks with legitimate-looking traffic that no single victim can defend against at the network edge.
Attack TypeMechanismTarget ResourceCountermeasure
SYN FloodSend thousands of TCP SYN packets with spoofed source IPs. Server allocates state for each (half-open connection) waiting for the third ACK that never arrives. TCP half-open connection table fills up; legitimate connections refused.Server memory / connection table (Layer 4)SYN cookies: server doesn't allocate state until ACK received (cryptographic token in SYN-ACK); rate limiting SYN per IP; firewall/IPS SYN rate limits
UDP FloodSend massive volumes of UDP packets to random ports. Target must process each packet, check if a service is listening, and send ICMP "unreachable" — exhausting CPU and bandwidth.CPU + bandwidth (Layer 4)Rate limiting; cloud scrubbing; upstream black-hole routing (RTBH); ISP coordination
ICMP Flood (Ping Flood)Overwhelm target with ICMP Echo Request packets (pings), forcing continuous Echo Reply responses and consuming bandwidth/CPU.Bandwidth + CPURate limit ICMP at firewall; block ICMP from untrusted sources; cloud scrubbing
Smurf AttackSend ICMP Echo Requests with the victim's IP spoofed as source to a network broadcast address. Every host on the broadcast domain replies to the victim — amplifying the attack by the number of hosts on that network.Victim bandwidth (amplification)Disable broadcast forwarding on routers (no ip directed-broadcast); ingress filtering (BCP38) prevents IP spoofing; deprecated by modern ISPs
HTTP FloodSend high volumes of legitimate-looking HTTP GET/POST requests to exhaust web server resources (CPU for request processing, database queries). Harder to filter than volumetric UDP/ICMP — looks like real user traffic.Web server CPU/application resources (Layer 7)WAF rate limiting; CAPTCHA challenges; CDN absorption; bot detection (JavaScript challenge)
Slow HTTP (Slowloris)Open many HTTP connections to a server but send headers very slowly — keeping the connection open indefinitely. Server's connection limit fills with half-completed requests; legitimate users cannot connect. Requires very low bandwidth from attacker.Web server connection slotsConnection timeout configuration; reverse proxy (nginx); limiting connections per IP; WAF

4. DDoS Amplification Attacks

📖 Amplification Attack: The attacker sends a small request with a spoofed source IP (victim's IP) to a reflector server that generates a much larger response. The large response is directed to the victim — not the attacker. Amplification factor = response size / request size. The attacker generates a small amount of traffic and the reflectors generate the massive attack volume.
ProtocolAmplification FactorMechanismDefence
DNS Amplification28–73×Small DNS ANY query (60 bytes) → large DNS response (3,000+ bytes). Use open DNS resolvers as reflectors.Block open DNS resolvers; BCP38 (prevent IP spoofing at ISP); Response Rate Limiting (RRL) on DNS servers
NTP Amplification (monlist)200–556×NTP monlist command returns list of last 600 clients — small request generates massive response. NTP servers as reflectors.Disable NTP monlist (ntpdc -c monlist should fail); update NTP to version 4.2.7p26+; BCP38
SSDP Amplification30×Simple Service Discovery Protocol (upnP devices); M-SEARCH request → large device descriptor responseBlock inbound SSDP (UDP 1900) from internet; disable UPnP on routers
Memcached AmplificationUp to 51,000×Memcached UDP (port 11211) — stores cached data; forged request returns entire cache entry. 15-byte request → 750KB response.Disable Memcached UDP (TCP only); bind to localhost; never expose Memcached to internet
🔑 Root Cause of All Amplification Attacks: Two requirements: (1) IP spoofing allowed by ISP (violates BCP38), (2) servers running services with asymmetric request/response sizes that respond to unauthenticated queries. BCP38 (ingress filtering at ISP network edge) eliminates spoofed source IPs — the fundamental fix at the internet infrastructure level.

5. Man-in-the-Middle (MitM) Attacks

📖 MitM (Man-in-the-Middle): The attacker secretly positions themselves between two communicating parties, intercepting and possibly altering communications. Both parties believe they are communicating directly with each other, while the attacker reads, modifies, or injects traffic.
MitM TechniqueLayerMechanismDefence
ARP Poisoning MitML2Attacker sends gratuitous ARP replies mapping attacker's MAC to gateway's IP. Victims send traffic to attacker's MAC. Attacker forwards to real gateway (transparent relay). Full traffic interception on LAN.Dynamic ARP Inspection (DAI); static ARP entries for gateways; 802.1X; encrypted protocols (TLS)
SSL StrippingL7Attacker (as intercepting proxy) downgrades HTTPS connection: responds to client with HTTP while maintaining HTTPS to the server. Client sees HTTP; traffic is plaintext to attacker.HSTS (HTTP Strict Transport Security) — browser refuses HTTP after first HTTPS visit; HSTS preloading; only HTTPS links in application
BGP HijackingL3Attacker announces more specific route for victim's IP prefix — internet routes traffic to attacker's AS. Attacker can inspect (and forward) traffic before routing it onward. (See Day 59)RPKI + ROA; BGP authentication; AS-path filtering
DNS Spoofing MitML7 (DNS)Poisoned DNS resolution sends victim to attacker's IP for a legitimate domain. TLS certificate validation becomes the last line of defence.DNSSEC; HTTPS certificate pinning; DNS over TLS/HTTPS
Evil TwinL2 (Wi-Fi)Rogue AP with matching SSID — all wireless traffic through attacker. (See Day 63)802.1X mutual auth; VPN overlay; certificate validation
💡 Exam Tip — MitM Defence: The fundamental defence against MitM is mutual authentication with cryptographic verification — TLS with proper certificate validation, DNSSEC, RPKI. Encryption alone without authentication only prevents passive eavesdropping, not active MitM (the attacker can establish two separate encrypted sessions, one with each party). Certificate pinning goes further by rejecting even valid-looking certificates from unexpected issuers.

6. Replay Attacks

📖 Replay Attack: An attacker captures a valid authentication or transaction message and retransmits it later to achieve unauthorized access or repeat a transaction. The captured data was legitimate at the time — the attack exploits the lack of session uniqueness controls.
AspectDetail
Classic exampleAttacker captures a valid login credential hash (Kerberos TGT, NTLM hash) and replays it to authenticate without knowing the password. "Pass-the-Hash" and "Pass-the-Ticket" Kerberos attacks are replay attacks.
Transaction replayCapture a "transfer $100" request and resend it multiple times — each replay executes an additional transfer. Financial systems must use unique transaction IDs to prevent this.
CountermeasuresNonce (Number Used Once): unique random value included in each message; server rejects reuse of nonces. Timestamp: Kerberos requires synchronized clocks (within 5 minutes); old timestamps are rejected. Sequence numbers: TLS/IPSec use sequence numbers; replayed packets are out-of-order and rejected. Challenge-response: each authentication requires a fresh unique challenge.
KerberosKerberos tickets have a limited lifetime (typically 10 hours). The 5-minute clock skew limit and ticket expiry together prevent replay. NTP synchronization is therefore a Kerberos security dependency.

7. Session Hijacking

📖 Session Hijacking: An attacker takes over an already-authenticated session by stealing or predicting the session token used to identify the session, then using that token to impersonate the legitimate user without needing to know the credentials.
TechniqueHowDefence
TCP Session HijackingAttacker predicts TCP sequence numbers, injects forged packets into an active TCP session. Effective on unencrypted protocols (Telnet, HTTP). TLS encryption prevents this by ensuring packets that don't match the session key are rejected.Encrypt all sessions with TLS; TLS sequence numbers are inside the encrypted record
Cookie Theft (XSS)Attacker injects JavaScript into a web page that steals the victim's session cookie and sends it to the attacker's server. Attacker uses the cookie to access the session. Most common web session hijacking method.HttpOnly cookie flag (JavaScript cannot access cookie); Secure flag (HTTPS only); CSP headers; XSS prevention (input validation, output encoding)
Cookie SniffingAttacker on the same LAN uses ARP poisoning or a passive sniffer to capture session cookies transmitted over HTTP (not HTTPS).Enforce HTTPS everywhere; HSTS; Secure cookie flag
Session FixationAttacker forces a known Session ID on the victim before authentication. When victim authenticates, the attacker's pre-known session ID is now authenticated. Attacker uses the same ID to access the authenticated session.Regenerate session ID on successful authentication; never accept externally-supplied session IDs

8. IP Spoofing

📖 IP Spoofing: Forging the source IP address in IP packet headers to impersonate another host or to hide the attacker's true identity. Used to bypass IP-based access controls, enable amplification/reflection attacks, and in combination with other attacks.
Use in AttacksDescription
DDoS/amplificationSpoof victim's IP as source in requests to reflectors — response floods the victim. Enables attribution evasion and amplification simultaneously. (See Section 4)
Trust exploitationOld systems using rsh/rlogin relied on source IP for authentication. Spoofing a trusted IP bypassed authentication. Same principle applies to IP-based firewall rules that trust specific source IPs without additional authentication.
SYN floodSpoofed source IPs prevent the server from tracing the attack and prevent SYN-ACK responses from reaching the attacker (which would complete the 3-way handshake).

Countermeasures Against IP Spoofing

ControlHow It Works
BCP38 Ingress FilteringISPs filter packets at network ingress — if a packet arrives on your upstream link with a source IP that could not legitimately originate from that direction, drop it. Prevents internet-scale spoofing but requires ISP cooperation.
Firewall ingress filteringDrop any inbound packet from internet with a source IP in your internal RFC 1918 space (impossible to legitimately originate from internet). Drop inbound with your own external IP as source.
Cryptographic authenticationUse TLS, IPSec, or application-level authentication — validates identity cryptographically rather than relying on IP address. IP spoofing is irrelevant if credentials are cryptographically verified.
Unicast Reverse Path Forwarding (uRPF)Router checks that the source IP of an arriving packet is reachable via the same interface it arrived on. If not, the packet is rejected (asymmetric routing complicates this).

9. DNS Attacks

AttackMechanismDefence
DNS Cache Poisoning (Kaminsky Attack)Attacker floods a DNS resolver with forged responses containing a malicious IP for a legitimate domain, racing to get the poisoned record cached before the legitimate response arrives. Kaminsky (2008) showed this was practical against unpatched resolvers by randomizing additional data.DNSSEC (cryptographically signed records); DNS source port randomization; 0x20 encoding; DNSSEC validation in resolvers
DNS AmplificationSpoofed ANY queries to open resolvers generate large responses directed at victim. (Covered in Section 4)Block open resolvers; BCP38; DNS RRL
DNS TunnelingEncodes arbitrary data (command & control traffic, exfiltrated data) inside DNS query/response fields (subdomain labels, TXT records). DNS is usually permitted outbound through firewalls and rarely inspected deeply.DNS query monitoring (anomalous query length, high frequency, NXDOMAIN rate); DNS traffic analysis tools; DNS over HTTPS to internal resolver only
DNS HijackingModify DNS records at registrar level (compromising registrar account) or at hosting provider for authoritative records. Traffic redirected to attacker-controlled server.Registrar account security (MFA, registry lock); domain locking; DNSSEC; CAA records limit certificate issuance
Fast Flux DNSMalware C2 domains rapidly change IP addresses (TTL of seconds) to evade IP blocklists and attribution. Hundreds of compromised hosts host the C2 service at any time.Domain reputation / threat intelligence; behavioral detection (high NX rate, short TTLs); DNS RPZ (Response Policy Zones) blocking

10. ARP Poisoning

📖 ARP Poisoning (ARP Spoofing): The attacker sends unsolicited (gratuitous) ARP Reply messages that map the attacker's MAC address to a legitimate IP address (typically the gateway's IP). Hosts cache this mapping and send traffic for that IP to the attacker's MAC instead of the legitimate host's MAC.
ARP Poisoning – MitM on LAN: Legitimate State: Host A (10.0.0.1) ─── ARP Cache: 10.0.0.254 → [Gateway's real MAC: AA:BB:CC:DD:EE:FF] Traffic to internet: → sent to AA:BB:CC:DD:EE:FF → [Gateway] After ARP Poisoning by Attacker (10.0.0.100): Attacker sends: "I am 10.0.0.254; My MAC is 11:22:33:44:55:66" (Gratuitous ARP) Host A ARP Cache: 10.0.0.254 → [Attacker's MAC: 11:22:33:44:55:66] Host A traffic → [Attacker's MAC] → Attacker intercepts → forwards to real gateway ^ Attacker performs MitM; sees all plaintext traffic
CountermeasureHow It Works
Dynamic ARP Inspection (DAI)Switch validates ARP packets against a trusted DHCP snooping binding table (IP→MAC→port mappings). ARP replies that don't match the binding table are dropped. Prevents gratuitous ARP poisoning on the switch.
Static ARP entriesManually configure ARP entries for the gateway on all hosts — not scalable in large environments but effective for critical hosts
ARP spoofing detection toolsARPwatch, XArp — monitor ARP cache changes; alert on IP→MAC mapping changes that indicate poisoning
Encrypted protocolsEven if ARP poisoning succeeds, TLS/HTTPS/SSH encryption of all sessions means the attacker sees only encrypted traffic — MitM without SSL inspection yields no useful data
802.1XOnly authenticated devices on the network; reduces ARP poisoning attack surface by restricting who can send frames

11. Botnet & C2 Infrastructure

📖 Botnet: A network of compromised devices (bots/zombies) remotely controlled by an attacker through Command and Control (C2) infrastructure. Used to launch DDoS attacks, send spam, distribute malware, mine cryptocurrency, or commit click fraud — all at scale without direct exposure of the attacker.
C2 Channel TypeDescriptionDetection
IRC-based C2Early botnets used IRC channels for commands. Bot connects to IRC server and waits for commands. Detectable by outbound IRC on unusual ports.Block outbound IRC (port 6667); detect IRC protocol on non-standard ports
HTTP/HTTPS C2Bot beacons to a web server (or compromised website) for commands. Blends in with normal web traffic. HTTP responses contain encoded commands.Proxy with URL reputation filtering; beacon detection (periodic outbound connections at regular intervals); threat intelligence feeds
DNS C2Commands encoded in DNS TXT/subdomain records. Bot polls attacker's DNS server for commands. Bypasses many proxies that allow DNS.DNS traffic analysis; anomalous query rates; subdomain entropy analysis; block external DNS (force internal DNS resolver)
P2P C2No central C2 server — bots communicate peer-to-peer. Highly resilient; taking down one C2 server doesn't disrupt the botnet. Used by sophisticated botnets (Conficker, Sality).Peer-to-peer traffic analysis; anomalous connection patterns; reputation-based blocking
Domain Generation Algorithm (DGA)Bots and C2 server use the same DGA (seeded by date/time) to generate hundreds of domain names daily. Only attacker registers a few; bot tries all until one resolves. Defeats domain blocklists.DGA detection via machine learning on DNS query patterns; NXDOMAIN spike detection; threat intel; Threat hunting for DGA domains

12. DDoS Mitigation Strategies

Distribute the targeted service across many anycast nodes globally. Attack traffic is naturally spread across all nodes based on BGP routing — no single node absorbs the full volume.
StrategyDescriptionScope
Cloud Scrubbing / CDNTraffic is routed through a cloud scrubbing service (Cloudflare, Akamai, AWS Shield Advanced) that absorbs volumetric DDoS via globally distributed capacity. Malicious traffic is dropped at the edge; clean traffic forwarded to origin.Volumetric DDoS (Tbps scale)
ISP-Level Black-Holing (RTBH)Remote Triggered Black Hole routing — ISP drops all traffic destined for the targeted IP prefix. Stops attack traffic but also stops legitimate traffic to that IP. Last resort when the attack volume exceeds all other mitigations.Saves network infrastructure; sacrifices the targeted IP
Firewall Rate LimitingLimit connection rate per source IP; drop excessive ICMP; limit SYN rate per source. Effective against smaller, non-distributed attacks. Ineffective against large-volume DDoS where attack comes from millions of IPs.Small to medium DoS
SYN CookiesServer doesn't allocate connection state until the 3-way handshake completes. The initial SYN-ACK encodes state in the TCP sequence number cryptographically. Only legitimate connections that complete the handshake consume server resources.TCP SYN flood
Anycast DistributionLarge DDoS absorbed by distributed infrastructure
Ingress Filtering (BCP38)ISPs drop packets with impossible source IPs at network ingress. Eliminates IP-spoofed UDP/ICMP amplification attacks at internet scale.Spoofed-source amplification attacks

13. Memory Tricks & Mnemonics

🧠 DoS vs. DDoS — "Solo vs. Swarm":
DoS = one attacker, one victim (solo attacker; scalable via amplification)
DDoS = botnet swarm of thousands → victim flooded from every direction simultaneously
DDoS at Tbps scale requires cloud scrubbing — no single on-premise device can handle it.
🧠 Smurf Attack — "Smurf Hides Behind the Crowd":
Smurf = spoof victim's IP + send ICMP to broadcast address → entire network replies to victim.
Fixed by: disabling directed broadcasts on routers. Historical attack — mostly irrelevant on modern networks. Still on exams.
🧠 Replay vs. Session Hijacking:
Replay = record-and-playback of a previously captured auth/transaction token. The session may no longer be active. Defence: nonce, timestamp, sequence number.
Session Hijacking = steal a currently active session token and impersonate the active user in real time. Defence: HTTPOnly/Secure cookie, HTTPS everywhere, session regeneration.
🧠 ARP Poisoning Defence — "DAI Stands Guard":
DAI (Dynamic ARP Inspection) = switch checks every ARP reply against DHCP snooping binding table; rejects unauthorized IP→MAC claims.
DHCP Snooping (trusted/untrusted port config) is a prerequisite for DAI → both are deployed together.
