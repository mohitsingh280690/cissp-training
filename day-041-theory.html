3. CPU Protection Rings

📖 Protection Rings: A hierarchical privilege model built into CPU architecture (Intel x86) that separates code into different privilege levels. Lower ring number = higher privilege. 
  ┌──────────────────────────────────────┐
  │        Ring 3 (User Mode)            │   Applications, user programs
  │  ┌──────────────────────────────┐    │
  │  │    Ring 2 (Device Drivers)   │    │   (Rarely used in modern OS)
  │  │  ┌──────────────────────┐    │    │
  │  │  │  Ring 1 (OS Services) │   │    │   (Rarely used in modern OS)
  │  │  │  ┌──────────────┐    │    │    │
  │  │  │  │ Ring 0       │    │    │    │   OS Kernel (full HW access)
  │  │  │  │ (Kernel)     │    │    │    │
  │  │  │  └──────────────┘    │    │    │
  │  │  └──────────────────────┘    │    │
  │  └──────────────────────────────┘    │
  └──────────────────────────────────────┘
       Ring -1: Hypervisor (VMX root)
       Ring -2: SMM (System Management Mode)
       Ring -3: Intel ME / AMD PSP
RingPrivilegeComponentsAccess
Ring 0Highest (kernel mode)OS kernel, security kernelFull hardware access, all instructions
Ring 1HighOS services (rarely used)Limited privileged instructions
Ring 2MediumDevice drivers (rarely used)I/O access
Ring 3Lowest (user mode)Applications, user programsCannot execute privileged instructions
🔑 Key Facts:
• Most modern OS use only Ring 0 (kernel) and Ring 3 (user) — "two-ring model"
• Ring 0 = kernel mode = supervisor mode = privileged mode
• Ring 3 = user mode = unprivileged mode
• Hypervisors use Ring -1 (VMX root mode) below the OS kernel
• A process in Ring 3 CANNOT directly access Ring 0 resources — must use system calls
💡 Exam Tip: The exam loves asking: "In which ring does the kernel operate?" → Ring 0. "In which ring do applications run?" → Ring 3. Remember: lower number = MORE privilege (counterintuitive).

4. Process Isolation

📖 Process Isolation: The OS mechanism that ensures each process runs in its own separate memory space and cannot read or modify another process's memory without authorization. Implemented through hardware (MMU) and OS virtual memory.

Isolation Mechanisms

MechanismHow It Works
Virtual Address SpacesEach process has its own virtual memory space; addresses map to different physical locations
Memory Management Unit (MMU)Hardware translates virtual → physical addresses; prevents unauthorized access
Page TablesPer-process mapping of virtual pages to physical frames; kernel manages
Hardware EnforcementCPU raises exception if process accesses memory outside its pages
⚠️ When Isolation Fails: Spectre and Meltdown attacks (2018) demonstrated that CPU speculative execution could leak data across isolation boundaries. These are side-channel attacks that bypass logical isolation through hardware-level timing vulnerabilities.

5. Memory Protection Mechanisms

Segmentation

📖 Segmentation: Divides memory into variable-sized segments (code, data, stack, heap). Each segment has a base address and length limit. The CPU checks that memory accesses stay within segment bounds.

Paging

📖 Paging: Divides memory into fixed-size pages (typically 4 KB). The page table maps virtual pages to physical frames. Enables virtual memory and process isolation. Each page has permission bits (read, write, execute).
FeatureSegmentationPaging
Unit SizeVariable (logical divisions)Fixed (typically 4 KB)
GranularityCoarse (whole segments)Fine (individual pages)
FragmentationExternal fragmentationInternal fragmentation (within pages)
Modern UsageMostly deprecated in x86-64Primary mechanism in all modern OS
Permission ControlPer segmentPer page (R/W/X bits)

6. ASLR & DEP/NX

ASLR (Address Space Layout Randomization)

📖 ASLR: A memory protection technique that randomizes the location of key memory areas (stack, heap, libraries, executable) each time a program runs. Makes it difficult for attackers to predict where injected code or return addresses will be in memory.
🔑 ASLR Key Facts:
• Randomizes positions of stack, heap, shared libraries, and executable base
• Defeats return-to-libc attacks and ROP (Return-Oriented Programming) by making addresses unpredictable
• Enabled by default in modern OS (Windows Vista+, Linux, macOS, iOS, Android)
• Can be weakened by information disclosure vulnerabilities that leak addresses

DEP/NX (Data Execution Prevention / No-eXecute)

📖 DEP/NX Bit: A hardware-enforced memory protection that marks certain memory pages as non-executable. Data regions (stack, heap) are marked NX — the CPU refuses to execute code from these regions, preventing most buffer overflow code injection attacks.
🔑 DEP/NX Key Facts:
• NX bit (AMD) = XD bit (Intel) = DEP (Microsoft Windows term)
• Marks data memory pages as "no execute" — CPU hardware refuses to run code from these pages
• Prevents classic stack/heap buffer overflow exploitation (injecting shellcode into data, then executing it)
• Attackers bypass DEP using ROP (Return-Oriented Programming) — chaining existing executable code snippets
💡 Exam Tip: ASLR randomizes WHERE things are in memory. DEP/NX prevents WHAT can execute. Together they form a strong defense: even if an attacker finds a buffer overflow, they can't inject+execute code (DEP) and can't predict addresses (ASLR).

7. Virtual Memory & Swapping Security

📖 Virtual Memory: A technique where the OS uses disk space (swap file/partition) to extend physical RAM. Allows processes to use more memory than physically available by swapping pages between RAM and disk.

Security Concerns

ConcernRiskMitigation
Swap file residual dataSensitive data (keys, passwords) swapped to disk in cleartextEncrypt swap partition; clear swap on shutdown
Memory reuseNew process gets physical memory with previous process's dataZero-fill new pages before allocation (object reuse protection)
Core dumpsCrash dumps contain full memory contents including secretsRestrict core dump creation; encrypt crash dumps
Memory-mapped filesAttackers could access shared memory regionsProper access controls on shared memory
⚠️ Object Reuse: When memory (or any resource) is deallocated from one process and given to another, the new process must NOT be able to read the previous process's data. This is called object reuse protection — implemented by zeroing memory before reallocation. Required at TCSEC C2 and above.

8. Buffer Overflow Prevention

📖 Buffer Overflow: A vulnerability where a program writes data beyond the boundary of an allocated buffer, overwriting adjacent memory. Attackers exploit this to inject malicious code or redirect program execution (e.g., overwriting the return address on the stack).

Types of Buffer Overflows

TypeLocationMechanism
Stack Buffer OverflowStack memoryOverwrite return address → redirect execution to attacker's code
Heap Buffer OverflowHeap memoryCorrupt heap metadata → arbitrary write when free() is called
Integer OverflowAnyArithmetic overflow causes incorrect buffer size calculation
Format String AttackStackUncontrolled format specifiers (%s, %x, %n) read/write memory

Prevention Techniques

TechniqueLevelHow It Helps
ASLROSRandomizes memory layout — attacker can't predict target addresses
DEP/NXHardware + OSPrevents code execution in data regions
Stack CanariesCompilerPlaces sentinel values before return address; detects overwrite
Bounds CheckingLanguage/CompilerValidates array/buffer access stays within bounds
Safe FunctionsDeveloperUse strncpy() instead of strcpy(), snprintf() instead of sprintf()
Memory-Safe LanguagesDeveloperRust, Go, Java, Python — prevent manual memory management errors
💡 Exam Tip: The CISSP exam expects you to know that buffer overflows are the #1 software vulnerability class. Prevention is multi-layered: ASLR + DEP at the OS/HW level, stack canaries at the compiler level, and bounds checking/safe languages at the development level.

9. Memory Tricks & Mnemonics

🧠 "Ring 0 = King, Ring 3 = Peasant": In the medieval kingdom of your CPU, Ring 0 is the king (kernel, full power) and Ring 3 is a peasant (user applications, no privileges). Lower number = more power.
🧠 "ASLR = Address Shuffler, DEP = Data Execution Preventer":
• ASLR shuffles WHERE things are
• DEP prevents WHERE code can run
• Together: can't find it (ASLR) AND can't run it there (DEP)
🧠 "Buffer Overflow = Overfilling a Cup": A buffer overflow is like pouring water into a cup that's too small — the water overflows into nearby cups (memory), contaminating their contents (overwriting data/return addresses).
