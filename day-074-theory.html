2. Real-World Organizational Relevance

IssueContext
Cloud SSOOrganizations use SAML 2.0 to federate their on-premises identity provider (Okta, Azure AD, ADFS) with cloud SaaS applications (Salesforce, Office 365, Workday, AWS Console). User authenticates to corporate IdP; SAML assertion grants access to SaaS without separate SaaS password.
"Sign in with Google/Apple/GitHub"OIDC powers social login. The identity provider (Google, Apple) authenticates the user and issues an ID token (JWT). The relying party (website) accepts this token to establish user identity. No need for separate account creation.
API authorizationOAuth 2.0 powers API authorization across the internet. When "Slack" wants to access your "Google Calendar," OAuth 2.0 grants Slack a scoped access token without sharing your Google password. The token scope limits what Slack can do.
B2B federationPartners, suppliers, auditors, contractors need access to internal systems. Rather than creating accounts in your IdP, you federate with the partner's IdP. Users authenticate with their own organization's credentials — their identity is trusted via federation agreement.

3. Federation Concepts & Trust Models

📖 Identity Federation: A system that allows users to use their identity from one domain (Identity Provider) to access resources in another domain (Service Provider / Relying Party) without creating a separate account, based on a pre-established trust relationship between the two parties.
RoleDefinitionExample
Identity Provider (IdP)The entity that authenticates the user and issues assertions/tokens about their identity. The authoritative source of identity.Enterprise ADFS; Okta; Azure AD; Google Accounts; Auth0
Service Provider (SP)The entity that provides the service/resource the user wants to access. Trusts assertions from the IdP without re-authenticating the user.Salesforce; AWS Console; GitHub Enterprise; your internal app behind Okta
Relying Party (RP)OAuth/OIDC equivalent of the Service Provider. Relies on the authorization server/IdP to authenticate and authorize the user.Third-party app using OIDC; API consuming OAuth tokens

Federation Trust Models

ModelDescriptionExample
Direct trustOne organization directly trusts another's IdP. Bilateral agreement. Simple; doesn't scale beyond a few partners.Company A trusts Company B's ADFS. One SAML metadata exchange, manual agreement.
Brokered / Hub-and-spokeA central federation broker (hub) that all parties trust. Any party in the federation can authenticate users from any other party via the broker. Scales to many IdPs/SPs.InCommon Federation (US higher education); REFEDS; national eID schemes
Identity meshEvery party trusts every other party — each has direct trust with all others. Maximum flexibility, exponentially complex to manage as parties scale.Rare in practice above a few parties; replaced by hub-and-spoke at scale

4. SAML 2.0

📖 SAML 2.0 (Security Assertion Markup Language): An XML-based, open standard for exchanging authentication and authorization data between an Identity Provider and a Service Provider. The primary standard for enterprise web SSO since 2005. Uses signed and optionally encrypted XML assertions.

SAML Assertion Types

Assertion TypePurpose
Authentication assertionStates that the user was authenticated by the IdP at a specific time using a specific method. Answers: WHO authenticated, WHEN, HOW.
Attribute assertionContains attributes about the user (email, role, group membership, department). Service Provider uses these for authorization decisions.
Authorization decision assertionStates whether the user is permitted to access a specific resource. Rarely used directly — most implementations derive authorization from attribute assertions.

SAML Security Controls

ControlPurpose
XML digital signatureEnsures assertion was issued by the legitimate IdP (integrity + authenticity). SP must validate signature against IdP's certificate.
Audience restrictionAssertion contains the SP's entity ID as the intended audience. Prevents token re-use at a different SP ("audience confusion")
Time conditions (NotBefore / NotOnOrAfter)Assertion is only valid within a defined time window. Prevents replay attacks using old assertions.
InResponseToTies the response to a specific SP-originated request. Prevents unsolicited IdP-initiated assertion injection attacks.
Assertion encryption (optional)Encrypts attribute assertions in transit so SP proxy servers cannot read user attributes.

5. OAuth 2.0

🔑 Critical: OAuth 2.0 is an AUTHORIZATION framework, NOT an authentication protocol.
OAuth 2.0 answers: "Can app X access resource Y on behalf of user Z?"
It does NOT answer: "Is this person who they claim to be?" (that's OIDC or SAML)
Core concept: delegated authorization — grant access without sharing credentials

OAuth 2.0 Key Roles

RoleDefinitionExample
Resource OwnerThe user who owns the data/resource and grants accessUser "Alice" who owns Google Calendar data
ClientThe application requesting access to the resource on behalf of the resource ownerSlack (wants to read Alice's calendar)
Authorization ServerIssues access tokens after authenticating the resource owner and obtaining consentGoogle Auth server (accounts.google.com)
Resource ServerThe API/server hosting the protected resource. Validates access tokens.Google Calendar API

OAuth 2.0 Grant Types

Grant TypeUse CaseSecurity Notes
Authorization CodeServer-side web apps with a back end that can keep secretsMost secure standard grant; code exchanged for token on back channel; PKCE extension required even for server-side apps
Authorization Code + PKCEMobile/SPA/native apps that cannot store client secretsCurrent recommended replacement for Implicit; code verifier/challenge prevents code interception attacks
Client CredentialsMachine-to-machine (no user involved); service-to-serviceClient authenticates with its own credentials (client_id + client_secret); no user context
Device CodeDevices with limited input (Smart TV, printer, CLI)Device shows code; user authenticates on secondary device (phone/laptop)
Implicit (deprecated)SPAs (legacy) — access token returned directly in URL fragment⚠️ Deprecated (OAuth 2.1): token in URL = logged in server logs, accessible to scripts; replaced by Auth Code + PKCE
Resource Owner Password Credentials (ROPC) (deprecated)Legacy apps where client collects username/password directly⚠️ Deprecated (OAuth 2.1): app handles credentials directly — defeats OAuth's purpose of not sharing credentials; only acceptable for migration from legacy systems

OAuth 2.0 Tokens

Token TypePurposeNotes
Access tokenBearer token presented to resource server to access the resource. Scoped to specific permissions.Short-lived (minutes to hours); opaque or JWT; resource server validates by introspection (opaque) or signature (JWT)
Refresh tokenLong-lived token used to obtain new access tokens without re-prompting the user.Never sent to resource servers; stored securely by client; can be revoked by authorization server; rotation recommended

6. OpenID Connect (OIDC)

📖 OpenID Connect (OIDC): An authentication layer built on top of OAuth 2.0. While OAuth 2.0 handles authorization (resource access), OIDC adds identity — the ability to authenticate a user and receive identity information about them. OIDC = OAuth 2.0 + UserInfo endpoint + ID Token.
OIDC ConceptDescription
ID TokenA JSON Web Token (JWT) returned by the authorization server that contains claims about the authenticated user (sub, email, name, iat, exp, iss). The ID token IS the authentication credential. The relying party validates the JWT signature to verify identity.
UserInfo endpointAn OAuth 2.0 protected endpoint where the relying party can retrieve additional user attributes using the access token. Supplements ID token claims.
ClaimsKey-value pairs in the ID token (JWT payload). Standard claims: sub (subject/user ID), iss (issuer), aud (audience), exp (expiry), iat (issued at), email, name, preferred_username.
sub (Subject)Unique identifier for the user within the authorization server. Stable and non-reassignable. Used as the primary user identifier — more stable than email (email can change).
Scopesopenid scope is required to trigger OIDC (returns ID token). Additional scopes: profile, email, address, phone — each adds specific claims to the ID token/UserInfo response.
💡 Exam Tip: OAuth 2.0 vs. OIDC purpose:
OAuth 2.0: "Can this app access my calendar?" — authorization, resource access, delegated permissions
OIDC: "Who is this user?" — authentication, identity assertion, sign-in
OIDC requires the openid scope in the OAuth 2.0 request. Without it, you get OAuth 2.0 access tokens but no ID token.

7. WS-Federation

AspectDetail
DefinitionA WS-* (Web Services) protocol for cross-organizational identity federation. Uses WS-Trust for token exchange and SOAP-based messaging. Microsoft/IBM initiative.
Token formatUses SAML 1.1 or SAML 2.0 tokens (unlike SAML profiles which define binding). WS-Federation defines the federation protocol; SAML defines the token format.
Use in practicePrimarily found in Microsoft environments (ADFS — Active Directory Federation Services). WS-Federation passive profile used for browser-based SSO; WS-Federation active profile for thick-client/WCF service authentication.
Current relevanceBeing replaced by SAML 2.0 and OIDC in new deployments. Microsoft has shifted Azure AD toward SAML 2.0 and OIDC. WS-Federation legacy support remains in ADFS for backward compatibility.
CISSP exam useTested as a federation concept; know: WS-Federation = Microsoft environment, uses security tokens, predecessor to modern federation standards in M365/Azure ecosystems

9. Federation Attacks

AttackDescriptionDefence
Golden SAMLAnalogous to Kerberos Golden Ticket. Attacker steals the IdP's SAML signing private key (from compromised ADFS/identity server). Can forge arbitrary SAML assertions for any user, granting access to all federated services. Used in SolarWinds attack (2020).Protect IdP signing key with HSM; monitor for SAML assertions with unusual attributes; IP restrictions on SAML-consuming services; certificate rotation
SAML XML injection / wrapping attackOlder XML signature vulnerability allowing an attacker to wrap the signed portion of an XML assertion, replacing unsigned content while signature still validates.Use modern SAML libraries with full XML canonicalization; validate both signed element reference AND entire assertion integrity
OAuth token theftStealing access tokens from client storage (browser localStorage, logs, insecure apps).Store tokens in HttpOnly cookies or secure in-memory; short token lifetimes; token binding; mTLS sender-constrained tokens
OAuth redirect_uri manipulationAttacker registers a malicious redirect URI; if authorization server doesn't strictly validate redirect URIs, authorization code or token is redirected to attacker's server.Exact match redirect URI validation (no wildcards); PKCE prevents stolen auth codes from being used
Open redirector abuseOAuth authorization endpoint is an open redirector — attacker crafts malicious authorization URL that redirects victim to attacker's site after consent.Server validates redirect URI against pre-registered exact values; user education
IdP compromiseCompromising the Identity Provider gives access to all service providers that trust it. The IdP is the highest-value target in a federated architecture.Harden IdP to highest standard; MFA on IdP admin; monitor for anomalous assertion issuance; IdP key HSM protection

10. IdP Concepts & Trust Architecture

ConceptDetail
Identity Provider (IdP)The authoritative authentication authority. Authenticates users, manages their identity lifecycle, issues assertions/tokens. Examples: Okta, Azure AD, ADFS, Auth0, Ping Identity, Google Workspace.
Federation metadataXML document describing an IdP or SP's capabilities, entity ID, signing certificates, supported bindings. Exchanged between parties to establish trust. SAML metadata is imported at both ends before federation works.
Trust establishmentSPs must be configured to trust specific IdPs (import IdP metadata). IdPs must register SPs (import SP metadata, define which attributes to release). Manual or metadata-based automated trust.
Attribute release policyIdPs control which user attributes are shared with which SPs. A policy might release email to Service A but only employee number to Service B. Minimizes data exposure and supports privacy principles.
Cross-domain authenticationWhen User from Domain A accesses Service in Domain B: User authenticates to Domain A IdP → IdP issues credential → Domain B SP validates credential based on pre-established trust → User gets access without Domain B account.

11. Memory Tricks & Mnemonics

🧠 OAuth vs. OIDC — "Authorization vs Authentication":
OAuth = O for Open authorization = "Can you access this?" — delegation
OIDC = OAuth + identity = "Who are you?" — authentication on top of OAuth
If you see "Sign in with..." → OIDC
If you see "Allow app X to access your Y..." → OAuth 2.0
🧠 SAML Parties — "Idp Serves, SP Trusts":
IdP = Identity Provider = WHO ISSUED the assertion (authenticates user, signs XML)
SP = Service Provider = WHO TRUSTS the assertion (validates signature, grants access)
Flow: User → SP → redirect to IdP → authenticate → IdP sends SAML back → SP grants access
🧠 Golden SAML = Golden Ticket, different protocol:
Golden Ticket (Kerberos) = steal KRBTGT → forge Kerberos tickets
Golden SAML (federation) = steal IdP signing key → forge SAML assertions
Both require IdP/KDC-level compromise but grant access to ALL downstream services

IdPAuthenticates user, issues assertions/tokens; most critical node in federation — compromise = all SPs compromised
