2. Real-World Organizational Relevance

Organizational Ownership

Access control attack defense is owned jointly by the IAM team (credential policies, MFA enforcement), the SOC/SIEM team (detecting anomalous access patterns), and the Red Team / Pen Testing team (proactively testing for these weaknesses). The CISO is accountable to the board when an attack succeeds.

Implementation Reality

Most successful breaches involve compromised credentials — the 2021 Verizon DBIR found >80% of hacking-related breaches involved stolen or brute-forced credentials. Organizations deploy MFA, privileged access workstations (PAWs), SIEM rules, and endpoint detection (EDR) to detect and stop these attacks. Microsoft Credential Guard and Protected Users security group specifically harden Windows environments against pass-the-hash and Kerberoasting.

Failure Modes

  • Credential stuffing enablement: Password reuse from third-party breach dumps hits VPN, webmail, or cloud portals
  • Kerberoasting in AD: Service accounts with weak passwords and SPNs are trivially crackable offline — never caught by traditional logging
  • Privilege escalation via misconfigured SUDO: A Linux developer account with excessive sudo rights becomes a full root compromise

Business Impact

Credential attacks cost organizations an average of $4.5 million per breach (IBM Cost of a Data Breach 2023). The reputational damage from "your account data was used to attack us" is severe. Regulators under GDPR and HIPAA ding organizations that lack basic credential hygiene controls.

3. Credential-Based Attacks

Brute Force Attack

Brute Force: Systematically trying every possible password combination until the correct one is found. Exhaustive key search. The attacker has no prior knowledge of the password.

Modern GPUs can test billions of MD5 hashes per second. Defense: account lockout policies, long/complex passwords, MFA, rate limiting, CAPTCHA.

Dictionary Attack

Uses a pre-compiled wordlist of common passwords, phrases, and variations (e.g., "password1", "Welcome123"). Far faster than pure brute force because it targets human password patterns. Tools: Hashcat, John the Ripper.

Exam Tip: Salting passwords defeats rainbow tables but does NOT stop dictionary attacks — dictionary attacks hash each candidate password combined with the salt and compare. Only strong, unique passwords stop dictionary attacks.

Rainbow Table Attack

Pre-computed tables mapping plaintext passwords to their hash values. Allows instant lookup of common passwords from a captured hash — trading storage for speed. Defense: cryptographic salting. A salt is a random value unique per credential, prepended to the password before hashing, making pre-computation impractical.

⚠️ Exam Trap: "Encryption" does NOT defeat rainbow tables — only salting does. A salted hash with SHA-256 defeats rainbow tables even though SHA-256 itself is not "slow."

Credential Stuffing

Credential Stuffing: Automated injection of stolen username/password pairs (from prior data breaches) against login portals. Relies on password reuse across sites.

Attackers use breach databases (Have I Been Pwned aggregates these) and automated bots. Differs from brute force because each credential pair is a real, known-good password from another site — not a guess. Defense: MFA, anomalous login detection, IP reputation filtering, CAPTCHA.

Password Spraying

The inverse of brute force: tries one or a few very common passwords (e.g., "Spring2024!") across many accounts simultaneously. Avoids lockout thresholds by staying below the per-account limit. Particularly effective against O365 and Azure AD tenants. Defense: smart lockout policies that track by source IP (not just per account), MFA, monitoring for low-frequency multi-account failures.

4. Pass-the-Hash & Pass-the-Ticket

Pass-the-Hash (PtH)

Pass-the-Hash: An attacker captures the NTLM hash of a user's password (from memory, the SAM database, or domain controller) and uses the hash directly to authenticate — without ever cracking it to obtain the plaintext password.

NTLM authentication passes a hash (not the plaintext) over the network. If an attacker obtains the hash (via Mimikatz from lsass.exe memory), they can present it as proof of identity. Kerberos-only environments reduce PtH risk, but NTLM fallback often remains enabled.

Tools: Mimikatz (sekurlsa::pth), Impacket's psexec.py, CrackMapExec

Defenses: Microsoft Credential Guard (virtualizes LSA secrets), Protected Users AD security group (disables NTLM for members), disabling NTLMv1, enabling Protected Process Light (PPL) for lsass, Privileged Access Workstations (PAWs)

Pass-the-Ticket (PtT)

Pass-the-Ticket: An attacker steals a Kerberos Ticket Granting Ticket (TGT) or service ticket from memory and replays it to gain access — without needing the password or hash.

Kerberos tickets are valid for 10 hours by default (TGT). Attackers extract them from LSASS memory. Unlike PtH which abuses NTLM, PtT abuses Kerberos. Defense: Short ticket lifetimes, Privileged Access Workstations, Protected Users security group (forces Kerberos-only and limits ticket caching).

5. Kerberos Attacks

Kerberoasting

Kerberoasting: An attacker (with any valid domain account) requests Kerberos service tickets for accounts that have Service Principal Names (SPNs). The ticket is encrypted with the service account's password hash. The attacker takes the encrypted ticket offline and cracks it.
Domain User (low privilege) │ ▼ Request TGS ticket for SPN (e.g., MSSQLSvc/dbserver:1433) Kerberos KDC │ ▼ Returns ticket encrypted with SERVICE ACCOUNT's password hash Attacker extracts ticket → cracks offline with Hashcat/John │ ▼ If service account password is weak → plaintext recovered

Why it's dangerous: The attack is legitimate Kerberos behavior — no brute-force against AD, no lockout, no unusual network traffic. Service accounts often have domain admin rights and stale/weak passwords.

Defense: Use long random passwords (25+ chars) for service accounts; use Group Managed Service Accounts (gMSA) which auto-rotate 120-character passwords; monitor for excessive SPN ticket requests (4769 events).

AS-REP Roasting

Similar to Kerberoasting but targets accounts with "Do not require Kerberos pre-authentication" enabled. The attacker requests a TGT without providing credentials; the KDC responds with a credential-encrypted message that can be cracked offline. Defense: enable pre-authentication (the default) on all accounts.

Golden Ticket Attack

Golden Ticket: Forged Kerberos TGT created using the KRBTGT account's NTLM hash. Valid for any user, any service, with any lifetime — the attacker essentially becomes the KDC.

Requires prior Domain Admin or Domain Controller compromise to dump the KRBTGT hash. Provides unlimited domain persistence — survives password resets of regular accounts. Defense: Reset the KRBTGT password twice (it has two versions); detect with event ID 4769 for unusual ticket attributes (lifetime, encryption type, unusual SIDs).

Silver Ticket Attack

Forged Kerberos service ticket using a service account's hash (not KRBTGT). More limited scope (access to that service only) but harder to detect because it doesn't touch the KDC — no KDC event logs generated. Defense: Privileged Attribute Certificate (PAC) validation; service accounts using gMSA.

Exam Tip — Golden vs. Silver Ticket:
Golden = uses KRBTGT hash → any resource in the domain → highest privilege
Silver = uses service account hash → only that service → lower privilege but stealthier

6. Privilege Escalation

Privilege Escalation: An attacker gains higher-level access rights than were originally granted. Can be vertical (low user → admin) or horizontal (user A accesses user B's resources).

Vertical Privilege Escalation

Low-privileged account gains elevated rights (e.g., standard user → Administrator/root). Exploits vulnerabilities in OS, applications, or misconfigurations.

  • SUDO misconfiguration (Linux): Wildcard entries or ALL=(ALL) NOPASSWD grants
  • UAC bypass (Windows): Techniques to bypass User Account Control prompts
  • Unquoted service paths (Windows): Spaces in service executable paths allow executable path hijacking
  • SUID/SGID binaries (Linux): Binaries running as root exploited via shell escape
  • Token impersonation (Windows): Using stolen token via Incognito/Potato attacks

Horizontal Privilege Escalation

User accesses data or functions of a peer user without elevated rights. Common in web applications as Insecure Direct Object Reference (IDOR) — e.g., changing ?user_id=1001 to ?user_id=1002 in a URL to access another user's profile.

Access Token Manipulation

Windows uses access tokens to represent security context. Attackers may impersonate tokens of higher-privileged processes using ImpersonateLoggedOnUser() API calls. Also: token theft from other processes via Mimikatz token::elevate.

IDOR vs. Forced Browsing vs. Path Traversal:
IDOR: Changing object reference (ID, filename) in a request to access unauthorized resources
Forced Browsing: Directly accessing URLs not linked in the UI (e.g., /admin/console)
Path Traversal: Using ../ sequences to escape the intended directory and read arbitrary files (e.g., /../../../etc/passwd)

7. Social Engineering for Credentials

Phishing

Fraudulent email impersonating a trusted entity to steal credentials. Variants: spear phishing (targeted, personalized), whaling (targeting executives/C-suite), smishing (SMS), vishing (voice/phone).

Defenses: SPF/DKIM/DMARC (email authentication), security awareness training, hardware MFA (phishing-resistant: FIDO2/passkeys), anti-phishing gateways, simulated phishing exercises.

⚠️ Phishing bypasses MFA when: Real-time adversary-in-the-middle (AiTM) proxies (e.g., Evilginx2) can relay credentials AND session cookies, defeating SMS and TOTP MFA. Only hardware/FIDO2 MFA is phishing-resistant because the credential is cryptographically bound to the legitimate origin.

Pretexting

Attacker creates a fabricated scenario (pretext) to manipulate the victim — e.g., impersonating IT support ("We need your VPN credentials to fix your account"). Defense: callback verification procedures, strict identity verification protocols.

Vishing (Voice Phishing)

Phone calls impersonating IT helpdesk, banks, or law enforcement. The 2020 Twitter VIP account hack started with vishing calls to Twitter employees to steal credentials. Defense: strict callback procedures, never give passwords over phone, supervisor approval for sensitive requests.

9. Memory Tricks & Mnemonics

🧠 Credential Attack Types — "BDC-SR"
Brute force — every combo
Dictionary — wordlist
Credential stuffing — stolen pairs from other breaches
Spraying — one password, many accounts
Rainbow tables — pre-computed hash lookup
Reminder: only salting defeats rainbow tables
🧠 Kerberos Attack Hierarchy — "AS-K-G-S"
AS-REP Roasting — pre-auth disabled, no creds needed
Kerberoasting — SPN ticket, offline crack, needs domain user
Golden Ticket — KRBTGT hash, total domain control, highest risk
Silver Ticket — service hash, single service, stealthiest
Privilege escalates: AS-REP < Kerberoasting < Silver < Golden
🧠 Pass-the-Hash vs. Pass-the-Ticket
Hash = NTLM (H for Hash)
Ticket = Kerberos (T for Ticket)
Hash → horizontal move via NTLM auth
Ticket → replay TGT without password
🧠 Web Access Control Attacks — "IFP"
IDOR — object reference manipulation (changing IDs)
Forced Browsing — going to hidden URLs directly
Path Traversal — using ../ to escape directories

Top Exam Points:
  • Salt defeats rainbow tables — not encryption, not hashing algorithm choice alone
  • Kerberoasting is entirely offline after ticket acquisition — no lockout possible
  • Golden Ticket persists even after user password resets — must reset KRBTGT twice
  • Protected Users security group in AD: disables NTLM, DES/RC4, credential caching — prevents both PtH and PtT
  • Password spraying and credential stuffing trigger only 1–2 failures per account → evade standard lockout policies
